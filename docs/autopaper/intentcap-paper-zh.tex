\begin{abstract}
LLM agents 正在从固定工具调用器演变为可扩展执行环境，可以加载 Skills、连接 MCP servers、运行本地脚本，并将任务委托给 subagents。
这些扩展组件不只执行操作，也把说明、工具结果、外部文档和运行时观察带入后续的 tool selection、sink selection、approval scope 和 delegation target。
现有权限机制主要约束已经成形的工具调用、参数、文件和网络端点，却难以表达哪类 context 可以证明一次 authority-bearing decision 的哪个字段。
结果是 agent 可能发出参数合法、provenance 看似可信、OS sandbox 也无法区分的调用，但权限状态已经被错误扩大、复用或委托。
\sys 的核心洞察是：LLM agent 的授权单位应是副作用、prompt placement 或 delegation 前提交的 protected-decision transition，而不是事后过滤的 tool call。
\sys 将该 transition 表示为 field-owned protected-decision lease；lease 的新点不是一个更复杂的 policy predicate，而是 commit-time field ownership、checker-owned lease state 和 same-transition lifecycle mutation 的线性化接口。
通过 \(\mathsf{check\_and\_consume}\)，runtime 在同一原子步骤中验证 issuer-owned proofs、当前 intent、provenance、budget、expiry 和 delegation state。
原型把这个 contract 接入 trace replay、local callable/env side effects、context and Skill placement、delegation handoff，并把 env/local-effect projection lowering 到 deterministic replay target 和 local bubblewrap namespace sandbox，而不是生产级 ActPlane/eBPF monitor。LLM 只提出 plans 和 candidate leases，不在 TCB 内。
评估支持一个有界结论：在已 instrumented protected-decision boundaries 内，\sys 可以阻断 owner substitution、context-class promotion 和 lease-lifecycle violations，同时保留 benign reference-action expressiveness。
Same-event ablations 固定 action、argument 和 provenance record，只替换 issuer ownership 或 lifecycle ownership，显示 no-owner collapse 会错误接受 3,593/3,823 个完整 checker 拒绝的 protected decisions。Local multi-boundary suite 和 integrated PDF-to-issue workflows 进一步显示同一 contract 可放到 side-effect、placement、handoff 和 local tool/MCP-style gateway event 前；paired data/control run 允许同一 PDF text 作为 issue body 数据，同时阻断它作为 instruction/repo control，并产生 0 个 unsafe executions/placements。
\end{abstract}

\section{引言}

LLM agents 的安全边界正在发生变化。传统应用扩展通常通过 API、插件接口或系统调用与宿主交互，因此权限系统可以围绕文件、网络、进程和工具调用来定义。现代 agent extension 的行为则由用户请求、系统策略、扩展说明、外部文档、工具结果和中间总结共同塑造，尤其是在 Skills、MCP servers 和 agent tool ecosystems 中~\cite{skillguard,skillscope,mcp-spec}。

然而，agent extension 打破了 data 和 authority 的传统边界。用户只要求从两个 PDF 提取表格，并在 \texttt{org/repo-x} 创建一个 summary issue；PDF 内容只应影响表格和 issue body。其中一个 PDF 的隐藏文本不需要修改 \texttt{repo} 参数。它可以诱导 agent 把一次性 issue-creation authority 交给 summarizer subagent，并让该 subagent 在后续步骤中复用这份 authority。最终 \texttt{create\_issue(repo=org/repo-x)} 的参数仍等于用户指定的 repository，issue body 也可能仍包含 PDF summary，所以普通 tool/argument guard、参数 provenance check 和 OS sandbox 都可能看见一个参数合法的调用。可见损害不是 repo 字符串错误，而是在合法参数下发生 unauthorized authority transfer 和 one-shot lease reuse。Approval widening、policy update 和 broader capability request 是同一结构问题的其他表现，但最小失败 trace 已经说明参数合法不代表权限状态合法。

根因不是不可信文本进入 prompt 本身，而是现有 agent runtime 在提交高影响决策时，通常没有一个副作用前的原子记录，同时绑定字段由谁证明、证明来自哪里、哪些 context 控制该字段，以及这次提交如何消费或更新权限状态。这产生三个结构性错配。第一个是 \textbf{authority-root collapse}，即用户意图、approval、sink 和 delegation root 与扩展文本共享 planning channel。第二个是 \textbf{procedure/interface/observation collapse}，即 Skill/manual procedure、tool or command interface metadata、以及 runtime observations 被当成同一种 trusted context，使 instruction、tool 和 env 可以互相代填 field。第三个是 \textbf{authority-state split}，即下游 guard 多数在 action materialized 后才检查 tool、arguments 或 resource，而一次性权限是否已被消费、是否过期、是否被委托给子任务、以及是谁铸造了权限，分散在不同状态中。这三个错配先推出两个接口需求：每个 protected field 必须有可审计 owner，且 authority state 必须在同一次 transition 中提交、消费或更新。在本文的 Skill、MCP、cmd 和 subagent workloads 中，这些需求实例化为四类 proof owners：agent-owned authority root、instruction-owned procedure endorsement、tool-owned interface contract 和 env-owned runtime observation。它们共同说明 authorization object 选错了。本文后文把需要检查的副作用前原子状态转移称为 protected-decision transition。

现有防线分别覆盖了这个问题的一部分。Tool guards、argument validators 和 action-time authority controls 可以拒绝错误对象或非法参数~\cite{authgraph,pact,airguard}。Static allowlists 和 server-level policies 可以限制可见工具。Human approval gates 可以重新引入用户确认。OS sandbox 或 monitor 可以约束文件、网络和进程 side effects~\cite{actplane}。Provenance、taint 和 IFC-style defenses 可以表达部分 flow policy~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}。这些机制是必要的；若它们显式暴露 issuer-owned field proofs 和 same-transition lifecycle mutation，就可以实现本文接口的一部分。但 agent runtime 默认把 field proofs、lease state 和 lifecycle update 分散在 prompt construction、tool gateway、approval UI、sandbox 和 subagent handoff 中。OS、sandbox 或 extension substrate 可以执行 local effects 的资源策略，但它们不决定哪个 issuer-owned field proof 可以改变 agent 的 authority state。EIM/bpftime 和 ActPlane 的细节留到相关工作讨论~\cite{eim-bpftime,actplane}。

\sys 的核心洞察是把这些分散对象显式化为一个 runtime-visible pre-effect authorization record。本文的主线分为三层：context influence 是要保护的安全对象，field-owned lease 是授权表达，authority-state commit object 是执行 lease 的运行时线性化机制。Lease 是可消耗的 authority object；commit object 是一次执行、放置或委托该 lease 时返回的线性化记录。非平凡点不是“再写一个 ABAC/IFC predicate”，而是 agent runtime 默认把 proof issuer、observation issuer 和 lifecycle writer 分散在 prompt construction、tool registry、runtime observer、approval UI 和 delegation adapter 中。本文的接口要求是：任何会产生 side effect、prompt authority placement 或 authority handoff 的 adapter，都必须在执行前暴露一个 authority-state transition API，让同一个 checker-state writer 同时看到三个 runtime objects：issuer-owned field proofs、checker-owned active-lease state、以及 same-transition consume/delegate mutation。若任一对象只存在于另一层，某些 required proof 会不可见，或会被错误 issuer 写入。这个 record 同时提交 issuer-owned field proofs、checker-owned active-lease version 和 same-transition consume/delegate mutation，并返回可审计的 commit binding。本文把这一 linearization record 称为 authority-state commit object。本文比较的是 runtime 是否暴露这个授权状态对象和 sole-writer discipline，而不是某个 policy DSL 理论上能否写出相同布尔谓词；它的 tested-removal claim 只限定在本文测试过的 owner-projection、lifecycle 和 local-boundary removal families 上。一个 DSL、IFC runtime 或 reference monitor 若暴露同一对象，就在覆盖的 fields 上实现了本文接口；缺少任一已测试字段，则需要另外证明不会出现 class substitution、stale reuse、double consume 或 over-delegation。Audit binding 是实现中的追踪与审计字段；当前实验把它作为 auditability gap，而不是已证明必要的核心安全字段。

实现这一洞察需要解决三个工程挑战。第一，checker 必须分离不同 authority fields 的证明来源。Skill/manual text 可以指导 workflow，但不能替代用户意图。MCP/tool metadata 可以描述接口和 credential scope，但不能授权 sink。Script output、tool result 和文件状态可以提供 runtime evidence，但不能成为 instruction 或 approval reason。第二，lease table、budget counters 和 consumption state 必须属于 checker state \(\sigma\)，而不能被 env context 或 adapter 本地状态替代。第三，同一个授权对象必须出现在 tool call 之外的 prompt placement、Skill instruction placement、本地环境执行和 delegation handoff 边界，否则 untrusted context 仍可通过间接边界影响 authority。

\sys 使用 field-owned protected-decision lease 实现这个授权单位。Lease 绑定 operation、object、arguments、intent derivation、control/data provenance、temporal state、budget 和 delegation，并从四类输入中取证。Agent-owned authority-root context 证明用户授权的目标、对象、sink、approval 和 delegation root；它来自 trusted issuer canonicalization，不是模型自己生成的 self-issued context。Instruction context 证明被 endorsement 的 workflow scope。Tool context 证明 callable、schema、credential scope 和 sandbox contract。Runtime-observation/env context 证明本次运行实际观察到的路径、stdout、tool result、文件状态和 side-effect evidence。\sys 的核心 system API 是 \(\mathsf{check\_and\_consume}\)。Adapter 提交的不是“请求是否允许”，而是一条 authority-state transition 的 commit record。LLM-assisted compiler 只提出 plans 和 candidate leases，不能铸造最终权限。Runtime adapters 将通过的 leases 落到 context construction、Skill instruction placement、local tool/MCP-style policy events、本地环境执行、delegation handoff 和 env-projection replay lowering。OS monitor 或 ActPlane-style backend 只接收 env/local-effect projection，即 file/process/network/object predicates、holder、budget 和 provenance/influence labels；它不能生成 agent-owned sink/approval proof、instruction endorsement、tool schema owner proof 或 delegation graph mutation。在 PDF workflow 中，\texttt{create\_issue(repo=org/repo-x)} 必须先提交 agent-owned repo/approval、instruction-owned workflow、tool-owned schema 和 env-owned summary evidence，并在 checker 接受后消耗 one-shot lease。

原型评估目标是 instrumented protected-decision boundaries 上的 safety 和 enforceability。E1 显示 event model 没有排除 benign reference behavior。E2 显示 issuer ownership 和 same-transition lifecycle update 不是命名偏好，因为 weakened variants 会重新接受完整 checker 拒绝的 protected decisions。E3 显示同一 commit API 可以放到 tool call 之外，并在本地 side-effect、prompt placement 和 delegation boundaries 上执行。因此，本文的安全 claim 是可证伪且有边界的。在已 instrumented protected-decision boundaries 内，checker 接受的事件必须满足 owner-field proof、active lease、intent derivation、provenance constraints 和同一次 transition 中的 lifecycle update。这个 claim 不覆盖未接入的 runtime boundary、LLM 内部因果归因、生产 sandbox 完整性、完整因果式 context influence 归因、或所有 prompt injection 变体。

因此，本文贡献限定在已 instrumented protected-decision boundaries 上的模型、runtime/system contract 和有界证据。核心贡献不是新的 policy predicate，而是 agent authority transition 的 linearization interface：field ownership、provenance、lease lifecycle 和 delegation state 必须在副作用前同一次提交。四类 context 是该接口在本文 prototype 中的 proof-owner instantiation，不是全局 component taxonomy；本文不声称 global taxonomy minimality、生产级 ActPlane integration 或 benchmark-scale utility。

\begin{enumerate}[leftmargin=*,label=\textbf{C\arabic*.}]
  \item \textbf{Field-owned lease model.} 一个 per-field owner protected-decision lease model，把 agent authority-changing decisions 表示为带 owner-typed proof projections、no-substitution/no-promotion rules 和 lifecycle mutation 的状态转移；agent/instruction/tool/env 是本文原型对这些 owners 的实例化（第~\ref{sec:formal} 节）。
  \item \textbf{Authority-state commit interface.} 一个 adapter-facing \(\mathsf{check\_and\_consume}\) interface 和 local authorization substrate，要求已接入 adapter 在 effect、placement 或 handoff 前提交 typed field proofs、checker-owned lease-state version 与 same-transition consume/delegate update，并在 local callable、env side-effect、prompt/Skill placement、delegation、env-projection deterministic replay target 和 local bubblewrap namespace sandbox 上实现同一 pre-effect contract；该实现不是生产级 OS/ActPlane integration（第~\ref{sec:design}--\ref{sec:implementation} 节）。
  \item \textbf{Bounded evidence for model and system claims.} 一个围绕 protected-decision counterexamples 组织的实验评估，显示 event model 覆盖 benign reference actions，已测试的 owner/lifecycle/local-boundary removals 会导致 false accepts；完整 checker 在 E2 中保持 0 unsafe accepts，local adapters 在 E3 和 integrated local workflows 中保持 0 unsafe executions/placements，并在 paired workflow 中区分同源 context 的 data use 和 control use（第~\ref{sec:evaluation}--\ref{sec:evidence-status} 节）。
\end{enumerate}

\section{背景与动机}

\subsection{Agent Extensions}

现代 agent runtime 通常不只暴露一个固定工具集合。Skills 将任务说明、参考资料和可选脚本打包为可复用工作流。MCP servers 将工具 schema、资源和 prompts 暴露给 agent。本地脚本和 shell 执行提供文件转换和项目操作能力，subagents 则把上下文和部分权限交给另一个模型或执行单元~\cite{skillguard,skillscope,mcp-spec}。因而，agent extension 同时包含自然语言指令、结构化 tool metadata、外部数据和可执行代码。这个组合使权限不能只绑定到 component identity：同一个 Skill、server 或 command 在不同 run 中会携带不同的过程、接口和运行时证据。

\subsection{Authority and Provenance}

传统权限系统主要围绕 subject、object 和 operation 建模，例如进程是否能读取文件、调用 API 或连接网络。Agent 场景还需要记录 provenance，因为同一个 tool call 的参数可能来自用户选择、扩展说明、工具结果或不可信文档。已有 provenance、taint 和 IFC-style 系统说明不可信输入不应自由流向高影响输出~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}，但它们通常把核心对象放在 observed action 或 data flow 上，而不是把未来 protected decision 本身作为可消耗、可衰减的授权对象。本文中的 protected decision 指会选择、扩大、消耗或转移 authority 的字段级决策，例如 sink selection、approval scope、capability request 或 delegation target。后续设计因此同时检查 provenance 和 lease lifecycle，而不是把 provenance label 当作完整授权。

\subsection{Motivating Workflow}
\label{sec:motivation}

本文使用一个文档处理工作流贯穿设计和评估。用户要求 agent 从两个选中的 PDF 中提取表格，保存 XLSX，并在指定 GitHub repository 中创建一个 issue。PDF text 可以影响表格单元格和 issue body，但不应影响 repository selection、approval scope、capability request 或 delegation target。这个例子把普通 data use 和 authority-bearing decision 分开，使问题不依赖某一种具体工具或 sandbox。

如果其中一个 PDF 包含隐藏指令要求 agent 请求 full GitHub access、加载另一个 Skill，或把 issue authority 交给 subagent，普通 argument check 只能看到最终调用是否仍然使用合法 repo。它无法表达 PDF text 不能成为 approval scope、fresh capability request 或 delegated capability set 的控制来源。\sys 的 running example 因此不是一个错误参数问题，而是一个 context-to-decision influence 问题。

即使没有恶意输入，同一个 Skill、MCP server 或 command 在不同 run 中也需要不同 authority。PDF Skill 用于本地 XLSX 转换时不需要网络或 GitHub issue lease。同一 Skill 用于生成 issue summary 时需要一次性 GitHub sink authority，但仍不应选择 repository。若权限只能在安装期、server 级别或静态 allowlist 中给出，系统通常只能声明覆盖多个 run 的 union authority，或把合法 run 拆成额外授权步骤。这是结构性授权粒度问题，而不只是 prompt injection 问题。例如，同一个 PDF Skill 在 local XLSX run 中只需要 selected-file read、本地 output write 和无网络 exec；在 GitHub issue run 中还需要一次性 issue lease。Skill manifest 若覆盖二者，只能声明二者的 union authority；若声明得更窄，又需要在第二个 run 重新由更高层 issuer 补齐缺失字段。

这个工作流暴露出四个不可代填的证明问题：谁授权了当前目标、对象、sink、approval 和 delegation root；哪个 Skill/manual instruction 被 endorsement 并可指导 workflow；哪个 tool、MCP method、command binary 或 sandbox contract 可以被调用；以及本次运行实际观察到了哪些路径、stdout、tool result、文件状态和 side-effect evidence。前三个问题只描述意图、过程和接口。缺少第四个 runtime-observation/env 问题时，checker 只能退化为静态 prompt/tool policy，无法证明本次执行绑定到哪些具体观测值。相反，若把 env 与 instruction 或 tool 合并，运行时文本就会继承过程说明或接口元数据的可信度。因此本文后续把 agent、instruction、tool 和 env 作为四类默认 authority-input boundaries，而不是三类 component contexts。

\subsection{Protected-Decision Transitions}

受保护转移（protected-decision transition）指一次会 create、select、widen、spend 或 transfer authority 的可观察状态转移。Tool selection、sink selection、approval scope、capability request、policy update 和 delegation target 都属于这类转移。实现中把这些转移编码为 protected events。一个 guard 如果只检查最终 tool name 或 repo argument，会漏掉 untrusted context 对 approval scope、fresh authority request、policy patch 或 delegated capability set 的控制。因此，本文要保护的对象不是“某个字符串能否进入 prompt”，而是“某个 context source 是否被授权影响某类未来决策”。

三个术语在后文保持固定含义。Decision class 是受保护字段的类型，例如 sink selection、approval scope 或 delegation target。Protected event 是 instrumentation 在 runtime 看到的具体事件，例如一次 MCP 调用、approval request 或 subagent spawn。受保护转移是 checker 对 protected event 执行的原子状态更新，包括 proof check、lease consumption、temporal update 和 delegation-state update。这些 decision classes 定义了后续 threat model、lease language 和 evaluation oracle 的共同边界。

\subsection{Problem Characterization}

我们先在已有 traces 和 controlled counterexample artifacts 上检查这个授权单位是否由问题本身导出，而不是由 \sys 的实现事后规定。该检查不运行模型，也不同步新数据；它只读取已有 provenance labels、event modes 和 protected-decision labels，并使用预先声明的 protected-decision oracle：authority-bearing fields 必须由对应 owner class 证明，不能由其他 class 代填。结果形成三条设计前观察。

第一，agent authority decisions 通常需要多个 issuer classes。8,696 个已有事件中，8,691 个需要不止一种 proof source，说明高影响转移不是单一“工具是否可调用”的问题。第二，runtime-observation/env proof 不能省略。198 个事件需要本次运行的路径、stdout、tool result 或 side-effect evidence，否则 checker 只能退化成静态 prompt/tool policy。第三，错误接受常来自 class substitution，而不是最终参数变化。3,593 个 protected-decision-oracle-denied events 存在某类 context 试图替另一类 owner 补齐 field 的情形；48 个 sampled labels 中，37 个暴露这种 substitution attempt。这些数字是 artifact-derived、author-adjudicated design evidence，用于检验设计粒度，不是自然频率统计或 independent-label study。

这些观察的作用不是证明自然出现频率，而是给出设计粒度的证据。它们把本文设计推向 protected-decision authority-state commit object：checker 必须在同一次 pre-effect transition 中看到 owner projections 和 lifecycle mutation。详细派生规则、sampled labels 和同事件 false-accept 结果放在 E2 中报告。

\section{威胁模型与非目标}

\sys 保护的是已经通过 adapter 暴露给 checker 的受保护决策。Agent 可以读取外部文档、加载 Skills、连接 MCP servers、运行本地脚本、调用工具，并将部分任务委托给 subagents。攻击者可以控制部分输入 context，例如 PDF、网页、邮件、issue comment、MCP tool result、unsigned or malicious Skill text、script output 或 subagent summary。Signed/trusted Skill instruction 只有在被 issuer endorsement 后才能作为 instruction context 指导声明范围内的 workflow。Installed-but-overbroad Skill package 也不能因此获得 sink、approval 或 delegation authority。

攻击者的目标不是单纯让模型输出某段文本，而是让不可信 context 控制 authority lifecycle event。典型目标包括选择错误 sink、请求更宽 approval、修改 capability request、复用已消耗 lease、把权限委托给错误 subagent，或把运行时输出提升为 workflow instruction。读取无关私有文件、创建错误 repository 的 issue、修改策略、跳过检查或委托给恶意组件，是这些 unauthorized influence 在具体边界上的表现形式。

TCB 包括 intent certificate issuer、deterministic checker、provenance tracker、audit log 和已经接入的 enforcement adapters。LLM、外部文档、工具结果、MCP annotations、unsigned Skill instructions、Skill/package comments、script static-analysis summaries 和自动 compiler proposals 都不在 TCB 内。Adapter 必须在 side effect、prompt authority placement 或 authority handoff 前提交结构化 field proofs。Checker 只相信 adapter 提交的结构化证明，不相信自然语言解释。

TCB 边界是 per-boundary 的。没有接入 adapter 的 side effect 不在安全性质覆盖范围内。OS sandbox 或 ActPlane-inspired monitor 可以作为本地 file/process/network enforcement 后端，但它们执行的是 checker 已经决定的 lease contract。\sys 的安全性质覆盖已建模的 decision classes 和已接入的 enforcement boundaries。Prompt injection 的语言层诱导、自动 compiler 的完整正确性、生产级 sandbox 替换，以及未 instrumented 的 agent 决策，属于这组性质之外的系统边界。

\section{系统设计}
\label{sec:design}

\subsection{系统概览}

\sys 的架构围绕一个 pre-effect checker transition 组织：untrusted compiler 提出 plan 和 candidate leases，trusted checker 消费 issuer-owned proofs 并返回 audit commit id。Agent 授权因此是副作用发生前的 multi-issuer commit protocol，而不是对已完成 action 的事后过滤器。LLM 可以读取任务、扩展说明和上下文，并提出 plan、effects 和 candidate leases，但它不能铸造最终权限。最终 authority 只来自 trusted intent certificate，并由 deterministic checker 根据 context provenance 和 global policy 验证。

图~\ref{fig:architecture} 展示了五个组件的边界。Trusted issuer 将用户请求、显式对象选择、sink 选择、管理员策略和 fresh approval canonicalize 成 intent certificate。Context labeler 为用户输入、文档、Skill instructions、MCP metadata、tool results、script outputs 和 subagent summaries 标注 origin、integrity、confidentiality 和 influence modes。

LLM-assisted compiler 从扩展元数据、tool schemas、local scripts 和当前 plan 中生成 security-relevant effect IR，并提出短期 leases。Compiler 不在 TCB 内，不能直接修改 authority state。Deterministic checker 验证每个 lease 和事件是否满足 intent derivation、argument constraints、control/data provenance、flow、temporal、budget 和 delegation rules。Adapter contract 要求 context construction、local tool/MCP-style policy events、本地执行和 subagent delegation 在对应 protected transition 前提交证明。

在 PDF workflow 中，issuer 首先把用户选择的两个 PDF 和 \texttt{org/repo-x} sink 写入 intent certificate。Labeler 将 PDF bytes/text 标为 env context，将 PDF Skill 说明标为 instruction context，将 GitHub-style method schema 标为 tool context。Compiler 可以提出 extraction effects 和 \texttt{create\_issue} lease，但 checker 只会 mint/consume repo 由 agent context 授权、body 可由 PDF data 影响、且 budget 尚未耗尽的 lease。Local tool/MCP-style、local env 和 delegation adapters 在副作用或 handoff 前调用同一个 checker 决策。

\subsection{设计目标}

Running example 中的失败轨迹对应四个设计需求。Wrong sink 或 approval widening 需要 intent root 限定用户真正授权的对象和 sink。合法数据流但非法 authority field 需要 context no-promotion 和 influence separation。第一次合法 issue call 与第二次 consumed-lease reuse、parent lease 被交给 child subagent 等轨迹拥有相同或相近 action arguments，却需要 stateful lease lifecycle 才能区分。合法 tool guard 仍可能漏掉 prompt placement、Skill instruction placement、env side effect 或 delegation handoff，因此需要 multi-boundary transition interface。

这四个目标直接对应引言中的两个根因。G1 修复 authority-root collapse：只有 agent-owned user intent、selection、policy 或 fresh approval 可以 mint 或 bound high-impact authority。G2 修复 field-owner collapse：即使 operation 和 object 合法，每个 protected field 也只能由对应 owner class 证明，且不能被其他 class 代填或提升。G3 修复 authority-state split：它要求 lease lifecycle 在同一次 transition 中检查并更新。G4 则把这些 invariant 放到已 instrumented 的 prompt placement、Skill placement、env side effect 和 delegation handoff 边界；对未实现的生产边界，本文只定义 adapter obligation。

Fail-closed denial feedback 和 adapter coverage 是实现这些目标的机制要求，而不是独立顶层目标。因此，设计目标包括四个可测试目标：

\begin{enumerate}[leftmargin=*,label=\textbf{G\arabic*.}]
  \item \textbf{Intent-bounded authority root.} High-impact authority 的 minting 和 upper bound 必须来自当前用户意图、显式对象/目标选择、管理员策略或 fresh approval。
  \item \textbf{Field-owner no-substitution.} 系统必须阻止 env、tool 或 instruction context 代填不属于自己 owner class 的 protected fields，并禁止 runtime text、tool metadata 或 workflow text 被提升为 authority root。
  \item \textbf{Atomic lease-state mutation.} Allow decision 必须和 lease consumption、expiry progress、holder scope 与 parent-child attenuation 的状态更新原子绑定，而不是分散在独立 guards 中。
  \item \textbf{Instrumented multi-boundary transition exposure.} 已接入的 local callable、env side effects、context placement、Skill instruction placement 和 authority handoff 必须在 effect、placement 或 handoff 前暴露同一种 protected transition；未接入的生产 MCP、prompt builder、subagent runtime 或 kernel boundary 只作为更强部署证据需求。
\end{enumerate}

\subsection{授权输入}

\sys 的 capability synthesis 不是从单一 prompt context 中抽取权限，而是从一组 proof-carrying issuer boundaries 中求解 lease。通用模型只要求每个 protected field 有唯一 owner，并要求 owner proof、checker-owned lease state 和 lifecycle mutation 在同一次 transition 中提交。在本文的 agent-extension prototype 中，这些 owners 实例化为四类授权输入类（authority-input classes）：agent、instruction、tool 和 runtime-observation/env。它们不是四个组件名，也不是全局 taxonomy，而是当前 workloads 中四个证明问题的 owner-equivalence partition：agent 回答谁可以授权，instruction 回答哪个过程可以指导，tool 回答哪个接口可以执行，env 回答本次运行实际发生或观察到了什么。这四个问题的 issuer、可伪造面、观测点和 state-write authority 不同，因此不能只作为一组普通 attributes 合并。Agent context 证明用户授权的 goal、objects、sinks、approval tokens 和 delegation root。Instruction context 证明被 endorsement 的 workflow scope 和 procedure guidance。Tool context 证明 callable name、schema、credential scope、binary descriptor 和 declared side-effect contract。Env context 证明本次运行观察到的 concrete args、cwd、路径存在性、tool results、script outputs 和 observed side effects。

用户原始文本不是第五类 authority input。Trusted issuer 会把用户文本 canonicalize 成不同 proof cells：goal、object、sink、approval 和 delegation root 进入 agent-owned intent fields；workflow preference、formatting preference 或过程约束进入 instruction-owned fields。即使某个 workflow preference 来自用户，它也不能单独替代 agent-owned sink、approval 或 delegation root。

Env context 是必要的执行证据边界，不是 Unix environment variables。若只使用 agent intent、instruction/Skill/manual 和 tool/MCP interface 三类上层来源，checker 只能知道“用户想做什么”“过程如何描述”“接口如何声明”，却不能证明本次运行实际看到了哪些路径、stdout、tool result、文件状态、资源状态或外部数据。缺少 env 类，系统会退化成 prompt/tool policy。把 env 合并进 tool 或 instruction，则会让运行时文本继承接口可信度或过程可信度。反过来，把 tool 和 env 合并也不安全：tool schema 是执行前的接口承诺，env evidence 是执行中或执行后的观测事实。前者可以证明参数形状和 credential scope，后者只能把观测值绑定到已经授权的 predicate，不能反向证明接口或授权范围。

这四类的 novelty 不在名称，而在 safe-merge derivation。推导从 protected fields 开始，而不是从 artifact 名称开始：先列出 sink、approval、workflow scope、callable schema、credential scope、observed path、stdout、delegation root 等 field families；再判断两个 field families 是否拥有相同 issuer、可伪造面、观测边界和 lifecycle authority；只有完全等价的 field families 才能共享 owner。形式化模型允许更细的 per-field owners；四类只是本文 workloads 和 adapter surface 经过 safe merge 后得到的 prototype owner set。E2 和 integrated workflow 只证明当前接入的 Skill/MCP/cmd/subagent/env workloads 中这些 owner projections 不能安全合并，不声称所有 agent 系统只能有四类 context。同一段文本如果来自被 endorsement 的 \texttt{SKILL.md} 或手册，可能在 workflow scope 内成为 instruction context。如果它来自普通 PDF、网页、tool result、script output 或 subagent summary，即使内容像指令，也仍是 env context。Command-line tools 也不是第五类。Binary、argv schema 和 sandbox profile 属于 tool context，实际 cwd、输入文件、stdout/stderr 和文件系统变化属于 env context。

E2 把 safe-merge criterion 转化为可证伪的 tested-removal contract。四类不是一个关于 agent 世界的 ontology，也不是声称所有系统都只能有四类 owner，而是本文接入边界中的四个不可代填证明问题：谁授权了当前目标和 sink，哪个过程说明被 endorsement，哪个接口或 command 可以被调用，以及本次运行实际观察到了什么。若把任意两个问题放进同一个 proof boundary，checker 就需要额外证明其中一个问题的答案不能被另一个问题的 issuer 伪造或代填。E2 正是按这个 contract 测试折叠：tool\(\rightarrow\)agent 让 interface metadata 供应 user authority，env\(\rightarrow\)agent 让 runtime text 供应 user selection，env\(\rightarrow\)tool 让 observed output 供应 interface proof，instruction\(\rightarrow\)agent 让 workflow text 供应 approval。

错误合并会产生不同的 false authority path。把 instruction 并入 agent 会把 Skill/manual 中的过程建议提升为用户意图或 approval。把 tool 并入 agent 会让 server/interface description 变成用户授权。把 env 并入 tool 会让 runtime observations、script outputs 和文档内容获得本不具备的接口可信度。把 tool 并入 env 又会丢失 schema、credential scope 和 declared side effect 这些接口约束。ABAC 问的是一组 attributes 是否满足 policy rule。\sys 问的是哪些 issuer-owned fields 可以共同提交一次 authority-state transition，并且禁止一类 issuer boundary 替另一类补齐缺失字段。

No-substitution rule 的作用是阻止一类 projection 替另一类 projection 证明 field。若把 instruction 和 tool 放进同一证明通道，Skill/manual 的过程文本就可能替代 MCP schema、command binary 或 credential scope。若把 env 和 tool 放进同一证明通道，运行时结果、stdout 和文件状态就可能继承工具接口的静态可信度。若把 instruction 或 tool 放进 agent-owned 证明通道，扩展说明或 server metadata 就可能替代用户选择、approval 或 delegation root。若一个实现内部仍保留 per-field owner projections、proof objects 和 same-transition lifecycle update，它就在这些 fields 上实现了本文所需的 commit interface，而不是本文反驳的 collapsed interface。

这四类也不是 component taxonomy。一个 Skill package 同时可能携带 instruction context（\texttt{SKILL.md} 和手册）、tool context（bundled script descriptor 和 argv schema）以及 env context（script stdout、生成文件和错误日志）。一个 MCP server 同时可能携带 tool context（schema、server identity 和 credential scope）以及 env context（tool result 和资源状态）。一个 agent run 同时持有 agent context（用户 intent、approval 和 delegation root）与 instruction context（用户给出的格式偏好或 workflow preference）。因此，\sys 不按“来源属于 Skill、MCP 还是 command”铸造 capability，而按每个 protected field 的 issuer owner 铸造 capability。文档手册和 Skill instruction 只约束过程和 prompt placement；MCP/tool/cmd metadata 约束接口、参数 schema、credential scope 和 sandbox lowering；runtime environment 证明本次执行实际观察到的路径、stdout、tool result 或 side effect；agent intent 给出目标、对象、sink、approval 和 delegation root。实际 lease 是这些 proof projections 对同一次 protected transition 的合取：少一类 required proof 就拒绝，任一类 proof 越权代填另一个 field 也拒绝。

实际 capability 必须由相关 protected fields、当前 intent、checker lifecycle state 与全局策略共同决定。令 \(C_q\) 表示 class \(q \in \{agent,inst,tool,env\}\) 的 canonicalized proof-projection set。四类 proof 的互斥和 \(C=C_{agent}\uplus C_{inst}\uplus C_{tool}\uplus C_{env}\) 只约束 canonicalized proof cells，不约束 raw artifact 或 component：同一个 Skill package、MCP server 或 agent run 可以被 adapter 拆成多个 proof cells，但每个 protected field proof 必须有唯一 issuer owner。未被某个 decision class 要求的输入类可以为空，但被要求的 field 必须从自己的 owner class 取证：

\[
K_d = Solve_\Gamma(I,\sigma,d,C_{agent}, C_{inst}, C_{tool}, C_{env}) .
\]

这里的 \(Solve\) 是合取约束而不是权限并集。\(I\) 给出当前 run 的 authority root，\(\sigma\) 给出 active/exhausted lease、budget、expiry 和 delegation graph，\(d\) 给出本次 protected decision 的 required fields。Agent context 给出目标、对象和 sink 的上界。Instruction context 只能在该上界内约束过程。Tool context 只能暴露可调用接口、凭证范围和 sandbox contract。Env context 只能把本次运行实际观测到的值绑定到这些 lease predicates。任一 required field 缺失都会使 lease 不可铸造。相反，若把四类 context 做成 union authority，最强或最会写自然语言的一类输入就可能替其他类补齐字段。

这个定义也给出 safe-merge test。任何实现若要合并两个 proof projections，都必须证明该合并不会让 workflow text 获得 user-owned sink、approval 或 delegation root，不会让 runtime stdout/tool result 继承静态 schema/credential trust，也不会丢失 callable schema 和 sandbox contract 这类接口约束。若该合并产生 \(Accept_h\Rightarrow Accept\) 的反例，它就不是等价 proof boundary。

纯 approval event、纯 context-use event 和本地 evidence event 可能只需要其中几类字段。关键规则是：对某个 protected decision 所需的字段，checker 必须从该字段指定的输入类读取，不能由其他输入类代填。例如，agent context 可以授权“处理这两个用户选中的 PDF 并在 \texttt{org/repo-x} 创建一个 issue”，但不能指定 PDF Skill 的脚本路径。Instruction context 可以指导 extraction workflow，但不能选择 GitHub sink。Tool context 可以说明 \texttt{github.create\_issue} 的参数结构和 credential scope，但不能把任意 repo 变成授权 sink。Env context 可以证明某个路径、参数值或 observed result 在运行时存在，但不能把存在性变成用户选择或 retroactively mint broader authority。

Lease 的作用是把这些 required fields 固定为一个可检查、可消耗、可衰减的 transition object。

\subsection{意图证书}

Intent certificate 是结构化安全对象，而不是 prompt 中的一段自然语言。它记录用户目标、选择的对象、允许的 sinks、禁止的 sinks、approval tokens 和 expiry。对于 PDF 处理任务，certificate 会命名选中的 PDF 路径、输出目录、唯一授权的 GitHub repository、禁止任意网络访问的约束，以及创建 issue 所需的 fresh approval。LLM 可以提出 certificate fields，但 trusted parser、UI、administrator policy 或 approval step 必须 canonicalize 这些字段。Certificate 是后续 capability derivation 的根。

Certificate 的粒度决定后续 leases 的最大 authority。若用户只选择了两个 PDF，后续 file-read leases 只能覆盖这些对象或它们的安全派生物。若用户只确认了一个 GitHub repository，repo selection 就不能由 PDF text、Skill instruction 或 MCP result 改写。模糊字段不会被解释为宽权限。系统要么将其保守地标记为 unknown，要么要求 fresh approval。

\subsection{上下文影响权限}

每个 context cell 都带 provenance label，记录 origin、integrity、confidentiality、purpose、time 和 endorsements。\sys 的关键增加是为 context source 指定 influence modes。Influence authority 是 decision-specific 的：一个 source 可以被允许影响 output content，却不能影响 recipient、repository、network destination、approval scope 或 delegated capability set。Signed Skill instructions 可以在 declared workflow scope 内 instruct extraction procedure，但不能为自己请求额外工具。MCP tool results 和 annotations 必须按照 server trust 处理。MCP specification 已经提醒 tool behavior annotations 不能默认可信~\cite{mcp-spec}，\sys 将这一原则变成 deterministic provenance rule。

PDF text 可以流向 issue body 和 spreadsheet cells。普通 IFC 可以表达这条数据流是否允许。\sys 还要求 repository field 的 issuer 是 agent 输入类，approval scope 的 control proof 不含 PDF provenance，并且对应 one-shot lease 尚未被消耗。只有当 influence authority 与 intent derivation、budget、expiry 和 delegation bounds 绑定在同一 lease 上时，checker 才能防止 context 在一个字段上被允许、又在另一个生命周期状态中被复用为授权依据。这种 decision-specific authority 允许系统保留有用的数据流，同时切断同一 context 对 authority-bearing fields 的控制流。

实现会把抽象 influence mode 映射为 adapter 提交的 decision-mode 字段。表述上，influence mode 指抽象能力，decision class 指被影响字段，事件字段记录本次使用所需的最低影响权限。Quote、summarize 和 parameterize 对应 output body、spreadsheet cells 或 issue body 的数据使用。Plan 和 instruct 对应 tool selection 与 workflow-step choice。Authorize 对应 sink selection、approval scope、capability request 和 policy update。Delegate 对应 delegation target 和 child capability set。这个映射使论文中的抽象权限集合与实验 trace 一致，同时避免把所有 modes 都解释成同一种 taint level。

\subsection{Effect IR 与 Leases}

Agent plan 被编译成安全相关 effect graph。IR 不追求表达完整程序语义，而是捕捉会影响 authority 的事件：context use、tool call、MCP call、file read/write、subprocess execution、network connection、approval request 和 subagent delegation。每个 effect 记录 data provenance 和 control provenance。这个区分是 \sys 的关键：不可信 PDF bytes 可以是 output data 的 data dependency，但不应是 sink selection 的 control dependency。

Control provenance 不是声称系统能读出 LLM 内部注意力，而是一个 protected-field proof policy。对于 sink、approval scope、capability request、delegation set、policy patch、local executable、network destination 和 high-impact arguments，runtime adapter 必须提交结构化来源证明。字段可以来自 trusted UI canonicalization、fresh approval、tool registry/schema、runtime observer、context placement gateway，或某个 explicit context cell。模型自由生成字段本身不产生 proof，只能触发 adapter 查找对应结构化来源。

若字段由模型自由生成、缺少结构化 proof，或来源不能被 adapter 归因，checker 会保守处理。它把该字段的 control provenance 近似为当前 prompt 中所有可见、可能影响该字段的 non-policy context。只要其中存在缺少所需 influence mode 的 context，checker 就保守拒绝。普通 output body 或 spreadsheet cell 可以带 data provenance。Authority-bearing fields 必须带 control proof。这一策略会带来可用性代价，因此 E1/E3 把 false denial、proof completeness 和大规模 recovery 评估与 same-event safety 分开报告，而不是假设 provenance 总是精确。

Lease 绑定 operation、object、arguments、intent、influence、flow、temporal、budget、expiry 和 delegation。一个 PDF extraction lease 可以允许脚本读取两个选中文件并写入输出目录，但禁止网络、secrets 和额外 subprocess。GitHub issue lease 可以允许一次 \texttt{create\_issue(repo=org/repo-x)}，但要求 repo control provenance 只能来自 trusted user intent。Lease solving 的目标不是全局证明所有可能 plan 的最小权限，而是在给定 effect plan、extension catalog 和 global policy 下生成风险更低、可检查、可衰减的 authority set。

\subsection{Compiler 与 Checker}

\sys 使用 LLM-assisted 但 checker-verified 的 pipeline。Compiler frontend 总结 Skill metadata、MCP schemas、local scripts、observed traces 和 policy。它推导 plan 与 security-relevant effects，并为当前 run 提出 candidate leases。Frontend 不在 TCB 内。Deterministic checker 验证每个 lease 是否满足 global policy、intent certificate、provenance constraints、flow rules、temporal guards、budgets 和 delegation monotonicity。

Compiler 的输出必须携带 proof object，说明每个 protected field 来自哪些 intent fields、哪些 context sources 可以作为 data/control dependency、哪些 sources 被排除、以及 lease 在什么 temporal state 和 budget 下有效。Checker 不信任 proof 的自然语言解释，只检查其中的结构化 predicates。若 proof 缺失、argument predicate 过宽、control provenance 不明、或 lease 试图把 parent 不具备的 authority 委托给 child，checker 返回结构化 denial。

Denial 是 runtime refinement 的输入，不是绕过授权的通道。LLM 可以根据 denial reason 重新规划，例如选择更窄的 sink、补充 read-only proof-gathering step、或请求用户 fresh approval。但新 proposal 仍要经过同一个 checker。这一设计使 recovery 可以改善 utility，而不会把失败路径变成宽权限回退。

\subsection{运行时降低}

单一 tool gateway 不足以表达 \sys 的安全边界，因为 protected decision 可能发生在工具调用之前的 context construction、计划生成和 approval request 中，也可能发生在调用之后的 delegation 或 runtime binding 中。不同 boundary adapters 共享同一个 checker transition；差别在于它们为同一次 transition 提交哪些 agent、instruction、tool 和 runtime-observation/env projections，以及 checker state \(\sigma\) 在该 transition 中读取或更新哪些字段。表~\ref{tab:runtime-lowering} 给出这些边界。

所有 adapter 调用同一个状态转移接口。这个接口的语义不是普通 reference monitor 的“check before use”。Adapter 必须提交一个完整 commit record，说明本次 transition 要读取哪些 issuer-owned fields、消耗哪个 lease、把哪些 runtime observations 绑定到参数、以及成功后如何更新 checker-owned state。Checker 接受和 state update 是同一原子事件；若只先检查 predicate、再由另一个组件更新 budget 或 delegation graph，就重新引入 authority-state split。

形式上，接口为：

\[
\mathsf{check\_and\_consume}(e,lease\_id,field\_proofs,prov,\sigma)
\rightarrow \mathsf{allow}(\sigma',audit) \mid \mathsf{deny}(reason).
\]

\(e\) 已包含 operation、object 和 argument values。\(lease\_id\) 指向候选 active lease。\(field\_proofs\) 是 issuer-typed map。Agent proofs 绑定 intent、object/sink selection、approval 和 parent authority。Instruction proofs 绑定 workflow scope。Tool proofs 绑定 schema、credential scope、binary 和 sandbox contract。Env proofs 绑定 runtime observations、tool results、路径状态和 side-effect evidence。Adapter 可以不同，但它们不能本地修改 lease table、budget、expiry 或 delegation graph。若 checker 返回 deny，adapter 不执行 side effect、不写 prompt section、不创建 handoff。若 checker 返回 allow，\(\sigma'\) 已经包含 budget consumption、temporal progress、audit record 和 delegation-state update。

\section{形式化模型}
\label{sec:formal}

\sys 的形式化目标不是为任意自然语言推理建模，而是为上一节 checker pipeline 的输入和输出提供可检查语义。Compiler、gateway 和 runtime adapters 负责把模型计划、工具调用、MCP 调用、本地执行和 delegation 转换成安全相关事件。Checker 只对这些事件、active leases、intent certificate 和 provenance labels 做确定性判断。因此，模型刻意不讨论 LLM 内部是否“理解”指令，而是讨论一个被 instrumentation 暴露出来的 authority-bearing event 是否应被接受。

形式化对象直接来自引言中的三个根因。Issuer collapse 对应 no-substitution 和 no-promotion：每个 protected field 必须由自己的 issuer owner 证明。Authority-state split 对应 atomic check/consume：proof check、budget consumption、expiry progress 和 delegation update 必须在同一次 transition 中完成。间接边界绕过对应 pre-effect adapter contract：tool call、prompt placement、env side effect 和 handoff 都必须在副作用或 authority transfer 前提交同一类 commit record。

模型围绕三个 invariant 展开。第一，\textbf{no substitution}: 每个 protected field 必须由自己的输入类证明，其他输入类不能代填。第二，\textbf{atomic check/consume}: protected event 的 predicate、provenance、budget、temporal 和 delegation state 必须在同一次 transition 中检查并更新。第三，\textbf{no promotion}: env/tool/instruction context 不能仅凭自然语言内容升级为更高 authority class，除非 trusted issuer 产生显式 endorsement 或 fresh lease mint event。

\subsection{Events and Proof Projections}

一个 agent execution 被表示为事件 trace：

\[
\tau = e_1, e_2, \ldots, e_n .
\]

事件语言只覆盖会影响 authority 的边界：

\[
\begin{aligned}
e ::=~& ctx(c,m,d)
  \mid call(t,args)
  \mid mcp(s,t,args)
  \mid read(o)
  \mid write(o) \\
  & \mid exec(p,args)
  \mid send(sink,msg)
  \mid spawn(a,K)
  \mid approve(u,a) \\
  & \mid request(\kappa,reason)
  \mid policy\_update(patch)
  \mid lease(\kappa).
\end{aligned}
\]

每个事件带有两类 provenance：

\[
prov(e)=\langle prov_D(e), prov_C(e) \rangle ,
\]

其中 \(prov_D\) 表示数据依赖，\(prov_C\) 表示控制依赖。\sys 只在 protected decisions 上约束 authority-bearing fields 的 \(prov_C\)，例如 tool selection、sink selection、approval scope、policy update 和 delegation target。这样可以表达同一段 PDF text 可以作为 issue body 的 data dependency，却不能作为 GitHub repository 参数的 owner proof 或 control proof。本文的主实验度量 owner substitution、class promotion 和 lifecycle split；paired data/control coverage 是解释性 guardrail，而不是单独顶层指标。

在 running example 中，repo field 需要 agent-owned proof，而 issue body 只需要 data-use authority。形式化模型先给每个 context cell \(c\) 标注来源和 endorsement：

\[
\ell(c)=\langle origin, integrity, confidentiality, purpose, time, endorsements\rangle .
\]

标签进一步映射到 context authority。这里需要区分 influence mode 和 decision class。Mode 描述 context 的使用方式，例如 quote、summarize、parameterize、plan、instruct、delegate 和 authorize。Decision class 描述被影响的安全相关字段，例如 output content、sink selection、approval scope、capability request、policy update 和 delegation target：

\[
auth(c) \subseteq InfluenceModes \times DecisionClasses .
\]

例如，PDF text 可以 parameterize spreadsheet cells 并 summarize issue body，但不能 authorize capability request、policy update 或 GitHub repository selection。实现可以将 \(InfluenceModes\) 做成偏序或命名集合。论文中的性质只要求 checker 能判断某个 context 是否被授权以所需 mode 影响某个 decision class。

Intent-carrying lease 的抽象形式为：

\[
\kappa =
\langle holder, op, obj, args, I, influence, flow, temporal, budget, expiry, delegation\rangle .
\]

其中 \(I\) 是当前 intent certificate，\(influence\) 约束 control/data provenance，\(flow\) 约束 confidentiality 到 sink 的流向，\(temporal\) 和 \(budget\) 约束事件顺序与调用次数，\(delegation\) 约束可传递给 subagent 的能力集合。

形式上，lease synthesis 可被看作从 per-decision issuer-typed fields、当前 intent 与 checker state 中求解可授权 lease。令 \(\mathcal{O}\) 是当前 runtime 暴露的 owner set，\(\pi_o(C_o)\) 是 owner \(o\) 可提交的 proof projection：

\[
\kappa \in Solve_\Gamma(I,\sigma,d,\{\pi_o(C_o)\}_{o\in \mathcal{O}}) .
\]

其中 \(I\) 是当前 intent certificate，\(d\) 是本次 protected decision class，\(\sigma\) 是 checker-owned lifecycle state。本文 prototype 取 \(\mathcal{O}=\{agent,inst,tool,env\}\)。\(C_{agent}\) 包含 intent certificate、authorized object/sink selections、approvals、parent authority，以及由 UI、policy 或 checker state 签发的 canonical plan stage 和 role；LLM/compiler 产生的 plan stage 或 role 只能作为 untrusted proposal，不能进入 agent-owned authority root。\(C_{inst}\) 包含 system/developer instructions、被 trusted issuer canonicalize 为过程约束的用户 workflow preferences、Skill/manual workflow text，以及由独立 trusted issuer endorsement 的 MCP prompts；默认 MCP prompts、tool descriptions 和 server annotations 仍是 tool metadata 或 env text，不能占用 instruction-owner fields。用户原始文本不会整体进入 \(C_{inst}\)：其中 goal、object、sink、approval 和 delegation root 进入 \(C_{agent}\)，只有格式偏好、过程偏好或 workflow constraints 在 canonicalization 后进入 \(C_{inst}\)。\(C_{tool}\) 包含 MCP/tool schemas、server identity、credential scope、command/script execution descriptors 和 sandbox contracts。\(C_{env}\) 包含 runtime object existence、cwd/filesystem state、concrete arguments、observed results、runtime evidence 和 resource availability。Active/exhausted lease table、per-lease counters 和 delegation graph 属于 checker state \(\sigma\)，不属于任何可由环境输出填充的 context class。

\subsection{Issuer Ownership and Safe Merge}

Owner projections 不是先验 taxonomy，而是从 protected fields 上的 safe-merge question 推导出来的。设 \(F\) 是所有会出现在 protected transition 中的 authority fields。每个 field \(f\in F\) 有五个可审计来源属性：\(issuer(f)\) 指谁能签发该 field，\(forge(f)\) 指哪些输入能伪造或模仿该 field 的值，\(observe(f)\) 指 runtime 在哪个边界能观察该 field，\(write(f)\) 指哪个 state writer 可以创建或更新该 field 对应的 authority state，\(life(f)\) 指该 field 是否参与 mint、consume、expire 或 delegate。语义上的 safe merge 不是“名字是否相近”，而是后文的 \(Accept_h\Rightarrow Accept\) obligation：合并两个 proof projections 后，不能让非 owner 补齐 required field，也不能丢失 schema、observation 或 lifecycle state。下面的五元相等是一个保守、可审计的充分条件，用于标注协议和 removal-family 设计；它不是声明所有系统必须按五元字面相等才能合并 fields：

\[
\begin{aligned}
f_i \equiv_{merge} f_j
\iff{}& issuer_i=issuer_j \land forge_i=forge_j\\
&\land observe_i=observe_j \land write_i=write_j\\
&\land life_i=life_j .
\end{aligned}
\]

四类 owner 是本文 workloads 中 \(F/\!\equiv_{merge}\) 的默认合并结果，而不是按 artifact 名称手工分组。Agent、instruction、tool 和 env 分别回答四个不可互换的证明问题：用户授权了什么，过程如何被允许约束，接口声明了什么，以及本次运行实际观察到了什么。

错误合并会让一个 issuer class 补齐另一个 class 才拥有的 field。若把 \(C_{inst}\) 合并进 \(C_{tool}\)，Skill/manual workflow text 就会和 callable schema、credential scope、binary descriptor 处在同一 issuer class，工具结果或 tool metadata 更容易被误解释为 procedure authority。若把 \(C_{inst}\) 合并进 \(C_{agent}\)，Skill instruction 就可能替用户选择 sink、扩大 approval scope 或 mint lease。若把 \(C_{env}\) 合并进 \(C_{tool}\)，runtime observation 会和静态接口描述混在一起，已执行 tool result、script output 或文件状态就可能替 schema 或 policy 补证明。划分的目的不是命名方便，而是让每个 authority field 只有一个可以签发它的 issuer boundary。

Checker 以四个 projection functions 读取这些输入类。这里的 projection 不是 parser convenience，而是 proof boundary：每个 projection 只返回该 issuer class 拥有的 fields。

\[
\begin{aligned}
intent(e) &\leftarrow \pi_{agent}(C_{agent}),\\
procedure(e) &\leftarrow \pi_{inst}(C_{inst}),\\
interface(e) &\leftarrow \pi_{tool}(C_{tool}),\\
evidence(e) &\leftarrow \pi_{env}(C_{env}) .
\end{aligned}
\]

令 \(F=F_A\uplus F_I\uplus F_T\uplus F_E\) 表示 authority fields 的不相交并集，并令
\[
owner:F\rightarrow\{agent,inst,tool,env\}
\]
给出每个 field 的唯一签发者。代表性 field 如下：
\begin{itemize}[leftmargin=*]
  \item \(F_A\)：\(goal\)、\(authorized\_object\)、\(authorized\_sink\)、\(approval\_scope\)、\(delegation\_root\)。
  \item \(F_I\)：\(workflow\_scope\)、\(procedure\_step\)、\(prompt\_section\)、\(formatting\_preference\)。
  \item \(F_T\)：\(callable\)、\(schema\)、\(credential\_scope\)、\(binary\_descriptor\)、\(sandbox\_contract\)。
  \item \(F_E\)：\(observed\_path\)、\(observed\_arg\)、\(tool\_result\)、\(stdout\)、\(file\_state\)、\(side\_effect\_evidence\)。
\end{itemize}
同一个自然语言片段可以被不同 adapter 观察到，但它能证明哪个 field 只由 \(owner(f)\) 和 endorsement state 决定。

一个 protected event \(e\) 只有在 \(Req(d)\) 要求的 projections 与同一个 lease \(\kappa\) 的 predicates 一致时才被接受。未被 \(Req(d)\) 要求的输入类可以为空，但 required field 不能从其他 context class 中补齐。例如 instruction 不能补齐 authorized sink，tool schema 不能补齐 user intent，env evidence 不能补齐 approval token。这个 typed field rule 使 \sys 不只是 MCP/tool-call guard，而是跨 Skill instruction、tool interface、runtime environment 和 agent workflow state 的统一 checker transition。

对应地，capability synthesis 可以写成一个 partial function，而不是最大权限选择：
\[
synth_\Gamma(d,\Phi,\sigma)=
\begin{cases}
\kappa & \text{if } \forall f\in Req(d).\; typed(\Phi(e,d),f,\kappa,\Gamma,\sigma)\\
\bot & \text{otherwise.}
\end{cases}
\]
其中 \(typed\) 同时检查 proof issuer、proof value、lease predicate、endorsement 和 lifecycle state。这个定义强调三点。第一，\(synth\) 不会把四类输入求并集后取最宽 authority。第二，缺失的 agent approval 不能由 instruction 或 tool metadata 自动补齐。第三，env observation 只能收窄或绑定已经授权的 predicate，例如把输出文件绑定到一个已授权 output prefix，而不能反向扩大 object set、sink set 或 approval scope。

也可以把“四类是否可以合并”写成 soundness question。形式化上可以使用任意更细的 owner set \(Q\)，而本文的 \(Q=\{agent,inst,tool,env\}\) 是从当前 workloads 的 protected fields 合并得到的默认 partition。令 \(h:Q\rightarrow Q'\) 是一个把若干输入类折叠到更少类别的映射，\(Accept_h(e)\) 是 checker 在折叠后只检查 \(h(owner(f))\) 是否匹配时的接受判定。一次 owner-projection 折叠是安全的，必须满足：

\[
\forall e,\sigma,\kappa.\quad Accept_h(e,\sigma,\kappa) \Rightarrow Accept(e,\sigma,\kappa).
\]

这给出一个可测试的 removal-family 反例判据。若存在事件 \(e\)、状态 \(\sigma\)、lease \(\kappa\) 和折叠映射 \(h\)，使得 \(Accept_h(e,\sigma,\kappa)\) 成立但 \(Accept(e,\sigma,\kappa)\) 不成立，则被 \(h\) 合并的输入类不是等价 proof boundary。这样的反例不需要改变 tool name、argument value 或 prompt slot，只需要把同一个 field proof 的 issuer class 从合法 owner 换成被合并后的其他来源。换言之，分类是否需要分开不是由名称决定，而是由接口合并是否会产生 class-substitution false accept 决定。

\textbf{判据 1（owner merge counterexample criterion）.}
对任意折叠映射 \(h\)，若存在 required field \(f\) 和两个输入类 \(q_1\neq q_2\)，满足 \(owner(f)=q_1\)、\(h(q_1)=h(q_2)\)，且 \(q_2\) 的 issuer 可以产生与 \(\kappa.f\) 匹配的 proof value 但没有 \(f\) 的签发权，则存在同 action/argument 的事件 \(e\)，使 \(Accept_h(e,\sigma,\kappa)\) 成立而 \(Accept(e,\sigma,\kappa)\) 不成立。
证明思路是构造一个只替换 proof issuer 的事件：operation、object、argument、prompt slot 和 data provenance 都保持不变，弱 checker 在折叠后的 class 中看到匹配 value，完整 checker 因 \(q_2\neq owner(f)\) 拒绝。
E2 中 tool\(\rightarrow\)agent、env\(\rightarrow\)agent、env\(\rightarrow\)tool、instruction\(\rightarrow\)agent、instruction\(\rightarrow\)env 和 instruction\(\leftrightarrow\)tool 的 rows 是该判据在本文 artifacts 和 controlled suites 上的实例化，而不是关于所有可能 taxonomy 的全局最小性证明。
这个前提不是纯逻辑假设。MCPTox-derived MCP metadata 给出 tool issuer 产生 matching policy/approval value 的 cases，AgentDojo/InjecAgent 和本地 env suites 给出 env text 或 observed output 产生 matching agent/tool fields 的 cases，Skill placement records 给出 unsigned or wrong-source text 产生 matching instruction slot 的 cases。因此 E2 测的是 artifact-derived issuer substitution，而不是只在纸面上构造的反例。

\textbf{推论 1（tested no three-class collapse）.}
在本文 prototype 的四个 owner classes 上，任何三类 owner partition 都至少合并 \(\{agent,inst,tool,env\}\) 中的一对 owner。
若 E2 对六个 pairwise merges 都给出同 action/argument 或同 prompt-slot shape 的 false-accept counterexample，则不存在一个只通过合并这些四类 owner 得到的三类 partition 能在本文 tested adapter surfaces 上满足 \(Accept_h\Rightarrow Accept\)。
这个推论的证明只用 pigeonhole argument 加判据 1：三类 partition 必须合并一对 owner，而该对 owner 在 E2 中已有一个折叠后接受、完整 checker 拒绝的反例。
它的范围也同样受限：它排除的是本文测试过的 collapsed owner interface，不排除一个内部仍保留 per-field owner projections、proof objects 和 same-transition lifecycle update 的实现把外部 API 命名为三类。

第二个等价条件是 lifecycle-linearization equivalence。令 \(Accept_{atomic}(e,\sigma,\kappa)\rightarrow\sigma'\) 表示 \sys 的 \(\mathsf{check\_and\_consume}\) 在一个 transition 中完成 proof check、lease lookup、budget consume、expiry progress 和 delegation attenuation，并返回 effect/handoff 可绑定的 audit record。令 \(Accept_{split}\) 表示一个把 predicate check、budget update、delegation comparison 或 audit write 分散到多个组件或多个时间点的 monitor。二者在本文 decision classes 上等价，必须满足：

\[
\forall \tau.\quad
Accept_{split}(\tau)=Accept_{atomic}(\tau)
\land final_\sigma(Accept_{split},\tau)=final_\sigma(Accept_{atomic},\tau).
\]

该条件要求 split monitor 不仅接受同一批 events，还必须产生同一个 checker-owned lease state：active/exhausted lease version、budget counter 和 parent-child capability relation 都一致。Audit id 需要绑定到 allow decision 以支持审计和事后归因，但本文尚未把它作为单独 false-accept 安全字段隔离证明。若 predicate check 在旧 \(\sigma\) 上允许事件、budget consume 在 side effect 后发生，或 child lease set 没有和 parent active set 在 handoff 前同 transition 比较，则会出现 stale reuse、double consume 或 over-delegation false accept。本文不声称这四类是所有 agent 系统的唯一 ontology；它声称在本文覆盖的 Skill/manual、MCP/tool/cmd、runtime-observation/env 和 delegation boundaries 中，四类是默认 owner-equivalence classes，而任何进一步合并或拆分生命周期的实现都必须证明 owner-projection equivalence 和 lifecycle-linearization equivalence，否则 E2 的 counterexamples 就会给出 unsafe merge 或 unsafe split。

\textbf{命题 2（split lifecycle unsoundness）.}
若一个实现把 allow predicate、budget consumption、expiry progress 或 parent-child attenuation 拆到不同 writer 或不同提交点，并且不能证明其最终 \(\sigma\) 与 \(Accept_{atomic}\) 在所有 interleavings 上相同，则存在同 action/argument 的 trace pair，使 split implementation 接受 stale reuse、double consume 或 over-delegation，而 atomic commit 拒绝。
证明思路是让第一次事件在旧 \(\sigma\) 上合法并应消耗 one-shot lease，再让第二个相同 action 或 child handoff 在消费/比较尚未线性化的状态窗口中提交。
Typed-provenance state guard 阻止 6/7 个 workflow residuals 但漏掉 over-delegation，正是该命题的边界例：继续加入 parent-child lease-set comparison、checker sole-writer state 和 same-transition child update 后，它与 \sys 在该 residual class 上收敛为同一接口。
该前提也由 workflow artifacts 支撑。Integrated workflow 和 local delegation records 包含合法 first issue、duplicate one-shot issue、parent-to-child handoff 和 child sink widening 的相同或相近 action shapes。它们说明 lifecycle split 的风险不是抽象并发 bug，而是 agent workflow 中自然出现的 authority reuse 和 delegation update 问题。

\textbf{Field-issuer linearization requirement.}
综合判据 1 和命题 2，任何在本文 decision classes 上与 \sys 等价的 monitor 都必须给出两个 obligation：owner-equivalence obligation，说明合并后的 issuer classes 不会让非 owner 补齐 required field；以及 lifecycle-equivalence obligation，说明 proof check、observation binding、budget/expiry consume 和 delegation attenuation 与 side effect 使用同一个 \(\sigma\) 版本线性化。缺少前者时，E2 的 owner-substitution artifacts 给出 false accept；缺少后者时，workflow lifecycle artifacts 给出 stale reuse 或 over-delegation false accept。这个要求是本文的主 theorem 形态；后续实验分别测试这两个 obligation 在 artifact-derived traces 和 local multi-boundary workflows 中是否被违反。

E2 的 counterexample suites 和 authority-input characterization evidence 就是在寻找该蕴含式的反例：同一 object、argument 或 prompt slot 保持不变，只替换 issuer class。若折叠后的 checker 接受而完整 checker 拒绝，说明被合并的 projections 拥有不同 authority fields，不能作为同一个 proof boundary。这个定义把 owner-projection claim 变成可测试的 false-accept 条件，而不是命名偏好。

更具体地，令 \(\Phi(e,d)\) 是事件 \(e\) 对 decision class \(d\) 提交的 field-proof set。每个证明项的形状为

\[
\phi=\langle f, issuer, value, source, prov\rangle .
\]

Checker 接受 \(e\) 的必要条件是：

\[
\forall f\in Req(d),\exists \phi\in\Phi(e,d):
\phi.f=f \land \phi.issuer=owner(f) \land match_\Gamma(\phi.value,\kappa.f).
\]

这个条件故意不是“存在一组可信 attributes 满足 policy”。它要求每个 required field 都由自己的 issuer class 证明，并且证明值与同一个 active lease 的 predicate 匹配。Env proof 可以证明某个 \texttt{/tmp/out.xlsx} 确实由本次脚本生成，但不能证明该路径是用户授权 output directory。Tool proof 可以证明 \texttt{pdftotext} 的 argv schema，但不能证明当前 PDF 是用户选择对象。Instruction proof 可以证明 Skill workflow 允许调用 extraction step，但不能证明 GitHub sink 或 approval scope。

不同 protected decision 对四类 context 的需求不同。令

\[
Req(d)=\langle A_d, I_d, T_d, E_d\rangle
\]

表示 decision class \(d\) 的 agent、instruction、tool 和 env field requirements。若 \(A_d\)、\(I_d\)、\(T_d\) 或 \(E_d\) 中任一 required field 缺失，checker 必须拒绝该 transition，而不能由其他 context class 代填。表~\ref{tab:decision-reqs} 给出 running example 中的代表性规则。

为了避免把四类输入退化成普通 ABAC attribute namespaces，\sys 将 required fields 视为 issuer-typed authority fields。令 \(owner(f)\in\{agent,inst,tool,env\}\) 表示 field \(f\) 的唯一授权 issuer class，令 \(proof(e,f)=\langle issuer,value,source\rangle\) 是 runtime adapter 为事件 \(e\) 提交的结构化证明。No-substitution typing rule 是：

\[
\frac{f \in Req(d) \quad proof(e,f)=\langle q,v,s\rangle \quad q = owner(f)}
     {\Gamma \vdash proof(e,f) : f}
\qquad
\frac{q \neq owner(f)}
     {\Gamma \nvdash \langle q,v,s\rangle : f}
\]

例如 \texttt{repo=org/repo-x} 的 sink authorization field 由 agent issuer 证明。Trusted GitHub tool metadata 可以证明 schema field，却不能证明该 repo 是用户授权 sink。于是，同一个 tool-issued proof 对 interface field 可以被接受：

\[
\Gamma \vdash \langle tool,\texttt{create\_issue.schema},mcp\_registry\rangle : github\_schema ,
\]

但对 agent-owned field 必须被拒绝：

\[
\Gamma \nvdash \langle tool,\texttt{org/repo-x},mcp\_metadata\rangle : authorized\_sink .
\]

同理，tool result 或 tool metadata 也不能证明 \(policy\_update\) field。这个 rule 的作用是给每个 protected field 建立唯一证明来源：checker 接受的不是一组满足谓词的可信 attributes，而是与 field owner、proof value 和 active lease predicate 同时匹配的 typed proof。

\subsection{Equivalence Boundary}

ABAC、IFC 或 policy-DSL system 若要在本文 decision classes 上达到同等语义，必须实现上一小节的 typed transition interface。区别不在于某个 DSL 是否能写出相似布尔谓词，而在于 runtime 是否把 issuer-owned field proofs 和 lifecycle updates 作为同一个 authority-state commit object 提交、检查和更新。

一个 stateful policy DSL、reference monitor 或 IFC runtime 若要在本文 decision classes 上与 \sys 等价，必须在同一个 pre-effect authority-state commit object 中暴露三个已测试的核心接口对象：\textbf{field-owner proof projections}，用于说明每个 required field 由哪个 issuer boundary 证明；\textbf{checker-owned lease state}，用于保存 active/exhausted leases、budget、expiry 和 parent-child graph；以及 \textbf{same-transition consume/delegate update}，用于把 allow decision 与 budget consumption、expiry progress 和 child-state update 原子绑定。实现还应返回 audit commit id，把后续 side effect、prompt placement 或 handoff 绑定到 \(\mathsf{allow}(\sigma',audit)\) 的线性化点。该 id 支持审计和调试，但当前实验尚未单独证明其缺失会导致 false accept。满足前三个接口对象并提供等价审计绑定的系统在覆盖 fields 上实现的是 \sys commit interface；缺少任一已测试核心对象的系统需要额外证明不会出现 cross-issuer substitution、stale reuse、double consume 或 over-delegation。E2 中的 typed-provenance state guard 正是一个 convergence-to-\sys example：它不是被当作“弱 baseline”击败，而是用来标出等价条件；当它继续加入 parent-child lease-set comparison、checker sole-writer state 和 same-transition child update 后，它就在该 residual class 上实现了同一 commit interface。

Context class 不是 raw artifact 的普通 attribute，而是 adapter canonicalize 后的 proof-projection owner。Checker 还维护一个 no-promotion invariant。令 \(class(c)\in\{agent,inst,tool,env\}\) 表示 proof cell \(c\) 的 canonical owner，令 \(endorse_q(c)\) 表示 trusted issuer 明确把 \(c\) endorsement 为 class \(q\)。若没有 endorsement，则：

\[
\begin{array}{ll}
c\in C_{env} \land \neg endorse_{inst}(c) & \Rightarrow c \notin C_{inst},\\
c\in C_{tool} \land \neg endorse_{agent}(c) & \Rightarrow c \notin C_{agent},\\
c\in C_{inst} \land \neg endorse_{agent}(c) & \Rightarrow c \notin C_{agent}.
\end{array}
\]

直观地说，tool result、script output 或 PDF text 不能因为包含祈使句就变成 instruction。Tool schema 不能因为声明“安全”就变成 user intent。Skill/manual instruction 不能因为要求更多权限就变成 approval。Promotion 只能由 trusted issuer、policy 或 fresh approval 显式产生，并作为新的 lease mint event 记录在 \(\sigma\) 中。No-promotion 还意味着任一 context class 都不能替另一个 class 补齐缺失的 authority field。Instruction 可以建议 workflow，但 binary/schema 必须由 tool context 验证。Tool context 可以定义接口，但 sink/intent 必须由 agent context 验证。Env context 可以提供 observed values，但不能把这些 values 转换成 approval 或 policy。

\subsection{Lease Lifecycle and Commit Semantics}

Lease lifecycle 是 \sys 与静态 policy rule 或单次 action provenance check 的主要区别。Agent workloads 中，同一个 action argument 可以在不同 authority state 下有不同安全含义：第一次 \texttt{create\_issue(repo=org/repo-x)} 可能由 fresh approval 授权。第二次相同参数可能是在复用已消耗的一次性 lease。同一 repo 参数交给 subagent 可能违反 delegation bound。同一 approval reason 若由 PDF text 控制，则可能变成 scope widening。Action-time provenance 或 object predicate 往往只看到 repo 值仍然匹配，无法区分这些 lifecycle violations。

形式上，可以构造两条 action/provenance 等价但 authority-state 不等价的 trace。Trace A 在 \(\kappa.budget=1\) 且 \(review\_complete\) 后执行
\(\texttt{create\_issue(repo=org/repo-x)}\)，并把 \(\kappa\) 消耗为 exhausted。Trace B 在同样的 tool、repo argument、intent proof 和 control provenance 下再次执行相同 call，或把同一 issue authority 交给未覆盖的 child capability set。Action guard、object predicate 和 ordinary provenance check 看到的是同一个 action shape。\sys 看到的是不同的 \(\sigma\)：A 有 active lease，B 只有 exhausted lease 或违反 \(K_c\sqsubseteq K_{parent(a)}\)。因此，被保护对象必须是读取并更新 \(\sigma\) 的 protected-decision transition，而不是 action tuple 本身。

Lease 因此是 future protected decision 的有状态授权对象，而不只是具体 operation 的 permission。它表达某个 context source 在 intent \(I\) 下可以影响 decision class \(d\)，直到某个状态转移使其被消耗、过期或衰减。Checker 将 lease 作为有状态对象处理，而不是把它当作一组静态 predicates。

Mint provenance、influence authority、consumption/expiration 和 delegation bound 必须在同一个 authorization object 上被检查并原子更新。若它们分散在独立 taint labels、ACL entries、counters 和 delegation records 中，系统就必须额外证明这些对象不会产生 stale authority reuse、cross-object widening 或 consumed-state bypass。Runtime state \(\sigma\) 至少包含 active/exhausted lease table、per-lease consumption counters、temporal automata state、mint proof records 和 parent-child delegation graph。Checker 是这些 lease-state fields 的唯一 writer。实现中的 checker state schema 直接保存这些字段，并在每个 accepted event 后原子更新。

Checker 暴露的 authority-changing interface 只有四类 transition：

\[
\begin{aligned}
&mint(\kappa), \quad check\_and\_consume(e,lease\_id,field\_proofs,prov,\sigma),\\
&attenuate(\kappa_{parent},K_c), \quad expire(\kappa).
\end{aligned}
\]

Runtime adapters 可以分布在 context、tool、MCP、local execution 和 delegation 边界，但它们不能本地修改 authority state。任何会产生外部副作用或改变 authority state 的 protected event，包括 tool call、MCP call、local command、approval request、lease mint、policy update 和 delegation，在副作用发生前都必须经过 central checker state 的 \(\mathsf{check\_and\_consume}\)。Accepted event、budget update、provenance record 和 delegation-state update 是同一个 transition：

Commit semantics 给出线性化规则。\(allow(\sigma',audit)\) 是唯一线性化点；adapter 不能缓存 allow 决策，也不能在 checker 外消耗或恢复 lease。每个外部 side effect、prompt placement 或 handoff 都必须绑定到返回 audit record 中的 commit id。并发 tool calls 或 parallel subagents 必须串行化到 checker，或使用等价的 compare-and-swap transition，使它们都以同一个 \(\sigma\) 版本为条件提交。这样 one-shot lease 不会被两个 adapter 同时消费，也不会在 handoff 和 tool execution 之间被复用。

\begin{itemize}[leftmargin=*]
  \item \textbf{Mint.} 只有 trusted issuer 可以通过 \(lease(\kappa)\) 事件把新 lease 加入 \(K\)。Mint rule 要求 \(\kappa.intent\) 可由 \(I\) 中的 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导。Untrusted context 不能成为 mint 或 widening 的 control dependency。
  \item \textbf{Check.} 每个 protected event 在 side effect 发生前必须引用一个 active lease \(\kappa \in K\)，且 operation、object、arguments、influence、flow 和 temporal predicates 同时满足。没有 active lease 的 event 即使 action arguments 看似可信也被拒绝。
  \item \textbf{Consume.} 若 \(\kappa.budget\) 是 one-shot 或 bounded count，accepted event 会递减 counter。Counter 为零时 \(\kappa\) 从 active set 中移除或进入 exhausted state。后续相同 tool/argument 的复用必须重新 mint。
  \item \textbf{Expire.} Temporal automata 可以在 task completion、approval window 结束、required predecessor missing 或 forbidden successor 出现时使 lease invalid。Expired lease 不再授权 event。
  \item \textbf{Attenuate.} Delegation event \(spawn(a,K_c)\) 只在 \(K_c \sqsubseteq K_{parent(a)}\) 时被接受，其中 \(K_{parent(a)}\) 是发起该 handoff 的 parent agent 在当前 \(\sigma\) 中仍 active 的 capability set。Child lease 的 objects、arguments、budgets、sinks、influence modes 和 delegation rights 均不得宽于 parent lease。
  \item \textbf{Deny amplification.} Skill instructions、MCP results、documents、script outputs 或 subagent summaries 可以作为 data/control provenance 参与普通 decisions，但除非被 trusted issuer endorsement，它们不能 mint fresh high-impact lease、扩大 existing lease，或把 exhausted/expired lease 恢复为 active。
\end{itemize}

这些规则也可以写成紧凑的状态转移。令 \(K_\sigma\) 表示 \(\sigma\) 中的 active leases，\(K_\sigma(p)\) 表示 parent \(p\) 在当前 \(\sigma\) 中持有的 active capability set，\(budget_\sigma(\kappa)\) 表示剩余调用次数，\(\sqsubseteq\) 表示 capability attenuation：

\[
\begin{array}{ll}
\textsc{Mint}:&
  \Gamma;I;\sigma \vdash trusted\_mint(\kappa)
  \Rightarrow \sigma[K_\sigma \leftarrow K_\sigma \cup \{\kappa\}] \\
\textsc{Check}:&
  \Gamma;I;K_\sigma;\sigma \vdash e:\kappa
  \land active_\sigma(\kappa)
  \Rightarrow \sigma_1 \\
\textsc{Consume}:&
  budget_{\sigma_1}(\kappa)=1
  \Rightarrow \sigma_1[K \leftarrow K_{\sigma_1}\setminus\{\kappa\}] \\
&
  budget_{\sigma_1}(\kappa)>1
  \Rightarrow \sigma_1[budget(\kappa)\leftarrow budget_{\sigma_1}(\kappa)-1] \\
\textsc{Expire}:&
  expired_\sigma(\kappa)
  \Rightarrow \sigma[K \leftarrow K_\sigma\setminus\{\kappa\}] \\
\textsc{Attenuate}:&
  spawn(a,K_c) \land parent(a)=p \land K_c \sqsubseteq K_{\sigma}(p)
  \Rightarrow \sigma[child(a)\leftarrow K_c] \\
\textsc{NoAmp}:&
  c \notin trusted\_issuers
  \Rightarrow \neg mint_c(\kappa') \land \neg widen_c(\kappa,\kappa').
\end{array}
\]

这些转移表达的不可约 invariant 是：authority state 不能通过 untrusted provenance 被创建、刷新、扩大或委托。若一个 provenance graph 或 stateful policy baseline 要表达同一性质，它必须引入等价的原子授权对象，同时覆盖 trusted mint、decision-specific influence、budget consumption、expiration 和 monotone delegation state。换言之，\sys 的结论不是这些检查项各自新颖，而是 agent protected decisions 需要一个把它们绑定为同一 transition 的授权单元。

一个直接的 lemma 是：若 trace 中每个 protected event 都通过上述 checker transition，则 no-substitution、no-promotion 和 atomic lifecycle 同时成立。Proof sketch 是逐事件归纳：\textsc{Check} 要求每个 required field 的 proof issuer 等于 \(owner(f)\)，\textsc{NoAmp} 和 endorsement guard 排除 context class promotion，\textsc{Consume}/\textsc{Expire}/\textsc{Attenuate} 在同一次 transition 中更新 \(\sigma\)。E2 的 ablations 反过来测试这些前提是否必要：去掉 owner check、promotion guard 或 atomic state update 时，构造的 class-substitution、stale reuse 或 over-delegation events 会被 false accept。

单个 argument check、taint label、one-shot counter 或 delegation rule 可以覆盖其中一部分，但它们必须在同一个 \(check\_and\_consume\) transition 中读取并更新。否则系统还需要额外证明多个 guard 之间不会出现 stale read、split-brain state 或 cross-object widening。例如 tool guard 看到 repo argument 合法后准备执行，而 budget counter 稍后才发现 one-shot lease 已耗尽。另一个例子是 provenance guard 允许 issue body 由 PDF text 控制，而 delegation guard 在另一份状态里仍看到 parent 拥有旧 issue authority。E2 把这些情况作为 ablation counterexamples 测试，正文形式化只保留它们共同依赖的状态转移规则。

Checker judgment is:

\[
\Gamma; I; K; \sigma \vdash e \Rightarrow \sigma' ,
\]

其中 \(\Gamma\) 是全局策略，\(I\) 是 intent certificate，\(K\) 是 active leases，\(\sigma\) 是 runtime state。事件被接受当且仅当存在 \(\kappa \in K\) 使得七个条件同时成立：\(1\) operation 匹配，\(2\) object 和 arguments 满足 lease predicate，\(3\) event purpose 可由 \(I\) 推导，\(4\) \(prov_C(e)\) 中的每个 context 都拥有影响该 decision class 所需的 authority，\(5\) \(prov_D(e)\) 满足 flow policy，\(6\) temporal guard 和 budget 尚未过期，\(7\) delegated leases 不宽于 parent leases。

对于 \(request(\kappa,reason)\) 和 \(policy\_update(patch)\)，checker 将其作为 protected decisions 处理。它们自身不能 mint authority，除非 trusted issuer 随后产生独立 \(lease(\kappa)\) event。它们的 control provenance 也必须拥有 \((authorize, capability\_request)\) 或 \((authorize, policy\_update)\)。

\subsection{Safety Properties}
一个 trace 被接受，表示其中每个 protected event 都按上述 judgment 顺序通过 checker，并且每个通过事件对 \(\sigma\) 的更新会消耗预算、推进 temporal state、记录 provenance，并更新 active leases。下面五个性质是这一 checker semantics 的直接安全后果。它们不是关于任意 prompt 的全局声明，而是关于 accepted traces over instrumented protected-decision events with adapter-supplied control-provenance proofs 的 trace-level properties。

\begin{enumerate}[leftmargin=*,label=\textbf{P\arabic*.}]
  \item \textbf{No issuer substitution.} 对任意 accepted protected event \(e\) 和其 decision class \(d\)，若 \(f \in Req(d)\)，则 checker 接受的 field proof 必须满足 \(proof(e,f).issuer=owner(f)\)，且 proof value 与同一个 active lease 的 predicate 匹配。其他 issuer class 提交的同值 proof 不能补齐该 field。特别是 tool metadata 不能证明 agent-owned sink 或 approval field，env evidence 不能证明 instruction-owned workflow field，Skill/manual instruction 不能证明 fresh approval。
  \item \textbf{No unauthorized owner/provenance influence.} 对任意被 instrumentation 暴露并被 checker 接受的 trace \(\tau\)，如果事件 \(e \in \tau\) 属于 protected decision class \(d\)，则每个 authority-bearing field 的 control proof 必须来自该 field 的 owner class，且 \(prov_C(e)\) 中用于证明该 field 的 context 必须拥有对应 \((mode,d)\) authority。换言之，未被授权证明或影响该 protected field 的 context 不能成为该字段的控制依赖。
  \item \textbf{Intent-bounded authority.} 对任意 accepted high-impact event \(e\)，存在 active lease \(\kappa\)，且 \(\kappa\) 的 intent predicate 可由 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导。系统不会仅凭 Skill identity、MCP server identity 或工具描述铸造 high-impact authority。
  \item \textbf{Delegation monotonicity.} 对任意 accepted delegation event \(spawn(a,K_c)\)，child capability set \(K_c\) 是 parent active capability set 的子集或更窄 attenuation。除非 trusted issuer 基于新的 intent certificate 创建 fresh lease，subagent、Skill、MCP tool 或 script 都不能扩大自身权限。
  \item \textbf{No context-class promotion.} 对任意 accepted trace，若 context cell \(c\) 的原始 class 是 \(env\)、\(tool\) 或 \(inst\)，则 \(c\) 不能在后续 checker transition 中以更高 authority class 使用，除非 trusted issuer 产生显式 endorsement 或 fresh lease mint event。特别是，tool result、script output、PDF text 和 subagent summary 不能仅凭自然语言内容变成 instruction context。Tool schema 不能变成 user intent。Skill/manual instruction 不能变成 approval。
\end{enumerate}

这些性质是 trace-level safety properties。它们依赖于两个工程前提：runtime 必须把 protected decisions 接入 checker，并且 provenance tracker 必须保守地标记事件控制依赖。若某个 OS-level side effect 没有经过任何 adapter，\sys 无法单独观察它。这正是 sandbox 或 ActPlane-style backend 仍然有价值的原因。实现层面的 falsifier 也很直接：missing adapter 会破坏事件覆盖，错误 control-provenance 会破坏 P2，stale lease version 会破坏 P3，错误 owner projection 会破坏 P1/P5，未原子比较 parent/child lease set 会破坏 P4。

\section{实现}
\label{sec:implementation}

我们的原型是一个 agent authorization runtime，用于验证三个边界性质：lease language 是否足以表达 intent/context authority，deterministic checker 是否能在统一 event model 上执行这些约束，以及这些约束能否落到 trace replay、live callable gateway、benchmark-derived workflows、context/delegation boundary experiments 和 env-projection backend。这里的系统贡献不是若干独立 adapter，而是一个统一的 pre-effect commit contract：任何 authority-changing boundary 都必须在副作用、prompt placement 或 handoff 前提交同一种 issuer-owned field-proof record，并由 checker 作为 lease-state sole writer 返回 audit commit id。一个 surface 被认为实现 C2，当且仅当它在 pre-effect blockpoint 提交 typed field proofs、versioned lease id、holder/scope state 和 checker-owned consume/delegate update。原型实现 checker core、tool/live callable gateway、本地 env pre-side-effect gateway、live context-placement gateway、section-aware prompt-builder adapter、live delegation monitor、runtime binding 和结果分析脚本。Env-projection replay backend 用于检查 local/env lease contract 能否被 lowered 为 deterministic replay rules；bubblewrap namespace backend 用于检查同一 env projection 是否能驱动真实本地 OS sandbox。生产级 MCP broker、生产 prompt/subagent runtime、ActPlane/eBPF integration 和 online user-simulator loop 是更强部署结论所需的接口。

\subsection{Checker Core}
Checker core 位于 \texttt{src/intentcap/}，实现 context labels、effect events、leases、checker state 和 proof objects。当前 checker 支持 exact、prefix、suffix 和 one-of argument predicates，支持 lease matching、holder scoping、control/data provenance checks、context-label influence-mode checks、temporal prerequisites、per-lease invocation budgets、proof-carrying intent/approval derivation，以及 delegated capability attenuation。这个 core 是本文的主要 TCB：LLM 输出、Skill 描述、MCP annotations 和 tool results 只能产生 candidate effects 或 candidate leases，不能绕过 checker 直接获得 authority。

Checker state 显式保存 active/exhausted lease table、budget counters、temporal prerequisites、holder identity、delegation parent-child relation 和 audit records。每次 protected event 进入 checker 时，runtime adapter 必须提交 event \(e\)（含 argument values）、\(lease\_id\)、\(field\_proofs\) 和 control/data provenance。Checker 在一个函数调用中完成 lease lookup、field-owner proof check、argument predicate check、budget/expiry check 和 attenuation check。若事件被接受，checker 同步写入 audit log 并更新 budget 或 child lease state。若被拒绝，它返回 structured denial，adapter 不执行 side effect、不写 prompt section、不创建 child handoff。

\subsection{Adapter-Facing Transition API}

\(\mathsf{check\_and\_consume}\) 是 runtime adapter 面向 checker 的唯一 authority-changing API。它不是一个普通 allow/deny predicate，而是一个带状态提交语义的 transition：

\[
\begin{aligned}
&\mathsf{check\_and\_consume}(e,lease\_id,field\_proofs,prov,version)\\
&\qquad \rightarrow allow(\sigma',audit)\mid deny(reason,\sigma).
\end{aligned}
\]

输入端，adapter 必须提交五类对象。第一，事件 \(e\) 包含 operation、object、arguments、holder、decision class 和 block point。第二，\(lease\_id\) 指向 checker 当前 active set 中的一条 lease。第三，\(field\_proofs\) 是按 \(owner(f)\) typed 的 proof map，覆盖本次 \(Req(d)\) 所需的 agent、instruction、tool 和 env fields。第四，\(prov\) 给出 data/control provenance 和 context influence modes。第五，\(version\) 是 adapter 观察到的 checker-state version，用于并发或 parallel subagent 提交时的 compare-and-swap 检查。

输出端，\(\mathsf{allow}\) 返回新的 checker-owned state \(\sigma'\) 和 audit commit id。该 commit id 是 side effect、prompt section write 或 delegation handoff 的线性化点。Adapter 只能执行绑定到该 id 的 effect，不能缓存旧 allow、不能在 checker 外消耗 budget、不能恢复 exhausted lease，也不能本地写 parent-child capability graph。\(\mathsf{deny}\) 返回结构化原因，但不改变外部世界；adapter 必须阻止 handler、placement 或 handoff。

这个 API 给出四个 runtime obligations。State ownership obligation 要求 active/exhausted leases、budgets、expiry 和 delegation graph 只有 checker 可以写。Linearization obligation 要求 field proof check、argument predicate、budget consume、expiry progress 和 delegation attenuation 在同一 state transition 内完成。Failure obligation 要求 deny 发生在 side effect、prompt authority placement 或 handoff 之前。Concurrency obligation 要求并发 tool calls 和 parallel subagents 通过 checker version 串行化或提供等价 compare-and-swap transition。E3 的每个 adapter 都按这四个 obligations 报告，而不是只报告某个本地测试是否通过。

表~\ref{tab:runtime-interface-audit} 是本文使用的 runtime-interface audit rubric。它不声称已有系统无法扩展出这些字段，而是检查默认 agent/protocol boundary 暴露给 monitor 的对象、默认 state writer，以及是否已有一个 pre-effect transition 同时 consume 和 delegate authority。这个 audit 将“stateful ABAC + CAS 也能做”的反驳转化为等价条件：若某个 runtime 在对应 boundary 上暴露表中最后一列的对象，它就在该 boundary 上实现 \sys 的 commit interface；若没有暴露，就需要额外证明不会出现 E2/E3 中的 stale reuse、issuer substitution 或 over-delegation。

\subsection{Trace and Live Gateways}
\texttt{TraceGateway} 用同一个 checker replay benchmark-derived traces，并把每个事件记录为 execute/block decision。它用于把 AgentDojo、MCPTox、InjecAgent 和 tau2-bench benchmark labels 转换成统一的 protected-decision events~\cite{agentdojo,mcptox,injecagent,tau2bench}，从而比较不同 policy variants 的 false accept、authority breadth 和与人工标注 lease 的距离。

\texttt{LiveToolGateway} 包裹本地 Python callables。只有 checker 接受的 event 才会真正调用注册函数。被拒绝的事件在 side effect 发生前停止。这一区分很重要：trace replay 提供大规模 policy evidence，live gateway 验证同一 lease semantics 可以保护真实 callable boundary。

本地 env gateway 使用相同调用路径，但 block point 位于 side-effect handler 之前。File read/write、script execution、network-like event 和 context-use event 都先被转换成 protected event。Adapter 为每个事件提交路径、argv、cwd、observed file state、script output 或 tool result provenance。被拒绝事件不进入 handler，因此不会创建文件、修改 secret fixture 或写入 context section。这条路径是 E3 中 local env suite 和 Qwen3.6 proposer loop 的执行边界。

\texttt{LiveContextPlacementGateway} 和 \texttt{LiveDelegationMonitor} 将同一个 stateful checker session 放在 prompt-section placement 与 subagent handoff 之前。前者只有在 \texttt{ctx.use} event 被允许后才把内容写入目标 section。后者只有在 \texttt{subagent.spawn} event 的 delegated capabilities 被 lease attenuation 覆盖后才记录 child handoff。Skill/manual substitution test 进一步把同一个合法 instruction slot 固定为 signed Skill issuer，测试 tool result、script output 和 unsigned Skill text 是否能替代 instruction issuer。这些 adapters 支撑本地 boundary experiments。

\subsection{Compiler Frontend and Runtime Binding}
原型还包含 local llama.cpp/Qwen frontend，用来模拟不可信 lease compiler 和 action proposer。模型可以根据 task text、tool schemas、已有 tool-result previews 或 checker feedback 生成 candidate leases/actions。系统只把通过 checker 的结果交给 gateway。

对于 tau2-bench-style stateful workflows，runtime binding 会从已经执行的 tool-result evidence 中铸造 exact one-shot leases，并重新检查参数值是否出现在当前可见 task/tool-result context 中~\cite{tau2bench}。High-impact write candidates 还必须满足 structured value proof：例如列表参数需要逐项绑定到结构化上下文，不能从原始文本窗口或 sibling objects 中借用 authority。

除表中 contract surfaces 外，原型还包含 benchmark replay adapters、runtime binding、structured value proof、read/write activation、lease-label audit，以及 E1 security/utility baseline policy comparisons。已实现的 runtime surfaces 支持 local side-effect、context/Skill placement 和 delegation handoff 前的同一 commit contract。Benchmark replay、local Qwen diagnostics 和 saved-model comparisons 只作为 replay/diagnostic evidence，不构成生产 runtime integration。

原型还包含 local Qwen/llama.cpp compiler experiments 和本地 env backend experiments。Env backend 在隔离目录中真实执行 Python script、file read/write 和 context-use handlers，并在 side effect handler 之前调用 checker。被拒绝的 exec/read/write/net/ctx events 不会进入本地 side-effect handler。模型闭环版本使用 Qwen3.6/llama.cpp 作为不可信 action proposer。模型在 adversarial env-context prompt 下返回 call/abort，只有 call 事件会继续进入 checker 和本地 side-effect handler。

原型还实现了一个 env-projection replay backend：从 active env leases 生成 deny-by-default monitor rules，并在 replay monitor 中检查 operation、object、holder、argument predicates、budget consumption、control/data provenance 和 context-label influence rules。该 lowering 证明 lease contract 可以被编码为 env/OS monitor policy 形态。当前 monitor 运行在 Python replay runtime 中，且 provenance labels 由 IntentCap adapter 提供。Lowering 只覆盖 env/local-effect projection：file/process/network/object predicates、holder、budget 和 provenance/influence labels 可以进入 monitor rules；agent-owned sink/approval proof、instruction endorsement、tool schema owner proof、lease mint/delegation graph update 仍由 checker 处理，不能由 OS monitor 或 ActPlane-style backend 推导。

这些实现支撑机制性结论：在已接入 checker 的边界上，\sys 能拒绝 owner substitution、class promotion 和 lifecycle violations，runtime exact leases 可以缩小 object/tool authority surface，checker 可以阻止 LLM-proposed broad/runtime authority 直接进入执行路径，context/delegation boundary 可以在 placement/handoff 前调用同一 checker，并且同一 env lease contract 可以被降低为包含 provenance/influence checks 的 monitor rules 或本地 namespace sandbox 配置。生产级 ActPlane/eBPF integration、完整 MCP broker、独立 human expert-oracle replication 和大规模 recovery evaluation 属于扩展结论所需的实验条件。

\section{评估}
\label{sec:evaluation}

评估按 claim dependency 组织，而不是按脚本或 artifact 罗列。正文回答三个问题。第一，protected-decision event model 是否仍能表达良性任务行为？E1 显示 \sys 覆盖 3,813/3,813 个 reference actions，并在 3,746 个 protected-event replay cases 上产生 0 个 unsafe accepts；这只是 representability / sanity gate，不是 end-to-end utility claim。第二，在本文测试的 owner/lifecycle removal families 中，缺失 field-owner proofs 或 same-transition lifecycle update 是否会重新打开 false authority paths？E2 显示 no-owner collapse 会 false-accept 3,593/3,823 个完整 checker 拒绝的 protected decisions，split-state variants 也会接受 workflow residuals。第三，同一个 transition contract 能否放到 tool call 之外的系统边界前？E3 在 local env、context placement、Skill placement、delegation handoff、integrated workflow 和 env-projection replay 上记录 0 个 unsafe executions/placements。

Supporting audit 只问 lease 是否是可机器审计的 authority object，并报告 author-adjudicated lease-distance 结果；它不承担独立 expert-oracle least-privilege 结论。内部 run ids、命令和 artifact provenance 保留在 evaluation ledger 中，正文只使用稳定实验名、聚合结果和 claim boundary。

\subsection{评估方法}

Workloads 分为两组。公开 trace 来自 AgentDojo、MCPTox、InjecAgent 和 tau2-style artifacts~\cite{agentdojo,mcptox,injecagent,tau2bench}。本地 suites 覆盖 Skill/manual injection、command/script side effects 和 delegation workflows。这些 workloads 覆盖 agent、instruction、tool 和 env 四类输入，而不是只测试 MCP/tool-call 参数。

对包含模型调用的实验，我们固定底层模型、任务顺序、工具描述和 prompt budget。对 replay experiments，我们固定 event set、oracle 和 baseline policy，只改变授权检查层。High-impact denials 使用同一个 protected-decision oracle 判定。若某个 protected decision 的 control provenance 来自缺少相应 influence mode 的 context source，该事件应被拒绝。

Baseline policies 分为两类。第一类是 implemented wrapper baselines，例如 vanilla/all-tool exposure、static ACL、exact-tool ACL、MCP/server allowlist、Skill manifest-style policy、tool-call guard 和 OS/sandbox-only policy，它们在同一事件模型上直接重放。第二类是 interface-derived removal studies 或 counterfactual predicates，例如 IFC/taint predicate、action-provenance predicate、prior-derived typed-provenance state guard 和 split-state composite policy；它们不是 AuthGraph、PACT、AIRGuard 或其他完整系统的复现，而是把这些方向默认暴露的 runtime object 映射成可判定 predicates 后测试缺失字段。Same-event comparison 表示不同策略在同一批 protected events 上重放，不代表 live replanning。Oracle sources 包括 benchmark security/utility labels、protected-decision labels、author-adjudicated lease labels 和 provenance-proof completeness records。

\subsection{E1: Event-Model Feasibility and Reference-Action Coverage}

E1 排除一个基本失败模式。如果 protected-decision event model 不能表达 benign reference behavior，那么后续安全结果即使为真也没有实用意义。为固定比较对象，实验把 3,746 个 benchmark-derived protected-event replay cases 输入 \sys 和 same-event policies，作为 unsafe-accept sanity check；同时在 3,813 个 benign reference actions 与 2,556 个 applicable tool-oracle checks 上评估良性动作是否仍能被表示和覆盖。Primary metric 是 reference-action coverage，unsafe accept 只是对 event extraction 和 replay policy 的 sanity check。E2 才承担 issuer/lifecycle removal-family 的反例证据。

在 protected-event replay 中，\sys 在 3,746 个 protected-event cases 上产生 0 个 unsafe accepts。Object/tool ACL policies 接受许多 protected-decision violations，因为它们不区分 context influence authority。在 benign reference-action proxy 上，\sys 覆盖 3,813/3,813 个 reference actions，并通过 2,554/2,556 个 applicable tool-oracle checks。

E1 还包含一个已完成本地 Qwen3.6 matched slice，用来检查 live model-loop 下的 authority surface。该比较合并一个 leased exposure run 和一个 all-tools exposure run 的已有结果；它不运行新模型、不执行新工具、不同步数据集。在同一 18 个 tau2-style tasks 上，leased exposure 平均向模型暴露 2.61 个 tool schemas，而 all-tools exposure 暴露 14.72 个；对应 model calls 为 62 vs 74。Task-level utility 在这个切片上没有改善：action-reward 均为 8/18，tool-oracle 均为 0/18。差别主要出现在 checker-visible authority friction：gateway blocks 为 2 vs 18，off-lease blocks 为 1 vs 9。该切片只支持一个窄结论：较宽 tool visibility 会显著扩大可见 authority surface 和被 checker 拦截的越界尝试，但不能作为大规模 benchmark utility 或 recovery 结论。

\subsection{Diagnostic: Recovery}

作为主指标之外的 supporting recovery diagnostic，本文还加入一个手写构造的 denial-targeted closed-loop recovery microbenchmark，用来检查 denial 后的 agent 是否必然只能 abort。该 microbenchmark 固定六个 recoverable authority-bearing tasks，覆盖 wrong sink、approval-scope widening、delegation attenuation、instruction-slot placement、Skill workflow scope 和 tool-result approval substitution。每个任务先强制提交一个已知 unsafe event；\sys gateway 阻断 6/6 个初始事件。

在语义候选名和 structured denial feedback 可见时，本地 Qwen3.6 在 6/6 个任务上选择预期 authorized alternative，产生 6 个 final correct executions、0 个 dangerous executions。将候选 ID 和候选描述盲化后，原始输出预算下恢复 5/6 个任务；把输出预算从 384 提高到 768 后恢复 6/6 个任务。No-feedback baseline 没有第二轮模型调用，因此 0/6 恢复。Generic feedback 和 candidate-only feedback 均恢复 6/6 个任务并保持 0 个 dangerous executions。这说明在该枚举候选微基准中，关键因素是第二轮 constrained reconsideration prompt 加上 lease/provenance/candidate scaffold；详细 checker denial reason 不是必要条件。但这个结果仍不证明 free-form replanning recovery、fresh online task-completion、自然 benchmark recovery、approval-burden reduction，或没有枚举候选时的恢复能力。

\subsection{E2: Issuer-Collapse and Lifecycle-Split Removal Study}

E2 是本文的 tested removal-family 实验：它问移除 issuer ownership 或 same-transition lifecycle update 后，同一个 event/action/provenance record 是否会重新出现 unsafe accepting path。E2 的假设是：owner-typed authority fields 和 stateful lease lifecycle 共同形成一个不可拆开的 authority-state commit object。在本文测试的 no-owner 和 split-state variants 中，object/action guards、IFC/provenance guards 和 split-state counters 会 false-accept 某些事件，因为它们允许一类输入替另一类输入补齐 field，或把 budget/delegation/provenance 分散到不同状态中。因此，E2 是 issuer-collapse/lifecycle-split removal study，而不是完整自然频率证明，也不是声称所有 policy DSL 都无法表达相同 predicate。本文只声明 tested owner/lifecycle/local-boundary removal families 上的必要性：tested-removal matrix 中 10/10 个 removals 都有 same-event 或 local-boundary false-accept 反例；audit-id removal 仍是 1 个 gap，并且只作为 auditability gap 报告。

E2 的第一步是 label-independence protocol，而不是 checker replay。为避免用 checker 结果反过来定义 owner class，实验先执行 field-owner adjudication，再执行 ablation。Adjudication 的输入不是 \sys 的 allow/deny，而是 task text、trusted object/sink selection、Skill/manual endorsement、tool or command schema、runtime observation 和 saved provenance label。协议有四步。第一，标注者从事件中抽取 protected field family，例如 sink、approval、workflow scope、callable schema、observed path 或 child lease bound。第二，标注者只根据外部 evidence 选择 allowed issuer：用户选择或 trusted UI 可证明 agent field，signed Skill/manual 可证明 instruction field，registry/schema 可证明 tool field，gateway observation 可证明 env field。第三，标注者为至少一个 plausible disallowed issuer 写出不可替代理由，例如“tool metadata can quote a repo but cannot authorize it”。第四，若同一 raw artifact 同时包含多类信息，标注者必须拆成多个 proof cells；若 evidence 不足以决定 owner，则样本标为 ambiguous，不进入 necessity ablation。只有完成这一步后，实验才把同一 event 交给 full checker 和 weakened variants。表~\ref{tab:field-owner-adjudication} 给出协议的代表性规则和 label-sheet columns。这个协议把“哪个 owner 合法”绑定到外部证据源，而不是绑定到某个 baseline 是否失败。当前 labels 仍是 author-adjudicated；blinded second-pass 和 disagreement reporting 是表~\ref{tab:current-evidence} 中列出的顶会级扩展证据。因此 E2 证明的是 predeclared labels 下存在 owner-collapse false accepts，而不是自然 prevalence 或独立专家共识。

E2 用两层证据把“四类输入是否只是 attribute namespaces”转化为 falsifiable false-accept question。三类设计等价于把四个 owner classes 中至少一对合并，因此 E2 检查这些合并是否保持形式化中的 \(Accept_h\Rightarrow Accept\)。Trace characterization 检查已有 artifacts 中的事件是否已经触发 agent、instruction、tool 和 env 的不同 proof boundary。Controlled counterexample replay 则固定 object、argument、prompt slot 或 action shape，只替换 issuer class 或 lifecycle state，测试 weakened variants 是否误接受。若折叠 issuer ownership 或拆开 lifecycle state 不改变结果，\sys 的 owner/lifecycle interface 就会是过度规定。

这些 counterexamples 不是任意编造的字符串。Tool-metadata promotions 来自 MCPTox-derived MCP tool/schema artifacts。Injected runtime/env cases 来自 AgentDojo、InjecAgent 和本地 env suites 的已有 provenance labels。Skill/manual substitutions 使用本地 Skill placement records。Workflow residuals 来自本地 workflow suite 的保存 traces。实验在这些 artifact-derived records 上固定 object、argument、action shape 或 prompt slot，只替换 issuer owner 或 lifecycle state；因此 false accept 归因到 authority-state commit object 被折叠，而不是 workload 改变。

Trace characterization 从已有 provenance labels 和 event modes 派生 issuer-class requirements，而不运行模型或同步新数据。结果显示，在 8,696 个已有事件中，8,691 个需要多个 issuer classes，198 个需要 env/runtime issuer，3,593 个 protected-decision-oracle-denied events 存在 class-substitution attempt。Substitution edges 包括 tool \(\rightarrow\) agent 1,928 次、env \(\rightarrow\) agent 1,663 次、env \(\rightarrow\) tool 1,662 次，以及 instruction \(\rightarrow\) agent 2 次。Edge 计数不是互斥事件数，因为同一事件可能暴露多个 collapse risk。对应的 48 个 sampled protected-decision labels 把这些派生字段落到可复核的事件级标注任务上，其中 45 个样本是 protected-decision labels，43 个需要多个 issuer classes，13 个需要 env/runtime proof，37 个暴露 class-substitution attempts。

\subsection{E3: Local Multi-Boundary Adapter Practicality}

E3 的系统问题是同一个 protected-transition API 能否作为 adapter contract 放到 tool call 之外的多个 authority-changing boundaries 前。E2 说明 interface 为什么需要这些 owner/lifecycle fields；E3 只检查 adapter 是否能在本地实现中暴露并执行这些 fields。Live callable gateway 是已实现 surface，用来说明 checker 可以挡在本地 callable 前；MCP-style JSON-RPC broker 检查 \texttt{tools/list} 暴露和 \texttt{tools/call} 进入 fake server handler 前的 authority；E3 的 primary claim 验证同一 commit contract 的六个性质：pre-side-effect correctness、pre-MCP-call correctness、pre-placement/handoff correctness、integrated workflow consistency、env-projection lowering equivalence 和 local OS-substrate containment boundary。Qwen3.6 loop 只是 env-boundary diagnostic：不可信 proposer 的 unsafe calls 仍要经过同一 checker。Baselines 是 object-only policy、server allowlist policy 和 lease-args/no-provenance policy；lowering audit 还加入去掉 holder、argument、provenance/influence、budget 和 default-deny 的 weakened monitor variants；bubblewrap probe 使用真实 namespace sandbox 检查 OS-visible side effects 和 semantic-authority gap。Primary metrics 是 unsafe executions/placements、unsafe JSON-RPC executions、blocked unsafe attempts、checker/monitor mismatches、weakened-lowering unsafe accepts 和 sandbox-contained unsafe attempts。这个实验验证一致的 pre-side-effect/call/placement/handoff contract；生产 MCP broker、生产 prompt runtime、subagent runtime 或 ActPlane/kernel mediation 属于表~\ref{tab:current-evidence} 的更强部署证据。

\subsection{Support: Lease Auditability}

Supporting audit 检查 intent-carrying leases 是否能被机器审计为结构化 authority objects，包含 run-specific intent、influence-mode、budget 和 delegation fields。该 lease-audit corpus 包含 24 个 author-adjudicated lease labels，覆盖 agent、tool、env 和部分 instruction context。Audit 检查 schema validity、sampled label consistency，以及 object scope、influence modes、argument predicates、budget/expiry、delegation bounds 和 approval scope 上的 policy-distance invariants。

Audit 通过 24/24 个 labels，并对 144 个 IntentCap/baseline policy rows 计算 authority distance。这些 checks 说明 lease representation 可以支持 structured authority-distance analysis，并能在该 corpus 中发现 schema/sample/policy-distance mistakes。

该审计的结论限定在 author-adjudicated lease-audit result。它是 supporting evidence，不是主实验块。若要声称 \sys 比 symbolic policy baselines 更接近 expert minimal policy，还需要 blinded independent adjudication、inter-rater agreement 和统计化 authority-distance comparison。

\subsection{局限}
本节只支持一个结论，即 protected-decision transitions 可以在已 instrumented 的边界上被形式化并执行。结果覆盖 instrumented protected-decision events。端到端 utility、approval burden、independent expert replication 和生产级 sandbox integration 需要单独实验。

第一，workloads 仍以 saved traces、reference-action proxies、controlled counterexample suites、benchmark-derived counterexamples 和 local experiments 为主。Authority-input characterization 使用已有 artifacts 做 derived trace annotation。对应的 48 个 sampled protected-decision labels 已经提供 natural protected-decision events 和 author-adjudicated labels，但顶会级 prevalence claim 仍需要独立标注者复核 field owner 与 unsafe substitutes。MCPTox-derived benchmark cases 使用已有 artifacts 生成 24 个 authentic tool-metadata counterexample cases，但它不是自然发生频率统计。更大规模的 online user-simulator evaluation 是端到端 utility claim 的证据来源。

第二，辅助 lease audit 使用 author-adjudicated labels。这能检查 lease schema 和 authority-distance invariants。Blinded independent expert replication 才能支撑 expert-oracle least-privilege claim。

第三，本地 env backend 在隔离目录中验证 pre-side-effect blocking。Env-projection replay backend 只验证同一 lease contract 可以降低成 monitor rule set。MCP-style broker probe 使用本地 JSON-RPC 形态和 fake server handlers 验证 \texttt{tools/list}/\texttt{tools/call} 前的 pre-handler check，但仍不是 deployed MCP server 或真实网络 integration。Bubblewrap probe 使用真实 namespace sandbox 执行本地 handlers，但仍不是 ActPlane/eBPF，也没有覆盖生产 prompt/subagent runtime。System-surface result 只是把这些本地边界结果汇总。生产级 ActPlane/kernel integration、真实 syscall mediation 和低开销 enforcement 仍需要单独工程与实验。

第四，本地 boundary tests 已经实现 live context-placement guard、section-aware prompt-builder assembly、Skill/manual instruction-source substitution test、live delegation monitor test、本地 MCP-style broker，以及一个五边界 integrated local workflow。若要声称完整多边界 enforcement，还需要接入 deployed MCP servers、生产 prompt runtime、真实 subagent runtime、更多自然 Skill/manual injection workloads 和端到端 approval/recovery metrics。

这些限制限定外推范围。这些结果支持的性质是：instrumented protected-decision events 可以由 intent-derived, provenance-gated, consumable leases 管理，并且在这些边界上暴露 issuer/lifecycle collapse 的 false-accept 路径。完整 workload utility、approval burden、independent expert-oracle least privilege 和生产级 sandbox integration 需要分别报告，而不能由这些结果自动推出。

\section{相关工作}

相关工作分为四组。第一组是应用扩展隔离和 OS-level enforcement，它们提供 lowering substrate，但不决定 agent decision authority 的来源。第二组是 prompt injection、provenance 和 information-flow defenses，它们解释不可信 context 的数据/控制影响。第三组是 agent policy languages 和 access-control systems，它们提供可编程策略和 checker。第四组是 Skill/MCP permission frameworks，它们直接处理 agent extension ecosystem。表~\ref{tab:closest-abstractions} 按授权对象和生命周期比较这些方向。

\subsection{应用扩展隔离与底层 Enforcement}
传统扩展系统和 sandbox 关注扩展代码能访问哪些宿主资源。EIM/bpftime 将应用扩展表示为 host resources 和 capabilities，并强调安全、高效的 userspace extension execution~\cite{eim-bpftime}。ActPlane 将 agent policy lowering 到 OS-level enforcement，以覆盖 tool interception 看不到的 file、process 和 network side effects~\cite{actplane}。在 \sys 的分层视角中，ActPlane-style monitor 是 env-projection backend：它执行已经认证的 file/process/network/local-env predicates，并提供本地执行观测。IntentCap 的授权对象位于 pre-OS authority decision 层：它在 tool choice、sink selection、approval scope、prompt placement 和 delegation handoff 发生前，把 agent intent、instruction endorsement、tool interface proof、env observation 和 lease lifecycle 线性化为一个 commit object。若 OS/local backend 接收这个 commit object 的 env projection，它就在本地效果子集上实现 \sys 的 lowering contract；若只接收 syscall/local event，则它仍然是强 enforcement backend，但不是 sink、approval、instruction endorsement、tool interface proof 或 delegation graph mutation 的签发者。

\subsection{Prompt Injection, Provenance, and Information Flow}
CaMeL、Prompt Flow Integrity、FIDES、RTBAS 和 NeuroTaint 等工作从 control/data flow、IFC、dependency screening 或 taint tracking 的角度约束 untrusted content~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}。AuthGraph、PACT 和 AIRGuard 进一步把 user intent、argument provenance 和 action-time authority control 引入 agent security~\cite{authgraph,pact,airguard}。这些工作是 \sys 最接近的思想背景。本文的新粒度不是单独的 intent/provenance authorization，而是会铸造、消耗、过期和委托的 protected-decision lease。

本文把差异放在授权对象的粒度和生命周期上。已有系统主要检查 observed action、argument 或 side effect 是否与 intent/provenance 对齐。\sys 则把未来决策权限表示为会被铸造、消耗、过期和委托的短期 lease。为此，\sys 把 mint 来源、influence mode、budget/expiry、no-promotion rule 和 parent-child delegation order 放进同一个授权对象。关键不是单独发明 provenance label、taint tracking 或 attenuable capability，而是把这些检查绑定为每次 protected decision 更新 authority state 的原子单位。

\subsection{Agent Policy Languages and Access Control}
Progent、PCAS、AgentSpec、AgentGuard 和 AgentBound 等工作提供 programmable privilege control、policy compilation、runtime DSL、ABAC 或 MCP access-control enforcement~\cite{progent,pcas,agentspec,agentguard,agentbound}。这些系统提供 policy/checker substrate，并回答“某个工具、参数、server 或 action 是否被 policy 允许”。\sys 选择不同授权对象：从当前 run 的 intent 和 context provenance 铸造短期 lease，并让 lease 随 protected decision 被消耗、过期或委托。因此，本文的主比较是 controlled protected-decision counterexamples 和 issuer/lifecycle ablations；lease audit 只检查 representation/auditability，不作为独立 expert-oracle least-privilege 结论。

\subsection{Skill 和 MCP 权限框架}
SkillGuard 和 SkillScope 直接面向 Agent Skills 的权限、行为规范和 least-privilege enforcement~\cite{skillguard,skillscope}。MCP 规范也强调 tool metadata 和 behavior annotations 不能默认被当作可信事实~\cite{mcp-spec}。这些工作说明 Skill/MCP ecosystem 已经需要权限边界。与 package-level manifest 或 server-level grant 不同，\sys 的 authority root 是当前 intent certificate：同一个 Skill 或 MCP server 在不同 run 中会得到不同、可消耗、不可自扩张的 leases。一个 lease 可以同时约束 Skill instruction 对 workflow 的影响、MCP result 对后续 sink 的影响、本地脚本的 file/network authority，以及 subagent 获得的 attenuated capability set。

更重要的区别是 field ownership。即使 Skill/MCP manifest 也按 run mint，如果 package 或 server 可以声明 sink、approval 或 delegation field，它仍然能替 agent issuer 证明 user-owned authority。例如用户只授权 \texttt{org/repo-x}，而 Skill manifest 声称 safe repo 是 \texttt{attacker/repo} 或需要 full GitHub scope。若 per-run manifest 把 package 声明当作 sink/scope owner，就会接受该扩大权限。\sys 拒绝这一路径，因为 sink 和 approval scope 是 agent-owned fields，不能由 Skill/MCP/tool/env 填充。这个定位把 \sys 放在 Skill/MCP permissions 之上，作为一个 intent-derived lease compiler/checker，而不是替代 Skill/MCP 的包级权限系统。

一个具体区别是 union authority。相同的 PDF Skill 若用于“只把两个选中文件转换为本地 XLSX”，\sys 只会给 selected-file read、本地 output write 和无网络 exec lease。若用于“提取表格并在指定 GitHub repo 创建 issue”，同一 Skill 仍不能选择 repo，但 workflow 会多出由 agent intent 和 GitHub tool context 共同证明的一次性 issue lease。Skill-level manifest 若要覆盖两个 run，通常必须声明二者的 union authority。\sys 则在每次 protected transition 上重新绑定 intent、Skill instruction、tool/cmd interface 和 env evidence。因此它不是 Skill permission 的子集，而是跨 Skill、MCP/cmd、env 和 subagent 的 run-time authorization layer。

\section{结论}

LLM agent extensions 改变了权限系统需要保护的对象。文档、工具结果和扩展说明不只携带数据，也可能影响 tool selection、sink selection、approval scope、capability request 和 delegation target。\sys 将这种影响权表示为 intent-derived、provenance-gated、可消耗、可过期、可衰减的 protected-decision leases，并用 deterministic checker 把 LLM proposals 保持在 TCB 之外。在已建模并接入 checker 的边界内，这些结果表明 issuer-owned field proofs、context provenance 和 lease lifecycle 可以作为同一个授权状态被检查和执行：event model 覆盖全部 reference actions，issuer/lifecycle ablations 暴露同事件 false accepts，多边界原型在本地 side-effect、placement 和 handoff 前执行同一个 commit contract。因此，本文不是发明 capability、taint、lease 或 monitor 本身，而是定义 agent authority-changing decision 的 runtime linearization object。Agent extension security 的关键不只是限制工具能做什么，也包括限制不同 context 能证明什么、影响什么，以及何时消耗或转移 authority。
