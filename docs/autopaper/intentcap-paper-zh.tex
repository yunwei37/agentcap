\begin{abstract}
LLM agents 正在从固定工具调用器演变为可扩展执行环境，可以加载 Skills、连接 MCP servers、运行本地脚本，并将任务委托给 subagents；现有权限机制约束工具、文件和网络端点，却很少约束不可信 context 能否影响未来的 tool selection、sink selection、approval scope 或 delegation target。
本文的核心 insight 是：agent authorization 的单位不应是 tool permission、context label 或已发生的 action，而应是由当前用户意图铸造、会改变未来 authority state 的 protected-decision transition（受保护的权限状态转移）。
\sys 将这种 transition 表示为短期 lease，把 agent、instruction、tool 和 env 四类 issuer-typed required fields 与 control/data provenance、temporal state、budget 和 delegation bounds 绑定，并由 deterministic checker 在副作用或权限转移前原子验证。
当前稿呈现设计、形式化和 pilot evidence：在已建模边界内，\sys 拒绝 ACL-style wrappers 会放行的 protected-decision violations，阻止本地 env model-loop 中的 unsafe side effects，并在 local-Qwen 对照中分离 authority surface 与 reward/oracle success；完整端到端 utility、recovery 和独立 expert-oracle 结论仍由 E1--E4 主实验闭环决定。
\end{abstract}

\section{引言}

LLM agents 的安全边界正在发生变化。传统应用扩展通常通过 API、插件接口或系统调用与宿主交互，因此权限系统可以围绕文件、网络、进程和工具调用来定义。现代 agent extension 的行为则由用户请求、系统策略、扩展说明、外部文档、工具结果和中间总结共同塑造。

这种执行过程打破了 data 和 authority 的传统边界。考虑一个具体 failure trace：用户只要求从两个 PDF 提取表格，并在 \texttt{org/repo-x} 创建一个 summary issue；其中一个 PDF 的隐藏文本诱导 agent 请求 full GitHub approval，再把一次性 issue authority 委托给一个 summarizer subagent，并在 approval 被消耗后复用同一权限。最终出现的 \texttt{create\_issue(repo=org/repo-x)} 参数仍然等于用户指定的 repository，所以普通 tool/argument guard 可能放行。可是实际事故已经发生：凭证范围被扩大，权限被转移，审计日志也无法回答“哪一个可信 intent 授权了这次扩权”。

根因不是不可信文本进入 prompt 本身，而是 agent runtime 把用户意图、扩展说明、tool metadata、工具结果和运行时输出放进同一个 planning channel；下游 guard 多数在 action materialized 之后才检查 tool、arguments 或 resource。于是，一个对输出内容而言安全的输入，可能对控制决策而言并不安全：PDF text 可以安全地填充 spreadsheet cells，却不应选择 sink、扩大 approval scope 或创建 delegation。本文称这一问题为 context privilege：context 获得了影响 authority-bearing decisions 的权力，而不仅是携带可被引用或总结的信息。

这个问题也说明为什么 agent authorization 不能把所有 context 合并成一个信任等级。Skill/manual text 可以指导 workflow，但不能替代用户 intent；MCP/tool metadata 可以描述接口和 credential scope，但不能授权 sink；script output 和 tool result 可以提供 runtime evidence，但不能升级成 instruction 或 approval reason。粗粒度地折叠这些来源会让过程建议、接口描述和环境观测互相补权，导致 overbroad permission、approval fatigue，或因无法表达合法 runtime evidence 而产生不必要的 false denial。

本文关注的 protected decisions 是任何会 create、select、widen、spend 或 transfer authority 的模型可见选择，包括 tool selection、sink selection、approval scope、policy/capability request 和 delegation target。边界也随之明确：本文只保护被 instrumentation 暴露的 authority lifecycle events，不追踪模型内部的任意语义影响，也不覆盖尚未接入 adapter 的 OS-level side effects。

现有防线分别覆盖了这个问题的一部分。Tool guards 和 argument validators 可以拒绝错误对象或非法参数；static allowlists 和 server-level policies 可以限制可见工具；human approval gates 可以重新引入用户确认；OS sandbox 或 monitor 可以约束文件、网络和进程 side effects；provenance、taint 和 IFC-style defenses 可以表达部分 flow policy。这些机制是必要的，但 residual gap 在于：它们通常不把 future decision authority 表示为一个可消耗、可过期、可委托的 transaction object，其有效性同时依赖 intent derivation、allowed control provenance、temporal state、budget 和 delegation bounds。

\sys 从这一差异出发，但不把“检查 action provenance 是否可信”作为 novelty claim；这一方向已经被多种 agent provenance、IFC 和 intent-policy 系统探索。本文的可检验观点更窄也更强：agent least privilege 的授权单位不应是 tool、resource、context source 或已发生的 action，而应是一次由 intent 铸造、会改变未来 authority state 的 protected-decision transition。这一观点要求一个最小 transaction interface：issuer-typed required fields 说明每个 protected field 必须由哪个 trusted issuer 证明；control-provenance proof 说明该 field 被哪些 context 控制；\texttt{mint}/\texttt{check\_and\_consume}/\texttt{attenuate}/\texttt{expire} 在同一个 checker state 上原子更新 authority；adapter 在副作用或权限转移前调用该接口。缺少 typed fields 会接受 cross-plane promotion；缺少 consume/expire 会接受 stale reuse；缺少 attenuation 会接受 over-delegation；缺少 control proof 会把 untrusted data 当作 authority。\sys 的 lease 是这类 transition 的原子授权对象：它把 trusted mint provenance、decision-specific influence authority、consumption/expiration 和 delegation attenuation 绑定在同一个状态更新里。一个 high-impact event 只有在两个条件同时成立时才允许执行：它必须由当前 intent certificate 推导出来；并且它的 control provenance 必须来自被授权影响对应 decision class 的 context source。

\sys 用 intent-carrying capability leases 表达这一生命周期。Lease 是短期、可消耗、可衰减的授权对象，绑定 holder、operation、object、arguments、intent derivation、control/data provenance、flow、temporal state、budget 和 delegation。它的输入不是单一 prompt，而是四类 agent-specific context 的交集：agent context 给出当前 intent 与 workflow state，instruction context 给出可被当作过程指导的文本，tool context 给出接口和 side-effect surface，env context 给出运行时对象、结果和资源事实。Lease table、budget counter 和 consumption state 属于 checker state \(\sigma\)，而不是 env context；env 可以报告观测事实，但不能伪造 active 或 exhausted authority。与只在 action time 比对参数来源的 guard 不同，\sys lease 先成为 future decision authority 的对象：它定义哪些 context 可以影响哪些安全相关字段、何时有效、可以被调用几次、能否传给 subagent、以及如何 lowering 到不同 extension boundary。因此，同一个 document 可以被授权 parameterize 输出数据，但不能 authorize 新工具、扩大 approval scope 或选择外部 sink。

这一设计带来两个实现挑战。第一，lease 不能由 LLM 单独决定，否则攻击 context 仍可能通过 prompt 影响授权本身。第二，系统不能只在工具调用口检查权限，因为 untrusted context 可能先影响 plan、prompt construction、delegation 或 approval request。\sys 因此采用 LLM-assisted compiler 与 deterministic checker 的结构：LLM 只提出 plan 和 candidate leases；checker 根据 intent certificate、context labels、effect IR 和 global policy 验证 proof-carrying leases；runtime adapters 再将 leases lowering 到 context constructor、tool gateway、MCP broker、本地执行 sandbox 和 delegation monitor。这个定位把 \sys 与参数级 provenance guard 区分开来：后者判断已观察到的 action 是否可信，\sys 还显式管理未来决策权限的 mint、check、consume、attenuate 和 delegate 过程。

本文贡献如下：

\begin{enumerate}[leftmargin=*,label=\textbf{C\arabic*.}]
  \item \textbf{Model.} 一个 stateful protected-decision lease model，将 agent 授权单位定义为 intent-minted、issuer-typed、可消耗且可衰减的 authority-state transition（第~\ref{sec:formal} 节）。
  \item \textbf{System.} 一个 checker-centered transaction API with cross-boundary adapters，实现 \texttt{mint}、\texttt{check\_and\_consume}、\texttt{attenuate} 和 \texttt{expire}，包含已实现的 tool/MCP 与 local-env gateways、trace-level instruction/delegation adapter contracts，以及可 lowering 到 ActPlane-style OS monitor 的 env-side contract（第~\ref{sec:design}--\ref{sec:implementation} 节）。
  \item \textbf{Evaluation protocol and pilot evidence.} 当前 full-paper scaffold 给出一个可证伪 evaluation protocol with pilot evidence，把已经有机制性证据的 checker/runtime claim 与尚需 E1--E4 闭环的 full-paper security、least-privilege 和 utility/recovery claims 分开；最终论文应将该贡献替换为完成后的 workload characterization 和 evaluation results。
\end{enumerate}

\section{背景与动机}

\subsection{Agent Extensions}

现代 agent runtime 通常不只暴露一个固定工具集合。Skills 将任务说明、参考资料和可选脚本打包为可复用工作流；MCP servers 将工具 schema、资源和 prompts 暴露给 agent；本地脚本和 shell 执行提供文件转换和项目操作能力；subagents 则把上下文和部分权限交给另一个模型或执行单元~\cite{skillguard,skillscope,mcp-spec}。因而，agent extension 同时包含自然语言指令、结构化 tool metadata、外部数据和可执行代码。

\subsection{Authority and Provenance}

传统权限系统主要围绕 subject、object 和 operation 建模，例如进程是否能读取文件、调用 API 或连接网络。Agent 场景还需要记录 provenance，因为同一个 tool call 的参数可能来自用户选择、扩展说明、工具结果或不可信文档。已有 provenance、taint 和 IFC-style 系统说明不可信输入不应自由流向高影响输出，但它们通常把核心对象放在 observed action 或 data flow 上，而不是把未来 protected decision 本身作为可消耗、可衰减的授权对象。

\subsection{Motivating Workflow}
\label{sec:motivation}

本文使用一个文档处理工作流贯穿设计和评估。用户要求 agent 从两个选中的 PDF 中提取表格，保存 XLSX，并在指定 GitHub repository 中创建一个 issue。PDF text 可以影响表格单元格和 issue body，但不应影响 repository selection、approval scope、capability request 或 delegation target。这个例子把普通 data use 和 authority-bearing decision 分开，使问题不依赖某一种具体工具或 sandbox。

如果其中一个 PDF 包含隐藏指令要求 agent 请求 full GitHub access、加载另一个 Skill，或把 issue authority 交给 subagent，普通 argument check 只能看到最终调用是否仍然使用合法 repo。它无法表达 PDF text 不能成为 approval scope、fresh capability request 或 delegated capability set 的控制来源。\sys 的 running example 因此不是一个错误参数问题，而是一个 context-to-decision influence 问题。

\subsection{Protected-Decision Transitions}

本文的主术语是 protected-decision transition：一次会 create、select、widen、spend 或 transfer authority 的可观察状态转移。Tool selection、sink selection、approval scope、capability request、policy update 和 delegation target 都属于这类 transition；实现中把它们编码为 protected events。一个 guard 如果只检查最终 tool name 或 repo argument，会漏掉 untrusted context 对 approval scope、fresh authority request、policy patch 或 delegated capability set 的控制。因此，本文要保护的对象不是“某个字符串能否进入 prompt”，而是“某个 context source 是否被授权影响某类未来决策”。

三个术语在后文保持固定含义。Decision class 是受保护字段的类型，例如 sink selection、approval scope 或 delegation target。Protected event 是 instrumentation 在 runtime 看到的具体事件，例如一次 MCP 调用、approval request 或 subagent spawn。Protected-decision transition 是 checker 对 protected event 执行的原子状态更新，包括 proof check、lease consumption、temporal update 和 delegation-state update。这些 decision classes 定义了后续 threat model、lease language 和 evaluation oracle 的共同边界。

\section{威胁模型与非目标}

\sys 面向可扩展 agent runtime。Agent 可以读取外部文档、加载 Skills、连接 MCP servers、运行本地脚本、调用工具，并将部分任务委托给 subagents。攻击者可以控制部分输入 context，例如 PDF、网页、邮件、issue comment、MCP tool result、unsigned or malicious Skill text、script output 或 subagent summary。Signed/trusted Skill instruction 只有在被 issuer endorsement 后才能作为 instruction context 指导声明范围内的 workflow；installed-but-overbroad Skill package 也不能因此获得 sink、approval 或 delegation authority。主要攻击目标是让不可信 context 控制某个 authority lifecycle event，例如选择错误 sink、请求更宽 approval、修改 capability request 或扩大 delegation。读取无关私有文件、创建错误 repository 的 issue、修改策略、跳过检查或委托给恶意组件，是这些 unauthorized influence 在具体边界上的表现形式。

我们不假设 LLM 会可靠遵守安全指令；不信任外部文档、工具结果、MCP annotations 或 unsigned Skill instructions；也不假设静态分析能完美理解任意脚本。\sys 依赖的 TCB 包括 intent certificate issuer、deterministic checker、provenance tracker、audit log 和已经接入的 enforcement adapters。这个 TCB statement 是 per-boundary 的：没有接入 adapter 的 side effect 不在安全性质覆盖范围内。OS sandbox 或 ActPlane-style monitor 可以作为执行后端，但不是本文的核心贡献；\sys 关注的是上游 policy synthesis 和 context-authority checking。

非目标同样重要。\sys 不声称解决 prompt injection，不声称自动 compiler 永远正确，不替代 sandbox，也不保证所有 agent 决策都能被完整追踪。它的安全性质只覆盖已建模的 decision classes 和已接入的 enforcement boundaries。

\section{系统设计}
\label{sec:design}

\subsection{System Overview}

\sys 的设计目标是把“模型想做什么”和“系统允许什么”分开。LLM 可以读取任务、扩展说明和上下文，并提出 plan、effects 和 candidate leases；但它不能铸造最终权限。最终 authority 只来自 trusted intent certificate，并由 deterministic checker 根据 context provenance 和 global policy 验证。

本文的可证伪 novelty claim 是：agent 扩权错误常常不是 object predicate 失败，而是四个 authority planes 被跨类补权，或旧 lease state 被复用。只有把 trusted mint、decision-specific influence、consumption/expiry 和 delegation attenuation 绑定为同一个原子 transition，checker 才能同时阻止 stale reuse、cross-plane promotion 和 delegation mismatch。E3 因此不是辅助 ablation，而是验证这个 claim 的主实验；若一个 stateful ABAC/IFC/capability baseline 采用了同样的原子 transition object，则它是 semantic-equivalent baseline，而不是本文要击败的弱对照。

系统的数据流分成五步。首先，trusted issuer 将用户请求、显式对象选择、sink 选择、管理员策略和 fresh approval canonicalize 成 intent certificate。其次，context labeler 为用户输入、文档、Skill instructions、MCP metadata、tool results、script outputs 和 subagent summaries 标注 origin、integrity、confidentiality 和 influence modes。

第三，LLM-assisted compiler 从扩展元数据、tool schemas、local scripts 和当前 plan 中生成 security-relevant effect IR，并提出短期 leases。第四，checker 验证每个 lease 和事件是否满足 intent derivation、argument constraints、control/data provenance、flow、temporal、budget 和 delegation rules。最后，runtime adapters 将通过的 leases lowering 到 context construction、tool/MCP invocation、本地执行和 subagent delegation 边界。

\subsection{Design Goals}

Running example 中的 failure trace 对应四个设计需求：approval widening 需要 intent root 限定用户真正授权的对象和 sink；PDF text 控制 approval 或 delegation 需要 context no-promotion 和 influence separation；一次性权限复用需要 stateful lease lifecycle；跨 tool、MCP、本地执行和 delegation 的副作用需要 multi-boundary transaction API。Fail-closed denial feedback 和 adapter coverage 是实现这些目标的机制要求，而不是独立顶层目标。因此，设计目标包括四个可测试目标，每个目标都映射到贡献和评估判定标准：

\begin{enumerate}[leftmargin=*,label=\textbf{G\arabic*.}]
  \item \textbf{Intent-bounded authority.} High-impact authority 必须来自当前用户意图、显式对象/目标选择、管理员策略或 fresh approval；该目标由 C1 建模，主要由 E1 检验，E2 作为 authority-breadth secondary evidence。
  \item \textbf{Context no-promotion and influence separation.} 系统必须区分 data dependency 和 control dependency，允许不可信 context 作为数据使用，同时阻止其补齐不属于自己 plane 的 protected fields；该目标由 C1/C2 支撑，主要由 E3 检验，E1 作为 end-to-end secondary evidence。
  \item \textbf{Stateful lease lifecycle.} 权限必须是短期、可消耗、可过期、可衰减、不可由不可信 context 自扩张的 run-time leases，而不是安装期或 server 级别 ambient permission；该目标由 C1/C2 支撑，主要由 E3 检验，E2 作为 lease-quality secondary evidence。
  \item \textbf{Multi-boundary transaction API.} 同一 lease semantics 必须通过 central checker 和 adapter contracts 落到 context placement、tool/MCP invocation、本地 env boundary 和 delegation boundary；该目标由 C2 支撑，主要由 E4 检验，E1/E3 作为 boundary-coverage secondary evidence。
\end{enumerate}

这个结构把 \sys 和普通 tool allowlist 区分开来。Allowlist 只决定哪个对象可见；\sys 还决定哪些 context source 可以影响哪个 protected decision。即使某个工具和参数在对象层面都匹配，checker 仍会拒绝 control provenance 来自无权影响该 decision class 的 context。

\sys 的系统边界因此不是一个单点 wrapper，而是一个 central transaction API 加一组 adapter contracts。Tool/MCP adapter、本地 env adapter、instruction placement adapter 和 delegation adapter 都必须在 side effect、prompt authority placement 或 authority transfer 前调用同一个 checker；ActPlane-style monitor 可以强化 env adapter 的 file/process/network enforcement，但它执行的是 checker 已经决定的 lease contract，而不是替代 protected-decision authorization model。

\subsection{Authority Inputs}

\sys 的 capability synthesis 不是从单一 prompt context 中抽取权限，而是从四类 issuer-typed fields 中求解 lease。表~\ref{tab:authority-inputs} 把它们定义为系统接口，而不是单纯分类。四类 plane 的划分准则是：issuer 是否不同、可伪造面是否不同、能证明的 authority field 是否不同、adapter 观测点是否不同；只有四项都相同时，两个来源才可以合并。这里的 agent context 不是模型隐藏状态，而是 trusted issuer 维护的 task intent、approval 和 workflow state；instruction context 是可被当作过程指导的 policy/Skill/manual 文本；tool context 是接口、凭证和 sandbox contract；env context 是运行时观测到的对象、值、结果和资源事实。这个边界是 agent-specific 的：同一段文本如果来自 \texttt{SKILL.md}，可能在 workflow scope 内成为 instruction context；如果来自 PDF、tool result 或 script output，即使内容像指令，也仍是低完整性的 env context，不能自动升级为 instruction。

这四类不能合并成三类。Agent plane 的 issuer 是 trusted UI/approval state，能证明“为什么做”和“哪个对象或 sink 被用户授权”；instruction plane 的 issuer 是 system/developer policy 或被 endorsement 的 Skill/manual source，能证明“按什么过程做”；tool plane 的 issuer 是 trusted registry、MCP broker 或 local command registry，能证明“通过哪个接口、凭证和 sandbox contract 做”；env plane 的 issuer 是 runtime observer 或 monitor，能证明“哪些对象、值和结果在本次执行中存在”。把 instruction 并入 agent 会错误地把 Skill/manual 中的过程建议提升为用户意图或 approval；把 tool 并入 env 会丢失 schema、credential scope 和 declared side effect 这些接口约束，使工具结果和工具定义共享同一 authority class；把 env 并入 tool 则会让 runtime observations、script outputs 和文档内容获得本不具备的接口可信度。

这也不是普通 ABAC attribute grouping。ABAC 问的是一组 attributes 是否满足 policy rule；\sys 问的是四个独立签发的 authority planes 是否共同授权一次 authority-state transition，并且禁止一个 plane 替另一个 plane 补齐缺失字段。若 sink selection 缺少 agent-plane authorization，tool metadata 不能补上；若 local exec 缺少 tool-plane binary/schema，instruction 不能补上；若 approval request 缺少 fresh issuer token，env evidence 不能补上。

每个 plane 都有独立 issuer 和不可由其他 plane 伪造的字段。Agent plane 的不可伪造字段是 trusted issuer canonicalize 的 goal、authorized objects/sinks 和 approval tokens；instruction plane 的不可伪造字段是 policy/Skill/manual 的 workflow scope；tool plane 的不可伪造字段是 registry 或 MCP broker 认可的 schema、credential scope、binary descriptor 和 sandbox contract；env plane 的不可伪造字段是 runtime observer 记录的路径存在性、concrete values、tool results 和 script outputs。E3 的 collapse baselines 必须分别测试这些字段被其他 plane 补齐时的 false accept。

实际 capability 必须由相关 protected fields 与全局策略共同决定；未被某个 decision class 要求的 plane 可以为空，但被要求的 field 必须从自己的 owner plane 取证：

\[
K = Solve_\Gamma(C_{agent}, C_{inst}, C_{tool}, C_{env}) .
\]

纯 approval event、纯 context-use event 和本地 evidence probe 可能只需要其中几类字段。关键规则是：对某个 protected decision 所需的字段，checker 必须从该字段指定的 plane 读取，不能由其他 plane 代填。Agent context 可以授权“处理这两个用户选中的 PDF 并在 org/repo-x 创建一个 issue”，但不能指定 PDF Skill 的脚本路径；instruction context 可以指导 extraction workflow，但不能选择 GitHub sink 或扩大 approval scope；tool context 可以说明 \texttt{github.create\_issue} 的参数结构和 credential scope，但不能把任意 repo 变成授权 sink；env context 可以证明某个路径、参数值、PDF text 或 observed result 在运行时存在，但不能把存在性变成用户选择或 retroactively mint broader authority。来自用户的自然语言也会被拆分：被 trusted issuer canonicalize 成 goal、object、sink 或 approval 的部分进入 agent context；描述 workflow preference 或 formatting preference 的部分进入 instruction context，即使它来自用户，也不能单独授权 sink、approval 或 delegation。Lease 的作用是把这些 required fields 固定为一个可检查、可消耗、可衰减的 transition object。

\subsection{Intent Certificates}

Intent certificate 是结构化安全对象，而不是 prompt 中的一段自然语言。它记录用户目标、选择的对象、允许的 sinks、禁止的 sinks、approval tokens 和 expiry。对于 PDF 处理任务，certificate 会命名选中的 PDF 路径、输出目录、唯一授权的 GitHub repository、禁止任意网络访问的约束，以及创建 issue 所需的 fresh approval。LLM 可以提出 certificate fields，但 trusted parser、UI、administrator policy 或 approval step 必须 canonicalize 这些字段。Certificate 是后续 capability derivation 的根。

Certificate 的粒度决定后续 leases 的最大 authority。若用户只选择了两个 PDF，后续 file-read leases 只能覆盖这些对象或它们的安全派生物；若用户只确认了一个 GitHub repository，repo selection 就不能由 PDF text、Skill instruction 或 MCP result 改写。模糊字段不会被解释为宽权限；系统要么将其保守地标记为 unknown，要么要求 fresh approval。

\subsection{Context Authority}

每个 context cell 都带 provenance label，记录 origin、integrity、confidentiality、purpose、time 和 endorsements。\sys 的关键增加是为 context source 指定 influence modes。PDF text 可以被允许 quote、summarize 或 parameterize spreadsheet cells，但禁止 plan、delegate 或 authorize。Signed Skill instructions 可以在其 declared workflow scope 内 instruct extraction procedure，但不能为自己请求额外工具。MCP tool results 和 annotations 必须按照 server trust 处理；MCP specification 已经提醒 tool behavior annotations 不能默认可信~\cite{mcp-spec}，\sys 将这一原则变成 deterministic provenance rule。

Influence modes 总是相对于 decision class 解释。一个 source 可以被允许影响 output content，却不能影响 recipient、repository、network destination、approval scope 或 delegated capability set。这不是给 taint label 增加一个 policy 维度，而是规定某个 context 是否有权参与一次 authority-state transition。PDF text 可以流向 issue body 和 spreadsheet cells；普通 IFC 可以表达这条数据流是否允许。但 \sys 还要求 repository field 的 issuer 是 agent plane、approval scope 的 control proof 不含 PDF provenance、并且对应 one-shot lease 尚未被消耗。只有当 influence authority 与 intent derivation、budget、expiry 和 delegation bounds 绑定在同一 lease 上时，checker 才能防止 context 在一个字段上被允许、又在另一个生命周期状态中被复用为授权依据。这种 decision-specific authority 是 \sys 的核心：它允许系统保留有用的数据流，同时切断同一 context 对 authority-bearing fields 的控制流。

\subsection{Effect IR and Leases}

Agent plan 被编译成安全相关 effect graph。IR 不追求表达完整程序语义，而是捕捉会影响 authority 的事件：context use、tool call、MCP call、file read/write、subprocess execution、network connection、approval request 和 subagent delegation。每个 effect 记录 data provenance 和 control provenance。这个区分是 \sys 的关键：不可信 PDF bytes 可以是 output data 的 data dependency，但不应是 sink selection 的 control dependency。

Control provenance 不是声称系统能读出 LLM 内部注意力，而是一个 protected-field proof policy。对于 sink、approval scope、capability request、delegation set、policy patch、local executable、network destination 和 high-impact arguments，runtime adapter 必须提交结构化来源证明：字段来自 trusted intent certificate、fresh approval、tool schema、runtime evidence，或某个 explicit context cell。若字段由模型自由生成、缺少结构化 proof，或来源不能被 adapter 归因，checker 保守地把该字段的 control provenance 近似为当前 prompt 中所有可见、可能影响该字段的 non-policy context；只要其中存在缺少所需 influence mode 的 context，就 fail closed。普通 output body 或 spreadsheet cell 可以带 data provenance；authority-bearing fields 必须带 control proof。这一策略会带来 utility cost，因此 E1/E4 必须报告 false denial、proof completeness 和 recovery，而不是假设 provenance always precise。

Lease 绑定 operation、object、arguments、intent、influence、flow、temporal、budget、expiry 和 delegation。一个 PDF extraction lease 可以允许脚本读取两个选中文件并写入输出目录，但禁止网络、secrets 和额外 subprocess；GitHub issue lease 可以允许一次 \texttt{create\_issue(repo=org/repo-x)}，但要求 repo control provenance 只能来自 trusted user intent。Lease solving 的目标不是全局证明所有可能 plan 的最小权限，而是在给定 effect plan、extension catalog 和 global policy 下生成风险更低、可检查、可衰减的 authority set。

\subsection{Compiler and Checker}

\sys 使用 LLM-assisted 但 checker-verified 的 pipeline。Compiler frontend 总结 Skill metadata、MCP schemas、local scripts、observed traces 和 policy；推导 plan 与 security-relevant effects；并为当前 run 提出 candidate leases。Frontend 不在 TCB 内。Deterministic checker 验证每个 lease 是否满足 global policy、intent certificate、provenance constraints、flow rules、temporal guards、budgets 和 delegation monotonicity。

Compiler 的输出必须携带 proof object，说明每个 protected field 来自哪些 intent fields、哪些 context sources 可以作为 data/control dependency、哪些 sources 被排除、以及 lease 在什么 temporal state 和 budget 下有效。Checker 不信任 proof 的自然语言解释，只检查其中的结构化 predicates。若 proof 缺失、argument predicate 过宽、control provenance 不明、或 lease 试图把 parent 不具备的 authority 委托给 child，checker 返回结构化 denial。

Denial 是 runtime refinement 的输入，而不是 authority escape hatch。LLM 可以根据 denial reason 重新规划，例如选择更窄的 sink、补充 read-only proof probe、或请求用户 fresh approval；但新 proposal 仍要经过同一个 checker。这一设计使 recovery 可以改善 utility，而不会把失败路径变成 broad permission fallback。

\subsection{Runtime Lowering}

单一 tool gateway 不足以表达 \sys 的安全边界，因为 protected decision 可能发生在工具调用之前的 context construction、计划生成和 approval request 中，也可能发生在调用之后的 delegation 或 runtime binding 中。Accepted leases 因此被 lowering 到五类 adapter contracts：

\begin{itemize}[leftmargin=*]
  \item \textbf{Agent adapter.} 维护 intent certificate、approval tokens、workflow stage、active/exhausted leases 和 parent-child delegation state。它是 \(C_{agent}\) 的 trusted issuer，并且是唯一能把 approval 转换为 fresh lease mint event 的组件。
  \item \textbf{Instruction adapter.} 对 system/developer/user instructions、Skill/manual files、\texttt{SKILL.md}、workflow notes 和 MCP prompts 做 provenance labeling，决定哪些文本可以进入 trusted instruction section，哪些只能进入 untrusted data section。它执行 no-promotion rule：env outputs 不会因为包含命令式文本而变成 instruction。
  \item \textbf{Tool adapter.} 将 MCP/tool schema、本地 command descriptor、credential scope、argv/env schema 和 declared side effects 转换为 \(C_{tool}\)。Tool gateway 和 MCP broker 只暴露 active leases 覆盖的 callable surface，并在 side effect 前验证 arguments、control provenance 和 budget。
  \item \textbf{Env adapter.} 观察 cwd、selected files、tool results、script outputs、runtime evidence 和 resource availability。它只产生 \(C_{env}\) facts，并把 file、process、network 和 secrets events 交给 checker；active/exhausted lease table 与 consumption counters 只在 checker state \(\sigma\) 中维护。Execution sandbox 将 file、process、network 和 secrets constraints 下发到底层环境；ActPlane-style backend 可以作为这一 adapter 的 OS-level lowering target，用来覆盖 local script、process 和 network side effects。
  \item \textbf{Delegation adapter.} 在 subagent spawn 前调用 \(attenuate(\kappa_{parent},K_c)\)，确保 child context 和 capabilities 都不宽于 parent active leases，并记录 delegated authority 的 consumption state。
\end{itemize}

这些 adapters 是设计层 contracts，不需要共享同一种底层 sandbox；它们共享的是 lease language 和 checker verdict。第~\ref{sec:implementation} 节只报告当前 prototype 实例化的边界，而不是声称所有 contracts 都已有生产级实现。若某个 boundary 尚未接入 adapter，\sys 不声称覆盖该 side effect。这个限制保留了 OS-level monitor 的价值，也使本文的贡献保持在 policy synthesis and checking，而不是新的 syscall monitor。

\section{形式化模型}
\label{sec:formal}

\sys 的形式化目标不是为任意自然语言推理建模，而是为上一节 checker pipeline 的输入和输出提供可检查语义。Compiler、gateway 和 runtime adapters 负责把模型计划、工具调用、MCP 调用、本地执行和 delegation 转换成安全相关事件；checker 只对这些事件、active leases、intent certificate 和 provenance labels 做确定性判断。因此，模型刻意不讨论 LLM 内部是否“理解”指令，而是讨论一个被 instrumentation 暴露出来的 authority-bearing event 是否应被接受。

一个 agent execution 被表示为事件 trace：

\[
\tau = e_1, e_2, \ldots, e_n .
\]

事件语言只覆盖会影响 authority 的边界：

\[
\begin{aligned}
e ::=~& ctx(c,m,d)
  \mid call(t,args)
  \mid mcp(s,t,args)
  \mid read(o)
  \mid write(o) \\
  & \mid exec(p,args)
  \mid send(sink,msg)
  \mid spawn(a,K)
  \mid approve(u,a) \\
  & \mid request(\kappa,reason)
  \mid policy\_update(patch)
  \mid lease(\kappa).
\end{aligned}
\]

每个事件带有两类 provenance：

\[
prov(e)=\langle prov_D(e), prov_C(e) \rangle ,
\]

其中 \(prov_D\) 表示数据依赖，\(prov_C\) 表示控制依赖。\sys 只在 protected decisions 上约束 \(prov_C\)，例如 tool selection、sink selection、approval scope、policy update 和 delegation target。这样可以表达同一段 PDF text 可以作为 issue body 的 data dependency，却不能作为 GitHub repository 参数的 control dependency。

每个 context cell \(c\) 具有标签：

\[
\ell(c)=\langle origin, integrity, confidentiality, purpose, time, endorsements\rangle .
\]

标签进一步映射到 context authority。这里需要区分 influence mode 和 decision class：mode 描述 context 的使用方式，例如 quote、summarize、parameterize、plan、instruct、delegate 和 authorize；decision class 描述被影响的安全相关字段，例如 output content、sink selection、approval scope、capability request、policy update 和 delegation target：

\[
auth(c) \subseteq InfluenceModes \times DecisionClasses .
\]

例如，PDF text 可以 parameterize spreadsheet cells 并 summarize issue body，但不能 authorize capability request、policy update 或 GitHub repository selection。实现可以将 \(InfluenceModes\) 做成偏序或命名集合；论文中的性质只要求 checker 能判断某个 context 是否被授权以所需 mode 影响某个 decision class。

Intent-carrying lease 的抽象形式为：

\[
\kappa =
\langle holder, op, obj, args, I, influence, flow, temporal, budget, expiry, delegation\rangle .
\]

其中 \(I\) 是当前 intent certificate，\(influence\) 约束 control/data provenance，\(flow\) 约束 confidentiality 到 sink 的流向，\(temporal\) 和 \(budget\) 约束事件顺序与调用次数，\(delegation\) 约束可传递给 subagent 的能力集合。

形式上，lease synthesis 可被看作从 per-decision issuer-typed fields 中求解可授权 lease：

\[
\kappa \in Solve_\Gamma(C_{agent}, C_{inst}, C_{tool}, C_{env}) .
\]

其中 \(C_{agent}\) 包含 intent certificate、authorized object/sink selections、approvals、plan stage、agent role 和 parent authority；\(C_{inst}\) 包含 system/developer/user instructions、Skill/manual workflow text、MCP prompts 和 trust endorsements；\(C_{tool}\) 包含 MCP/tool schemas、server identity、credential scope、command/script execution descriptors 和 sandbox contracts；\(C_{env}\) 包含 runtime object existence、cwd/filesystem state、concrete arguments、observed results、runtime evidence 和 resource availability。Active/exhausted lease table、per-lease counters 和 delegation graph 属于 checker state \(\sigma\)，不属于任何可由环境输出填充的 context class。

Checker 以四个 projection functions 读取这些输入面：

\[
\begin{aligned}
intent(e) &\leftarrow \pi_{agent}(C_{agent}),\\
procedure(e) &\leftarrow \pi_{inst}(C_{inst}),\\
interface(e) &\leftarrow \pi_{tool}(C_{tool}),\\
evidence(e) &\leftarrow \pi_{env}(C_{env}) .
\end{aligned}
\]

一个 protected event \(e\) 只有在 \(Req(d)\) 要求的 projections 与同一个 lease \(\kappa\) 的 predicates 一致时才被接受。未被 \(Req(d)\) 要求的 plane 可以为空；但 required field 不能从其他 context class 中补齐。例如 instruction 不能补齐 authorized sink，tool schema 不能补齐 user intent，env evidence 不能补齐 approval token。这个 issuer-typed required-field 形式化使 \sys 的系统贡献不只是 MCP/tool-call guard，而是一个跨 Skill instruction、tool interface、runtime environment 和 agent workflow state 的统一 authorization transaction。

不同 protected decision 对四类 context 的需求不同。令

\[
Req(d)=\langle A_d, I_d, T_d, E_d\rangle
\]

表示 decision class \(d\) 的 agent、instruction、tool 和 env field requirements。若 \(A_d\)、\(I_d\)、\(T_d\) 或 \(E_d\) 中任一 required field 缺失，checker 必须拒绝该 transition，而不能由其他 context class 代填。表~\ref{tab:decision-reqs} 给出 running example 中的代表性规则。

为了避免把四类 plane 退化成普通 ABAC attribute namespaces，\sys 将 required fields 视为 issuer-typed authority fields。令 \(owner(f)\in\{agent,inst,tool,env\}\) 表示 field \(f\) 的唯一授权 issuer class，令 \(proof(e,f)=\langle issuer,value,source\rangle\) 是 runtime adapter 为事件 \(e\) 提交的结构化证明。No-substitution typing rule 是：

\[
\frac{f \in Req(d) \quad proof(e,f)=\langle q,v,s\rangle \quad q = owner(f)}
     {\Gamma \vdash proof(e,f) : f}
\qquad
\frac{q \neq owner(f)}
     {\Gamma \nvdash \langle q,v,s\rangle : f}
\]

例如 \texttt{repo=org/repo-x} 的 sink authorization field 由 agent issuer 证明；trusted GitHub tool metadata 可以证明 schema field，却不能证明该 repo 是用户授权 sink。于是，同一个 tool-issued proof 对 interface field 可以被接受：

\[
\Gamma \vdash \langle tool,\texttt{create\_issue.schema},mcp\_registry\rangle : github\_schema ,
\]

但对 agent-owned field 必须被拒绝：

\[
\Gamma \nvdash \langle tool,\texttt{org/repo-x},mcp\_metadata\rangle : authorized\_sink .
\]

同理，tool result 或 tool metadata 也不能证明 \(policy\_update\) field。ABAC 或 IFC system 若要与 \sys 等价，必须实现这种 typed transaction API，并且在同一次 accepted transition 中原子检查 required fields、control-provenance proof、lease consumption 和 delegation state。

Context class 不是普通 attribute。Checker 还维护一个 no-promotion invariant。令 \(class(c)\in\{agent,inst,tool,env\}\)，令 \(endorse_q(c)\) 表示 trusted issuer 明确把 \(c\) endorsement 为 class \(q\)。若没有 endorsement，则：

\[
\begin{array}{ll}
c\in C_{env} \land \neg endorse_{inst}(c) & \Rightarrow c \notin C_{inst},\\
c\in C_{tool} \land \neg endorse_{agent}(c) & \Rightarrow c \notin C_{agent},\\
c\in C_{inst} \land \neg endorse_{agent}(c) & \Rightarrow c \notin C_{agent}.
\end{array}
\]

直观地说，tool result、script output 或 PDF text 不能因为包含祈使句就变成 instruction；tool schema 不能因为声明“安全”就变成 user intent；Skill/manual instruction 不能因为要求更多权限就变成 approval。Promotion 只能由 trusted issuer、policy 或 fresh approval 显式产生，并作为新的 lease mint event 记录在 \(\sigma\) 中。No-promotion 还意味着任一 context class 都不能替另一个 class 补齐缺失的 authority field：instruction 可以建议 workflow，但 binary/schema 必须由 tool context 验证；tool context 可以定义接口，但 sink/intent 必须由 agent context 验证；env context 可以提供 observed values，但不能把这些 values 转换成 approval 或 policy。

Lease lifecycle 是 \sys 与静态 policy rule 或单次 action provenance check 的主要区别。Agent workloads 中，同一个 action argument 可以在不同 authority state 下有不同安全含义：第一次 \texttt{create\_issue(repo=org/repo-x)} 可能由 fresh approval 授权；第二次相同参数可能是在复用已消耗的一次性 lease；同一 repo 参数交给 subagent 可能违反 delegation bound；同一 approval reason 若由 PDF text 控制，则可能变成 scope widening。Action-time provenance 或 object predicate 往往只看到 repo 值仍然匹配，无法区分这些 lifecycle violations。Lease 因此是 future protected decision 的有状态授权对象，而不只是具体 operation 的 permission：它表达某个 context source 在 intent \(I\) 下可以影响 decision class \(d\)，直到某个状态转移使其被消耗、过期或衰减。更重要的是，mint provenance、influence authority、consumption/expiration 和 delegation bound 必须在同一个 authorization object 上被检查并原子更新；若它们分散在独立 taint labels、ACL entries、counters 和 delegation records 中，系统就必须额外证明这些对象不会产生 stale authority reuse、cross-object widening 或 consumed-state bypass。Runtime state \(\sigma\) 至少包含 active/exhausted lease table、per-lease consumption counters、temporal automata state、mint proof records 和 parent-child delegation graph；实现中的 checker state schema 直接保存这些字段，并在每个 accepted event 后原子更新。Checker 将 lease 作为有状态对象处理，而不是把它当作一组静态 predicates。

Checker 暴露的 authority-changing interface 只有四类 transition：

\[
\begin{aligned}
&mint(\kappa), \quad check\_and\_consume(e,\kappa),\\
&attenuate(\kappa_{parent},K_c), \quad expire(\kappa).
\end{aligned}
\]

Runtime adapters 可以分布在 context、tool、MCP、local execution 和 delegation 边界，但它们不能本地修改 authority state。任何会产生外部副作用或改变 authority state 的 protected event，包括 tool call、MCP call、local command、approval request、lease mint、policy update 和 delegation，在副作用发生前都必须经过 central checker state 的 \(check\_and\_consume\)。Accepted event、budget update、provenance record 和 delegation-state update 是同一个 transition：

\begin{itemize}[leftmargin=*]
  \item \textbf{Mint.} 只有 trusted issuer 可以通过 \(lease(\kappa)\) 事件把新 lease 加入 \(K\)。Mint rule 要求 \(\kappa.intent\) 可由 \(I\) 中的 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导；untrusted context 不能成为 mint 或 widening 的 control dependency。
  \item \textbf{Check.} 每个 protected event 在 side effect 发生前必须引用一个 active lease \(\kappa \in K\)，且 operation、object、arguments、influence、flow 和 temporal predicates 同时满足。没有 active lease 的 event 即使 action arguments 看似可信也被拒绝。
  \item \textbf{Consume.} 若 \(\kappa.budget\) 是 one-shot 或 bounded count，accepted event 会递减 counter；counter 为零时 \(\kappa\) 从 active set 中移除或进入 exhausted state。后续相同 tool/argument 的复用必须重新 mint。
  \item \textbf{Expire.} Temporal automata 可以在 task completion、approval window 结束、required predecessor missing 或 forbidden successor 出现时使 lease invalid。Expired lease 不再授权 event。
  \item \textbf{Attenuate.} Delegation event \(spawn(a,K_c)\) 只在 \(K_c \sqsubseteq K_{parent}\) 时被接受；child lease 的 objects、arguments、budgets、sinks、influence modes 和 delegation rights 均不得宽于 parent lease。
  \item \textbf{Deny amplification.} Skill instructions、MCP results、documents、script outputs 或 subagent summaries 可以作为 data/control provenance 参与普通 decisions，但除非被 trusted issuer endorsement，它们不能 mint fresh high-impact lease、扩大 existing lease，或把 exhausted/expired lease 恢复为 active。
\end{itemize}

这些规则也可以写成紧凑的状态转移。令 \(K_\sigma\) 表示 \(\sigma\) 中的 active leases，\(budget_\sigma(\kappa)\) 表示剩余调用次数，\(\sqsubseteq\) 表示 capability attenuation：

\[
\begin{array}{ll}
\textsc{Mint}:&
  \Gamma;I;\sigma \vdash trusted\_mint(\kappa)
  \Rightarrow \sigma[K_\sigma \leftarrow K_\sigma \cup \{\kappa\}] \\
\textsc{Check}:&
  \Gamma;I;K_\sigma;\sigma \vdash e:\kappa
  \land active_\sigma(\kappa)
  \Rightarrow \sigma_1 \\
\textsc{Consume}:&
  budget_{\sigma_1}(\kappa)=1
  \Rightarrow \sigma_1[K \leftarrow K_{\sigma_1}\setminus\{\kappa\}] \\
&
  budget_{\sigma_1}(\kappa)>1
  \Rightarrow \sigma_1[budget(\kappa)\leftarrow budget_{\sigma_1}(\kappa)-1] \\
\textsc{Expire}:&
  expired_\sigma(\kappa)
  \Rightarrow \sigma[K \leftarrow K_\sigma\setminus\{\kappa\}] \\
\textsc{Attenuate}:&
  spawn(a,K_c) \land K_c \sqsubseteq K_\sigma
  \Rightarrow \sigma[child(a)\leftarrow K_c] \\
\textsc{NoAmp}:&
  c \notin trusted\_issuers
  \Rightarrow \neg mint_c(\kappa') \land \neg widen_c(\kappa,\kappa').
\end{array}
\]

这些转移表达的不可约 invariant 是：authority state 不能通过 untrusted provenance 被创建、刷新、扩大或委托；若一个 provenance graph 或 stateful policy baseline 要表达同一性质，它必须引入等价的原子授权对象，同时覆盖 trusted mint、decision-specific influence、budget consumption、expiration 和 monotone delegation state。换言之，\sys 的 claim 不是这些检查项各自新颖，而是 agent protected decisions 需要一个把它们绑定为同一 transition 的授权单元。

Checker judgment is:

\[
\Gamma; I; K; \sigma \vdash e \Rightarrow \sigma' ,
\]

其中 \(\Gamma\) 是全局策略，\(I\) 是 intent certificate，\(K\) 是 active leases，\(\sigma\) 是 runtime state。事件被接受当且仅当存在 \(\kappa \in K\) 使得所有条件同时成立。Operation 必须匹配；object 和 arguments 必须满足 lease predicate；event purpose 必须可由 \(I\) 推导；\(prov_C(e)\) 中的每个 context 都必须拥有影响该 decision class 所需的 authority；\(prov_D(e)\) 必须满足 flow policy；temporal guard 和 budget 必须尚未过期；delegated leases 必须不宽于 parent leases。对于 \(request(\kappa,reason)\) 和 \(policy\_update(patch)\)，checker 将其作为 protected decisions 处理。它们自身不能 mint authority，除非 trusted issuer 随后产生独立 \(lease(\kappa)\) event；其 control provenance 也必须拥有 \((authorize, capability\_request)\) 或 \((authorize, policy\_update)\)。

一个 trace 被接受，表示其中每个 protected event 都按上述 judgment 顺序通过 checker，并且每个通过事件对 \(\sigma\) 的更新会消耗预算、推进 temporal state、记录 provenance，并更新 active leases。下面四个性质是这一 checker semantics 的直接安全后果；它们不是关于任意 prompt 的全局声明，而是关于 accepted traces over instrumented protected-decision events with adapter-supplied control-provenance proofs 的 trace-level properties。

\subsection{Property 1: No Unauthorized Influence}
对任意被 instrumentation 暴露并被 checker 接受的 trace \(\tau\)，如果事件 \(e \in \tau\) 属于 protected decision class \(d\)，令 \(req(d)\) 是 \(d\) 所需的最低 influence mode，则对所有 \(c \in prov_C(e)\)，都有 \((req(d),d) \in auth(c)\)。换言之，未被授权以所需 mode 影响 \(d\) 的 context 不能成为该决策的控制依赖。

\subsection{Property 2: Intent-Bounded Authority}
对任意 accepted high-impact event \(e\)，存在 active lease \(\kappa\)，且 \(\kappa\) 的 intent predicate 可由 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导。系统不会仅凭 Skill identity、MCP server identity 或工具描述铸造 high-impact authority。

\subsection{Property 3: Delegation Monotonicity}
对任意 accepted delegation event \(spawn(a,K_c)\)，child capability set \(K_c\) 是 parent active capability set 的子集或更窄 attenuation。除非 trusted issuer 基于新的 intent certificate 创建 fresh lease，subagent、Skill、MCP tool 或 script 都不能扩大自身权限。

\subsection{Property 4: No Context-Class Promotion}
对任意 accepted trace，若 context cell \(c\) 的原始 class 是 \(env\)、\(tool\) 或 \(inst\)，则 \(c\) 不能在后续 checker transition 中以更高 authority class 使用，除非 trusted issuer 产生显式 endorsement 或 fresh lease mint event。特别是，tool result、script output、PDF text 和 subagent summary 不能仅凭自然语言内容变成 instruction context；tool schema 不能变成 user intent；Skill/manual instruction 不能变成 approval。

这些性质是 trace-level safety properties。它们依赖于两个工程前提：runtime 必须把 protected decisions 接入 checker，并且 provenance tracker 必须保守地标记事件控制依赖。若某个 OS-level side effect 没有经过任何 adapter，\sys 无法单独观察它；这正是 sandbox 或 ActPlane-style backend 仍然有价值的原因。

\section{实现}
\label{sec:implementation}

我们的 prototype 是一个 Python-based agent authorization harness。它的目标不是替换生产级 MCP server、agent runtime 或 OS sandbox，也不是声称设计中的每个 lowering contract 都已完整工程化；它验证三件事：lease language 是否足以表达 intent/context authority，deterministic checker 是否能在统一 event model 上执行这些约束，以及这些约束能否被 lowering 到 trace replay、live callable gateway 和 benchmark-derived workflows。

\subsection{Checker Core}
Checker core 位于 \texttt{src/intentcap/}，实现 context labels、effect events、leases、checker state 和 proof objects。当前 checker 支持 exact、prefix、suffix 和 one-of argument predicates，支持 lease matching、holder scoping、control/data provenance checks、context-label influence-mode checks、temporal prerequisites、per-lease invocation budgets、proof-carrying intent/approval derivation，以及 delegated capability attenuation。这个 core 是本文的主要 TCB：LLM 输出、Skill 描述、MCP annotations 和 tool results 只能产生 candidate effects 或 candidate leases，不能绕过 checker 直接获得 authority。

\subsection{Trace and Live Gateways}
\texttt{TraceGateway} 用同一个 checker replay benchmark-derived traces，并把每个事件记录为 execute/block decision。它用于把 AgentDojo、MCPTox、InjecAgent 和 tau2-bench reference artifacts 转换成统一的 protected-decision events~\cite{agentdojo,mcptox,injecagent,tau2bench}，从而比较不同 policy variants 的 false accept、authority breadth 和 oracle distance。

\texttt{LiveToolGateway} 包裹本地 Python callables：只有 checker 接受的 event 才会真正调用注册函数；被拒绝的事件在 side effect 发生前停止。这一区分很重要：trace replay 提供大规模 policy evidence，live gateway 验证同一 lease semantics 可以保护真实 callable boundary。

\subsection{Compiler Probes and Runtime Binding}
Prototype 还包含 local llama.cpp/Qwen frontend，用来模拟不可信 lease compiler 和 action proposer。模型可以根据 task text、tool schemas、已有 tool-result previews 或 checker feedback 生成 candidate leases/actions；系统只把通过 checker 的结果交给 gateway。

对于 tau2-bench-style stateful workflows，runtime binding 会从已经执行的 tool-result evidence 中铸造 exact one-shot leases，并重新检查参数值是否出现在当前可见 task/tool-result context 中~\cite{tau2bench}。High-impact write candidates 还必须满足 structured value proof：例如列表参数需要逐项绑定到结构化上下文，不能从原始文本窗口或 sibling objects 中借用 authority。

\subsection{Implemented Surface}
Prototype 已经实现 checker/gateway、live local callable boundary、benchmark-derived replay adapters、local Qwen/llama.cpp compiler probes、runtime binding、structured value proof、read/write activation、expert-oracle labeling/scoring/audit pipeline、E1 security/utility wrapper matrices，以及本地 env backend probes。Env backend 在隔离目录中真实执行 Python script、file read/write 和 context-use handlers，并在 side effect handler 之前调用 checker；被拒绝的 exec/read/write/net/ctx events 不会进入本地 side-effect handler。模型闭环版本使用 Qwen3.6/llama.cpp 作为不可信 action proposer：模型在 adversarial env-context prompt 下返回 call/abort，只有 call 事件会继续进入 checker 和本地 side-effect handler。它不是生产级 OS sandbox，但比 trace replay 更接近 env adapter 的系统边界。

这些实现足以支撑机制性 claim：在已建模 traces 上，\sys 能拒绝 unauthorized context-to-decision influence；run-time exact leases 可以缩小 object/tool authority surface；checker 可以阻止 LLM-proposed broad/runtime authority 直接进入执行路径。它还不能单独支撑更强的端到端 claim：full MCP broker、生产级 execution sandbox、fresh user-simulator loop、独立 human expert-oracle replication 和 benchmark-scale utility/recovery evaluation 仍是剩余评估要求。

\section{评估}
\label{sec:evaluation}

本节是 Evaluation Design：它定义 full-paper claim 需要通过的四个主实验，而不是把当前 pilot runs 写成最终结果。E1 回答系统是否有端到端安全和 utility 信号；E2 回答 leases 是否更接近 least privilege；E3 是 novelty 的旗舰实验，检验没有 four-plane no-promotion 和 check-and-consume lifecycle 的最强替代机制是否仍会 false accept；E4 回答 transaction API 是否能覆盖关键边界并支持安全 recovery。每个实验都按 hypothesis、baselines、metrics、success criterion 和 interpretation boundary 解读。已完成数据只在第~\ref{sec:evidence-status} 节作为 Current Evidence Status 汇总；具体运行编号、探索性 probes 和 residual analyses 只作为这些问题的证据来源，不作为论文主实验编号。

评估的共同原则是固定底层模型、任务顺序、工具描述和 prompt budget，尽量只改变授权层。当某个 baseline 只有 trace-level 或概念性复现时，结果必须标记为 partial comparison，不能写成完整数值胜利。所有 high-impact denials 使用 protected-decision oracle 判定：若某个 protected decision 的 control provenance 来自缺少相应 influence mode 的 context source，则该事件应被拒绝。

\subsection{Evaluation Methodology}

Full-paper evaluation 应集中报告 workload、baseline、oracle 和 runtime setup，而不是按历史运行编号组织。Workloads 覆盖 AgentDojo、MCPTox、InjecAgent、tau2-style stateful tasks、Skill/manual injection、本地 command/script side effects 和 delegation workflows。Baselines 覆盖 vanilla agent、static ACL、exact-tool ACL、MCP/server allowlist、Skill manifest-style policy、tool-call guard、OS/sandbox-only wrapper、IFC/taint labeler、action-provenance labeler 和 strongest split-state composite policy。Model/runtime setup 固定底层模型、prompt budget、tool descriptions、task order、local llama.cpp/Qwen settings 和 gateway implementation。Oracle sources 包括 benchmark security/utility labels、protected-decision labels、expert lease adjudication 和 provenance-proof completeness records。

\subsection{E1: End-to-End Security and Utility}

E1 检验 \sys 是否在真实 agent workflows 中降低攻击成功率，同时不把 benign task progress 降到不可用。Primary comparison 是同一模型和同一任务集下的 vanilla agent、static ACL、exact-tool ACL、MCP/server allowlist、tool-call guard、OS/sandbox-only wrapper 和 evaluated \sys configuration。Workloads 必须覆盖四类边界，而不是只覆盖 MCP/tool calls：Skill/manual injection 检验 instruction context，MCPTox-style MCP poisoning 检验 tool context，document/tool-result/script-output injection 检验 env context，tau2-bench-style stateful workflows 和 subagent delegation cases 检验 agent context 与 delegation state~\cite{agentdojo,injecagent,mcptox,tau2bench}。

Primary outcome 是 bounded-utility setting 下的 attack success rate。Secondary outcomes 包括 wrong-sink rate、approval/delegation violations、approval-scope breadth、benign completion、false denial/recovery rate 和 provenance-proof completeness。Proof completeness 报告 protected fields 中有多少带 precise structured proof、多少退化为 prompt-wide conservative proof、以及这种退化造成多少 false denial 或 recovery turn。Oracle 来自 benchmark security/utility checks 与 \sys protected-decision labels。成功条件是：\sys 相比 access-control baselines 明显降低危险动作和 broad approval requests，并且 utility loss 有界、可由 denial/recovery 日志解释；若 utility 大幅下降，论文只能主张 \sys 是安全 enforcement layer；若攻击仍通过，则必须定位到缺失的 provenance 或 enforcement boundary。

E1 的解释必须同时报告安全收益和 utility cost。若 \sys 降低攻击成功率但造成 benign completion 大幅下降，结论应降级为 safety enforcement layer，而不是完整端到端 agent 安全结果。若攻击仍通过，结果必须定位到缺失的 provenance source、decision class 或 enforcement boundary，而不是把失败归因于模型“没有遵守提示”。

\subsection{E2: Lease Quality and Authority Reduction}

E2 检验 \sys 生成的 run-time leases 是否比 static extension/server policies 更接近 least privilege。实验单位不是单个工具调用，而是一个 run 在完成任务时暴露的 authority surface。样本必须同时覆盖 agent、instruction、tool 和 env context：stateful utility tasks 提供 agent intent/workflow leases；Skill-style workflows 提供 instruction leases；MCP poisoning 提供 tool/context authority leases；local command/script 或 document-processing workflows 提供 env/sandbox leases。目标方法是 blinded expert leases：专家在不看 \sys 输出的情况下写 minimal leases，经 adjudication 后形成 reference policy。若 reference policy 只来自作者标注，结果只能写成 author-adjudicated oracle；只有完成独立复核后，论文才能主张 expert-oracle least-privilege result。

这里不能直接把现有 trace 标注当作 expert lease oracle。Benchmark 标注通常说明攻击是否成功、reference action 是否应该出现、或模型是否调用了某个工具；它们很少标注最小授权对象本身，例如 context source 可影响的 decision class、argument predicate 的最小范围、temporal/budget guards、delegation bounds 或 approval scope。因此现有标注适合 E1/E3 的安全与 utility 判断，但 E2 的 least-privilege claim 仍需要独立 lease 标注。

比较对象包括 domain/global ACL、exact-tool ACL、MCP server allowlist、Skill manifest-style policy、Progent/PCAS/AgentSpec-style policy 和 \sys。Expert labeling rubric 必须覆盖 minimal object scope、allowed influence modes、argument predicates、budget/expiry、delegation bounds 和 approval scope；adjudication 后再计算 risk-weighted authority score、context-influence breadth、delegation breadth 和 oracle distance。成功条件是独立复核后 \sys 仍在保持授权 data use 的同时降低 authority breadth 和 oracle distance；若 symbolic policy baselines 同样窄，则论文应把 least-privilege claim 收窄为 proof-carrying checker operationalization。

\subsection{E3: Context-Influence Mechanism Ablation}

E3 隔离 \sys 的核心机制是否真的是 context influence capability，而不是普通 tool permission 或 taint/provenance tracking。E3 不衡量 task utility；它只在同一 protected-event schema 上检验哪些机制缺失会接受 full checker 拒绝的 violation。实验比较 full-mechanism \sys checker、no intent certificate、no influence lattice、no control/data split、object-only checker、no-provenance checker，以及 AuthGraph/PACT/AIRGuard-style provenance-authority 和 IFC/taint-style labelers。E3 的第一步应是 workload characterization：在 AgentDojo、MCPTox、InjecAgent 和 tau2-style traces 中统计 protected decisions 的类型、每类 decision 需要的 context planes、以及 observed/prompted violation 是否属于 wrong sink、approval widening、cross-plane promotion、stale lease reuse、delegation mismatch 或 policy update。没有这一步，residual suite 只能说明机制可行，不能说明问题在 workload 中自然出现。

最强替代解释是一个 composite baseline：stateful ABAC + taint labels + capability counters + delegation table。E3 因此还必须包含 split-state baseline：它拥有这些表，但没有统一的 \(check\_and\_consume\) transition，也不在同一次 protected-decision transaction 中绑定 issuer-typed required fields、control-provenance proof、lease consumption 和 delegation state。若该 baseline 接受 stale-state reuse、cross-plane promotion、policy-update no-promotion 或 delegation mismatch，而 full checker 拒绝，就说明 novelty 来自 authorization unit，而不只是更多 predicates。若该 baseline 加入统一 lease API 后通过这些 cases，论文应承认它采用了本文的 abstraction，而不是把它作为弱对照。

新增两个直接检验四类 context 的对照：\textbf{合并型对照} 将 agent/instruction/tool/env 合并成 generic context，只检查 provenance 是否 trusted；\textbf{误分类对照} 允许 env output 或 tool metadata 在没有 endorsement 时升级为 instruction/agent context。更强的 novelty test 是 incremental lifecycle ablation：每个对照只加入以下机制的真子集：trusted mint、decision-specific influence、consumption/expiry、delegation partial order 和 no-promotion。实验检查这些对照是否仍接受 full checker 拒绝的 residual violation。Workloads 包括 document injection、MCP poisoning、malicious Skill/tool-result、本地 command/script side-effect cases、policy-update attempts 和 residual workflow suites。

关键指标是 protected decisions 中 unauthorized context control 的接受率，以及每个 denial 是否只能由 lease semantics、temporal guards、budget、delegation 或 proof constraints 解释。成功条件是 ablations 或 access-control baselines 接受 full-mechanism \sys checker 拒绝的 protected-decision violations，并且每类 residual case 都需要完整 lifecycle set 中缺失的某一项才能解释。若最强 baseline 在同一事件 schema 上加入等价的原子授权对象后解释所有 denials，论文应承认其为 semantic-equivalent baseline，并将“新的 security boundary”表述收窄为 proof-carrying lease compiler/checker organization。

为避免 baseline 只停留在名称层面，E3 将每个对照实现为明确的 checker variant。若无法复现某个完整系统，论文应把它写成 over-the-same-event-schema 的 semantic variant，而不是声称复现原系统。Object ACL 只维护 operation/object/argument predicates。Taint/IFC-style baseline 维护 label 和 flow constraints，但不维护 lease consumption 或 delegation partial order。Action-provenance baseline 检查 observed action 的 intent/provenance alignment，但不允许 consumed lease state 影响后续授权。Collapsed-context baseline 不区分 instruction/tool/env/agent class，因此会把“可信来源的工具描述”和“当前用户授权”混为一类。Misclassified-context baseline 允许 PDF text、tool result 或 script output 被当作 instruction 使用，因此用于测试 prompt-like env output 的 promotion attack。Split-state stateful policy baseline 加入 temporal/budget/delegation tables，但不提供统一 \(check\_and\_consume\) transaction API，因此用于测试 stale reuse、split-state widening 和 delegation mismatch。Lease-API policy baseline 若实现等价的 intent-minted、consumed、attenuated protected-decision transition，则不是弱对照，而是 semantic-equivalent baseline；这时结论应表述为它采用了本文主张的授权单位。Full-mechanism \sys checker 同时维护 trusted mint provenance、per-lease consumption/expiration、decision-specific influence modes、no context-class promotion 和 parent-child capability partial order。

当前稿只把 local workflow-policy sanity check 作为 mechanism evidence，而不是 E3 的最终 novelty proof；其结果统一放在第~\ref{sec:evidence-status} 节。Full-paper E3 需要 workload characterization 和 strongest composite baseline 闭环后，才能把该机制性信号升级为 benchmark-scale novelty evidence。

\subsection{E4: Transaction API and Recovery Practicality}

E4 检验 transaction API 在多边界 adapter 和 denial/recovery loop 中是否可用。它有两个 primary outcomes：第一，context placement、tool/MCP invocation、本地 env side effect 和 delegation handoff 是否都在 side effect 或 authority transfer 前调用同一个 checker，并保持 invalid accepted \(=0\)、dangerous execute \(=0\)；第二，denial feedback 是否能在不激活 broad authority 的条件下恢复有效任务进展。Secondary outcomes 是 LLM-assisted compiler/recovery 的细分行为：LLM 生成 candidate plans/leases，checker 拒绝 broad/runtime policies 或 missing-proof writes，runtime 返回 structured denial，planner 再生成更窄的替代动作。比较对象包括 LLM-only accept、checker strict、checker+feedback、proof-only/rank-only/filter-only variants 和 checker+CEGAR planner。

Secondary metrics 包括 invalid leases accepted、valid leases rejected、broad/runtime proposals rejected、broad approval avoided、provenance-proof completeness、tool errors 和 benchmark task reward。Proof completeness 需要区分 precise structured proof、conservative prompt-wide proof 和 missing proof，并把它们分别关联到 accepted actions、false denials 和 successful recoveries。成功条件不是简单增加 executed calls，而是在不激活 broad/runtime authority、零 dangerous execute、零 tool error 的前提下，提高 task-correct executions 或 reward，并减少不必要的 broad approval。若 recovery 仍停滞，论文应把 utility claim 收窄为 bounded safety/proof completeness，并将 planner/candidate generation 明确列为 prototype limitation。

\section{证据与主张边界}
\label{sec:evidence-status}

本节将 prototype evidence 与最终主张强度分开，限定哪些结论已经有机制性证据，哪些必须等待 E1--E4 的主实验闭环。\sys 的核心解释风险不是机制讲不清，而是把 trace replay、oracle replay 或局部 compiler probes 误写成完整 end-to-end agent 结果。

\subsection{Evidence Boundary}
已有 prototype evidence 足以支撑一个保守的 system claim：\sys 可以作为 agent decision authorization layer，把 LLM-generated plans/leases 放在 TCB 之外，并在 checker-enforced boundaries 内拒绝违反 intent、argument、provenance、temporal、budget 或 delegation constraints 的事件。Saved benchmark traces 和 live callable gateway 证明同一 checker semantics 可以作用于 replay events 与真实本地调用边界；local Qwen/llama.cpp probes 证明模型输出可以作为 untrusted compiler input，由 checker 拒绝 broad/runtime authority、missing-proof writes 和 env-context 诱导的 side-effect calls；runtime binding 与 structured value proof 证明短期 exact leases 可以从已执行的 tool-result evidence 中安全铸造，而不是从模型自由文本中直接产生。

\subsection{Claim Boundaries}
本文采用条件式 claim ladder，而不是把 pilot evidence 写成最终结论。当前 scaffold claim 是：protected-decision transition 是一个可实现、可形式化、可在已接入边界执行的 agent authorization unit。这个 claim 由 checker semantics、local gateway/env probes、residual workflow suites 和 saved/local-Qwen pilot evidence 支撑。

Full-paper security claim 还需要 E1 和 E3 同时闭环：benchmark-scale workflows 中 attack success、wrong-sink、approval widening、delegation mismatch 和 policy-update violations 显著下降，并且 workload characterization 显示这些 protected decisions 不是人工构造的孤例。Full-paper least-privilege claim 需要 E2 的 independent expert-oracle replication；author-adjudicated labels、policy-distance scoring 和机器审计只能作为 pilot evidence。Full-paper utility/recovery claim 需要 E4 显示 denial feedback 和 evidence-bound repair 在不激活 broad/runtime authority、零 dangerous execute 和零 tool error 的前提下提升 task-correct executions 或 reward。

这一级别划分也约束负结果的写法。若 E1 utility 不足，\sys 只能主张 safety enforcement layer，而不是完整端到端 agent 安全；若 E2 oracle distance 不优于 symbolic policy baselines，least-privilege synthesis claim 应收窄为 proof-carrying checker operationalization；若 E3 的 strongest composite baseline 采用统一 lease API 后同样通过，结论应是该 baseline 实现了本文识别出的 transaction boundary，而不是被 \sys “击败”；若 E4 recovery 停滞，recovery 只能作为 diagnostic/refinement mechanism。

\subsection{应用扩展隔离与底层 Enforcement}
传统扩展系统和 sandbox 关注扩展代码能访问哪些宿主资源。EIM/bpftime 将应用扩展表示为 host resources 和 capabilities，并强调安全、高效的 userspace extension execution~\cite{eim-bpftime}。ActPlane 将 agent policy lowering 到 OS-level enforcement，以覆盖 tool interception 看不到的 file、process 和 network side effects~\cite{actplane}。这些机制适合作为 \sys 的 enforcement backend 或系统对照，但它们不决定 policy 的来源。\sys 的问题在更上层：某个 tool choice、sink selection、approval scope 或 delegation decision 是否由当前用户 intent 授权，并且是否被不可信 context 非法影响。

\subsection{Prompt Injection, Provenance, and Information Flow}
CaMeL、Prompt Flow Integrity、FIDES、RTBAS 和 NeuroTaint 等工作从 control/data flow、IFC、dependency screening 或 taint tracking 的角度约束 untrusted content~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}。AuthGraph、PACT 和 AIRGuard 进一步把 user intent、argument provenance 和 action-time authority control 引入 agent security~\cite{authgraph,pact,airguard}。这些工作是 \sys 最接近的思想背景；本文不把“intent/provenance authorization”本身作为新颖性主张。

本文把差异放在授权对象的粒度和生命周期上：这些系统主要检查 observed action、argument 或 side effect 是否与 intent/provenance 对齐；\sys 额外把未来决策权限表示为短期 lease，显式记录 authority 如何由 intent mint、如何在 protected decision 前被 check/consume、如何按 temporal/budget guard 缩小、如何在 delegation 中 attenuation，以及如何 lowering 到 Skills、MCP tools、本地执行和 subagents。一个足够扩展的 ABAC/IFC system 可以编码相同语义，但前提是它采用同一个授权单位：由 intent 铸造、被 check-and-consume、可过期、可衰减的 protected-decision lease。如果某个 baseline 实现了同样的 transaction object，实验结果应解释为 semantic equivalence，而不是 \sys 对它的击败；本文的 claim 将变成识别并评估这个 transaction boundary。关键不是单独发明 provenance label、taint tracking 或 attenuable capability，而是把 trusted mint provenance、per-lease consumption/expiration、protected-decision influence modes、no context-class promotion 和 parent-child capability partial order 绑定为同一个 authority transition，并把每个 protected decision 的 required fields 绑定到 issuer-typed agent、instruction、tool 或 env plane。Provenance label 是 checker 的输入，lease 是可被铸造、消耗和委托的授权对象，也是每次 protected decision 更新 authority state 的原子单位。

\subsection{Agent Policy Languages and Access Control}
Progent、PCAS、AgentSpec、AgentGuard 和 AgentBound 等工作提供 programmable privilege control、policy compilation、runtime DSL、ABAC 或 MCP access-control enforcement~\cite{progent,pcas,agentspec,agentguard,agentbound}。这些系统回答“某个工具、参数、server 或 action 是否被 policy 允许”。\sys 与它们的关系是互补和竞争并存：它同样需要确定性 checker，但 policy object 不是静态工具规则或 server policy，而是从当前 run 的 intent 和 context provenance 推导出的短期 leases。E2 和 E3 因此必须比较 \sys 与这些 policy families 在 authority breadth、oracle distance 和 protected-decision false accepts 上的差异。

\subsection{Skill and MCP Permission Frameworks}
SkillGuard 和 SkillScope 直接面向 Agent Skills 的权限、行为规范和 least-privilege enforcement~\cite{skillguard,skillscope}；MCP 规范也强调 tool metadata 和 behavior annotations 不能默认被当作可信事实~\cite{mcp-spec}。这些工作说明 Skill/MCP ecosystem 已经需要权限边界；本文不把 \sys 定位为第一个 agent permission system。\sys 的范围是 run-centric 和 cross-extension 的：一个 lease 可以同时约束 Skill instruction 对 workflow 的影响、MCP result 对后续 sink 的影响、本地脚本的 file/network authority，以及 subagent 获得的 attenuated capability set。与 package-level manifest 或 server-level grant 不同，\sys 的 authority root 是当前 intent certificate；同一个 Skill 或 MCP server 在不同 run 中会得到不同、可消耗、不可自扩张的 leases。这个定位把 \sys 放在 Skill/MCP permissions 之上，作为一个 intent-derived lease compiler/checker，而不是替代 Skill/MCP 的包级权限系统。

\section{结论}

LLM agent extensions 改变了权限系统需要保护的对象。文档、工具结果和扩展说明不只携带数据，也可能影响 tool selection、sink selection、approval scope、capability request 和 delegation target。\sys 将这种影响权表示为 intent-derived、provenance-gated、可消耗、可过期、可衰减的 leases，并用 deterministic checker 把 LLM proposals 保持在 TCB 之外。E1--E4 决定这一机制最终能支撑多强的端到端安全、utility 和 least-privilege claim；在已建模边界内，\sys 的核心结论是把“context 可以影响什么未来决策”变成可检查、可消耗、可审计的授权状态，并要求每个 protected field 都由自己的 issuer plane 证明。
