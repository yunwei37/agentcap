\begin{abstract}
LLM agents 正在从固定工具调用器演变为可扩展执行环境，可以加载 Skills、连接 MCP servers、运行本地脚本，并将任务委托给 subagents。
这些 extensions 不只执行操作，也让用户意图、Skill/manual instructions、tool metadata、文档内容、工具结果和环境状态共同影响未来的 tool selection、sink selection、approval scope 和 delegation target。
现有权限机制主要约束工具、文件和网络端点，难以表达哪个 context 被允许控制一次 authority-bearing decision。
结果是 agent 可能发出参数看似合法的调用，却已经通过不可信 context 扩大 approval、复用一次性权限或错误委托 authority。
\sys 的核心观点是：agent authorization 的单位不应是 tool permission、context label 或已发生的 action，而应是会选择工具、选择 sink、扩大 approval 或委托权限的受保护转移（protected-decision transition）。
普通 provenance policy 追踪值来源；\sys 检查该来源是否拥有本次 authority-state transition 中对应字段的签发权。
\sys 将这种转移表示为短期 lease。Lease 绑定当前用户意图、来自 agent/instruction/tool/env 四类 issuer-typed authority fields 的结构化证明，以及 provenance、预算、过期和委托约束，并由 deterministic checker 在副作用、prompt placement 或权限转移前原子验证。当前原型执行 tool/live-callable 和 local-env 边界，并以本地 live probes 验证 context placement、delegation 和 Skill instruction placement。
在 same-event protected-event replay 上，\sys 对 3,746 个危险 protected events 产生 0 个误接受。
在 tau2-style reference-action proxy 中，\sys 覆盖 3,813/3,813 个 reference actions，并通过 2,554/2,556 个适用的 tool-oracle checks。
在 issuer-substitution residuals 中，\sys 拒绝 48/48 个 MCPTox-derived policy/approval promotions 和 3/3 个 Skill instruction-source substitutions，而去掉 issuer ownership 或 provenance checking 的 variants 会接受这些 unsafe transitions。
在 isolated local env gateway 中，\sys 执行 4 个授权副作用，阻止 6 个违规事件，并在副作用发生前阻止 4 个 Qwen3.6 提出的 unsafe calls。
\end{abstract}

\section{引言}

LLM agents 的安全边界正在发生变化。传统应用扩展通常通过 API、插件接口或系统调用与宿主交互，因此权限系统可以围绕文件、网络、进程和工具调用来定义。现代 agent extension 的行为则由用户请求、系统策略、扩展说明、外部文档、工具结果和中间总结共同塑造，尤其是在 Skills、MCP servers 和 agent tool ecosystems 中~\cite{skillguard,skillscope,mcp-spec}。

然而，agent extension 打破了 data 和 authority 的传统边界。一个失败轨迹可以说明这个缺口。用户只要求从两个 PDF 提取表格，并在 \texttt{org/repo-x} 创建一个 summary issue，系统因此只应铸造一个 one-shot issue lease。随后，其中一个 PDF 的隐藏文本诱导 agent 请求完整 GitHub 授权范围，把可写 issue authority 委托给 summarizer subagent，在 lease 被消耗后复用同一权限，甚至建议更新 policy 以信任该 workflow。最终出现的 \texttt{create\_issue(repo=org/repo-x)} 参数仍然等于用户指定的 repository，所以普通 tool/argument guard 可能放行。此时，损害不是 repo 字符串错误，而是 authority lifecycle 已经被扩大、转移、复用或改写。后续步骤可能读取或写入用户未授权的 repository，把提取数据带到未授权 sink，或把 future runs 的 policy 改坏。参数合法不代表权限状态合法。

根因不是不可信文本进入 prompt 本身，而是两个结构性错配。第一是 \textbf{issuer collapse}: agent runtime 把用户意图、Skill/manual instructions、tool metadata、工具结果和运行时输出放进同一个 planning channel，使一类 context 可以替另一类 context 补齐 authority field。Wrong sink 和 approval widening 属于这种错配。第二是 \textbf{authority-state split}: 下游 guard 多数在 action materialized 后才检查 tool、arguments 或 resource，而 lease consumption、expiry、delegation 和 mint provenance 分散在不同状态中。Stale reuse 和 over-delegation 属于这种错配。于是，一个对输出内容而言安全的输入，可能对控制决策而言并不安全。PDF text 可以安全地填充 spreadsheet cells，却不应选择 sink、扩大 approval scope 或创建 delegation。本文称这一问题为 context privilege，因为 context 获得了影响 authority-bearing decisions 的权力，而不仅是携带可被引用或总结的信息。

这个根因要求系统区分四类 proof boundary。以本地 PDF extraction 为例，agent context 证明用户选中了哪些 PDF 和哪个 GitHub sink，instruction/Skill/manual context 证明 extraction workflow，tool/MCP/cmd context 证明 binary、schema、credential scope 和 sandbox contract，env/runtime context 证明本次运行实际看到的路径、stdout、tool result 和文件状态。后三类不能合并成一个 generic context。把 env 合并进 tool 或 instruction，会让 tool result、script output、PDF text 和 subagent summary 获得本不具备的接口权威或过程权威。把 tool metadata 合并进 agent，又会让 server description 变成用户授权。第~\ref{sec:design} 节给出完整 taxonomy 和形式化规则。

现有防线分别覆盖了这个问题的一部分。Tool guards、argument validators 和 action-time authority controls 可以拒绝错误对象或非法参数~\cite{authgraph,pact,airguard}。Static allowlists 和 server-level policies 可以限制可见工具。Human approval gates 可以重新引入用户确认。OS sandbox 或 monitor 可以约束文件、网络和进程 side effects~\cite{actplane}。Provenance、taint 和 IFC-style defenses 可以表达部分 flow policy~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}。这些机制是必要的，但本文比较的是默认 authorization object，而不是声称这些系统无法额外编码类似规则。若一个 policy DSL 要等价支持 \sys，它必须显式加入 issuer-owned fields、结构化 proof object，以及原子更新的 lease consumption、expiry 和 delegation state。\sys 的贡献是把这组约束作为 agent protected-decision 的统一授权对象。

最接近的 provenance/action-authority systems 已经能问“这个参数或动作是否来自可信来源”。\sys 要回答的是更窄但更状态化的问题：当参数合法、provenance 可信时，提交证明的 issuer class 是否拥有该 authority field，相关 lease 是否仍然 active，预算是否已消耗，child delegation 是否仍然窄于 parent。普通 provenance policy 问“这个值从哪里来”。\sys 问“这个来源是否拥有本次 lease mint、consume 或 delegate transition 中对应 authority field 的签发权”。因此，安全边界不是 action、flow 或 capability 本身，而是带 issuer ownership 和 lifecycle update 的原子 protected-decision transition。

\sys 从这一差异出发。本文的贡献在于把 future decision authority 作为可消耗、可过期、可委托的授权状态，而不是只检查已发生 action 的 provenance。Agent least privilege 的授权单位不应是 tool、resource、context source 或已发生的 action，而应是一次由 intent 铸造、会改变未来 authority state 的受保护转移。换言之，问题从“工具能不能调用”转为“哪个 context 能不能控制一次权限状态转移”。

实现这一授权单位需要分离不同 authority fields 的证明来源。Skill/manual text 可以指导 workflow，但不能替代用户意图。MCP/tool metadata 可以描述接口和 credential scope，但不能授权 sink。Script output 和 tool result 可以提供 runtime evidence，但不能成为 instruction 或 approval reason。Checker 因此必须在同一次 transition 中验证 field ownership、control provenance、lease consumption、expiry 和 delegation。缺少任一检查都会允许 cross-class promotion、stale reuse、over-delegation，或把 untrusted data 当成 authority。

为实现这个授权单位，\sys 使用 intent-carrying capability lease 绑定 operation、object、arguments、intent derivation、control/data provenance、temporal state、budget 和 delegation。Lease 的证明来自四类输入：agent context、instruction context、tool context 和 env context。Lease table、budget counters 和 consumption state 属于 checker state \(\sigma\)，而不是 env context。

\sys 的实现包含一个 LLM-assisted compiler 和一个 deterministic checker。LLM 提出 plans 和 candidate leases，checker 根据 intent certificates、context labels、effect IR 和 global policy 验证这些 proposal。Runtime adapters 将通过的 leases 落到 context construction、tool/MCP policy events、本地环境执行和 delegation 边界。当前证据支持的是 instrumented protected-decision boundaries 上的 safety 和 enforceability，不声称完整 production agent runtime、production ActPlane integration 或 benchmark-scale end-to-end utility。在 same-event protected-event replay 上，\sys 对 3,746 个危险 protected events 产生 0 个误接受。在 tau2-style reference-action proxy 中，\sys 覆盖 3,813/3,813 个 reference actions，并通过 2,554/2,556 个适用的 tool-oracle checks。在 controlled issuer/lifecycle ablations 中，去掉 issuer ownership 或 provenance checking 的 variants 会接受 MCPTox-derived policy/approval promotions、instruction-source substitutions 或 delegation probes，而 \sys 拒绝这些 unsafe transitions。在 isolated local env gateway 中，\sys 在副作用发生前阻止 6 个 scripted violations 和 4 个 Qwen3.6 unsafe calls。

因此，本文贡献限定在已 instrumented protected-decision boundaries 上的模型、系统接口和有界证据：

\begin{enumerate}[leftmargin=*,label=\textbf{C\arabic*.}]
  \item \textbf{Model.} 一个 issuer-owned atomic protected-decision lease model，定义 field owner、no-substitution typing rule 和原子 lifecycle transition（第~\ref{sec:formal} 节）。
  \item \textbf{System.} 一个 checker-centered protected-transition API 和 adapter contract，要求 tool/MCP policy events、local-env side effects、context placement 和 delegation handoff 在 side effect、prompt placement 或 handoff 前提交同一种 issuer-owned proof（第~\ref{sec:design}--\ref{sec:implementation} 节）。
  \item \textbf{Evidence.} 一个有界评估，展示 same-event protected-decision safety、去掉 issuer ownership 或 atomic lifecycle 后的 controlled false accepts，以及本地多边界 pre-side-effect/placement/handoff enforcement；lease auditability 作为辅助证据报告（第~\ref{sec:evaluation}--\ref{sec:evidence-status} 节）。
\end{enumerate}

\section{背景与动机}

\subsection{Agent Extensions}

现代 agent runtime 通常不只暴露一个固定工具集合。Skills 将任务说明、参考资料和可选脚本打包为可复用工作流。MCP servers 将工具 schema、资源和 prompts 暴露给 agent。本地脚本和 shell 执行提供文件转换和项目操作能力，subagents 则把上下文和部分权限交给另一个模型或执行单元~\cite{skillguard,skillscope,mcp-spec}。因而，agent extension 同时包含自然语言指令、结构化 tool metadata、外部数据和可执行代码。

\subsection{Authority and Provenance}

传统权限系统主要围绕 subject、object 和 operation 建模，例如进程是否能读取文件、调用 API 或连接网络。Agent 场景还需要记录 provenance，因为同一个 tool call 的参数可能来自用户选择、扩展说明、工具结果或不可信文档。已有 provenance、taint 和 IFC-style 系统说明不可信输入不应自由流向高影响输出~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}，但它们通常把核心对象放在 observed action 或 data flow 上，而不是把未来 protected decision 本身作为可消耗、可衰减的授权对象。本文中的 protected decision 指会选择、扩大、消耗或转移 authority 的字段级决策，例如 sink selection、approval scope、capability request 或 delegation target。

\subsection{Motivating Workflow}
\label{sec:motivation}

本文使用一个文档处理工作流贯穿设计和评估。用户要求 agent 从两个选中的 PDF 中提取表格，保存 XLSX，并在指定 GitHub repository 中创建一个 issue。PDF text 可以影响表格单元格和 issue body，但不应影响 repository selection、approval scope、capability request 或 delegation target。这个例子把普通 data use 和 authority-bearing decision 分开，使问题不依赖某一种具体工具或 sandbox。

如果其中一个 PDF 包含隐藏指令要求 agent 请求 full GitHub access、加载另一个 Skill，或把 issue authority 交给 subagent，普通 argument check 只能看到最终调用是否仍然使用合法 repo。它无法表达 PDF text 不能成为 approval scope、fresh capability request 或 delegated capability set 的控制来源。\sys 的 running example 因此不是一个错误参数问题，而是一个 context-to-decision influence 问题。

即使没有恶意输入，同一个 Skill、MCP server 或 command 在不同 run 中也需要不同 authority。PDF Skill 用于本地 XLSX 转换时不需要网络或 GitHub issue lease；同一 Skill 用于生成 issue summary 时需要一次性 GitHub sink authority，但仍不应选择 repository。若权限只能在安装期、server 级别或静态 allowlist 中给出，系统通常只能在 union permission、频繁 broad approval 和 false denial 之间折中。这是一个运维和可用性问题，而不只是 prompt injection 问题。

\subsection{Protected-Decision Transitions}

受保护转移（protected-decision transition）指一次会 create、select、widen、spend 或 transfer authority 的可观察状态转移。Tool selection、sink selection、approval scope、capability request、policy update 和 delegation target 都属于这类转移。实现中把这些转移编码为 protected events。一个 guard 如果只检查最终 tool name 或 repo argument，会漏掉 untrusted context 对 approval scope、fresh authority request、policy patch 或 delegated capability set 的控制。因此，本文要保护的对象不是“某个字符串能否进入 prompt”，而是“某个 context source 是否被授权影响某类未来决策”。

三个术语在后文保持固定含义。Decision class 是受保护字段的类型，例如 sink selection、approval scope 或 delegation target。Protected event 是 instrumentation 在 runtime 看到的具体事件，例如一次 MCP 调用、approval request 或 subagent spawn。受保护转移是 checker 对 protected event 执行的原子状态更新，包括 proof check、lease consumption、temporal update 和 delegation-state update。这些 decision classes 定义了后续 threat model、lease language 和 evaluation oracle 的共同边界。

\section{威胁模型与非目标}

\sys 保护的是已经通过 adapter 暴露给 checker 的受保护决策。Agent 可以读取外部文档、加载 Skills、连接 MCP servers、运行本地脚本、调用工具，并将部分任务委托给 subagents。攻击者可以控制部分输入 context，例如 PDF、网页、邮件、issue comment、MCP tool result、unsigned or malicious Skill text、script output 或 subagent summary。Signed/trusted Skill instruction 只有在被 issuer endorsement 后才能作为 instruction context 指导声明范围内的 workflow。Installed-but-overbroad Skill package 也不能因此获得 sink、approval 或 delegation authority。主要攻击目标是让不可信 context 控制某个 authority lifecycle event，例如选择错误 sink、请求更宽 approval、修改 capability request 或扩大 delegation。读取无关私有文件、创建错误 repository 的 issue、修改策略、跳过检查或委托给恶意组件，是这些 unauthorized influence 在具体边界上的表现形式。

威胁模型把 LLM、外部文档、工具结果、MCP annotations、unsigned Skill instructions 和任意脚本静态分析结果放在 TCB 之外。\sys 依赖的 TCB 包括 intent certificate issuer、deterministic checker、provenance tracker、audit log 和已经接入的 enforcement adapters。这个 TCB statement 是 per-boundary 的：没有接入 adapter 的 side effect 不在安全性质覆盖范围内。OS sandbox 或 ActPlane-style monitor 可以作为执行后端。\sys 关注的是上游 policy synthesis 和 context-authority checking。

\sys 的安全性质覆盖已建模的 decision classes 和已接入的 enforcement boundaries。Prompt injection 的语言层诱导、自动 compiler 的完整正确性、生产级 sandbox 替换，以及未 instrumented 的 agent 决策，属于这组性质之外的系统边界。

\section{系统设计}
\label{sec:design}

\subsection{System Overview}

\sys 的设计目标是把“模型想做什么”和“系统允许什么”分开。LLM 可以读取任务、扩展说明和上下文，并提出 plan、effects 和 candidate leases，但它不能铸造最终权限。最终 authority 只来自 trusted intent certificate，并由 deterministic checker 根据 context provenance 和 global policy 验证。

图~\ref{fig:architecture} 展示了五个组件的边界。Trusted issuer 将用户请求、显式对象选择、sink 选择、管理员策略和 fresh approval canonicalize 成 intent certificate。Context labeler 为用户输入、文档、Skill instructions、MCP metadata、tool results、script outputs 和 subagent summaries 标注 origin、integrity、confidentiality 和 influence modes。

LLM-assisted compiler 从扩展元数据、tool schemas、local scripts 和当前 plan 中生成 security-relevant effect IR，并提出短期 leases。Compiler 不在 TCB 内，不能直接修改 authority state。Deterministic checker 验证每个 lease 和事件是否满足 intent derivation、argument constraints、control/data provenance、flow、temporal、budget 和 delegation rules。Adapter contract 要求 context construction、tool/MCP policy events、本地执行和 subagent delegation 在对应 protected transition 前提交证明；当前原型完整实现 tool/live-callable 与 local-env adapters，并以本地 live probes 验证 context placement、Skill instruction placement 和 delegation handoff。Benchmark-derived traces 用于大规模语义 replay。

在 PDF workflow 中，issuer 首先把用户选择的两个 PDF 和 \texttt{org/repo-x} sink 写入 intent certificate。Labeler 将 PDF bytes/text 标为 env context，将 PDF Skill 说明标为 instruction context，将 GitHub MCP schema 标为 tool context。Compiler 可以提出 extraction effects 和 \texttt{create\_issue} lease，但 checker 只会 mint/consume repo 由 agent context 授权、body 可由 PDF data 影响、且 budget 尚未耗尽的 lease。Tool/MCP、local env 和 delegation adapters 在副作用或 handoff 前调用同一个 checker 决策。

\subsection{Design Goals}

Running example 中的失败轨迹对应四个设计需求。Approval widening 需要 intent root 限定用户真正授权的对象和 sink。PDF text 控制 approval 或 delegation 需要 context no-promotion 和 influence separation。一次性权限复用需要 stateful lease lifecycle。跨 tool、MCP、本地执行和 delegation 的副作用需要 multi-boundary transition interface。

Fail-closed denial feedback 和 adapter coverage 是实现这些目标的机制要求，而不是独立顶层目标。因此，设计目标包括四个可测试目标：

\begin{enumerate}[leftmargin=*,label=\textbf{G\arabic*.}]
  \item \textbf{Intent-bounded authority.} High-impact authority 必须来自当前用户意图、显式对象/目标选择、管理员策略或 fresh approval。
  \item \textbf{Context no-promotion and influence separation.} 系统必须区分 data dependency 和 control dependency，允许不可信 context 作为数据使用，同时阻止其补齐不属于自己输入类的 protected fields。
  \item \textbf{Stateful lease lifecycle.} 权限必须是短期、可消耗、可过期、可衰减、不可由不可信 context 自扩张的 run-time leases，而不是安装期或 server 级别 ambient permission。
  \item \textbf{Multi-boundary transition interface.} 同一 checker transition 必须能在已实现或本地 probe 的边界上，在 side effect、prompt placement 或 authority handoff 前验证 issuer-owned proof；当前原型覆盖 tool/live-callable、local-env、context placement、delegation 和 Skill instruction placement，并只把 ActPlane 作为 env lowering target。
\end{enumerate}

这个结构把 \sys 和普通 tool allowlist 区分开来。Allowlist 只决定哪个对象可见。\sys 还决定哪些 context source 可以影响哪个 protected decision。即使某个工具和参数在对象层面都匹配，checker 仍会拒绝 control provenance 来自无权影响该 decision class 的 context。

\sys 的系统边界因此不是一个单点 wrapper，而是一个 checker 仲裁的状态转移接口。设计上，tool/MCP、本地环境、instruction placement 和 delegation 边界都在 side effect、prompt authority placement 或 authority transfer 前提交同一类 field proofs。当前实现执行 tool/live-callable 与 local-env 边界，并用 live context-placement、Skill instruction placement 和 delegation probes 验证同一 checker session 可以放在 prompt section 和 handoff 之前。ActPlane-style monitor 可以强化本地环境边界的 file/process/network enforcement，但它执行的是 checker 已经决定的 lease contract，而不是替代 protected-decision authorization model。

\subsection{Authority Inputs}

\sys 的 capability synthesis 不是从单一 prompt context 中抽取权限，而是从四类授权输入类（authority-input classes）中求解 lease。本文将 agent、instruction、tool 和 env 称为四类输入。若只从语义上看，agent intent、instruction/Skill/manual 和 tool/MCP interface 是三类上层 authority source；但系统执行时还必须有第四类 env/runtime source，用来证明路径、进程、输出、资源状态和工具结果等运行时事实。缺少 env 类，论文会退化成 prompt/tool policy；把 env 合并进 tool 或 instruction，又会让运行时文本继承接口或过程可信度。表~\ref{tab:authority-inputs} 因此把四类定义为系统接口，而不是单纯分类。四类输入不是 attribute namespaces，而是四个互不替代的签发接口：每个接口有独立 issuer、独立失效模式和不可由其他接口补齐的 authority fields。两个来源只有在 issuer、可伪造面、可证明的 authority field 和观测边界都相同时才可以合并。

Agent context 不是模型隐藏状态，而是 trusted issuer 维护的 task intent、approval 和 workflow state。Instruction context 是可被当作过程指导的 policy/Skill/manual 文本。Tool context 是接口、凭证和 sandbox contract。Env context 是运行时观测到的对象、值、结果和资源事实。这四类输入的边界是 agent-specific 的。同一段文本如果来自被 endorsement 的 \texttt{SKILL.md} 或手册，可能在 workflow scope 内成为 instruction context；如果来自普通 PDF、网页、tool result、script output 或 subagent summary，即使内容像指令，也仍是低完整性的 env context，不能自动升级为 instruction。Command-line tools 同时涉及 tool context 和 env context：binary、argv schema 和 sandbox profile 属于 tool context；实际 cwd、输入文件、stdout/stderr 和文件系统变化属于 env context。

这四类不能合并成三类。三类说法通常把 agent intent、instruction procedure 和 tool interface 作为上层语义来源，这对解释工作流足够，但对执行授权不够。Checker 需要第四类 env/runtime issuer，因为很多 protected transitions 依赖的事实不是 intent、procedure 或 interface，而是本次运行中实际观察到的路径、stdout、tool result、文件状态、资源状态和 side-effect evidence。若没有这类 issuer，系统只能二选一：要么相信模型把 runtime facts 正确复述进 prompt，要么让 tool/instruction metadata 替 runtime observer 证明事实。两者都会破坏 proof boundary。

因此，本文把四类定义为 proof-carrying system interfaces，而不是自然语言来源分类。Agent 输入的 issuer 是 trusted UI/approval state，能证明“为什么做”和“哪个对象或 sink 被用户授权”。Instruction 输入的 issuer 是 system/developer policy 或被 endorsement 的 Skill/manual source，能证明“按什么过程做”。Tool 输入的 issuer 是 trusted registry、MCP broker 或 local command registry，能证明“通过哪个接口、凭证和 sandbox contract 做”。Env 输入的 issuer 是 runtime observer 或 monitor，能证明“哪些对象、值和结果在本次执行中存在”。Env issuer 不能授权目标或流程，但它是本地执行、ActPlane-style lowering 和 tool-result-bound leases 的必要证明来源。

错误合并会产生不同的 false authority path。把 instruction 并入 agent 会把 Skill/manual 中的过程建议提升为用户意图或 approval。把 tool 并入 env 会丢失 schema、credential scope 和 declared side effect 这些接口约束，使工具结果和工具定义共享同一 authority class。把 env 并入 tool 则会让 runtime observations、script outputs 和文档内容获得本不具备的接口可信度。

这也不是普通 ABAC attribute grouping。ABAC 问的是一组 attributes 是否满足 policy rule。\sys 问的是四类独立签发的输入是否共同授权一次 authority-state transition，并且禁止一类输入替另一类输入补齐缺失字段。若 sink selection 缺少 agent authorization，tool metadata 不能补上。若 local exec 缺少 tool binary/schema，instruction 不能补上。若 approval request 缺少 fresh issuer token，env evidence 不能补上。

每类输入都有独立 issuer 和不可由其他输入伪造的字段。Agent 输入的不可伪造字段是 trusted issuer canonicalize 的 goal、authorized objects/sinks 和 approval tokens。Instruction 输入的不可伪造字段是 policy/Skill/manual 的 workflow scope。Tool 输入的不可伪造字段是 registry 或 MCP broker 认可的 schema、credential scope、binary descriptor 和 sandbox contract。Env 输入的不可伪造字段是 runtime observer 记录的路径存在性、concrete values、tool results 和 script outputs。E3 的 collapse baselines 必须分别测试这些字段被其他输入补齐时的 false accept。

实际 capability 必须由相关 protected fields 与全局策略共同决定。未被某个 decision class 要求的输入类可以为空，但被要求的 field 必须从自己的 owner class 取证：

\[
K = Solve_\Gamma(C_{agent}, C_{inst}, C_{tool}, C_{env}) .
\]

这里的 \(Solve\) 是合取约束而不是权限并集。Agent context 给出目标、对象和 sink 的上界；instruction context 只能在该上界内约束过程；tool context 只能暴露可调用接口、凭证范围和 sandbox contract；env context 只能把本次运行实际观测到的值绑定到这些 lease predicates。任一 required field 缺失都会使 lease 不可铸造。相反，若把四类 context 做成 union authority，最强或最会写自然语言的一类输入就可能替其他类补齐字段。

纯 approval event、纯 context-use event 和本地 evidence event 可能只需要其中几类字段。关键规则是：对某个 protected decision 所需的字段，checker 必须从该字段指定的输入类读取，不能由其他输入类代填。

Agent context 可以授权“处理这两个用户选中的 PDF 并在 org/repo-x 创建一个 issue”，但不能指定 PDF Skill 的脚本路径。Instruction context 可以指导 extraction workflow，但不能选择 GitHub sink 或扩大 approval scope。Tool context 可以说明 \texttt{github.create\_issue} 的参数结构和 credential scope，但不能把任意 repo 变成授权 sink。Env context 可以证明某个路径、参数值、PDF text 或 observed result 在运行时存在，但不能把存在性变成用户选择或 retroactively mint broader authority。Local command 不是第五类 context：例如 \texttt{pdftotext} 的 binary path、argv schema 和 sandbox profile 属于 tool context；cwd、input file existence、stdout/stderr 和 created files 属于 env context。

用户原始文本不是第五类 authority input。Trusted issuer 会将其拆分为 agent-owned intent fields 和 instruction-owned workflow preferences。被 canonicalize 成 goal、object、sink 或 approval 的部分进入 agent context。描述 workflow preference 或 formatting preference 的部分进入 instruction context。即使 instruction preference 来自用户，它也不能单独授权 sink、approval 或 delegation。Lease 的作用是把这些 required fields 固定为一个可检查、可消耗、可衰减的 transition object。

\subsection{Intent Certificates}

Intent certificate 是结构化安全对象，而不是 prompt 中的一段自然语言。它记录用户目标、选择的对象、允许的 sinks、禁止的 sinks、approval tokens 和 expiry。对于 PDF 处理任务，certificate 会命名选中的 PDF 路径、输出目录、唯一授权的 GitHub repository、禁止任意网络访问的约束，以及创建 issue 所需的 fresh approval。LLM 可以提出 certificate fields，但 trusted parser、UI、administrator policy 或 approval step 必须 canonicalize 这些字段。Certificate 是后续 capability derivation 的根。

Certificate 的粒度决定后续 leases 的最大 authority。若用户只选择了两个 PDF，后续 file-read leases 只能覆盖这些对象或它们的安全派生物。若用户只确认了一个 GitHub repository，repo selection 就不能由 PDF text、Skill instruction 或 MCP result 改写。模糊字段不会被解释为宽权限。系统要么将其保守地标记为 unknown，要么要求 fresh approval。

\subsection{Context Authority}

每个 context cell 都带 provenance label，记录 origin、integrity、confidentiality、purpose、time 和 endorsements。\sys 的关键增加是为 context source 指定 influence modes。Influence authority 是 decision-specific 的：一个 source 可以被允许影响 output content，却不能影响 recipient、repository、network destination、approval scope 或 delegated capability set。Signed Skill instructions 可以在 declared workflow scope 内 instruct extraction procedure，但不能为自己请求额外工具。MCP tool results 和 annotations 必须按照 server trust 处理。MCP specification 已经提醒 tool behavior annotations 不能默认可信~\cite{mcp-spec}，\sys 将这一原则变成 deterministic provenance rule。

PDF text 可以流向 issue body 和 spreadsheet cells。普通 IFC 可以表达这条数据流是否允许。\sys 还要求 repository field 的 issuer 是 agent 输入类，approval scope 的 control proof 不含 PDF provenance，并且对应 one-shot lease 尚未被消耗。只有当 influence authority 与 intent derivation、budget、expiry 和 delegation bounds 绑定在同一 lease 上时，checker 才能防止 context 在一个字段上被允许、又在另一个生命周期状态中被复用为授权依据。这种 decision-specific authority 允许系统保留有用的数据流，同时切断同一 context 对 authority-bearing fields 的控制流。

实现会把抽象 influence mode 映射为 adapter 提交的 decision-mode 字段。表述上，influence mode 指抽象能力，decision class 指被影响字段，事件字段记录本次使用所需的最低影响权限。Quote、summarize 和 parameterize 对应 output body、spreadsheet cells 或 issue body 的数据使用。Plan 和 instruct 对应 tool selection 与 workflow-step choice。Authorize 对应 sink selection、approval scope、capability request 和 policy update。Delegate 对应 delegation target 和 child capability set。这个映射使论文中的抽象权限集合与实验 trace 一致，同时避免把所有 modes 都解释成同一种 taint level。

\subsection{Effect IR and Leases}

Agent plan 被编译成安全相关 effect graph。IR 不追求表达完整程序语义，而是捕捉会影响 authority 的事件：context use、tool call、MCP call、file read/write、subprocess execution、network connection、approval request 和 subagent delegation。每个 effect 记录 data provenance 和 control provenance。这个区分是 \sys 的关键：不可信 PDF bytes 可以是 output data 的 data dependency，但不应是 sink selection 的 control dependency。

Control provenance 不是声称系统能读出 LLM 内部注意力，而是一个 protected-field proof policy。对于 sink、approval scope、capability request、delegation set、policy patch、local executable、network destination 和 high-impact arguments，runtime adapter 必须提交结构化来源证明。字段可以来自 trusted intent certificate、fresh approval、tool schema、runtime evidence，或某个 explicit context cell。

若字段由模型自由生成、缺少结构化 proof，或来源不能被 adapter 归因，checker 会保守处理。它把该字段的 control provenance 近似为当前 prompt 中所有可见、可能影响该字段的 non-policy context。只要其中存在缺少所需 influence mode 的 context，checker 就保守拒绝。普通 output body 或 spreadsheet cell 可以带 data provenance。Authority-bearing fields 必须带 control proof。这一策略会带来可用性代价，因此 E1/E4 把 false denial、proof completeness 和大规模 recovery 评估与 same-event safety 分开报告，而不是假设 provenance 总是精确。

Lease 绑定 operation、object、arguments、intent、influence、flow、temporal、budget、expiry 和 delegation。一个 PDF extraction lease 可以允许脚本读取两个选中文件并写入输出目录，但禁止网络、secrets 和额外 subprocess。GitHub issue lease 可以允许一次 \texttt{create\_issue(repo=org/repo-x)}，但要求 repo control provenance 只能来自 trusted user intent。Lease solving 的目标不是全局证明所有可能 plan 的最小权限，而是在给定 effect plan、extension catalog 和 global policy 下生成风险更低、可检查、可衰减的 authority set。

\subsection{Compiler and Checker}

\sys 使用 LLM-assisted 但 checker-verified 的 pipeline。Compiler frontend 总结 Skill metadata、MCP schemas、local scripts、observed traces 和 policy。它推导 plan 与 security-relevant effects，并为当前 run 提出 candidate leases。Frontend 不在 TCB 内。Deterministic checker 验证每个 lease 是否满足 global policy、intent certificate、provenance constraints、flow rules、temporal guards、budgets 和 delegation monotonicity。

Compiler 的输出必须携带 proof object，说明每个 protected field 来自哪些 intent fields、哪些 context sources 可以作为 data/control dependency、哪些 sources 被排除、以及 lease 在什么 temporal state 和 budget 下有效。Checker 不信任 proof 的自然语言解释，只检查其中的结构化 predicates。若 proof 缺失、argument predicate 过宽、control provenance 不明、或 lease 试图把 parent 不具备的 authority 委托给 child，checker 返回结构化 denial。

Denial 是 runtime refinement 的输入，不是绕过授权的通道。LLM 可以根据 denial reason 重新规划，例如选择更窄的 sink、补充 read-only proof-gathering step、或请求用户 fresh approval。但新 proposal 仍要经过同一个 checker。这一设计使 recovery 可以改善 utility，而不会把失败路径变成宽权限回退。

\section{形式化模型}
\label{sec:formal}

\sys 的形式化目标不是为任意自然语言推理建模，而是为上一节 checker pipeline 的输入和输出提供可检查语义。Compiler、gateway 和 runtime adapters 负责把模型计划、工具调用、MCP 调用、本地执行和 delegation 转换成安全相关事件。Checker 只对这些事件、active leases、intent certificate 和 provenance labels 做确定性判断。因此，模型刻意不讨论 LLM 内部是否“理解”指令，而是讨论一个被 instrumentation 暴露出来的 authority-bearing event 是否应被接受。

模型围绕三个 invariant 展开。第一，\textbf{no substitution}: 每个 protected field 必须由自己的输入类证明，其他输入类不能代填。第二，\textbf{atomic check/consume}: protected event 的 predicate、provenance、budget、temporal 和 delegation state 必须在同一次 transition 中检查并更新。第三，\textbf{no promotion}: env/tool/instruction context 不能仅凭自然语言内容升级为更高 authority class，除非 trusted issuer 产生显式 endorsement 或 fresh lease mint event。

一个 agent execution 被表示为事件 trace：

\[
\tau = e_1, e_2, \ldots, e_n .
\]

事件语言只覆盖会影响 authority 的边界：

\[
\begin{aligned}
e ::=~& ctx(c,m,d)
  \mid call(t,args)
  \mid mcp(s,t,args)
  \mid read(o)
  \mid write(o) \\
  & \mid exec(p,args)
  \mid send(sink,msg)
  \mid spawn(a,K)
  \mid approve(u,a) \\
  & \mid request(\kappa,reason)
  \mid policy\_update(patch)
  \mid lease(\kappa).
\end{aligned}
\]

每个事件带有两类 provenance：

\[
prov(e)=\langle prov_D(e), prov_C(e) \rangle ,
\]

其中 \(prov_D\) 表示数据依赖，\(prov_C\) 表示控制依赖。\sys 只在 protected decisions 上约束 \(prov_C\)，例如 tool selection、sink selection、approval scope、policy update 和 delegation target。这样可以表达同一段 PDF text 可以作为 issue body 的 data dependency，却不能作为 GitHub repository 参数的 control dependency。

在 running example 中，repo field 需要 agent-owned proof，而 issue body 只需要 data-use authority。形式化模型先给每个 context cell \(c\) 标注来源和 endorsement：

\[
\ell(c)=\langle origin, integrity, confidentiality, purpose, time, endorsements\rangle .
\]

标签进一步映射到 context authority。这里需要区分 influence mode 和 decision class。Mode 描述 context 的使用方式，例如 quote、summarize、parameterize、plan、instruct、delegate 和 authorize。Decision class 描述被影响的安全相关字段，例如 output content、sink selection、approval scope、capability request、policy update 和 delegation target：

\[
auth(c) \subseteq InfluenceModes \times DecisionClasses .
\]

例如，PDF text 可以 parameterize spreadsheet cells 并 summarize issue body，但不能 authorize capability request、policy update 或 GitHub repository selection。实现可以将 \(InfluenceModes\) 做成偏序或命名集合。论文中的性质只要求 checker 能判断某个 context 是否被授权以所需 mode 影响某个 decision class。

Intent-carrying lease 的抽象形式为：

\[
\kappa =
\langle holder, op, obj, args, I, influence, flow, temporal, budget, expiry, delegation\rangle .
\]

其中 \(I\) 是当前 intent certificate，\(influence\) 约束 control/data provenance，\(flow\) 约束 confidentiality 到 sink 的流向，\(temporal\) 和 \(budget\) 约束事件顺序与调用次数，\(delegation\) 约束可传递给 subagent 的能力集合。

形式上，lease synthesis 可被看作从 per-decision issuer-typed fields 中求解可授权 lease：

\[
\kappa \in Solve_\Gamma(C_{agent}, C_{inst}, C_{tool}, C_{env}) .
\]

其中 \(C_{agent}\) 包含 intent certificate、authorized object/sink selections、approvals、plan stage、agent role 和 parent authority。\(C_{inst}\) 包含 system/developer/user instructions、Skill/manual workflow text、MCP prompts 和 trust endorsements。\(C_{tool}\) 包含 MCP/tool schemas、server identity、credential scope、command/script execution descriptors 和 sandbox contracts。\(C_{env}\) 包含 runtime object existence、cwd/filesystem state、concrete arguments、observed results、runtime evidence 和 resource availability。Active/exhausted lease table、per-lease counters 和 delegation graph 属于 checker state \(\sigma\)，不属于任何可由环境输出填充的 context class。

这四类不能自然合并成三类。若把 \(C_{inst}\) 合并进 \(C_{tool}\)，Skill/manual workflow text 就会和 callable schema、credential scope、binary descriptor 处在同一 issuer class，工具结果或 tool metadata 更容易被误解释为 procedure authority。若把 \(C_{inst}\) 合并进 \(C_{agent}\)，Skill instruction 就可能替用户选择 sink、扩大 approval scope 或 mint lease。若把 \(C_{env}\) 合并进 \(C_{tool}\)，runtime observation 会和静态接口描述混在一起，已执行 tool result、script output 或文件状态就可能替 schema 或 policy 补证明。四类划分的目的不是命名方便，而是让每个 authority field 只有一个可以签发它的 issuer boundary。

Checker 以四个 projection functions 读取这些输入类。这里的 projection 不是 parser convenience，而是 proof boundary：每个 projection 只返回该 issuer class 拥有的 fields。

\[
\begin{aligned}
intent(e) &\leftarrow \pi_{agent}(C_{agent}),\\
procedure(e) &\leftarrow \pi_{inst}(C_{inst}),\\
interface(e) &\leftarrow \pi_{tool}(C_{tool}),\\
evidence(e) &\leftarrow \pi_{env}(C_{env}) .
\end{aligned}
\]

一个 protected event \(e\) 只有在 \(Req(d)\) 要求的 projections 与同一个 lease \(\kappa\) 的 predicates 一致时才被接受。未被 \(Req(d)\) 要求的输入类可以为空，但 required field 不能从其他 context class 中补齐。例如 instruction 不能补齐 authorized sink，tool schema 不能补齐 user intent，env evidence 不能补齐 approval token。这个 typed field rule 使 \sys 不只是 MCP/tool-call guard，而是跨 Skill instruction、tool interface、runtime environment 和 agent workflow state 的统一 checker transition。

也可以把“四类是否可以合并”写成 soundness question。令 \(h:\{agent,inst,tool,env\}\rightarrow Q'\) 是一个把若干输入类折叠到更少类别的映射，\(Accept_h(e)\) 是 checker 在折叠后只检查 \(h(owner(f))\) 是否匹配时的接受判定。一次折叠是安全的，必须满足：

\[
\forall e,\sigma,\kappa.\quad Accept_h(e,\sigma,\kappa) \Rightarrow Accept(e,\sigma,\kappa).
\]

E3 的 residual suites 和 R220/R224 的 authority-input evidence 就是在寻找该蕴含式的反例：同一 object、argument 或 prompt slot 保持不变，只替换 issuer class。若折叠后的 checker 接受而四类 checker 拒绝，说明被合并的两类拥有不同 authority fields，不能作为同一个 proof boundary。这个定义把“四类不是三类”的主张变成可测试的 false-accept 条件，而不是命名偏好。

更具体地，令 \(\Phi(e,d)\) 是事件 \(e\) 对 decision class \(d\) 提交的 field-proof set。每个证明项的形状为

\[
\phi=\langle f, issuer, value, source, prov\rangle .
\]

Checker 接受 \(e\) 的必要条件是：

\[
\forall f\in Req(d),\exists \phi\in\Phi(e,d):
\phi.f=f \land \phi.issuer=owner(f) \land match_\Gamma(\phi.value,\kappa.f).
\]

这个条件故意不是“存在一组可信 attributes 满足 policy”。它要求每个 required field 都由自己的 issuer class 证明，并且证明值与同一个 active lease 的 predicate 匹配。Env proof 可以证明某个 \texttt{/tmp/out.xlsx} 确实由本次脚本生成；它不能证明该路径是用户授权 output directory。Tool proof 可以证明 \texttt{pdftotext} 的 argv schema；它不能证明当前 PDF 是用户选择对象。Instruction proof 可以证明 Skill workflow 允许调用 extraction step；它不能证明 GitHub sink 或 approval scope。

不同 protected decision 对四类 context 的需求不同。令

\[
Req(d)=\langle A_d, I_d, T_d, E_d\rangle
\]

表示 decision class \(d\) 的 agent、instruction、tool 和 env field requirements。若 \(A_d\)、\(I_d\)、\(T_d\) 或 \(E_d\) 中任一 required field 缺失，checker 必须拒绝该 transition，而不能由其他 context class 代填。表~\ref{tab:decision-reqs} 给出 running example 中的代表性规则。

为了避免把四类输入退化成普通 ABAC attribute namespaces，\sys 将 required fields 视为 issuer-typed authority fields。令 \(owner(f)\in\{agent,inst,tool,env\}\) 表示 field \(f\) 的唯一授权 issuer class，令 \(proof(e,f)=\langle issuer,value,source\rangle\) 是 runtime adapter 为事件 \(e\) 提交的结构化证明。No-substitution typing rule 是：

\[
\frac{f \in Req(d) \quad proof(e,f)=\langle q,v,s\rangle \quad q = owner(f)}
     {\Gamma \vdash proof(e,f) : f}
\qquad
\frac{q \neq owner(f)}
     {\Gamma \nvdash \langle q,v,s\rangle : f}
\]

例如 \texttt{repo=org/repo-x} 的 sink authorization field 由 agent issuer 证明。Trusted GitHub tool metadata 可以证明 schema field，却不能证明该 repo 是用户授权 sink。于是，同一个 tool-issued proof 对 interface field 可以被接受：

\[
\Gamma \vdash \langle tool,\texttt{create\_issue.schema},mcp\_registry\rangle : github\_schema ,
\]

但对 agent-owned field 必须被拒绝：

\[
\Gamma \nvdash \langle tool,\texttt{org/repo-x},mcp\_metadata\rangle : authorized\_sink .
\]

同理，tool result 或 tool metadata 也不能证明 \(policy\_update\) field。ABAC、IFC 或 policy-DSL system 若要与 \sys 等价，必须实现这种 typed transition interface：它需要显式暴露 \(C_{agent}\)、\(C_{inst}\)、\(C_{tool}\) 和 \(C_{env}\) 四个 projections，维护每个 authority field 的 owner，检查 proof object 与 owner 的一致性，并在同一次 accepted transition 中原子检查 required fields、control-provenance proof、lease consumption、budget/expiry 和 delegation state。这里的区别不是某个 sufficiently expressive DSL 能否编码同一规则，而是授权对象是否把 issuer-owned field proofs 和 lifecycle updates 暴露为同一个原子 transition。若没有这个对象，等价安全性需要额外证明多个 guard 之间不会出现 stale read、state split 或 cross-guard widening。若一个 DSL 增加这些接口和状态，它就在模拟 \sys 的授权对象；若缺少其中任一项，E3 residuals 会对应地产生 false accepts。

Context class 不是普通 attribute。Checker 还维护一个 no-promotion invariant。令 \(class(c)\in\{agent,inst,tool,env\}\)，令 \(endorse_q(c)\) 表示 trusted issuer 明确把 \(c\) endorsement 为 class \(q\)。若没有 endorsement，则：

\[
\begin{array}{ll}
c\in C_{env} \land \neg endorse_{inst}(c) & \Rightarrow c \notin C_{inst},\\
c\in C_{tool} \land \neg endorse_{agent}(c) & \Rightarrow c \notin C_{agent},\\
c\in C_{inst} \land \neg endorse_{agent}(c) & \Rightarrow c \notin C_{agent}.
\end{array}
\]

直观地说，tool result、script output 或 PDF text 不能因为包含祈使句就变成 instruction。Tool schema 不能因为声明“安全”就变成 user intent。Skill/manual instruction 不能因为要求更多权限就变成 approval。Promotion 只能由 trusted issuer、policy 或 fresh approval 显式产生，并作为新的 lease mint event 记录在 \(\sigma\) 中。No-promotion 还意味着任一 context class 都不能替另一个 class 补齐缺失的 authority field。Instruction 可以建议 workflow，但 binary/schema 必须由 tool context 验证。Tool context 可以定义接口，但 sink/intent 必须由 agent context 验证。Env context 可以提供 observed values，但不能把这些 values 转换成 approval 或 policy。

Lease lifecycle 是 \sys 与静态 policy rule 或单次 action provenance check 的主要区别。Agent workloads 中，同一个 action argument 可以在不同 authority state 下有不同安全含义：第一次 \texttt{create\_issue(repo=org/repo-x)} 可能由 fresh approval 授权。第二次相同参数可能是在复用已消耗的一次性 lease。同一 repo 参数交给 subagent 可能违反 delegation bound。同一 approval reason 若由 PDF text 控制，则可能变成 scope widening。Action-time provenance 或 object predicate 往往只看到 repo 值仍然匹配，无法区分这些 lifecycle violations。

Lease 因此是 future protected decision 的有状态授权对象，而不只是具体 operation 的 permission。它表达某个 context source 在 intent \(I\) 下可以影响 decision class \(d\)，直到某个状态转移使其被消耗、过期或衰减。Checker 将 lease 作为有状态对象处理，而不是把它当作一组静态 predicates。

Mint provenance、influence authority、consumption/expiration 和 delegation bound 必须在同一个 authorization object 上被检查并原子更新。若它们分散在独立 taint labels、ACL entries、counters 和 delegation records 中，系统就必须额外证明这些对象不会产生 stale authority reuse、cross-object widening 或 consumed-state bypass。Runtime state \(\sigma\) 至少包含 active/exhausted lease table、per-lease consumption counters、temporal automata state、mint proof records 和 parent-child delegation graph。实现中的 checker state schema 直接保存这些字段，并在每个 accepted event 后原子更新。

Checker 暴露的 authority-changing interface 只有四类 transition：

\[
\begin{aligned}
&mint(\kappa), \quad check\_and\_consume(e,\kappa),\\
&attenuate(\kappa_{parent},K_c), \quad expire(\kappa).
\end{aligned}
\]

Runtime adapters 可以分布在 context、tool、MCP、local execution 和 delegation 边界，但它们不能本地修改 authority state。任何会产生外部副作用或改变 authority state 的 protected event，包括 tool call、MCP call、local command、approval request、lease mint、policy update 和 delegation，在副作用发生前都必须经过 central checker state 的 \(check\_and\_consume\)。Accepted event、budget update、provenance record 和 delegation-state update 是同一个 transition：

\begin{itemize}[leftmargin=*]
  \item \textbf{Mint.} 只有 trusted issuer 可以通过 \(lease(\kappa)\) 事件把新 lease 加入 \(K\)。Mint rule 要求 \(\kappa.intent\) 可由 \(I\) 中的 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导。Untrusted context 不能成为 mint 或 widening 的 control dependency。
  \item \textbf{Check.} 每个 protected event 在 side effect 发生前必须引用一个 active lease \(\kappa \in K\)，且 operation、object、arguments、influence、flow 和 temporal predicates 同时满足。没有 active lease 的 event 即使 action arguments 看似可信也被拒绝。
  \item \textbf{Consume.} 若 \(\kappa.budget\) 是 one-shot 或 bounded count，accepted event 会递减 counter。Counter 为零时 \(\kappa\) 从 active set 中移除或进入 exhausted state。后续相同 tool/argument 的复用必须重新 mint。
  \item \textbf{Expire.} Temporal automata 可以在 task completion、approval window 结束、required predecessor missing 或 forbidden successor 出现时使 lease invalid。Expired lease 不再授权 event。
  \item \textbf{Attenuate.} Delegation event \(spawn(a,K_c)\) 只在 \(K_c \sqsubseteq K_{parent}\) 时被接受。Child lease 的 objects、arguments、budgets、sinks、influence modes 和 delegation rights 均不得宽于 parent lease。
  \item \textbf{Deny amplification.} Skill instructions、MCP results、documents、script outputs 或 subagent summaries 可以作为 data/control provenance 参与普通 decisions，但除非被 trusted issuer endorsement，它们不能 mint fresh high-impact lease、扩大 existing lease，或把 exhausted/expired lease 恢复为 active。
\end{itemize}

这些规则也可以写成紧凑的状态转移。令 \(K_\sigma\) 表示 \(\sigma\) 中的 active leases，\(budget_\sigma(\kappa)\) 表示剩余调用次数，\(\sqsubseteq\) 表示 capability attenuation：

\[
\begin{array}{ll}
\textsc{Mint}:&
  \Gamma;I;\sigma \vdash trusted\_mint(\kappa)
  \Rightarrow \sigma[K_\sigma \leftarrow K_\sigma \cup \{\kappa\}] \\
\textsc{Check}:&
  \Gamma;I;K_\sigma;\sigma \vdash e:\kappa
  \land active_\sigma(\kappa)
  \Rightarrow \sigma_1 \\
\textsc{Consume}:&
  budget_{\sigma_1}(\kappa)=1
  \Rightarrow \sigma_1[K \leftarrow K_{\sigma_1}\setminus\{\kappa\}] \\
&
  budget_{\sigma_1}(\kappa)>1
  \Rightarrow \sigma_1[budget(\kappa)\leftarrow budget_{\sigma_1}(\kappa)-1] \\
\textsc{Expire}:&
  expired_\sigma(\kappa)
  \Rightarrow \sigma[K \leftarrow K_\sigma\setminus\{\kappa\}] \\
\textsc{Attenuate}:&
  spawn(a,K_c) \land K_c \sqsubseteq K_\sigma
  \Rightarrow \sigma[child(a)\leftarrow K_c] \\
\textsc{NoAmp}:&
  c \notin trusted\_issuers
  \Rightarrow \neg mint_c(\kappa') \land \neg widen_c(\kappa,\kappa').
\end{array}
\]

这些转移表达的不可约 invariant 是：authority state 不能通过 untrusted provenance 被创建、刷新、扩大或委托。若一个 provenance graph 或 stateful policy baseline 要表达同一性质，它必须引入等价的原子授权对象，同时覆盖 trusted mint、decision-specific influence、budget consumption、expiration 和 monotone delegation state。换言之，\sys 的结论不是这些检查项各自新颖，而是 agent protected decisions 需要一个把它们绑定为同一 transition 的授权单元。

Checker judgment is:

\[
\Gamma; I; K; \sigma \vdash e \Rightarrow \sigma' ,
\]

其中 \(\Gamma\) 是全局策略，\(I\) 是 intent certificate，\(K\) 是 active leases，\(\sigma\) 是 runtime state。事件被接受当且仅当存在 \(\kappa \in K\) 使得七个条件同时成立：\(1\) operation 匹配，\(2\) object 和 arguments 满足 lease predicate，\(3\) event purpose 可由 \(I\) 推导，\(4\) \(prov_C(e)\) 中的每个 context 都拥有影响该 decision class 所需的 authority，\(5\) \(prov_D(e)\) 满足 flow policy，\(6\) temporal guard 和 budget 尚未过期，\(7\) delegated leases 不宽于 parent leases。

对于 \(request(\kappa,reason)\) 和 \(policy\_update(patch)\)，checker 将其作为 protected decisions 处理。它们自身不能 mint authority，除非 trusted issuer 随后产生独立 \(lease(\kappa)\) event。它们的 control provenance 也必须拥有 \((authorize, capability\_request)\) 或 \((authorize, policy\_update)\)。

\subsection{Safety Properties}
一个 trace 被接受，表示其中每个 protected event 都按上述 judgment 顺序通过 checker，并且每个通过事件对 \(\sigma\) 的更新会消耗预算、推进 temporal state、记录 provenance，并更新 active leases。下面五个性质是这一 checker semantics 的直接安全后果。它们不是关于任意 prompt 的全局声明，而是关于 accepted traces over instrumented protected-decision events with adapter-supplied control-provenance proofs 的 trace-level properties。

\begin{enumerate}[leftmargin=*,label=\textbf{P\arabic*.}]
  \item \textbf{No issuer substitution.} 对任意 accepted protected event \(e\) 和其 decision class \(d\)，若 \(f \in Req(d)\)，则 checker 接受的 field proof 必须满足 \(proof(e,f).issuer=owner(f)\)，且 proof value 与同一个 active lease 的 predicate 匹配。其他 issuer class 提交的同值 proof 不能补齐该 field。特别是 tool metadata 不能证明 agent-owned sink 或 approval field，env evidence 不能证明 instruction-owned workflow field，Skill/manual instruction 不能证明 fresh approval。
  \item \textbf{No unauthorized influence.} 对任意被 instrumentation 暴露并被 checker 接受的 trace \(\tau\)，如果事件 \(e \in \tau\) 属于 protected decision class \(d\)，令 \(req(d)\) 是 \(d\) 所需的最低 influence mode，则对所有 \(c \in prov_C(e)\)，都有 \((req(d),d) \in auth(c)\)。换言之，未被授权以所需 mode 影响 \(d\) 的 context 不能成为该决策的控制依赖。
  \item \textbf{Intent-bounded authority.} 对任意 accepted high-impact event \(e\)，存在 active lease \(\kappa\)，且 \(\kappa\) 的 intent predicate 可由 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导。系统不会仅凭 Skill identity、MCP server identity 或工具描述铸造 high-impact authority。
  \item \textbf{Delegation monotonicity.} 对任意 accepted delegation event \(spawn(a,K_c)\)，child capability set \(K_c\) 是 parent active capability set 的子集或更窄 attenuation。除非 trusted issuer 基于新的 intent certificate 创建 fresh lease，subagent、Skill、MCP tool 或 script 都不能扩大自身权限。
  \item \textbf{No context-class promotion.} 对任意 accepted trace，若 context cell \(c\) 的原始 class 是 \(env\)、\(tool\) 或 \(inst\)，则 \(c\) 不能在后续 checker transition 中以更高 authority class 使用，除非 trusted issuer 产生显式 endorsement 或 fresh lease mint event。特别是，tool result、script output、PDF text 和 subagent summary 不能仅凭自然语言内容变成 instruction context。Tool schema 不能变成 user intent。Skill/manual instruction 不能变成 approval。
\end{enumerate}

这些性质是 trace-level safety properties。它们依赖于两个工程前提：runtime 必须把 protected decisions 接入 checker，并且 provenance tracker 必须保守地标记事件控制依赖。若某个 OS-level side effect 没有经过任何 adapter，\sys 无法单独观察它。这正是 sandbox 或 ActPlane-style backend 仍然有价值的原因。

\section{实现}
\label{sec:implementation}

我们的原型是一个 Python-based agent authorization harness，用于验证三个边界性质：lease language 是否足以表达 intent/context authority，deterministic checker 是否能在统一 event model 上执行这些约束，以及这些约束能否落到 trace replay、live callable gateway、benchmark-derived workflows、context/delegation boundary probes 和 env/OS-monitor-style policy target。当前完整实现的是 checker core、tool/live callable gateway、本地 env pre-side-effect gateway、live context-placement gateway、live delegation monitor、runtime binding 和结果分析脚本；ActPlane 是 policy-target lowering，而不是 production kernel integration。Production MCP broker、production prompt builder/subagent runtime、production ActPlane integration 和 fresh user-simulator loop 是扩展实验所需的工程边界。

\subsection{Checker Core}
Checker core 位于 \texttt{src/intentcap/}，实现 context labels、effect events、leases、checker state 和 proof objects。当前 checker 支持 exact、prefix、suffix 和 one-of argument predicates，支持 lease matching、holder scoping、control/data provenance checks、context-label influence-mode checks、temporal prerequisites、per-lease invocation budgets、proof-carrying intent/approval derivation，以及 delegated capability attenuation。这个 core 是本文的主要 TCB：LLM 输出、Skill 描述、MCP annotations 和 tool results 只能产生 candidate effects 或 candidate leases，不能绕过 checker 直接获得 authority。

\subsection{Trace and Live Gateways}
\texttt{TraceGateway} 用同一个 checker replay benchmark-derived traces，并把每个事件记录为 execute/block decision。它用于把 AgentDojo、MCPTox、InjecAgent 和 tau2-bench benchmark labels 转换成统一的 protected-decision events~\cite{agentdojo,mcptox,injecagent,tau2bench}，从而比较不同 policy variants 的 false accept、authority breadth 和与人工标注 lease 的距离。

\texttt{LiveToolGateway} 包裹本地 Python callables。只有 checker 接受的 event 才会真正调用注册函数。被拒绝的事件在 side effect 发生前停止。这一区分很重要：trace replay 提供大规模 policy evidence，live gateway 验证同一 lease semantics 可以保护真实 callable boundary。

\texttt{LiveContextPlacementGateway} 和 \texttt{LiveDelegationMonitor} 将同一个 stateful checker session 放在 prompt-section placement 与 subagent handoff 之前。前者只有在 \texttt{ctx.use} event 被允许后才把内容写入目标 section；后者只有在 \texttt{subagent.spawn} event 的 delegated capabilities 被 lease attenuation 覆盖后才记录 child handoff。Skill/manual probe 进一步把同一个合法 instruction slot 固定为 signed Skill issuer，测试 tool result、script output 和 unsigned Skill text 是否能替代 instruction issuer。这些 adapters 是本地 probes，不是 production prompt builder 或 subagent runtime。

\subsection{Compiler Probes and Runtime Binding}
原型还包含 local llama.cpp/Qwen frontend，用来模拟不可信 lease compiler 和 action proposer。模型可以根据 task text、tool schemas、已有 tool-result previews 或 checker feedback 生成 candidate leases/actions。系统只把通过 checker 的结果交给 gateway。

对于 tau2-bench-style stateful workflows，runtime binding 会从已经执行的 tool-result evidence 中铸造 exact one-shot leases，并重新检查参数值是否出现在当前可见 task/tool-result context 中~\cite{tau2bench}。High-impact write candidates 还必须满足 structured value proof：例如列表参数需要逐项绑定到结构化上下文，不能从原始文本窗口或 sibling objects 中借用 authority。

\subsection{Implemented Surface}
当前原型实现了 checker/gateway、live callable、context/delegation probes、Skill/manual instruction-source substitution probe、benchmark replay adapters、runtime binding、structured value proof、read/write activation、lease-label audit，以及 E1 security/utility wrapper matrices。这些组件共同验证同一 lease semantics 可以同时服务 trace replay、真实本地 callable/context/delegation/instruction-placement boundary 和 benchmark-derived protected-decision events。

当前原型还包含 local Qwen/llama.cpp compiler experiments 和本地 env backend experiments。Env backend 在隔离目录中真实执行 Python script、file read/write 和 context-use handlers，并在 side effect handler 之前调用 checker。被拒绝的 exec/read/write/net/ctx events 不会进入本地 side-effect handler。模型闭环版本使用 Qwen3.6/llama.cpp 作为不可信 action proposer。模型在 adversarial env-context prompt 下返回 call/abort，只有 call 事件会继续进入 checker 和本地 side-effect handler。

原型还实现了一个 ActPlane-style lowering target：从 active env leases 生成 deny-by-default monitor rules，并在 replay monitor 中检查 operation、object、holder、argument predicates、budget consumption、control/data provenance 和 context-label influence rules。该 lowering 证明 lease contract 可以被编码为 env/OS monitor policy 形态。它不是 production ActPlane integration，因为当前 monitor 运行在 Python replay harness 中，且 provenance labels 仍由 IntentCap adapter 提供。

这些实现支撑机制性结论：在已接入 checker 的边界上，\sys 能拒绝 unauthorized context-to-decision influence，run-time exact leases 可以缩小 object/tool authority surface，checker 可以阻止 LLM-proposed broad/runtime authority 直接进入执行路径，context/delegation boundary 可以在 placement/handoff 前调用同一 checker，并且同一 env lease contract 可以被降低为包含 provenance/influence checks 的 monitor rules。生产级 sandbox、full MCP broker、独立 human expert-oracle replication 和 benchmark-scale recovery evaluation 属于扩展结论所需的实验条件。

\section{评估}
\label{sec:evaluation}

评估围绕四个 questions 组织。E1 测试 protected decisions 的安全性和良性动作覆盖率。E2 提供辅助 auditability evidence，测试 lease 是否支持机器审计和 structured policy-distance analysis，但不承担 expert-oracle least-privilege claim。E3 测试去掉 issuer ownership 或 atomic lifecycle 后是否产生 false accepts。E4 测试 checker transition 是否能在本地环境、context placement、delegation handoff 和 Qwen3.6 proposer 前阻止副作用。

所有实验固定底层模型、任务顺序、工具描述和 prompt budget，尽量只改变授权层。High-impact denials 使用同一个 protected-decision oracle 判定：若某个 protected decision 的 control provenance 来自缺少相应 influence mode 的 context source，则该事件应被拒绝。本文中的 closest baseline labelers 是在同一事件模型上实现的 trace-level policy-family abstractions，用来表示 provenance/IFC/policy-style baselines 的可判定谓词；它们不是 AuthGraph、PACT、AIRGuard 或其他完整系统的 artifact reproduction。表~\ref{tab:evaluation-matrix} 给出四个实验与结论的对应关系。

\subsection{E1: Protected-Event Safety and Utility Proxy}

E1 的假设是：若安全边界是 context influence 而不只是 tool visibility，\sys 应拒绝 protected-event violations，同时仍表达 benign reference behavior。E1 主要测试 ACL/tool-visibility gap：object/tool ACL baselines 会失败，因为它们只看工具或参数，不看控制 provenance。E3 再测试 provenance/action-authority families 之外的 residual lease semantics。实验把 3,746 个 benchmark-derived protected events 输入 \sys 和 same-event wrappers，并在 3,813 个 benign reference actions 与 2,556 个 applicable tool-oracle tasks 上评估良性动作重放。

在 protected-event replay 中，\sys 在 3,746-event matrix 上接受 0 个 dangerous events。Object/tool ACL wrappers 接受许多 protected-decision violations，因为它们不区分 context influence authority。在 benign proxy 上，\sys 覆盖 3,813/3,813 个 reference actions，并通过 2,554/2,556 个 applicable tool-oracle tasks。

这些 replay results 支持 protected-event replay 的安全性和良性动作覆盖率，但不推出 live agent 在窄 lease 下的端到端完成率。一个 matched 18-task local-Qwen 局部轨迹显示当前 utility boundary：all-tool exposure 将平均可见 schemas 从 2.61 提高到 14.72，将 model calls 从 62 提高到 74，但 action-reward 仍为 8/18，tool-oracle 仍为 0/18。因此，live utility、approval burden 和大规模 recovery 是独立结论，需要用 online agent runs 单独评估。

\subsection{E2: Lease Auditability}

E2 的假设是：intent-carrying leases 应能被机器审计为结构化 authority objects，包含 run-specific intent、influence-mode、budget 和 delegation fields。当前 corpus 包含 24 个 author-adjudicated lease labels，覆盖 agent、tool、env 和部分 instruction context。Audit 检查 schema validity、sampled label consistency，以及 object scope、influence modes、argument predicates、budget/expiry、delegation bounds 和 approval scope 上的 policy-distance invariants。

Audit 通过 24/24 个 labels。这个结果说明 lease representation 可以支持 structured authority-distance analysis，并能在当前 corpus 中发现 schema/sample/policy-distance mistakes。

E2 的结论限定在 author-adjudicated lease-audit result。若要声称 \sys 比 symbolic policy baselines 更接近 expert minimal policy，还需要 blinded independent adjudication。

\subsection{E3: Ablation Evidence for Issuer-Owned Atomic Transitions}

E3 的假设是：四类 authority-input fields 和 stateful lease lifecycle 共同形成一个不可拆开的机制。Object/action guards、IFC/provenance guards 和 split-state counters 会失败，因为它们允许一类输入替另一类输入补齐 field，或把 budget/delegation/provenance 分散到不同状态中。因此 E3 是 necessity-oriented ablation，而不是完整自然频率证明。它的比较对象是同一个 policy family 的三个版本：完整 \sys；去掉 issuer ownership、把 \(C_{agent},C_{inst},C_{tool},C_{env}\) 折叠成 generic trusted context 的版本；以及保留 predicates 但把 proof、budget、expiry 和 delegation state 分散检查的 split-state 版本。

完整 necessity experiment 需要四步：从已有或 live traces 抽取 protected-decision events；为每个 required field 标注 owner issuer；在 full/no-owner/split-state 三个 checker 上重放同一事件；报告自然事件和 controlled residuals 上的 false accepts。当前结果已经完成前三类证据的一部分。R220/R221 把已有 traces 转换成可复核的 protected-decision/field-owner 标注任务。R217/R219/R224 等 residual suites 固定 object、argument 或 prompt slot，只替换 issuer class 或 lifecycle state，测试 no-owner 与 split-state variants 的 false accepts。缺口也明确：顶会级 necessity claim 还需要独立标注者复核 R221 labels，并在更多自然 protected-decision events 上报告 full/no-owner/split-state 的 false accept rate，而不只依赖 controlled residuals。

R220 读取已有本地 artifacts，没有同步或下载新数据。输入包括 local env suite、residual workflow-policy suite、AgentDojo、MCPTox、InjecAgent、tau2-style traces 和 R136 runtime-binding summary。Analyzer 从已有 provenance labels 和 event modes 派生 issuer-class annotation，因此这是 trace-level annotation，不是独立人工标注。``Requires multiple issuer classes'' 指 analyzer 根据事件的 \(op\)、mode、decision class 和 provenance labels 推导出的 required-field class set 包含至少两个 owner classes，而不是人工断言该事件真实使用了两个独立 UI/API。结果显示，在 8,696 个事件中，8,691 个需要多个 issuer classes；198 个需要 env/runtime issuer；3,593 个 checker-denied events 存在 class-substitution attempt。Substitution edges 包括 tool \(\rightarrow\) agent 1,928 次、env \(\rightarrow\) agent 1,663 次、env \(\rightarrow\) tool 1,662 次，以及 instruction \(\rightarrow\) agent 2 次。Edge 计数不是互斥事件数，因为同一事件可能同时暴露多个 collapse risk。R136 还显示 29/29 次 runtime binding 成功，并有 25 个 runtime-evidence hint steps，说明当前原型确实把已执行 tool-result evidence 作为 env issuer 来铸造 one-shot leases，而不是让 prompt 或 tool metadata 代替 runtime observer。

R220 的作用是回答“四类是否只是作者任意分类”。它不证明所有真实 agent workloads 都天然包含这些 substitution patterns，但它说明已有 benchmark-derived traces 和 local workflows 已经同时触发 agent、instruction、tool 和 env 的不同证明边界。特别是，MCPTox 中 tool metadata 试图影响 authority，InjecAgent 和 AgentDojo 中 env/tool-result 或 injected goal 试图影响 agent-owned decisions，本地 env suite 和 R136 runtime binding 则需要 runtime observer 证明实际路径、输出和 tool-result values。

为把这个结论从自动派生统计推进到可复核的人工标注接口，R221 从 R220 的 8,696 个事件中按 source、decision mode、substitution edge 和 env-runtime requirement 抽样 48 个 concrete events，并回填原始 event、control/data provenance 和 label summaries。样本覆盖 AgentDojo、MCPTox、InjecAgent-style online trace、tau2-style trace、本地 env suite 和 residual workflow-policy suite；其中 45 个是 protected-decision labels，43 个需要多个 issuer classes，13 个需要 env/runtime proof，37 个暴露 class-substitution attempts。R221 同时输出 label schema、labeling protocol、input digests、复现命令和一版 Codex/project-author first-pass labels。该结果说明 R220 的字段边界可以落到事件级标注任务上，但它仍不是 blinded independent agreement；它的用途是固定下一轮 expert/adjudication protocol。

我们随后用 trace-level baseline audit 界定剩余边界：普通 saved-trace denials 大多可由 provenance/IFC abstractions 解释；因此 E3 的 residual 部分不是自然频率统计，而是 mechanism stress test。每个 residual case 都对应表~\ref{tab:decision-reqs} 中的 owner-field violation；例如 R219 是 tool-issued proof 试图补齐 agent/admin-owned policy 或 approval fields。

当前实验使用 concrete Skill、MCP、subagent 和 local-env workflow 的 controlled residual suites，并把其中一类 residual lift 到已有 MCPTox artifact。三个对照族是 object/action guard、IFC/provenance guard 和 split-state composite guard。Residual cases 覆盖 context-class collapse、instruction-source substitution、env/tool metadata misclassification、lifecycle reuse、over-delegation，以及 trusted tool metadata \(\rightarrow\) policy update。

\sys 拒绝这些 suites 中的 lifecycle/no-promotion residual violations。Closest labelers 会接受 trusted tool metadata 供应 policy authority 的 cases，也会接受 generic provenance 折叠 agent/instruction/tool/env fields 的 cases。在 R217 workflow-policy suite 中，\sys 执行 2 个合法 workflow events，并阻止 7 个 residual violations。Closest labelers 接受全部 9 个 events，因为它们不区分 trusted tool metadata 和 agent/admin policy authority。

R219 将同一四类 no-substitution residual lift 到已有 MCPTox artifact，而没有同步或下载新数据。实验读取本地 R007 successful-call trace 和 \texttt{response\_all.json} 的 authentic tool catalog，确定性选择 24 个在 trace 中真实出现且属于 authentic catalog 的 MCP tool objects。对每个 object，构造一个合法 schema-use 事件和两个非法 authority-promotion 事件：trusted tool metadata \(\rightarrow\) \texttt{policy.update(agent.policy)}，以及 trusted tool metadata \(\rightarrow\) approval widening。\sys 允许 24/24 个 schema-use 事件，拒绝 48/48 个 policy/approval residual events。Trace-level abstractions of closest provenance/IFC/policy baseline families 接受这 48 个 residual events，因为控制来源是 trusted metadata 且事件匹配 policy-shaped lease；\sys 拒绝它们，因为 tool context 只能证明接口/schema，不能证明 agent/admin policy 或 approval scope。该结果把 R217 的本地 controlled residual 从自定义 workflow 提升到 benchmark-derived MCP tool metadata objects，但仍不是 naturally occurring prevalence 或 fresh online utility 结论。

R224 将 instruction class 单独做成 live placement stress test，而不是只在表格中声明。实验固定一个合法 \texttt{ctx.use(pdf.skill.instructions)} instruction slot：\texttt{artifact}、\texttt{destination} 和 \texttt{scope} 参数完全相同，唯一变化是 issuer/provenance class。Signed Skill manifest/instruction 的 placement 被接受，tool result 作为 issue-body data use 也被接受；但 tool result、script output 和 unsigned Skill text 试图占用同一个 trusted instruction slot 的 3 个 substitution probes 全部被拒绝。Object-only 和 lease-args/no-provenance baselines 都接受 3/3 个 unsafe substitution probes。该结果隔离了 \(C_{inst}\) 不能由 \(C_{tool}\) 或 \(C_{env}\) 补齐的机制；它仍不是真实 Skill supply-chain prevalence study。

这些结果支持一个窄机制结论：在这些 residual tests 中，no-substitution、no-promotion 与原子 check/consume 必须一起检查。

这是 authority-input trace annotation、controlled mechanism evidence 加 benchmark-derived stress test，不是 blinded human-label study、benchmark-scale prevalence 结论或 online utility 结论。若要声明这些 residual patterns 在 benchmark 中普遍出现，还需要在 AgentDojo、MCPTox、InjecAgent 和 tau2-style traces 上扩大 naturally occurring protected-decision labels，并由独立标注者确认每个 field owner。

\subsection{E4: Local Multi-Boundary Practicality}

E4 的假设是：同一 checker transition 可以在真实本地边界、untrusted model proposer 和 OS-monitor-style target 前阻止副作用。Object-only wrappers 会失败，因为它们不携带 context labels、control provenance 和 influence predicates。Local env suite 包含 10 个 script/file/network/context events。\sys 在 side-effect handlers 运行前保留 4 个 authorized effects，并拒绝 6 个 violations。

本地实验执行 4 个 \sys-authorized effects，在副作用前阻止 6 个 violations，并执行 0 个 unsafe effects。Object-only wrapper 执行 8 个 effects，其中包括 4 个 unsafe effects，并修改 isolated secret file。这个结果验证了 local env events 的 pre-side-effect gateway contract。

在同一组 10 个 events 的 Qwen3.6 adversarial model loop 中，模型返回 8 个 calls，其中包括 4 个 unsafe calls。\sys 只执行 4 个 authorized calls，并在副作用前阻止全部 4 个 unsafe calls。Object-only wrapper 执行 7 个 calls，其中包括 3 个 unsafe effects，并修改 isolated secret file。该实验支持 local env adapter 和 untrusted model proposer 场景下的接口可行性。Production sandbox 需要将同一 lease contract 继续降低到 OS-level monitor。

R218 进一步把同一组 active env leases 降低为 4 条 deny-by-default ActPlane-style monitor rules，分别覆盖 \texttt{process.exec}、\texttt{filesystem.read}、\texttt{filesystem.write} 和 \texttt{context.use}。对同一 10-event env suite replay 时，lowered monitor 允许 4 个 authorized events、拒绝 6 个 violations，与 checker 有 0 个 decision mismatches，并执行 0 个 unsafe monitor allows。重要的是，lowered rules 不只包含 object/path predicates；它们还携带 context labels、control/data provenance 和 influence-mode predicates。否则 \texttt{script\_output} 这类 env context 会被错误提升为 instruction authority。这个结果的系统含义是 lease contract 可以保留 issuer/provenance/influence predicates 降到 OS-monitor policy 形态；它不是 production ActPlane/kernel integration claim。

R222 对 instruction placement 和 delegation boundary 做一个小型 live adapter probe。Context placement gateway 在 3 个 \texttt{ctx.use} attempts 中放置 2 个授权内容，并在 prompt-section placement 前阻止 1 个 \texttt{script\_output} 提升为 instruction 的事件。Delegation monitor 在 2 个 handoff attempts 中允许 1 个 summary-only subagent capability，并阻止 1 个把 email sink capability 交给 calendar subagent 的 over-delegation。两个 unsafe counters 都为 0。该结果把 instruction/delegation 从纯 trace-level semantics 推进到本地 live adapter evidence，但它仍不是 production prompt builder、真实 subagent runtime 或用户研究。

R223 在同一 5 个 boundary events 上比较两个简单 baseline。Object-only baseline 只检查 operation/object；lease-args/no-provenance baseline 检查 operation/object/declared argument constraints，但忽略 provenance、influence modes 和 delegation attenuation。两个 baseline 都接受 2/2 个 unsafe boundary probes：\texttt{script\_output} 被放进 instruction placement，以及 calendar subagent 获得 email sink capability。\sys checker 的 unsafe accepts 为 0。这个对比说明 boundary adapter 的关键不是单纯拦在 prompt builder 或 subagent runtime 前，而是在该边界执行同一 proof-carrying lease semantics。

R224 进一步把 Skill/manual instruction placement 单独拿出来测试。Live placement gateway 在 5 个 attempts 中接受 1 个 signed Skill instruction placement 和 1 个 tool-result-as-data summary use，阻止 3 个 instruction-source substitution probes。由于这些 unsafe probes 保持同一 \texttt{ctx.use} object 和 argument shape，object-only 与 lease-args/no-provenance baselines 都会接受 3/3 个 unsafe placements；完整 checker 的 unsafe accepts 为 0。这个结果说明 Skill/manual context 与 tool/env context 的区别已经落到 placement-time enforcement，而不只是形式化分类。

R225 将上述本地系统边界合并成一个只读 evidence matrix。它读取 R211、R212、R218、R222、R223 和 R224 的保存结果，不运行模型、不执行副作用、不同步数据。合并后的 6 个边界行覆盖 local env side effects、Qwen proposer、ActPlane-style env lowering、context placement、delegation handoff 和 Skill instruction placement。38 个 boundary attempts 中，\sys 执行或放置 17 个授权 effects，阻止 21 个 attempts，并产生 0 个 unsafe effects/placements；object-only baselines 在对应边界上观察到 12 个 unsafe accepts，lease-args/no-provenance baselines 观察到 5 个 unsafe accepts，monitor/checker mismatch 为 0。这个结果支撑的是系统表面覆盖结论，而不是 production ActPlane integration 或 benchmark-scale utility。

\section{范围与局限}
\label{sec:evidence-status}

本节限定当前实验结果的解释范围。\sys 的核心结论是一个可执行的 protected-decision authorization unit。Trace replay、oracle replay 和局部 compiler/model-loop experiments 分别覆盖不同边界，不能合并成对 sandbox、MCP broker、human approval UX 或所有 agent planning logic 的完整替代结论。

\subsection{Evidence Boundary}

Saved benchmark traces 和 live callable gateway 说明，同一 checker semantics 可以作用于 replay events 与真实本地调用边界。Local Qwen/llama.cpp experiments 说明，模型输出可以作为 untrusted compiler input，由 checker 拒绝 broad/runtime authority、missing-proof writes 和 env-context 诱导的 side-effect calls。Runtime binding 与 structured value proof 说明，短期 exact leases 可以从已执行的 tool-result evidence 中安全铸造，而不是从模型自由文本中直接产生。

\subsection{Limitations}
当前评估支持的结论是：protected-decision transitions 可以在已 instrumented 的边界上被形式化并执行。当前结果覆盖 instrumented protected-decision events。端到端 utility、approval burden、independent expert replication 和 production sandbox integration 需要单独实验。

第一，当前 workloads 仍以 saved traces、reference-action proxies、controlled residual suites、benchmark-derived residual lift 和 local experiments 为主。R220 使用已有 artifacts 做 issuer-class characterization，但它是 derived trace annotation；R221 已经把其中 48 个 natural protected-decision events 做成可复核标注包和 project-author first-pass labels，但顶会级 novelty claim 仍需要独立标注者复核 field owner 与 unsafe substitutes。R219 使用已有 MCPTox artifacts 生成 24 个 authentic tool-metadata residual cases，但它不是自然发生频率统计。更大规模的 online user-simulator evaluation 是端到端 utility claim 的证据来源。第二，E2 使用 author-adjudicated lease labels。这能检查 lease schema 和 authority-distance invariants。Blinded independent expert replication 才能支撑 expert-oracle least-privilege claim。第三，本地 env backend 在隔离目录中验证 pre-side-effect blocking，R218 只验证同一 lease contract 可以降低成 ActPlane-style monitor rule set，R225 只是把这些本地结果汇总成 system-surface matrix。Production ActPlane/kernel integration、真实 syscall mediation 和低开销 enforcement 仍需要单独工程与实验。第四，R222/R224 已经实现本地 live context-placement guard、Skill/manual instruction-source substitution probe 和 live delegation monitor probe；若要声称完整多边界 enforcement，还需要接入 production prompt builder、真实 subagent runtime、更多自然 Skill/manual injection workloads 和端到端 approval/recovery metrics。

这些限制限定外推范围。\sys 证明的是 instrumented protected-decision events 可以由 intent-derived, provenance-gated, consumable leases 管理。完整 workload utility、approval burden、independent expert-oracle least privilege 和 production sandbox integration 需要分别报告，而不能由当前结果自动推出。

\subsection{应用扩展隔离与底层 Enforcement}
传统扩展系统和 sandbox 关注扩展代码能访问哪些宿主资源。EIM/bpftime 将应用扩展表示为 host resources 和 capabilities，并强调安全、高效的 userspace extension execution~\cite{eim-bpftime}。ActPlane 将 agent policy lowering 到 OS-level enforcement，以覆盖 tool interception 看不到的 file、process 和 network side effects~\cite{actplane}。这些机制适合作为 \sys 的 enforcement backend 或系统对照，但它们不决定 policy 的来源。ActPlane-style monitor 可以阻止 \texttt{net.connect(attacker.com)} 或未授权 file write，但仅靠 OS event 不能判断 \texttt{repo=org/repo-x} 是用户授权、tool metadata 建议，还是 PDF text 诱导。\sys 先生成带 issuer proof 的 lease contract；ActPlane-style backend 只执行其中 local/env 子集。\sys 的问题在更上层：某个 tool choice、sink selection、approval scope 或 delegation decision 是否由当前用户 intent 授权，并且是否被不可信 context 非法影响。

\subsection{Prompt Injection, Provenance, and Information Flow}
CaMeL、Prompt Flow Integrity、FIDES、RTBAS 和 NeuroTaint 等工作从 control/data flow、IFC、dependency screening 或 taint tracking 的角度约束 untrusted content~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}。AuthGraph、PACT 和 AIRGuard 进一步把 user intent、argument provenance 和 action-time authority control 引入 agent security~\cite{authgraph,pact,airguard}。这些工作是 \sys 最接近的思想背景。本文的新粒度不是单独的 intent/provenance authorization，而是会铸造、消耗、过期和委托的 protected-decision lease。

本文把差异放在授权对象的粒度和生命周期上。已有系统主要检查 observed action、argument 或 side effect 是否与 intent/provenance 对齐。\sys 则把未来决策权限表示为会被铸造、消耗、过期和委托的短期 lease。为此，\sys 把 mint 来源、influence mode、budget/expiry、no-promotion rule 和 parent-child delegation order 放进同一个授权对象。关键不是单独发明 provenance label、taint tracking 或 attenuable capability，而是把这些检查绑定为每次 protected decision 更新 authority state 的原子单位。

\subsection{Agent Policy Languages and Access Control}
Progent、PCAS、AgentSpec、AgentGuard 和 AgentBound 等工作提供 programmable privilege control、policy compilation、runtime DSL、ABAC 或 MCP access-control enforcement~\cite{progent,pcas,agentspec,agentguard,agentbound}。这些系统提供 policy/checker substrate，并回答“某个工具、参数、server 或 action 是否被 policy 允许”。\sys 选择不同授权对象：从当前 run 的 intent 和 context provenance 铸造短期 lease，并让 lease 随 protected decision 被消耗、过期或委托。因此，评估通过 authority breadth、oracle-distance audit 和 controlled protected-decision residuals 比较这些 policy families。

\subsection{Skill and MCP Permission Frameworks}
SkillGuard 和 SkillScope 直接面向 Agent Skills 的权限、行为规范和 least-privilege enforcement~\cite{skillguard,skillscope}。MCP 规范也强调 tool metadata 和 behavior annotations 不能默认被当作可信事实~\cite{mcp-spec}。这些工作说明 Skill/MCP ecosystem 已经需要权限边界。\sys 的范围是 run-centric 和 cross-extension 的：一个 lease 可以同时约束 Skill instruction 对 workflow 的影响、MCP result 对后续 sink 的影响、本地脚本的 file/network authority，以及 subagent 获得的 attenuated capability set。与 package-level manifest 或 server-level grant 不同，\sys 的 authority root 是当前 intent certificate。同一个 Skill 或 MCP server 在不同 run 中会得到不同、可消耗、不可自扩张的 leases。更重要的是 field ownership：即使 Skill/MCP manifest 也按 run mint，如果 package 或 server 可以声明 sink、approval 或 delegation field，它仍然能替 agent issuer 证明 user-owned authority。例如用户只授权 \texttt{org/repo-x}，而 Skill manifest 声称 safe repo 是 \texttt{attacker/repo} 或需要 full GitHub scope；若 per-run manifest 把 package 声明当作 sink/scope owner，就会接受该扩大权限。\sys 拒绝这一路径，因为 sink 和 approval scope 是 agent-owned fields，不能由 Skill/MCP/tool/env 填充。这个定位把 \sys 放在 Skill/MCP permissions 之上，作为一个 intent-derived lease compiler/checker，而不是替代 Skill/MCP 的包级权限系统。

一个具体区别是 union authority。相同的 PDF Skill 若用于“只把两个选中文件转换为本地 XLSX”，\sys 只会给 selected-file read、本地 output write 和无网络 exec lease；若用于“提取表格并在指定 GitHub repo 创建 issue”，同一 Skill 仍不能选择 repo，但 workflow 会多出由 agent intent 和 GitHub tool context 共同证明的一次性 issue lease。Skill-level manifest 若要覆盖两个 run，通常必须声明二者的 union authority；\sys 则在每次 protected transition 上重新绑定 intent、Skill instruction、tool/cmd interface 和 env evidence。因此它不是 Skill permission 的子集，而是跨 Skill、MCP/cmd、env 和 subagent 的 run-time authorization layer。

\section{结论}

LLM agent extensions 改变了权限系统需要保护的对象。文档、工具结果和扩展说明不只携带数据，也可能影响 tool selection、sink selection、approval scope、capability request 和 delegation target。\sys 将这种影响权表示为 intent-derived、provenance-gated、可消耗、可过期、可衰减的 leases，并用 deterministic checker 把 LLM proposals 保持在 TCB 之外。在保存的安全 trace 上，\sys 对 3,746 个危险 protected events 产生 0 个误接受。在 tau2-style 良性动作重放中，\sys 覆盖 3,813/3,813 个 reference actions，并通过 2,554/2,556 个适用的 tool-oracle checks。在 issuer-substitution residuals 中，\sys 拒绝 48/48 个 MCPTox-derived policy/approval promotions 和 3/3 个 Skill instruction-source substitutions，而去掉 issuer ownership 或 provenance checking 的 variants 会接受这些 unsafe transitions。在本地环境实验中，\sys 在副作用发生前阻止 6 个违规副作用和 4 个 Qwen3.6 unsafe calls。这些结果表明，在已建模并接入 checker 的边界内，context influence 可以作为可检查、可消耗、可审计的授权状态来管理。
