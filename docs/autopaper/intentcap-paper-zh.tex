\begin{abstract}
LLM agents 正在从固定工具调用器演变为可扩展执行环境，可以加载 Skills、连接 MCP servers、运行本地脚本，并将任务委托给 subagents。
这些扩展组件不只执行操作，也把说明、工具结果、外部文档和运行时观察带入后续的 tool selection、sink selection、approval scope 和 delegation target。
现有权限机制主要约束已经成形的工具调用、参数、文件和网络端点，却难以表达哪类 context 可以证明一次 authority-bearing decision 的哪个字段。
结果是 agent 可能发出参数合法、provenance 看似可信、OS sandbox 也无法区分的调用，但权限状态已经被错误扩大、复用或委托。
\sys 的核心洞察是：LLM agent 的授权单位应是副作用、prompt placement 或 delegation 前提交的 protected-decision transition，而不是事后过滤的 tool call。
\sys 将该 transition 表示为 field-owned protected-decision lease，并通过 \(\mathsf{check\_and\_consume}\) 在同一原子步骤中验证 issuer-owned proofs、当前 intent、provenance、budget、expiry 和 delegation state。
原型把这个 contract 接入 trace replay、local callable/env side effects、context and Skill placement、delegation handoff，以及 OS-monitor-style replay lowering。LLM 只提出 plans 和 candidate leases，不在 TCB 内。
评估支持一个有界结论：在已 instrumented protected-decision boundaries 内，\sys 可以阻断 unauthorized context-to-decision influence，同时保留 benign reference-action expressiveness。
Event-model guardrail 覆盖 3,813/3,813 个 tau2-style reference actions。Same-event ablations 显示 no-owner collapse 会错误接受 3,593/3,823 个完整 checker 拒绝的 protected decisions。Local boundary suite 在 38 个 checker-submitted events 上产生 0 个 unsafe executions/placements。
\end{abstract}

\section{引言}

LLM agents 的安全边界正在发生变化。传统应用扩展通常通过 API、插件接口或系统调用与宿主交互，因此权限系统可以围绕文件、网络、进程和工具调用来定义。现代 agent extension 的行为则由用户请求、系统策略、扩展说明、外部文档、工具结果和中间总结共同塑造，尤其是在 Skills、MCP servers 和 agent tool ecosystems 中~\cite{skillguard,skillscope,mcp-spec}。

然而，agent extension 打破了 data 和 authority 的传统边界。用户只要求从两个 PDF 提取表格，并在 \texttt{org/repo-x} 创建一个 summary issue。其中一个 PDF 的隐藏文本不直接修改 \texttt{repo} 参数，而是影响 agent 把 issue 创建权交给一个 summarizer subagent。最终 \texttt{create\_issue(repo=org/repo-x)} 的参数仍等于用户指定的 repository，issue body 也来自 PDF summary，所以普通 tool/argument guard、参数 provenance check 和 OS sandbox 都可能放行。此时损害不是 repo 字符串错误，而是 PDF text 控制了一次权限转移。Approval widening、stale reuse 和 policy-update 变体说明，参数合法不代表权限状态合法。

根因不是不可信文本进入 prompt 本身，而是现有 agent runtime 在提交高影响决策时，通常没有同时暴露“哪个 issuer 拥有哪个 protected field”和“这次提交如何消费或更新权限状态”。这产生两个结构性错配。第一个是 \textbf{issuer collapse}，即用户意图、Skill/manual instructions、tool metadata、工具结果和运行时输出被放进同一个 planning channel，使一类 context 可以替另一类 context 补齐 authority field。第二个是 \textbf{authority-state split}，即下游 guard 多数在 action materialized 后才检查 tool、arguments 或 resource，而一次性权限是否已被消费、是否过期、是否被委托给子任务、以及是谁铸造了权限，分散在不同状态中。两个错配共同说明 authorization object 选错了。系统需要检查的不是 action 本身，而是带 field ownership 和 lifecycle update 的 protected-decision transition。

现有防线分别覆盖了这个问题的一部分。Tool guards、argument validators 和 action-time authority controls 可以拒绝错误对象或非法参数~\cite{authgraph,pact,airguard}。Static allowlists 和 server-level policies 可以限制可见工具。Human approval gates 可以重新引入用户确认。OS sandbox 或 monitor 可以约束文件、网络和进程 side effects~\cite{actplane}。Provenance、taint 和 IFC-style defenses 可以表达部分 flow policy~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}。这些机制是必要的，但 agent runtime 默认把 field proofs、lease state 和 lifecycle update 分散在 prompt construction、tool gateway、approval UI、sandbox 和 subagent handoff 中。OS、sandbox 或 extension substrate 可以执行 local effects 的资源策略，但它们不决定哪个 issuer-owned field proof 可以改变 agent 的 authority state。EIM/bpftime 和 ActPlane 的细节留到相关工作讨论~\cite{eim-bpftime,actplane}。

\sys 的核心洞察是把这些分散对象显式化为一个 pre-effect protected-decision transition。这个 transition 同时提交 issuer-owned field proofs、checker-owned lease state、same-transition consume/delegate update 和 audit commit id。本文把这一 runtime-exposed linearization record 称为 commit object。本文比较的是 runtime 是否暴露这个 commit object，而不是某个 policy DSL 理论上能否写出相同布尔谓词。一个 DSL、IFC runtime 或 reference monitor 若暴露同一对象，就在覆盖的 fields 上实现了 \sys 的接口。

实现这一洞察需要解决三个工程挑战。第一，checker 必须分离不同 authority fields 的证明来源。Skill/manual text 可以指导 workflow，但不能替代用户意图。MCP/tool metadata 可以描述接口和 credential scope，但不能授权 sink。Script output、tool result 和文件状态可以提供 runtime evidence，但不能成为 instruction 或 approval reason。第二，lease table、budget counters 和 consumption state 必须属于 checker state \(\sigma\)，而不能被 env context 或 adapter 本地状态替代。第三，同一个授权对象必须出现在 tool call 之外的 prompt placement、Skill instruction placement、本地环境执行和 delegation handoff 边界，否则 untrusted context 仍可通过间接边界影响 authority。

\sys 使用 field-owned protected-decision lease 实现这个授权单位。Lease 绑定 operation、object、arguments、intent derivation、control/data provenance、temporal state、budget 和 delegation，并从四类输入中取证。Agent context 证明用户授权的目标、对象、sink、approval 和 delegation root。Instruction context 证明被 endorsement 的 workflow scope。Tool context 证明 callable、schema、credential scope 和 sandbox contract。Runtime-observation/env context 证明本次运行实际观察到的路径、stdout、tool result、文件状态和 side-effect evidence。\sys 的核心 system API 是 \(\mathsf{check\_and\_consume}\)。Adapter 提交的不是“请求是否允许”，而是一条 authority-state transition 的 commit record。LLM-assisted compiler 只提出 plans 和 candidate leases，不能铸造最终权限。Runtime adapters 将通过的 leases 落到 context construction、Skill instruction placement、tool/MCP policy events、本地环境执行、delegation handoff 和 OS-monitor-style replay lowering。在 PDF workflow 中，\texttt{create\_issue(repo=org/repo-x)} 必须先提交 agent-owned repo/approval、instruction-owned workflow、tool-owned schema 和 env-owned summary evidence，并在 checker 接受后消耗 one-shot lease。

原型评估目标是 instrumented protected-decision boundaries 上的 safety 和 enforceability。E1 显示 event model 没有排除 benign reference behavior，覆盖 3,813/3,813 个 tau2-style reference actions。E2 显示 issuer ownership 和 same-transition lifecycle update 不是命名偏好，因为 no-owner collapse 会错误接受 3,593/3,823 个完整 checker 拒绝的 protected decisions。E3 显示同一 commit API 可以放到 tool call 之外，并在本地 side-effect、prompt placement 和 delegation boundaries 上产生 0 个 unsafe executions/placements。因此，本文的安全 claim 是可证伪且有边界的。在已 instrumented protected-decision boundaries 内，checker 接受的事件必须满足 owner-field proof、active lease、intent derivation、provenance constraints 和同一次 transition 中的 lifecycle update。这个 claim 不覆盖未接入的 runtime boundary、LLM 内部因果归因、生产 sandbox 完整性、或所有 prompt injection 变体。

因此，本文贡献限定在已 instrumented protected-decision boundaries 上的模型、系统接口、多边界原型和有界证据：

\begin{enumerate}[leftmargin=*,label=\textbf{C\arabic*.}]
  \item \textbf{Model.} 一个 multi-issuer protected-decision transition model，定义 field owner、proof projection、no-substitution/no-promotion 和 lifecycle mutation（第~\ref{sec:formal} 节）。
  \item \textbf{Runtime contract.} 一个 \(\mathsf{check\_and\_consume}\) linearization contract，要求 adapter 在 effect、placement 或 handoff 前提交 field proofs、lease-state version、consume/delegate update 和 audit id（第~\ref{sec:design}--\ref{sec:formal} 节）。
  \item \textbf{System.} 一个 local authorization substrate，在 implemented local callable、env side-effect、prompt/Skill placement、delegation boundaries 和 OS-monitor-style replay lowering 上实现同一 contract（第~\ref{sec:implementation} 节）。
  \item \textbf{Evidence.} 一个 same-event ablation 和 local-boundary evaluation，证明 issuer collapse/lifecycle split 会导致 false accepts，并验证 commit contract 可执行（第~\ref{sec:evaluation}--\ref{sec:evidence-status} 节）。
\end{enumerate}

\section{背景与动机}

\subsection{Agent Extensions}

现代 agent runtime 通常不只暴露一个固定工具集合。Skills 将任务说明、参考资料和可选脚本打包为可复用工作流。MCP servers 将工具 schema、资源和 prompts 暴露给 agent。本地脚本和 shell 执行提供文件转换和项目操作能力，subagents 则把上下文和部分权限交给另一个模型或执行单元~\cite{skillguard,skillscope,mcp-spec}。因而，agent extension 同时包含自然语言指令、结构化 tool metadata、外部数据和可执行代码。这个组合使权限不能只绑定到 component identity：同一个 Skill、server 或 command 在不同 run 中会携带不同的过程、接口和运行时证据。

\subsection{Authority and Provenance}

传统权限系统主要围绕 subject、object 和 operation 建模，例如进程是否能读取文件、调用 API 或连接网络。Agent 场景还需要记录 provenance，因为同一个 tool call 的参数可能来自用户选择、扩展说明、工具结果或不可信文档。已有 provenance、taint 和 IFC-style 系统说明不可信输入不应自由流向高影响输出~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}，但它们通常把核心对象放在 observed action 或 data flow 上，而不是把未来 protected decision 本身作为可消耗、可衰减的授权对象。本文中的 protected decision 指会选择、扩大、消耗或转移 authority 的字段级决策，例如 sink selection、approval scope、capability request 或 delegation target。后续设计因此同时检查 provenance 和 lease lifecycle，而不是把 provenance label 当作完整授权。

\subsection{Motivating Workflow}
\label{sec:motivation}

本文使用一个文档处理工作流贯穿设计和评估。用户要求 agent 从两个选中的 PDF 中提取表格，保存 XLSX，并在指定 GitHub repository 中创建一个 issue。PDF text 可以影响表格单元格和 issue body，但不应影响 repository selection、approval scope、capability request 或 delegation target。这个例子把普通 data use 和 authority-bearing decision 分开，使问题不依赖某一种具体工具或 sandbox。

如果其中一个 PDF 包含隐藏指令要求 agent 请求 full GitHub access、加载另一个 Skill，或把 issue authority 交给 subagent，普通 argument check 只能看到最终调用是否仍然使用合法 repo。它无法表达 PDF text 不能成为 approval scope、fresh capability request 或 delegated capability set 的控制来源。\sys 的 running example 因此不是一个错误参数问题，而是一个 context-to-decision influence 问题。

即使没有恶意输入，同一个 Skill、MCP server 或 command 在不同 run 中也需要不同 authority。PDF Skill 用于本地 XLSX 转换时不需要网络或 GitHub issue lease。同一 Skill 用于生成 issue summary 时需要一次性 GitHub sink authority，但仍不应选择 repository。若权限只能在安装期、server 级别或静态 allowlist 中给出，系统通常只能在 union permission、频繁 broad approval 和 false denial 之间折中。这是一个运维和可用性问题，而不只是 prompt injection 问题。

后文只保留两张核心表来支撑这个例子：表~\ref{tab:boundary-owner-classes} 说明这些字段为什么形成不同 owner-equivalence classes，表~\ref{tab:decision-reqs} 说明每类 protected decision 需要哪些 proof projections。

\subsection{Protected-Decision Transitions}

受保护转移（protected-decision transition）指一次会 create、select、widen、spend 或 transfer authority 的可观察状态转移。Tool selection、sink selection、approval scope、capability request、policy update 和 delegation target 都属于这类转移。实现中把这些转移编码为 protected events。一个 guard 如果只检查最终 tool name 或 repo argument，会漏掉 untrusted context 对 approval scope、fresh authority request、policy patch 或 delegated capability set 的控制。因此，本文要保护的对象不是“某个字符串能否进入 prompt”，而是“某个 context source 是否被授权影响某类未来决策”。

三个术语在后文保持固定含义。Decision class 是受保护字段的类型，例如 sink selection、approval scope 或 delegation target。Protected event 是 instrumentation 在 runtime 看到的具体事件，例如一次 MCP 调用、approval request 或 subagent spawn。受保护转移是 checker 对 protected event 执行的原子状态更新，包括 proof check、lease consumption、temporal update 和 delegation-state update。这些 decision classes 定义了后续 threat model、lease language 和 evaluation oracle 的共同边界。

\section{威胁模型与非目标}

\sys 保护的是已经通过 adapter 暴露给 checker 的受保护决策。Agent 可以读取外部文档、加载 Skills、连接 MCP servers、运行本地脚本、调用工具，并将部分任务委托给 subagents。攻击者可以控制部分输入 context，例如 PDF、网页、邮件、issue comment、MCP tool result、unsigned or malicious Skill text、script output 或 subagent summary。Signed/trusted Skill instruction 只有在被 issuer endorsement 后才能作为 instruction context 指导声明范围内的 workflow。Installed-but-overbroad Skill package 也不能因此获得 sink、approval 或 delegation authority。

攻击者的目标不是单纯让模型输出某段文本，而是让不可信 context 控制 authority lifecycle event。典型目标包括选择错误 sink、请求更宽 approval、修改 capability request、复用已消耗 lease、把权限委托给错误 subagent，或把运行时输出提升为 workflow instruction。读取无关私有文件、创建错误 repository 的 issue、修改策略、跳过检查或委托给恶意组件，是这些 unauthorized influence 在具体边界上的表现形式。

TCB 包括 intent certificate issuer、deterministic checker、provenance tracker、audit log 和已经接入的 enforcement adapters。LLM、外部文档、工具结果、MCP annotations、unsigned Skill instructions、Skill/package comments、script static-analysis summaries 和自动 compiler proposals 都不在 TCB 内。Adapter 必须在 side effect、prompt authority placement 或 authority handoff 前提交结构化 field proofs。Checker 只相信 adapter 提交的结构化证明，不相信自然语言解释。

TCB 边界是 per-boundary 的。没有接入 adapter 的 side effect 不在安全性质覆盖范围内。OS sandbox 或 ActPlane-inspired monitor 可以作为本地 file/process/network enforcement 后端，但它们执行的是 checker 已经决定的 lease contract。\sys 的安全性质覆盖已建模的 decision classes 和已接入的 enforcement boundaries。Prompt injection 的语言层诱导、自动 compiler 的完整正确性、生产级 sandbox 替换，以及未 instrumented 的 agent 决策，属于这组性质之外的系统边界。

\section{系统设计}
\label{sec:design}

\subsection{系统概览}

\sys 的架构围绕一个 pre-effect checker transition 组织：untrusted compiler 提出 plan 和 candidate leases，trusted checker 消费 issuer-owned proofs 并返回 audit commit id。Agent 授权因此是副作用发生前的 multi-issuer commit protocol，而不是对已完成 action 的事后过滤器。LLM 可以读取任务、扩展说明和上下文，并提出 plan、effects 和 candidate leases，但它不能铸造最终权限。最终 authority 只来自 trusted intent certificate，并由 deterministic checker 根据 context provenance 和 global policy 验证。

图~\ref{fig:architecture} 展示了五个组件的边界。Trusted issuer 将用户请求、显式对象选择、sink 选择、管理员策略和 fresh approval canonicalize 成 intent certificate。Context labeler 为用户输入、文档、Skill instructions、MCP metadata、tool results、script outputs 和 subagent summaries 标注 origin、integrity、confidentiality 和 influence modes。

LLM-assisted compiler 从扩展元数据、tool schemas、local scripts 和当前 plan 中生成 security-relevant effect IR，并提出短期 leases。Compiler 不在 TCB 内，不能直接修改 authority state。Deterministic checker 验证每个 lease 和事件是否满足 intent derivation、argument constraints、control/data provenance、flow、temporal、budget 和 delegation rules。Adapter contract 要求 context construction、tool/MCP policy events、本地执行和 subagent delegation 在对应 protected transition 前提交证明。

在 PDF workflow 中，issuer 首先把用户选择的两个 PDF 和 \texttt{org/repo-x} sink 写入 intent certificate。Labeler 将 PDF bytes/text 标为 env context，将 PDF Skill 说明标为 instruction context，将 GitHub MCP schema 标为 tool context。Compiler 可以提出 extraction effects 和 \texttt{create\_issue} lease，但 checker 只会 mint/consume repo 由 agent context 授权、body 可由 PDF data 影响、且 budget 尚未耗尽的 lease。Tool/MCP、local env 和 delegation adapters 在副作用或 handoff 前调用同一个 checker 决策。

\subsection{设计目标}

Running example 中的失败轨迹对应四个设计需求。合法参数但非法 approval widening 需要 intent root 限定用户真正授权的对象和 sink。合法数据流但非法 authority field 需要 context no-promotion 和 influence separation。第一次合法 issue call 与第二次 consumed-lease reuse 拥有相同 action arguments，却需要 stateful lease lifecycle 才能区分。合法 tool guard 仍可能漏掉 prompt placement、Skill instruction placement 或 delegation handoff，因此需要 multi-boundary transition interface。

这四个目标直接对应引言中的两个根因。G1 和 G2 修复 issuer collapse：它们规定哪些 issuer 可以证明 intent、sink、workflow、interface 和 runtime evidence。G3 修复 authority-state split：它要求 lease lifecycle 在同一次 transition 中检查并更新。G4 则把这些 invariant 放到 tool call 之外的 prompt placement、Skill placement、env side effect 和 delegation handoff 边界。

Fail-closed denial feedback 和 adapter coverage 是实现这些目标的机制要求，而不是独立顶层目标。因此，设计目标包括四个可测试目标：

\begin{enumerate}[leftmargin=*,label=\textbf{G\arabic*.}]
  \item \textbf{Intent-bounded authority.} High-impact authority 必须来自当前用户意图、显式对象/目标选择、管理员策略或 fresh approval。
  \item \textbf{Context no-promotion and influence separation.} 系统必须区分 data dependency 和 control dependency，允许不可信 context 作为数据使用，同时阻止其补齐不属于自己输入类的 protected fields。
  \item \textbf{Atomic lease-state mutation.} Allow decision 必须和 lease consumption、expiry progress、holder scope 与 parent-child attenuation 的状态更新原子绑定，而不是分散在独立 guards 中。
  \item \textbf{Multi-boundary transition exposure.} Local callable、env side effects、context placement、Skill instruction placement 和 authority handoff 必须在 effect、placement 或 handoff 前暴露同一种 protected transition。
\end{enumerate}

图~\ref{fig:architecture} 的架构把 \sys 和普通 tool allowlist 区分开来。Allowlist 只决定哪个对象可见。\sys 还决定哪些 context source 可以影响哪个 protected decision。即使某个工具和参数在对象层面都匹配，checker 仍会拒绝 control provenance 来自无权影响该 decision class 的 context。

\sys 的系统边界因此不是一个单点拦截器，而是一个 checker 仲裁的状态转移接口。设计上，tool/MCP、本地环境、instruction placement 和 delegation 边界都在 side effect、prompt authority placement 或 authority transfer 前提交同一类 field proofs。OS/runtime monitor 可以强化本地环境边界的 file/process/network enforcement，但它执行的是 checker 已经决定的 lease contract，而不是替代 protected-decision authorization model。

\subsection{授权输入}

\sys 的 capability synthesis 不是从单一 prompt context 中抽取权限，而是从一组 proof-carrying issuer boundaries 中求解 lease。在本文的 agent-extension setting 中，这组边界实例化为四类授权输入类（authority-input classes）：agent、instruction、tool 和 runtime-observation/env。Agent context 证明用户授权的 goal、objects、sinks、approval tokens 和 delegation root。Instruction context 证明被 endorsement 的 workflow scope 和 procedure guidance。Tool context 证明 callable name、schema、credential scope、binary descriptor 和 declared side-effect contract。Env context 证明本次运行观察到的 concrete args、cwd、路径存在性、tool results、script outputs 和 observed side effects。

Env context 是必要的执行证据边界，不是 Unix environment variables。若只使用 agent intent、instruction/Skill/manual 和 tool/MCP interface 三类上层来源，checker 只能知道“用户想做什么”“过程如何描述”“接口如何声明”，却不能证明本次运行实际看到了哪些路径、stdout、tool result、文件状态、资源状态或外部数据。缺少 env 类，系统会退化成 prompt/tool policy。把 env 合并进 tool 或 instruction，则会让运行时文本继承接口可信度或过程可信度。

这四类的 novelty 不在名称，而在 safe-merge criterion。两个来源只有在 issuer、可伪造面、可证明的 authority field、观测边界和 lifecycle authority 都相同时才可以合并。同一段文本如果来自被 endorsement 的 \texttt{SKILL.md} 或手册，可能在 workflow scope 内成为 instruction context。如果它来自普通 PDF、网页、tool result、script output 或 subagent summary，即使内容像指令，也仍是 env context。Command-line tools 也不是第五类。Binary、argv schema 和 sandbox profile 属于 tool context，实际 cwd、输入文件、stdout/stderr 和文件系统变化属于 env context。

E2 把 safe-merge criterion 转化为可证伪的 minimality contract。四类不是一个关于 agent 世界的 ontology，而是本文接入边界中的四个不可代填证明问题：谁授权了当前目标和 sink，哪个过程说明被 endorsement，哪个接口或 command 可以被调用，以及本次运行实际观察到了什么。若把任意两个问题放进同一个 proof boundary，checker 就需要额外证明其中一个问题的答案不能被另一个问题的 issuer 伪造或代填。E2 正是按这个 contract 测试折叠：tool\(\rightarrow\)agent 让 interface metadata 供应 user authority，env\(\rightarrow\)agent 让 runtime text 供应 user selection，env\(\rightarrow\)tool 让 observed output 供应 interface proof，instruction\(\rightarrow\)agent 让 workflow text 供应 approval。

错误合并会产生不同的 false authority path。把 instruction 并入 agent 会把 Skill/manual 中的过程建议提升为用户意图或 approval。把 tool 并入 agent 会让 server/interface description 变成用户授权。把 env 并入 tool 会让 runtime observations、script outputs 和文档内容获得本不具备的接口可信度。把 tool 并入 env 又会丢失 schema、credential scope 和 declared side effect 这些接口约束。ABAC 问的是一组 attributes 是否满足 policy rule。\sys 问的是哪些 issuer-owned fields 可以共同提交一次 authority-state transition，并且禁止一类 issuer boundary 替另一类补齐缺失字段。

No-substitution rule 的作用是阻止一类 projection 替另一类 projection 证明 field。若把 instruction 和 tool 放进同一证明通道，Skill/manual 的过程文本就可能替代 MCP schema、command binary 或 credential scope。若把 env 和 tool 放进同一证明通道，运行时结果、stdout 和文件状态就可能继承工具接口的静态可信度。若把 instruction 或 tool 放进 agent-owned 证明通道，扩展说明或 server metadata 就可能替代用户选择、approval 或 delegation root。若一个实现内部仍保留 per-field owner projections、proof objects 和 same-transition lifecycle update，它就在这些 fields 上实现了本文所需的 commit interface，而不是本文反驳的 collapsed interface。

这四类也不是 component taxonomy。一个 Skill package 同时可能携带 instruction context（\texttt{SKILL.md} 和手册）、tool context（bundled script descriptor 和 argv schema）以及 env context（script stdout、生成文件和错误日志）。一个 MCP server 同时可能携带 tool context（schema、server identity 和 credential scope）以及 env context（tool result 和资源状态）。一个 agent run 同时持有 agent context（用户 intent、approval 和 delegation root）与 instruction context（用户给出的格式偏好或 workflow preference）。因此，\sys 不按“来源属于 Skill、MCP 还是 command”铸造 capability，而按每个 protected field 的 issuer owner 铸造 capability。文档手册和 Skill instruction 只约束过程和 prompt placement；MCP/tool/cmd metadata 约束接口、参数 schema、credential scope 和 sandbox lowering；runtime environment 证明本次执行实际观察到的路径、stdout、tool result 或 side effect；agent intent 给出目标、对象、sink、approval 和 delegation root。实际 lease 是这些 proof projections 对同一次 protected transition 的合取：少一类 required proof 就拒绝，任一类 proof 越权代填另一个 field 也拒绝。

实际 capability 必须由相关 protected fields、当前 intent、checker lifecycle state 与全局策略共同决定。令 \(C_q\) 表示 class \(q \in \{agent,inst,tool,env\}\) 的 canonicalized proof-projection set。四类 proof 的互斥和 \(C=C_{agent}\uplus C_{inst}\uplus C_{tool}\uplus C_{env}\) 只约束 canonicalized proof cells，不约束 raw artifact 或 component：同一个 Skill package、MCP server 或 agent run 可以被 adapter 拆成多个 proof cells，但每个 protected field proof 必须有唯一 issuer owner。未被某个 decision class 要求的输入类可以为空，但被要求的 field 必须从自己的 owner class 取证：

\[
K_d = Solve_\Gamma(I,\sigma,d,C_{agent}, C_{inst}, C_{tool}, C_{env}) .
\]

这里的 \(Solve\) 是合取约束而不是权限并集。\(I\) 给出当前 run 的 authority root，\(\sigma\) 给出 active/exhausted lease、budget、expiry 和 delegation graph，\(d\) 给出本次 protected decision 的 required fields。Agent context 给出目标、对象和 sink 的上界。Instruction context 只能在该上界内约束过程。Tool context 只能暴露可调用接口、凭证范围和 sandbox contract。Env context 只能把本次运行实际观测到的值绑定到这些 lease predicates。任一 required field 缺失都会使 lease 不可铸造。相反，若把四类 context 做成 union authority，最强或最会写自然语言的一类输入就可能替其他类补齐字段。

这个定义也给出 safe-merge test。任何实现若要合并两个 proof projections，都必须证明该合并不会让 workflow text 获得 user-owned sink、approval 或 delegation root，不会让 runtime stdout/tool result 继承静态 schema/credential trust，也不会丢失 callable schema 和 sandbox contract 这类接口约束。若该合并产生 \(Accept_h\Rightarrow Accept\) 的反例，它就不是等价 proof boundary。

纯 approval event、纯 context-use event 和本地 evidence event 可能只需要其中几类字段。关键规则是：对某个 protected decision 所需的字段，checker 必须从该字段指定的输入类读取，不能由其他输入类代填。例如，agent context 可以授权“处理这两个用户选中的 PDF 并在 \texttt{org/repo-x} 创建一个 issue”，但不能指定 PDF Skill 的脚本路径。Instruction context 可以指导 extraction workflow，但不能选择 GitHub sink。Tool context 可以说明 \texttt{github.create\_issue} 的参数结构和 credential scope，但不能把任意 repo 变成授权 sink。Env context 可以证明某个路径、参数值或 observed result 在运行时存在，但不能把存在性变成用户选择或 retroactively mint broader authority。

用户原始文本不是第五类 authority input。Trusted issuer 会将其拆分为 agent-owned intent fields 和 instruction-owned workflow preferences。被 canonicalize 成 goal、object、sink 或 approval 的部分进入 agent context。描述 workflow preference 或 formatting preference 的部分进入 instruction context。即使 instruction preference 来自用户，它也不能单独授权 sink、approval 或 delegation。Lease 的作用是把这些 required fields 固定为一个可检查、可消耗、可衰减的 transition object。

\subsection{意图证书}

Intent certificate 是结构化安全对象，而不是 prompt 中的一段自然语言。它记录用户目标、选择的对象、允许的 sinks、禁止的 sinks、approval tokens 和 expiry。对于 PDF 处理任务，certificate 会命名选中的 PDF 路径、输出目录、唯一授权的 GitHub repository、禁止任意网络访问的约束，以及创建 issue 所需的 fresh approval。LLM 可以提出 certificate fields，但 trusted parser、UI、administrator policy 或 approval step 必须 canonicalize 这些字段。Certificate 是后续 capability derivation 的根。

Certificate 的粒度决定后续 leases 的最大 authority。若用户只选择了两个 PDF，后续 file-read leases 只能覆盖这些对象或它们的安全派生物。若用户只确认了一个 GitHub repository，repo selection 就不能由 PDF text、Skill instruction 或 MCP result 改写。模糊字段不会被解释为宽权限。系统要么将其保守地标记为 unknown，要么要求 fresh approval。

\subsection{上下文影响权限}

每个 context cell 都带 provenance label，记录 origin、integrity、confidentiality、purpose、time 和 endorsements。\sys 的关键增加是为 context source 指定 influence modes。Influence authority 是 decision-specific 的：一个 source 可以被允许影响 output content，却不能影响 recipient、repository、network destination、approval scope 或 delegated capability set。Signed Skill instructions 可以在 declared workflow scope 内 instruct extraction procedure，但不能为自己请求额外工具。MCP tool results 和 annotations 必须按照 server trust 处理。MCP specification 已经提醒 tool behavior annotations 不能默认可信~\cite{mcp-spec}，\sys 将这一原则变成 deterministic provenance rule。

PDF text 可以流向 issue body 和 spreadsheet cells。普通 IFC 可以表达这条数据流是否允许。\sys 还要求 repository field 的 issuer 是 agent 输入类，approval scope 的 control proof 不含 PDF provenance，并且对应 one-shot lease 尚未被消耗。只有当 influence authority 与 intent derivation、budget、expiry 和 delegation bounds 绑定在同一 lease 上时，checker 才能防止 context 在一个字段上被允许、又在另一个生命周期状态中被复用为授权依据。这种 decision-specific authority 允许系统保留有用的数据流，同时切断同一 context 对 authority-bearing fields 的控制流。

实现会把抽象 influence mode 映射为 adapter 提交的 decision-mode 字段。表述上，influence mode 指抽象能力，decision class 指被影响字段，事件字段记录本次使用所需的最低影响权限。Quote、summarize 和 parameterize 对应 output body、spreadsheet cells 或 issue body 的数据使用。Plan 和 instruct 对应 tool selection 与 workflow-step choice。Authorize 对应 sink selection、approval scope、capability request 和 policy update。Delegate 对应 delegation target 和 child capability set。这个映射使论文中的抽象权限集合与实验 trace 一致，同时避免把所有 modes 都解释成同一种 taint level。

\subsection{Effect IR 与 Leases}

Agent plan 被编译成安全相关 effect graph。IR 不追求表达完整程序语义，而是捕捉会影响 authority 的事件：context use、tool call、MCP call、file read/write、subprocess execution、network connection、approval request 和 subagent delegation。每个 effect 记录 data provenance 和 control provenance。这个区分是 \sys 的关键：不可信 PDF bytes 可以是 output data 的 data dependency，但不应是 sink selection 的 control dependency。

Control provenance 不是声称系统能读出 LLM 内部注意力，而是一个 protected-field proof policy。对于 sink、approval scope、capability request、delegation set、policy patch、local executable、network destination 和 high-impact arguments，runtime adapter 必须提交结构化来源证明。字段可以来自 trusted UI canonicalization、fresh approval、tool registry/schema、runtime observer、context placement gateway，或某个 explicit context cell。模型自由生成字段本身不产生 proof，只能触发 adapter 查找对应结构化来源。

若字段由模型自由生成、缺少结构化 proof，或来源不能被 adapter 归因，checker 会保守处理。它把该字段的 control provenance 近似为当前 prompt 中所有可见、可能影响该字段的 non-policy context。只要其中存在缺少所需 influence mode 的 context，checker 就保守拒绝。普通 output body 或 spreadsheet cell 可以带 data provenance。Authority-bearing fields 必须带 control proof。这一策略会带来可用性代价，因此 E1/E3 把 false denial、proof completeness 和大规模 recovery 评估与 same-event safety 分开报告，而不是假设 provenance 总是精确。

Lease 绑定 operation、object、arguments、intent、influence、flow、temporal、budget、expiry 和 delegation。一个 PDF extraction lease 可以允许脚本读取两个选中文件并写入输出目录，但禁止网络、secrets 和额外 subprocess。GitHub issue lease 可以允许一次 \texttt{create\_issue(repo=org/repo-x)}，但要求 repo control provenance 只能来自 trusted user intent。Lease solving 的目标不是全局证明所有可能 plan 的最小权限，而是在给定 effect plan、extension catalog 和 global policy 下生成风险更低、可检查、可衰减的 authority set。

\subsection{Compiler 与 Checker}

\sys 使用 LLM-assisted 但 checker-verified 的 pipeline。Compiler frontend 总结 Skill metadata、MCP schemas、local scripts、observed traces 和 policy。它推导 plan 与 security-relevant effects，并为当前 run 提出 candidate leases。Frontend 不在 TCB 内。Deterministic checker 验证每个 lease 是否满足 global policy、intent certificate、provenance constraints、flow rules、temporal guards、budgets 和 delegation monotonicity。

Compiler 的输出必须携带 proof object，说明每个 protected field 来自哪些 intent fields、哪些 context sources 可以作为 data/control dependency、哪些 sources 被排除、以及 lease 在什么 temporal state 和 budget 下有效。Checker 不信任 proof 的自然语言解释，只检查其中的结构化 predicates。若 proof 缺失、argument predicate 过宽、control provenance 不明、或 lease 试图把 parent 不具备的 authority 委托给 child，checker 返回结构化 denial。

Denial 是 runtime refinement 的输入，不是绕过授权的通道。LLM 可以根据 denial reason 重新规划，例如选择更窄的 sink、补充 read-only proof-gathering step、或请求用户 fresh approval。但新 proposal 仍要经过同一个 checker。这一设计使 recovery 可以改善 utility，而不会把失败路径变成宽权限回退。

\subsection{运行时降低}

单一 tool gateway 不足以表达 \sys 的安全边界，因为 protected decision 可能发生在工具调用之前的 context construction、计划生成和 approval request 中，也可能发生在调用之后的 delegation 或 runtime binding 中。不同 boundary adapters 共享同一个 checker transition；差别在于它们为同一次 transition 提交哪些 agent、instruction、tool 和 runtime-observation/env projections，以及 checker state \(\sigma\) 在该 transition 中读取或更新哪些字段。表~\ref{tab:runtime-lowering} 给出这些边界。

所有 adapter 调用同一个状态转移接口。这个接口的语义不是普通 reference monitor 的“check before use”。Adapter 必须提交一个完整 commit record，说明本次 transition 要读取哪些 issuer-owned fields、消耗哪个 lease、把哪些 runtime observations 绑定到参数、以及成功后如何更新 checker-owned state。Checker 接受和 state update 是同一原子事件；若只先检查 predicate、再由另一个组件更新 budget 或 delegation graph，就重新引入 authority-state split。

形式上，接口为：

\[
\mathsf{check\_and\_consume}(e,lease\_id,field\_proofs,prov,\sigma)
\rightarrow \mathsf{allow}(\sigma',audit) \mid \mathsf{deny}(reason).
\]

\(e\) 已包含 operation、object 和 argument values。\(lease\_id\) 指向候选 active lease。\(field\_proofs\) 是 issuer-typed map。Agent proofs 绑定 intent、object/sink selection、approval 和 parent authority。Instruction proofs 绑定 workflow scope。Tool proofs 绑定 schema、credential scope、binary 和 sandbox contract。Env proofs 绑定 runtime observations、tool results、路径状态和 side-effect evidence。Adapter 可以不同，但它们不能本地修改 lease table、budget、expiry 或 delegation graph。若 checker 返回 deny，adapter 不执行 side effect、不写 prompt section、不创建 handoff。若 checker 返回 allow，\(\sigma'\) 已经包含 budget consumption、temporal progress、audit record 和 delegation-state update。

\section{形式化模型}
\label{sec:formal}

\sys 的形式化目标不是为任意自然语言推理建模，而是为上一节 checker pipeline 的输入和输出提供可检查语义。Compiler、gateway 和 runtime adapters 负责把模型计划、工具调用、MCP 调用、本地执行和 delegation 转换成安全相关事件。Checker 只对这些事件、active leases、intent certificate 和 provenance labels 做确定性判断。因此，模型刻意不讨论 LLM 内部是否“理解”指令，而是讨论一个被 instrumentation 暴露出来的 authority-bearing event 是否应被接受。

模型围绕三个 invariant 展开。第一，\textbf{no substitution}: 每个 protected field 必须由自己的输入类证明，其他输入类不能代填。第二，\textbf{atomic check/consume}: protected event 的 predicate、provenance、budget、temporal 和 delegation state 必须在同一次 transition 中检查并更新。第三，\textbf{no promotion}: env/tool/instruction context 不能仅凭自然语言内容升级为更高 authority class，除非 trusted issuer 产生显式 endorsement 或 fresh lease mint event。

\subsection{Events and Proof Projections}

一个 agent execution 被表示为事件 trace：

\[
\tau = e_1, e_2, \ldots, e_n .
\]

事件语言只覆盖会影响 authority 的边界：

\[
\begin{aligned}
e ::=~& ctx(c,m,d)
  \mid call(t,args)
  \mid mcp(s,t,args)
  \mid read(o)
  \mid write(o) \\
  & \mid exec(p,args)
  \mid send(sink,msg)
  \mid spawn(a,K)
  \mid approve(u,a) \\
  & \mid request(\kappa,reason)
  \mid policy\_update(patch)
  \mid lease(\kappa).
\end{aligned}
\]

每个事件带有两类 provenance：

\[
prov(e)=\langle prov_D(e), prov_C(e) \rangle ,
\]

其中 \(prov_D\) 表示数据依赖，\(prov_C\) 表示控制依赖。\sys 只在 protected decisions 上约束 \(prov_C\)，例如 tool selection、sink selection、approval scope、policy update 和 delegation target。这样可以表达同一段 PDF text 可以作为 issue body 的 data dependency，却不能作为 GitHub repository 参数的 control dependency。

在 running example 中，repo field 需要 agent-owned proof，而 issue body 只需要 data-use authority。形式化模型先给每个 context cell \(c\) 标注来源和 endorsement：

\[
\ell(c)=\langle origin, integrity, confidentiality, purpose, time, endorsements\rangle .
\]

标签进一步映射到 context authority。这里需要区分 influence mode 和 decision class。Mode 描述 context 的使用方式，例如 quote、summarize、parameterize、plan、instruct、delegate 和 authorize。Decision class 描述被影响的安全相关字段，例如 output content、sink selection、approval scope、capability request、policy update 和 delegation target：

\[
auth(c) \subseteq InfluenceModes \times DecisionClasses .
\]

例如，PDF text 可以 parameterize spreadsheet cells 并 summarize issue body，但不能 authorize capability request、policy update 或 GitHub repository selection。实现可以将 \(InfluenceModes\) 做成偏序或命名集合。论文中的性质只要求 checker 能判断某个 context 是否被授权以所需 mode 影响某个 decision class。

Intent-carrying lease 的抽象形式为：

\[
\kappa =
\langle holder, op, obj, args, I, influence, flow, temporal, budget, expiry, delegation\rangle .
\]

其中 \(I\) 是当前 intent certificate，\(influence\) 约束 control/data provenance，\(flow\) 约束 confidentiality 到 sink 的流向，\(temporal\) 和 \(budget\) 约束事件顺序与调用次数，\(delegation\) 约束可传递给 subagent 的能力集合。

形式上，lease synthesis 可被看作从 per-decision issuer-typed fields、当前 intent 与 checker state 中求解可授权 lease：

\[
\kappa \in Solve_\Gamma(I,\sigma,d,C_{agent}, C_{inst}, C_{tool}, C_{env}) .
\]

其中 \(I\) 是当前 intent certificate，\(d\) 是本次 protected decision class，\(\sigma\) 是 checker-owned lifecycle state。\(C_{agent}\) 包含 intent certificate、authorized object/sink selections、approvals、plan stage、agent role 和 parent authority。\(C_{inst}\) 包含 system/developer/user instructions、Skill/manual workflow text、MCP prompts 和 trust endorsements。\(C_{tool}\) 包含 MCP/tool schemas、server identity、credential scope、command/script execution descriptors 和 sandbox contracts。\(C_{env}\) 包含 runtime object existence、cwd/filesystem state、concrete arguments、observed results、runtime evidence 和 resource availability。Active/exhausted lease table、per-lease counters 和 delegation graph 属于 checker state \(\sigma\)，不属于任何可由环境输出填充的 context class。

\subsection{Issuer Ownership and Safe Merge}

Owner projections 只有在 issuer、可伪造面、可证明字段、观测边界和 lifecycle authority 完全相同时才可合并。Agent、instruction、tool 和 env 分别回答四个不可互换的证明问题：用户授权了什么，过程如何被允许约束，接口声明了什么，以及本次运行实际观察到了什么。

错误合并会让一个 issuer class 补齐另一个 class 才拥有的 field。若把 \(C_{inst}\) 合并进 \(C_{tool}\)，Skill/manual workflow text 就会和 callable schema、credential scope、binary descriptor 处在同一 issuer class，工具结果或 tool metadata 更容易被误解释为 procedure authority。若把 \(C_{inst}\) 合并进 \(C_{agent}\)，Skill instruction 就可能替用户选择 sink、扩大 approval scope 或 mint lease。若把 \(C_{env}\) 合并进 \(C_{tool}\)，runtime observation 会和静态接口描述混在一起，已执行 tool result、script output 或文件状态就可能替 schema 或 policy 补证明。划分的目的不是命名方便，而是让每个 authority field 只有一个可以签发它的 issuer boundary。

Checker 以四个 projection functions 读取这些输入类。这里的 projection 不是 parser convenience，而是 proof boundary：每个 projection 只返回该 issuer class 拥有的 fields。

\[
\begin{aligned}
intent(e) &\leftarrow \pi_{agent}(C_{agent}),\\
procedure(e) &\leftarrow \pi_{inst}(C_{inst}),\\
interface(e) &\leftarrow \pi_{tool}(C_{tool}),\\
evidence(e) &\leftarrow \pi_{env}(C_{env}) .
\end{aligned}
\]

令 \(F=F_A\uplus F_I\uplus F_T\uplus F_E\) 表示 authority fields 的不相交并集，并令
\[
owner:F\rightarrow\{agent,inst,tool,env\}
\]
给出每个 field 的唯一签发者。代表性 field 如下：
\begin{itemize}[leftmargin=*]
  \item \(F_A\)：\(goal\)、\(authorized\_object\)、\(authorized\_sink\)、\(approval\_scope\)、\(delegation\_root\)。
  \item \(F_I\)：\(workflow\_scope\)、\(procedure\_step\)、\(prompt\_section\)、\(formatting\_preference\)。
  \item \(F_T\)：\(callable\)、\(schema\)、\(credential\_scope\)、\(binary\_descriptor\)、\(sandbox\_contract\)。
  \item \(F_E\)：\(observed\_path\)、\(observed\_arg\)、\(tool\_result\)、\(stdout\)、\(file\_state\)、\(side\_effect\_evidence\)。
\end{itemize}
同一个自然语言片段可以被不同 adapter 观察到，但它能证明哪个 field 只由 \(owner(f)\) 和 endorsement state 决定。

一个 protected event \(e\) 只有在 \(Req(d)\) 要求的 projections 与同一个 lease \(\kappa\) 的 predicates 一致时才被接受。未被 \(Req(d)\) 要求的输入类可以为空，但 required field 不能从其他 context class 中补齐。例如 instruction 不能补齐 authorized sink，tool schema 不能补齐 user intent，env evidence 不能补齐 approval token。这个 typed field rule 使 \sys 不只是 MCP/tool-call guard，而是跨 Skill instruction、tool interface、runtime environment 和 agent workflow state 的统一 checker transition。

对应地，capability synthesis 可以写成一个 partial function，而不是最大权限选择：
\[
synth_\Gamma(d,\Phi,\sigma)=
\begin{cases}
\kappa & \text{if } \forall f\in Req(d).\; typed(\Phi(e,d),f,\kappa,\Gamma,\sigma)\\
\bot & \text{otherwise.}
\end{cases}
\]
其中 \(typed\) 同时检查 proof issuer、proof value、lease predicate、endorsement 和 lifecycle state。这个定义强调三点。第一，\(synth\) 不会把四类输入求并集后取最宽 authority。第二，缺失的 agent approval 不能由 instruction 或 tool metadata 自动补齐。第三，env observation 只能收窄或绑定已经授权的 predicate，例如把输出文件绑定到一个已授权 output prefix，而不能反向扩大 object set、sink set 或 approval scope。

也可以把“四类是否可以合并”写成 soundness question。令 \(h:\{agent,inst,tool,env\}\rightarrow Q'\) 是一个把若干输入类折叠到更少类别的映射，\(Accept_h(e)\) 是 checker 在折叠后只检查 \(h(owner(f))\) 是否匹配时的接受判定。一次折叠是安全的，必须满足：

\[
\forall e,\sigma,\kappa.\quad Accept_h(e,\sigma,\kappa) \Rightarrow Accept(e,\sigma,\kappa).
\]

这给出一个可测试的最小性命题。若存在事件 \(e\)、状态 \(\sigma\)、lease \(\kappa\) 和折叠映射 \(h\)，使得 \(Accept_h(e,\sigma,\kappa)\) 成立但 \(Accept(e,\sigma,\kappa)\) 不成立，则被 \(h\) 合并的输入类不是等价 proof boundary。这样的反例不需要改变 tool name、argument value 或 prompt slot，只需要把同一个 field proof 的 issuer class 从合法 owner 换成被合并后的其他来源。换言之，必要性不是由类别名称决定，而是由接口是否会产生 class-substitution false accept 决定。本文不声称这四类是所有 agent 系统的唯一 ontology；它声称在本文覆盖的 Skill/manual、MCP/tool/cmd、runtime-observation/env 和 delegation boundaries 中，四类是默认 owner equivalence classes，而任何进一步合并都必须显式保留 per-field owner projections 与 same-transition lifecycle update，并证明上述蕴含式，否则 E2 的 counterexample 就会给出 unsafe merge。

E2 的 counterexample suites 和 authority-input characterization evidence 就是在寻找该蕴含式的反例：同一 object、argument 或 prompt slot 保持不变，只替换 issuer class。若折叠后的 checker 接受而完整 checker 拒绝，说明被合并的 projections 拥有不同 authority fields，不能作为同一个 proof boundary。这个定义把 owner-projection claim 变成可测试的 false-accept 条件，而不是命名偏好。

更具体地，令 \(\Phi(e,d)\) 是事件 \(e\) 对 decision class \(d\) 提交的 field-proof set。每个证明项的形状为

\[
\phi=\langle f, issuer, value, source, prov\rangle .
\]

Checker 接受 \(e\) 的必要条件是：

\[
\forall f\in Req(d),\exists \phi\in\Phi(e,d):
\phi.f=f \land \phi.issuer=owner(f) \land match_\Gamma(\phi.value,\kappa.f).
\]

这个条件故意不是“存在一组可信 attributes 满足 policy”。它要求每个 required field 都由自己的 issuer class 证明，并且证明值与同一个 active lease 的 predicate 匹配。Env proof 可以证明某个 \texttt{/tmp/out.xlsx} 确实由本次脚本生成，但不能证明该路径是用户授权 output directory。Tool proof 可以证明 \texttt{pdftotext} 的 argv schema，但不能证明当前 PDF 是用户选择对象。Instruction proof 可以证明 Skill workflow 允许调用 extraction step，但不能证明 GitHub sink 或 approval scope。

不同 protected decision 对四类 context 的需求不同。令

\[
Req(d)=\langle A_d, I_d, T_d, E_d\rangle
\]

表示 decision class \(d\) 的 agent、instruction、tool 和 env field requirements。若 \(A_d\)、\(I_d\)、\(T_d\) 或 \(E_d\) 中任一 required field 缺失，checker 必须拒绝该 transition，而不能由其他 context class 代填。表~\ref{tab:decision-reqs} 给出 running example 中的代表性规则。

为了避免把四类输入退化成普通 ABAC attribute namespaces，\sys 将 required fields 视为 issuer-typed authority fields。令 \(owner(f)\in\{agent,inst,tool,env\}\) 表示 field \(f\) 的唯一授权 issuer class，令 \(proof(e,f)=\langle issuer,value,source\rangle\) 是 runtime adapter 为事件 \(e\) 提交的结构化证明。No-substitution typing rule 是：

\[
\frac{f \in Req(d) \quad proof(e,f)=\langle q,v,s\rangle \quad q = owner(f)}
     {\Gamma \vdash proof(e,f) : f}
\qquad
\frac{q \neq owner(f)}
     {\Gamma \nvdash \langle q,v,s\rangle : f}
\]

例如 \texttt{repo=org/repo-x} 的 sink authorization field 由 agent issuer 证明。Trusted GitHub tool metadata 可以证明 schema field，却不能证明该 repo 是用户授权 sink。于是，同一个 tool-issued proof 对 interface field 可以被接受：

\[
\Gamma \vdash \langle tool,\texttt{create\_issue.schema},mcp\_registry\rangle : github\_schema ,
\]

但对 agent-owned field 必须被拒绝：

\[
\Gamma \nvdash \langle tool,\texttt{org/repo-x},mcp\_metadata\rangle : authorized\_sink .
\]

同理，tool result 或 tool metadata 也不能证明 \(policy\_update\) field。这个 rule 的作用是给每个 protected field 建立唯一证明来源：checker 接受的不是一组满足谓词的可信 attributes，而是与 field owner、proof value 和 active lease predicate 同时匹配的 typed proof。

\subsection{Equivalence Boundary}

ABAC、IFC 或 policy-DSL system 若要在本文 decision classes 上达到同等语义，必须实现上一小节的 typed transition interface。区别不在于某个 DSL 是否能写出相似布尔谓词，而在于 runtime 是否把 issuer-owned field proofs 和 lifecycle updates 作为同一个 commit object 提交、检查和更新。

一个 stateful policy DSL、reference monitor 或 IFC runtime 若要在本文 decision classes 上与 \sys 等价，必须在同一个 pre-effect commit object 中暴露四个接口对象：\textbf{field-owner proof projections}，用于说明每个 required field 由哪个 issuer boundary 证明；\textbf{checker-owned lease state}，用于保存 active/exhausted leases、budget、expiry 和 parent-child graph；\textbf{same-transition consume/delegate update}，用于把 allow decision 与 budget consumption、expiry progress 和 child-state update 原子绑定；以及 \textbf{audit commit id}，用于把后续 side effect、prompt placement 或 handoff 绑定到 \(\mathsf{allow}(\sigma',audit)\) 的线性化点。满足这四个接口对象的系统在覆盖 fields 上实现的是 \sys commit interface；缺少任一对象的系统需要额外证明不会出现 cross-issuer substitution、stale reuse、double consume 或 over-delegation。E2 中的 typed-provenance state guard 正是一个 convergence-to-\sys example：它不是被当作“弱 baseline”击败，而是用来标出等价条件；当它继续加入 parent-child lease-set comparison、checker sole-writer state 和 same-transition child update 后，它就在该 residual class 上实现了同一 commit interface。

Context class 不是 raw artifact 的普通 attribute，而是 adapter canonicalize 后的 proof-projection owner。Checker 还维护一个 no-promotion invariant。令 \(class(c)\in\{agent,inst,tool,env\}\) 表示 proof cell \(c\) 的 canonical owner，令 \(endorse_q(c)\) 表示 trusted issuer 明确把 \(c\) endorsement 为 class \(q\)。若没有 endorsement，则：

\[
\begin{array}{ll}
c\in C_{env} \land \neg endorse_{inst}(c) & \Rightarrow c \notin C_{inst},\\
c\in C_{tool} \land \neg endorse_{agent}(c) & \Rightarrow c \notin C_{agent},\\
c\in C_{inst} \land \neg endorse_{agent}(c) & \Rightarrow c \notin C_{agent}.
\end{array}
\]

直观地说，tool result、script output 或 PDF text 不能因为包含祈使句就变成 instruction。Tool schema 不能因为声明“安全”就变成 user intent。Skill/manual instruction 不能因为要求更多权限就变成 approval。Promotion 只能由 trusted issuer、policy 或 fresh approval 显式产生，并作为新的 lease mint event 记录在 \(\sigma\) 中。No-promotion 还意味着任一 context class 都不能替另一个 class 补齐缺失的 authority field。Instruction 可以建议 workflow，但 binary/schema 必须由 tool context 验证。Tool context 可以定义接口，但 sink/intent 必须由 agent context 验证。Env context 可以提供 observed values，但不能把这些 values 转换成 approval 或 policy。

\subsection{Lease Lifecycle and Commit Semantics}

Lease lifecycle 是 \sys 与静态 policy rule 或单次 action provenance check 的主要区别。Agent workloads 中，同一个 action argument 可以在不同 authority state 下有不同安全含义：第一次 \texttt{create\_issue(repo=org/repo-x)} 可能由 fresh approval 授权。第二次相同参数可能是在复用已消耗的一次性 lease。同一 repo 参数交给 subagent 可能违反 delegation bound。同一 approval reason 若由 PDF text 控制，则可能变成 scope widening。Action-time provenance 或 object predicate 往往只看到 repo 值仍然匹配，无法区分这些 lifecycle violations。

形式上，可以构造两条 action/provenance 等价但 authority-state 不等价的 trace。Trace A 在 \(\kappa.budget=1\) 且 \(review\_complete\) 后执行
\(\texttt{create\_issue(repo=org/repo-x)}\)，并把 \(\kappa\) 消耗为 exhausted。Trace B 在同样的 tool、repo argument、intent proof 和 control provenance 下再次执行相同 call，或把同一 issue authority 交给未覆盖的 child capability set。Action guard、object predicate 和 ordinary provenance check 看到的是同一个 action shape。\sys 看到的是不同的 \(\sigma\)：A 有 active lease，B 只有 exhausted lease 或违反 \(K_c\sqsubseteq K_{parent(a)}\)。因此，被保护对象必须是读取并更新 \(\sigma\) 的 protected-decision transition，而不是 action tuple 本身。

Lease 因此是 future protected decision 的有状态授权对象，而不只是具体 operation 的 permission。它表达某个 context source 在 intent \(I\) 下可以影响 decision class \(d\)，直到某个状态转移使其被消耗、过期或衰减。Checker 将 lease 作为有状态对象处理，而不是把它当作一组静态 predicates。

Mint provenance、influence authority、consumption/expiration 和 delegation bound 必须在同一个 authorization object 上被检查并原子更新。若它们分散在独立 taint labels、ACL entries、counters 和 delegation records 中，系统就必须额外证明这些对象不会产生 stale authority reuse、cross-object widening 或 consumed-state bypass。Runtime state \(\sigma\) 至少包含 active/exhausted lease table、per-lease consumption counters、temporal automata state、mint proof records 和 parent-child delegation graph。Checker 是这些 lease-state fields 的唯一 writer。实现中的 checker state schema 直接保存这些字段，并在每个 accepted event 后原子更新。

Checker 暴露的 authority-changing interface 只有四类 transition：

\[
\begin{aligned}
&mint(\kappa), \quad check\_and\_consume(e,lease\_id,field\_proofs,prov,\sigma),\\
&attenuate(\kappa_{parent},K_c), \quad expire(\kappa).
\end{aligned}
\]

Runtime adapters 可以分布在 context、tool、MCP、local execution 和 delegation 边界，但它们不能本地修改 authority state。任何会产生外部副作用或改变 authority state 的 protected event，包括 tool call、MCP call、local command、approval request、lease mint、policy update 和 delegation，在副作用发生前都必须经过 central checker state 的 \(\mathsf{check\_and\_consume}\)。Accepted event、budget update、provenance record 和 delegation-state update 是同一个 transition：

Commit semantics 给出线性化规则。\(allow(\sigma',audit)\) 是唯一线性化点；adapter 不能缓存 allow 决策，也不能在 checker 外消耗或恢复 lease。每个外部 side effect、prompt placement 或 handoff 都必须绑定到返回 audit record 中的 commit id。并发 tool calls 或 parallel subagents 必须串行化到 checker，或使用等价的 compare-and-swap transition，使它们都以同一个 \(\sigma\) 版本为条件提交。这样 one-shot lease 不会被两个 adapter 同时消费，也不会在 handoff 和 tool execution 之间被复用。

\begin{itemize}[leftmargin=*]
  \item \textbf{Mint.} 只有 trusted issuer 可以通过 \(lease(\kappa)\) 事件把新 lease 加入 \(K\)。Mint rule 要求 \(\kappa.intent\) 可由 \(I\) 中的 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导。Untrusted context 不能成为 mint 或 widening 的 control dependency。
  \item \textbf{Check.} 每个 protected event 在 side effect 发生前必须引用一个 active lease \(\kappa \in K\)，且 operation、object、arguments、influence、flow 和 temporal predicates 同时满足。没有 active lease 的 event 即使 action arguments 看似可信也被拒绝。
  \item \textbf{Consume.} 若 \(\kappa.budget\) 是 one-shot 或 bounded count，accepted event 会递减 counter。Counter 为零时 \(\kappa\) 从 active set 中移除或进入 exhausted state。后续相同 tool/argument 的复用必须重新 mint。
  \item \textbf{Expire.} Temporal automata 可以在 task completion、approval window 结束、required predecessor missing 或 forbidden successor 出现时使 lease invalid。Expired lease 不再授权 event。
  \item \textbf{Attenuate.} Delegation event \(spawn(a,K_c)\) 只在 \(K_c \sqsubseteq K_{parent(a)}\) 时被接受，其中 \(K_{parent(a)}\) 是发起该 handoff 的 parent agent 在当前 \(\sigma\) 中仍 active 的 capability set。Child lease 的 objects、arguments、budgets、sinks、influence modes 和 delegation rights 均不得宽于 parent lease。
  \item \textbf{Deny amplification.} Skill instructions、MCP results、documents、script outputs 或 subagent summaries 可以作为 data/control provenance 参与普通 decisions，但除非被 trusted issuer endorsement，它们不能 mint fresh high-impact lease、扩大 existing lease，或把 exhausted/expired lease 恢复为 active。
\end{itemize}

这些规则也可以写成紧凑的状态转移。令 \(K_\sigma\) 表示 \(\sigma\) 中的 active leases，\(K_\sigma(p)\) 表示 parent \(p\) 在当前 \(\sigma\) 中持有的 active capability set，\(budget_\sigma(\kappa)\) 表示剩余调用次数，\(\sqsubseteq\) 表示 capability attenuation：

\[
\begin{array}{ll}
\textsc{Mint}:&
  \Gamma;I;\sigma \vdash trusted\_mint(\kappa)
  \Rightarrow \sigma[K_\sigma \leftarrow K_\sigma \cup \{\kappa\}] \\
\textsc{Check}:&
  \Gamma;I;K_\sigma;\sigma \vdash e:\kappa
  \land active_\sigma(\kappa)
  \Rightarrow \sigma_1 \\
\textsc{Consume}:&
  budget_{\sigma_1}(\kappa)=1
  \Rightarrow \sigma_1[K \leftarrow K_{\sigma_1}\setminus\{\kappa\}] \\
&
  budget_{\sigma_1}(\kappa)>1
  \Rightarrow \sigma_1[budget(\kappa)\leftarrow budget_{\sigma_1}(\kappa)-1] \\
\textsc{Expire}:&
  expired_\sigma(\kappa)
  \Rightarrow \sigma[K \leftarrow K_\sigma\setminus\{\kappa\}] \\
\textsc{Attenuate}:&
  spawn(a,K_c) \land parent(a)=p \land K_c \sqsubseteq K_{\sigma}(p)
  \Rightarrow \sigma[child(a)\leftarrow K_c] \\
\textsc{NoAmp}:&
  c \notin trusted\_issuers
  \Rightarrow \neg mint_c(\kappa') \land \neg widen_c(\kappa,\kappa').
\end{array}
\]

这些转移表达的不可约 invariant 是：authority state 不能通过 untrusted provenance 被创建、刷新、扩大或委托。若一个 provenance graph 或 stateful policy baseline 要表达同一性质，它必须引入等价的原子授权对象，同时覆盖 trusted mint、decision-specific influence、budget consumption、expiration 和 monotone delegation state。换言之，\sys 的结论不是这些检查项各自新颖，而是 agent protected decisions 需要一个把它们绑定为同一 transition 的授权单元。

一个直接的 lemma 是：若 trace 中每个 protected event 都通过上述 checker transition，则 no-substitution、no-promotion 和 atomic lifecycle 同时成立。Proof sketch 是逐事件归纳：\textsc{Check} 要求每个 required field 的 proof issuer 等于 \(owner(f)\)，\textsc{NoAmp} 和 endorsement guard 排除 context class promotion，\textsc{Consume}/\textsc{Expire}/\textsc{Attenuate} 在同一次 transition 中更新 \(\sigma\)。E2 的 ablations 反过来测试这些前提是否必要：去掉 owner check、promotion guard 或 atomic state update 时，构造的 class-substitution、stale reuse 或 over-delegation events 会被 false accept。

单个 argument check、taint label、one-shot counter 或 delegation rule 可以覆盖其中一部分，但它们必须在同一个 \(check\_and\_consume\) transition 中读取并更新。否则系统还需要额外证明多个 guard 之间不会出现 stale read、split-brain state 或 cross-object widening。例如 tool guard 看到 repo argument 合法后准备执行，而 budget counter 稍后才发现 one-shot lease 已耗尽。另一个例子是 provenance guard 允许 issue body 由 PDF text 控制，而 delegation guard 在另一份状态里仍看到 parent 拥有旧 issue authority。E2 把这些情况作为 ablation counterexamples 测试，正文形式化只保留它们共同依赖的状态转移规则。

Checker judgment is:

\[
\Gamma; I; K; \sigma \vdash e \Rightarrow \sigma' ,
\]

其中 \(\Gamma\) 是全局策略，\(I\) 是 intent certificate，\(K\) 是 active leases，\(\sigma\) 是 runtime state。事件被接受当且仅当存在 \(\kappa \in K\) 使得七个条件同时成立：\(1\) operation 匹配，\(2\) object 和 arguments 满足 lease predicate，\(3\) event purpose 可由 \(I\) 推导，\(4\) \(prov_C(e)\) 中的每个 context 都拥有影响该 decision class 所需的 authority，\(5\) \(prov_D(e)\) 满足 flow policy，\(6\) temporal guard 和 budget 尚未过期，\(7\) delegated leases 不宽于 parent leases。

对于 \(request(\kappa,reason)\) 和 \(policy\_update(patch)\)，checker 将其作为 protected decisions 处理。它们自身不能 mint authority，除非 trusted issuer 随后产生独立 \(lease(\kappa)\) event。它们的 control provenance 也必须拥有 \((authorize, capability\_request)\) 或 \((authorize, policy\_update)\)。

\subsection{Safety Properties}
一个 trace 被接受，表示其中每个 protected event 都按上述 judgment 顺序通过 checker，并且每个通过事件对 \(\sigma\) 的更新会消耗预算、推进 temporal state、记录 provenance，并更新 active leases。下面五个性质是这一 checker semantics 的直接安全后果。它们不是关于任意 prompt 的全局声明，而是关于 accepted traces over instrumented protected-decision events with adapter-supplied control-provenance proofs 的 trace-level properties。

\begin{enumerate}[leftmargin=*,label=\textbf{P\arabic*.}]
  \item \textbf{No issuer substitution.} 对任意 accepted protected event \(e\) 和其 decision class \(d\)，若 \(f \in Req(d)\)，则 checker 接受的 field proof 必须满足 \(proof(e,f).issuer=owner(f)\)，且 proof value 与同一个 active lease 的 predicate 匹配。其他 issuer class 提交的同值 proof 不能补齐该 field。特别是 tool metadata 不能证明 agent-owned sink 或 approval field，env evidence 不能证明 instruction-owned workflow field，Skill/manual instruction 不能证明 fresh approval。
  \item \textbf{No unauthorized influence.} 对任意被 instrumentation 暴露并被 checker 接受的 trace \(\tau\)，如果事件 \(e \in \tau\) 属于 protected decision class \(d\)，令 \(req(d)\) 是 \(d\) 所需的最低 influence mode，则对所有 \(c \in prov_C(e)\)，都有 \((req(d),d) \in auth(c)\)。换言之，未被授权以所需 mode 影响 \(d\) 的 context 不能成为该决策的控制依赖。
  \item \textbf{Intent-bounded authority.} 对任意 accepted high-impact event \(e\)，存在 active lease \(\kappa\)，且 \(\kappa\) 的 intent predicate 可由 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导。系统不会仅凭 Skill identity、MCP server identity 或工具描述铸造 high-impact authority。
  \item \textbf{Delegation monotonicity.} 对任意 accepted delegation event \(spawn(a,K_c)\)，child capability set \(K_c\) 是 parent active capability set 的子集或更窄 attenuation。除非 trusted issuer 基于新的 intent certificate 创建 fresh lease，subagent、Skill、MCP tool 或 script 都不能扩大自身权限。
  \item \textbf{No context-class promotion.} 对任意 accepted trace，若 context cell \(c\) 的原始 class 是 \(env\)、\(tool\) 或 \(inst\)，则 \(c\) 不能在后续 checker transition 中以更高 authority class 使用，除非 trusted issuer 产生显式 endorsement 或 fresh lease mint event。特别是，tool result、script output、PDF text 和 subagent summary 不能仅凭自然语言内容变成 instruction context。Tool schema 不能变成 user intent。Skill/manual instruction 不能变成 approval。
\end{enumerate}

这些性质是 trace-level safety properties。它们依赖于两个工程前提：runtime 必须把 protected decisions 接入 checker，并且 provenance tracker 必须保守地标记事件控制依赖。若某个 OS-level side effect 没有经过任何 adapter，\sys 无法单独观察它。这正是 sandbox 或 ActPlane-style backend 仍然有价值的原因。

\section{实现}
\label{sec:implementation}

我们的原型是一个 agent authorization runtime，用于验证三个边界性质：lease language 是否足以表达 intent/context authority，deterministic checker 是否能在统一 event model 上执行这些约束，以及这些约束能否落到 trace replay、live callable gateway、benchmark-derived workflows、context/delegation boundary experiments 和 env/OS-monitor-style replay backend。这里的系统贡献不是若干独立 adapter，而是一个统一的 pre-effect commit contract：任何 authority-changing boundary 都必须在副作用、prompt placement 或 handoff 前提交同一种 issuer-owned field-proof record，并由 checker 作为 lease-state sole writer 返回 audit commit id。原型实现 checker core、tool/live callable gateway、本地 env pre-side-effect gateway、live context-placement gateway、live delegation monitor、runtime binding 和结果分析脚本。OS-monitor-style replay backend 用于检查 local/env lease contract 能否被 lowered 为 monitor-style rules。生产级 MCP broker、prompt builder/subagent runtime、ActPlane integration 和 online user-simulator loop 是更强部署结论所需的接口。

\subsection{Checker Core}
Checker core 位于 \texttt{src/intentcap/}，实现 context labels、effect events、leases、checker state 和 proof objects。当前 checker 支持 exact、prefix、suffix 和 one-of argument predicates，支持 lease matching、holder scoping、control/data provenance checks、context-label influence-mode checks、temporal prerequisites、per-lease invocation budgets、proof-carrying intent/approval derivation，以及 delegated capability attenuation。这个 core 是本文的主要 TCB：LLM 输出、Skill 描述、MCP annotations 和 tool results 只能产生 candidate effects 或 candidate leases，不能绕过 checker 直接获得 authority。

Checker state 显式保存 active/exhausted lease table、budget counters、temporal prerequisites、holder identity、delegation parent-child relation 和 audit records。每次 protected event 进入 checker 时，runtime adapter 必须提交 event \(e\)（含 argument values）、\(lease\_id\)、\(field\_proofs\) 和 control/data provenance。Checker 在一个函数调用中完成 lease lookup、field-owner proof check、argument predicate check、budget/expiry check 和 attenuation check。若事件被接受，checker 同步写入 audit log 并更新 budget 或 child lease state。若被拒绝，它返回 structured denial，adapter 不执行 side effect、不写 prompt section、不创建 child handoff。

\subsection{Trace and Live Gateways}
\texttt{TraceGateway} 用同一个 checker replay benchmark-derived traces，并把每个事件记录为 execute/block decision。它用于把 AgentDojo、MCPTox、InjecAgent 和 tau2-bench benchmark labels 转换成统一的 protected-decision events~\cite{agentdojo,mcptox,injecagent,tau2bench}，从而比较不同 policy variants 的 false accept、authority breadth 和与人工标注 lease 的距离。

\texttt{LiveToolGateway} 包裹本地 Python callables。只有 checker 接受的 event 才会真正调用注册函数。被拒绝的事件在 side effect 发生前停止。这一区分很重要：trace replay 提供大规模 policy evidence，live gateway 验证同一 lease semantics 可以保护真实 callable boundary。

本地 env gateway 使用相同调用路径，但 block point 位于 side-effect handler 之前。File read/write、script execution、network-like event 和 context-use event 都先被转换成 protected event。Adapter 为每个事件提交路径、argv、cwd、observed file state、script output 或 tool result provenance。被拒绝事件不进入 handler，因此不会创建文件、修改 secret fixture 或写入 context section。这条路径是 E3 中 local env suite 和 Qwen3.6 proposer loop 的执行边界。

\texttt{LiveContextPlacementGateway} 和 \texttt{LiveDelegationMonitor} 将同一个 stateful checker session 放在 prompt-section placement 与 subagent handoff 之前。前者只有在 \texttt{ctx.use} event 被允许后才把内容写入目标 section。后者只有在 \texttt{subagent.spawn} event 的 delegated capabilities 被 lease attenuation 覆盖后才记录 child handoff。Skill/manual substitution test 进一步把同一个合法 instruction slot 固定为 signed Skill issuer，测试 tool result、script output 和 unsigned Skill text 是否能替代 instruction issuer。这些 adapters 支撑本地 boundary experiments。

\subsection{Compiler Frontend and Runtime Binding}
原型还包含 local llama.cpp/Qwen frontend，用来模拟不可信 lease compiler 和 action proposer。模型可以根据 task text、tool schemas、已有 tool-result previews 或 checker feedback 生成 candidate leases/actions。系统只把通过 checker 的结果交给 gateway。

对于 tau2-bench-style stateful workflows，runtime binding 会从已经执行的 tool-result evidence 中铸造 exact one-shot leases，并重新检查参数值是否出现在当前可见 task/tool-result context 中~\cite{tau2bench}。High-impact write candidates 还必须满足 structured value proof：例如列表参数需要逐项绑定到结构化上下文，不能从原始文本窗口或 sibling objects 中借用 authority。

除表中 contract surfaces 外，原型还包含 benchmark replay adapters、runtime binding、structured value proof、read/write activation、lease-label audit，以及 E1 security/utility baseline policy comparisons。这些组件共同验证同一 lease semantics 可以同时服务 trace replay、真实本地 callable/context/delegation/instruction-placement boundary 和 benchmark-derived protected-decision events。

原型还包含 local Qwen/llama.cpp compiler experiments 和本地 env backend experiments。Env backend 在隔离目录中真实执行 Python script、file read/write 和 context-use handlers，并在 side effect handler 之前调用 checker。被拒绝的 exec/read/write/net/ctx events 不会进入本地 side-effect handler。模型闭环版本使用 Qwen3.6/llama.cpp 作为不可信 action proposer。模型在 adversarial env-context prompt 下返回 call/abort，只有 call 事件会继续进入 checker 和本地 side-effect handler。

原型还实现了一个 OS-monitor-style replay backend：从 active env leases 生成 deny-by-default monitor rules，并在 replay monitor 中检查 operation、object、holder、argument predicates、budget consumption、control/data provenance 和 context-label influence rules。该 lowering 证明 lease contract 可以被编码为 env/OS monitor policy 形态。当前 monitor 运行在 Python replay runtime 中，且 provenance labels 由 IntentCap adapter 提供。

这些实现支撑机制性结论：在已接入 checker 的边界上，\sys 能拒绝 unauthorized context-to-decision influence，run-time exact leases 可以缩小 object/tool authority surface，checker 可以阻止 LLM-proposed broad/runtime authority 直接进入执行路径，context/delegation boundary 可以在 placement/handoff 前调用同一 checker，并且同一 env lease contract 可以被降低为包含 provenance/influence checks 的 monitor rules。生产级 sandbox、完整 MCP broker、独立 human expert-oracle replication 和大规模 recovery evaluation 属于扩展结论所需的实验条件。

\section{评估}
\label{sec:evaluation}

评估按 claim dependency 组织，而不是按脚本或 artifact 罗列。E1 先检查 protected-decision event model 是否仍能表达良性 reference actions，避免后续安全结果建立在不可用模型上。E2 测试本文的核心机制 claim，即 issuer ownership 和 atomic lifecycle update 是否是必要的 runtime invariant。E3 再验证同一个 transition object 能否放到 tool-call 之外的本地系统边界前。Supporting audit 只问 lease 是否是可机器审计的 authority object，并报告 author-adjudicated lease-distance 结果；它不承担独立 expert-oracle least-privilege 结论。

\subsection{评估方法}

Workloads 分为两组。公开 trace 来自 AgentDojo、MCPTox、InjecAgent 和 tau2-style artifacts~\cite{agentdojo,mcptox,injecagent,tau2bench}。本地 suites 覆盖 Skill/manual injection、command/script side effects 和 delegation workflows。这些 workloads 覆盖 agent、instruction、tool 和 env 四类输入，而不是只测试 MCP/tool-call 参数。

对包含模型调用的实验，我们固定底层模型、任务顺序、工具描述和 prompt budget。对 replay experiments，我们固定 event set、oracle 和 baseline policy，只改变授权检查层。High-impact denials 使用同一个 protected-decision oracle 判定。若某个 protected decision 的 control provenance 来自缺少相应 influence mode 的 context source，该事件应被拒绝。

Baseline policies 在同一事件模型上实现各自可判定的 predicates，用来近似 provenance、IFC 和 policy-style baselines 的可检查部分。它们不是 AuthGraph、PACT、AIRGuard 或其他完整系统的复现。Baselines 包括 vanilla/all-tool exposure、static ACL、exact-tool ACL、MCP/server allowlist、Skill manifest-style policy、tool-call guard、OS/sandbox-only policy、IFC/taint predicate、action-provenance predicate 和 split-state composite policy。Same-event comparison 表示不同策略在同一批 protected events 上重放，不代表 live replanning。Oracle sources 包括 benchmark security/utility labels、protected-decision labels、author-adjudicated lease labels 和 provenance-proof completeness records。

\subsection{E1: Event-Model Feasibility and Reference-Action Coverage}

E1 排除一个基本失败模式。如果 protected-decision event model 不能表达 benign reference behavior，那么后续安全结果即使为真也没有实用意义。为固定比较对象，实验把 3,746 个 benchmark-derived protected-event replay cases 输入 \sys 和 same-event policies，作为 unsafe-accept sanity check；同时在 3,813 个 benign reference actions 与 2,556 个 applicable tool-oracle checks 上评估良性动作是否仍能被表示和覆盖。Primary metric 是 reference-action coverage，unsafe accept 只是对 event extraction 和 replay policy 的 sanity check。E2 才承担 issuer/lifecycle 机制必要性的证明责任。

在 protected-event replay 中，\sys 在 3,746 个 protected-event cases 上产生 0 个 unsafe accepts。Object/tool ACL policies 接受许多 protected-decision violations，因为它们不区分 context influence authority。在 benign reference-action proxy 上，\sys 覆盖 3,813/3,813 个 reference actions，并通过 2,554/2,556 个 applicable tool-oracle checks。

E1 还包含一个已完成本地 Qwen3.6 matched slice，用来检查 live model-loop 下的 authority surface。该比较合并一个 leased exposure run 和一个 all-tools exposure run 的已有结果；它不运行新模型、不执行新工具、不同步数据集。在同一 18 个 tau2-style tasks 上，leased exposure 平均向模型暴露 2.61 个 tool schemas，而 all-tools exposure 暴露 14.72 个；对应 model calls 为 62 vs 74。Task-level utility 在这个切片上没有改善：action-reward 均为 8/18，tool-oracle 均为 0/18。差别主要出现在 checker-visible authority friction：gateway blocks 为 2 vs 18，off-lease blocks 为 1 vs 9。该切片只支持一个窄结论：较宽 tool visibility 会显著扩大可见 authority surface 和被 checker 拦截的越界尝试，但不能作为大规模 benchmark utility 或 recovery 结论。

\subsection{E2: Issuer-Collapse and Lifecycle-Split Ablation}

E2 是本文的机制必要性实验：它问移除 issuer ownership 或 same-transition lifecycle update 后，同一个 event/action/provenance record 是否会重新出现 unsafe accepting path。E2 的假设是：四类 authority-input fields 和 stateful lease lifecycle 共同形成一个不可拆开的 commit object。在本文测试的 no-owner 和 split-state variants 中，object/action guards、IFC/provenance guards 和 split-state counters 会 false-accept 某些事件，因为它们允许一类输入替另一类输入补齐 field，或把 budget/delegation/provenance 分散到不同状态中。因此，E2 是 issuer-collapse/lifecycle-split test，而不是完整自然频率证明，也不是声称所有 policy DSL 都无法表达相同 predicate。

比较对象包括两类自然工程捷径和一个强收敛基线。第一类去掉 issuer ownership，并把 \(C_{agent},C_{inst},C_{tool},C_{env}\) 折叠成 generic trusted context。第二类保留 predicates，但把 proof、budget、expiry 和 delegation state 分散检查。Typed-provenance state guard 则已经保留 source/field owner、approval、holder、temporal 和 budget predicates，用来回答最强反驳：若一个 policy DSL 继续加入同一 pre-effect transition 中的 parent-child lease comparison、checker sole-writer state 和 child-state update，它就不再是被 \sys 反驳的外部 baseline，而是在该 boundary 上实现了 \sys 的 commit interface。

E2 用两层证据把“四类输入是否只是 attribute namespaces”转化为 falsifiable false-accept question。Trace characterization 检查已有 artifacts 中的事件是否已经触发 agent、instruction、tool 和 env 的不同 proof boundary。Controlled counterexample replay 则固定 object、argument、prompt slot 或 action shape，只替换 issuer class 或 lifecycle state，测试 weakened variants 是否误接受。若折叠 issuer ownership 或拆开 lifecycle state 不改变结果，\sys 的核心机制就没有必要。

这些 counterexamples 不是任意编造的字符串。Tool-metadata promotions 来自 MCPTox-derived MCP tool/schema artifacts。Injected runtime/env cases 来自 AgentDojo、InjecAgent 和本地 env suites 的已有 provenance labels。Skill/manual substitutions 使用本地 Skill placement records。Workflow residuals 来自本地 workflow suite 的保存 traces。实验在这些 artifact-derived records 上固定 object、argument、action shape 或 prompt slot，只替换 issuer owner 或 lifecycle state；因此 false accept 归因到 commit object 被折叠，而不是 workload 改变。

Trace characterization 从已有 provenance labels 和 event modes 派生 issuer-class requirements，而不运行模型或同步新数据。结果显示，在 8,696 个已有事件中，8,691 个需要多个 issuer classes，198 个需要 env/runtime issuer，3,593 个 checker-denied events 存在 class-substitution attempt。Substitution edges 包括 tool \(\rightarrow\) agent 1,928 次、env \(\rightarrow\) agent 1,663 次、env \(\rightarrow\) tool 1,662 次，以及 instruction \(\rightarrow\) agent 2 次。Edge 计数不是互斥事件数，因为同一事件可能暴露多个 collapse risk。对应的 48 个 sampled protected-decision labels 把这些派生字段落到可复核的事件级标注任务上，其中 45 个样本是 protected-decision labels，43 个需要多个 issuer classes，13 个需要 env/runtime proof，37 个暴露 class-substitution attempts。

这些数字的解释是边界存在性，而不是自然流行率。MCPTox 中 tool metadata 试图影响 authority，InjecAgent 和 AgentDojo 中 env/tool-result 或 injected goal 试图影响 agent-owned decisions，本地 env suite 和 runtime binding 则需要 runtime observer 证明实际路径、输出和 tool-result values。Sampled labels 固定了后续 expert/adjudication protocol，但 labels 仍是 author-adjudicated labels，不是 blinded independent agreement。

\subsection{E3: Local Multi-Boundary Adapter Practicality}

\subsection{Support: Lease Auditability}

Supporting audit 检查 intent-carrying leases 是否能被机器审计为结构化 authority objects，包含 run-specific intent、influence-mode、budget 和 delegation fields。该 lease-audit corpus 包含 24 个 author-adjudicated lease labels，覆盖 agent、tool、env 和部分 instruction context。Audit 检查 schema validity、sampled label consistency，以及 object scope、influence modes、argument predicates、budget/expiry、delegation bounds 和 approval scope 上的 policy-distance invariants。

Audit 通过 24/24 个 labels，并对 144 个 IntentCap/baseline policy rows 计算 authority distance。这些 checks 说明 lease representation 可以支持 structured authority-distance analysis，并能在该 corpus 中发现 schema/sample/policy-distance mistakes。

该审计的结论限定在 author-adjudicated lease-audit result。它是 supporting evidence，不是主实验块。若要声称 \sys 比 symbolic policy baselines 更接近 expert minimal policy，还需要 blinded independent adjudication、inter-rater agreement 和统计化 authority-distance comparison。

\subsection{局限}
本节只支持一个结论，即 protected-decision transitions 可以在已 instrumented 的边界上被形式化并执行。结果覆盖 instrumented protected-decision events。端到端 utility、approval burden、independent expert replication 和生产级 sandbox integration 需要单独实验。

第一，workloads 仍以 saved traces、reference-action proxies、controlled counterexample suites、benchmark-derived counterexamples 和 local experiments 为主。Authority-input characterization 使用已有 artifacts 做 derived trace annotation。对应的 48 个 sampled protected-decision labels 已经提供 natural protected-decision events 和 author-adjudicated labels，但顶会级 prevalence claim 仍需要独立标注者复核 field owner 与 unsafe substitutes。MCPTox-derived benchmark cases 使用已有 artifacts 生成 24 个 authentic tool-metadata counterexample cases，但它不是自然发生频率统计。更大规模的 online user-simulator evaluation 是端到端 utility claim 的证据来源。

第二，辅助 lease audit 使用 author-adjudicated labels。这能检查 lease schema 和 authority-distance invariants。Blinded independent expert replication 才能支撑 expert-oracle least-privilege claim。

第三，本地 env backend 在隔离目录中验证 pre-side-effect blocking。OS-monitor-style replay backend 只验证同一 lease contract 可以降低成 monitor rule set。System-surface result 只是把这些本地边界结果汇总。生产级 ActPlane/kernel integration、真实 syscall mediation 和低开销 enforcement 仍需要单独工程与实验。

第四，本地 boundary tests 已经实现 live context-placement guard、Skill/manual instruction-source substitution test 和 live delegation monitor test。若要声称完整多边界 enforcement，还需要接入生产级 prompt builder、真实 subagent runtime、更多自然 Skill/manual injection workloads 和端到端 approval/recovery metrics。

这些限制限定外推范围。这些结果支持的性质是：instrumented protected-decision events 可以由 intent-derived, provenance-gated, consumable leases 管理，并且在这些边界上暴露 issuer/lifecycle collapse 的 false-accept 路径。完整 workload utility、approval burden、independent expert-oracle least privilege 和生产级 sandbox integration 需要分别报告，而不能由这些结果自动推出。

\section{相关工作}

\subsection{应用扩展隔离与底层 Enforcement}
传统扩展系统和 sandbox 关注扩展代码能访问哪些宿主资源。EIM/bpftime 将应用扩展表示为 host resources 和 capabilities，并强调安全、高效的 userspace extension execution~\cite{eim-bpftime}。ActPlane 将 agent policy lowering 到 OS-level enforcement，以覆盖 tool interception 看不到的 file、process 和 network side effects~\cite{actplane}。这些机制适合作为 \sys 的 enforcement backend 或系统对照，但它们不决定 policy 的来源。OS-level monitor 可以阻止 \texttt{net.connect(attacker.com)} 或未授权 file write，但仅靠 OS event 不能判断 \texttt{repo=org/repo-x} 是用户授权、tool metadata 建议，还是 PDF text 诱导。\sys 先生成带 issuer proof 的 lease contract。ActPlane-style backend 接收的是已经由 checker 认证的 env lease contract，并执行其中 file/process/network/local-env 子集；它不能生成 agent intent、Skill instruction、tool interface 或 env evidence 的 field proofs。\sys 的问题在更上层：某个 tool choice、sink selection、approval scope 或 delegation decision 是否由当前用户 intent 授权，并且是否被不可信 context 非法影响。

\subsection{Prompt Injection, Provenance, and Information Flow}
CaMeL、Prompt Flow Integrity、FIDES、RTBAS 和 NeuroTaint 等工作从 control/data flow、IFC、dependency screening 或 taint tracking 的角度约束 untrusted content~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}。AuthGraph、PACT 和 AIRGuard 进一步把 user intent、argument provenance 和 action-time authority control 引入 agent security~\cite{authgraph,pact,airguard}。这些工作是 \sys 最接近的思想背景。本文的新粒度不是单独的 intent/provenance authorization，而是会铸造、消耗、过期和委托的 protected-decision lease。

本文把差异放在授权对象的粒度和生命周期上。已有系统主要检查 observed action、argument 或 side effect 是否与 intent/provenance 对齐。\sys 则把未来决策权限表示为会被铸造、消耗、过期和委托的短期 lease。为此，\sys 把 mint 来源、influence mode、budget/expiry、no-promotion rule 和 parent-child delegation order 放进同一个授权对象。关键不是单独发明 provenance label、taint tracking 或 attenuable capability，而是把这些检查绑定为每次 protected decision 更新 authority state 的原子单位。

\subsection{Agent Policy Languages and Access Control}
Progent、PCAS、AgentSpec、AgentGuard 和 AgentBound 等工作提供 programmable privilege control、policy compilation、runtime DSL、ABAC 或 MCP access-control enforcement~\cite{progent,pcas,agentspec,agentguard,agentbound}。这些系统提供 policy/checker substrate，并回答“某个工具、参数、server 或 action 是否被 policy 允许”。\sys 选择不同授权对象：从当前 run 的 intent 和 context provenance 铸造短期 lease，并让 lease 随 protected decision 被消耗、过期或委托。因此，本文的主比较是 controlled protected-decision counterexamples 和 issuer/lifecycle ablations；lease audit 只检查 representation/auditability，不作为独立 expert-oracle least-privilege 结论。

\subsection{Skill 和 MCP 权限框架}
SkillGuard 和 SkillScope 直接面向 Agent Skills 的权限、行为规范和 least-privilege enforcement~\cite{skillguard,skillscope}。MCP 规范也强调 tool metadata 和 behavior annotations 不能默认被当作可信事实~\cite{mcp-spec}。这些工作说明 Skill/MCP ecosystem 已经需要权限边界。与 package-level manifest 或 server-level grant 不同，\sys 的 authority root 是当前 intent certificate：同一个 Skill 或 MCP server 在不同 run 中会得到不同、可消耗、不可自扩张的 leases。一个 lease 可以同时约束 Skill instruction 对 workflow 的影响、MCP result 对后续 sink 的影响、本地脚本的 file/network authority，以及 subagent 获得的 attenuated capability set。

更重要的区别是 field ownership。即使 Skill/MCP manifest 也按 run mint，如果 package 或 server 可以声明 sink、approval 或 delegation field，它仍然能替 agent issuer 证明 user-owned authority。例如用户只授权 \texttt{org/repo-x}，而 Skill manifest 声称 safe repo 是 \texttt{attacker/repo} 或需要 full GitHub scope。若 per-run manifest 把 package 声明当作 sink/scope owner，就会接受该扩大权限。\sys 拒绝这一路径，因为 sink 和 approval scope 是 agent-owned fields，不能由 Skill/MCP/tool/env 填充。这个定位把 \sys 放在 Skill/MCP permissions 之上，作为一个 intent-derived lease compiler/checker，而不是替代 Skill/MCP 的包级权限系统。

一个具体区别是 union authority。相同的 PDF Skill 若用于“只把两个选中文件转换为本地 XLSX”，\sys 只会给 selected-file read、本地 output write 和无网络 exec lease。若用于“提取表格并在指定 GitHub repo 创建 issue”，同一 Skill 仍不能选择 repo，但 workflow 会多出由 agent intent 和 GitHub tool context 共同证明的一次性 issue lease。Skill-level manifest 若要覆盖两个 run，通常必须声明二者的 union authority。\sys 则在每次 protected transition 上重新绑定 intent、Skill instruction、tool/cmd interface 和 env evidence。因此它不是 Skill permission 的子集，而是跨 Skill、MCP/cmd、env 和 subagent 的 run-time authorization layer。

\section{结论}

LLM agent extensions 改变了权限系统需要保护的对象。文档、工具结果和扩展说明不只携带数据，也可能影响 tool selection、sink selection、approval scope、capability request 和 delegation target。\sys 将这种影响权表示为 intent-derived、provenance-gated、可消耗、可过期、可衰减的 protected-decision leases，并用 deterministic checker 把 LLM proposals 保持在 TCB 之外。在已建模并接入 checker 的边界内，这些结果表明 issuer-owned field proofs、context provenance 和 lease lifecycle 可以作为同一个授权状态被检查和执行：event model 覆盖全部 reference actions，issuer/lifecycle ablations 暴露同事件 false accepts，多边界原型在本地 side-effect、placement 和 handoff 前执行同一个 commit contract。因此，agent extension security 的关键不只是限制工具能做什么，也包括限制不同 context 能证明什么、影响什么，以及何时消耗或转移 authority。
