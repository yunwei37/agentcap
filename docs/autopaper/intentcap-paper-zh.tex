\begin{abstract}
LLM agents 正在从固定工具调用器演变为可扩展的执行环境。它们可以加载 Skills，连接 MCP servers，运行本地脚本，并将任务委托给 subagents。现有权限机制通常约束扩展能调用哪些工具、访问哪些文件或连接哪些网络端点；但在 agent 中，文本本身会影响未来的权限决策。一个文档、工具结果或 Skill instruction 可能改变 agent 选择的工具、目标 sink、approval scope 或 delegation target。因此，agent extension 的核心安全问题不只是 resource access，而是：哪些 context 可以影响哪些 authority-bearing decisions。

本文提出 \sys，一种面向 LLM agent extensions 的 intent-carrying capability model。\sys 不声称 intent/provenance checking 本身是新的；它的核心抽象是把 agent 的未来决策权限表示为 run-time lease lifecycle。系统首先从 trusted user input、explicit object/sink selection、policy 或 fresh approval 中生成结构化 intent certificate；随后把 agent plan 编译为短期、可衰减、可审计的 capability leases。每个 lease 不仅约束 operation、object 和 arguments，还记录 authority 如何由 intent mint、如何被 checker consume、如何随 delegation attenuation，并约束 control/data provenance、允许的 influence modes、data-flow sinks、temporal order 和 budget。LLM 可以提出 plan 和 candidate leases，但不属于 trusted computing base；deterministic checker 负责验证 intent derivation 和 provenance constraints，并将通过的 leases 下发到 context constructor、tool gateway、MCP broker、本地执行 sandbox 和 delegation monitor。

\sys 的目标不是消除所有 prompt injection，而是在已建模的 decision classes 和 enforcement boundaries 内阻断未授权 context-to-decision influence。我们形式化三个安全性质：no unauthorized influence、intent-bounded authority 和 delegation monotonicity。评估围绕四个主实验展开：端到端攻击阻断与任务效用、run-time lease quality 与 authority reduction、context-influence mechanism ablation，以及 LLM-assisted compiler 在 deterministic checker 和 structured recovery 下的实用性。本文的中心结论是，保护 agent extensions 需要控制的不只是代码能做什么，还包括 context 被允许影响什么。
\end{abstract}

\section{引言}

LLM agents 的安全边界正在发生变化。传统应用扩展通常通过 API、插件接口或系统调用与宿主交互，因此权限系统可以围绕文件、网络、进程和工具调用来定义。现代 agent extension 的边界更模糊：Skills 可以提供长程工作流说明和脚本；MCP servers 可以暴露工具、资源和 prompts；本地程序可以转换文件并读取环境状态；subagents 可以接收上下文和被委托的能力。最终的 agent 行为不是单个模型调用加一个固定 API，而是由用户请求、系统策略、扩展说明、外部文档、工具结果和中间总结共同塑造的执行过程。

这种执行过程打破了 data 和 authority 的传统边界。Agent runtime 会把 trusted instructions、用户请求、扩展说明和 untrusted documents/tool results 放进相邻的模型上下文；模型再输出 plan、tool arguments、approval request 或 delegation decision；下游 guard 往往只看到最终操作，而看不到哪个 context 控制了安全相关字段。于是，一个 PDF 或网页虽然在类型上只是输入数据，却可能成为选择工具、选择 sink、扩大 approval scope、跳过检查步骤或委托权限的控制信号。本文称这一问题为 context privilege：context 获得了影响 protected decisions 的权力，而不仅是携带可被引用或总结的信息。

一个典型例子是文档处理工作流。用户要求 agent 从两个选中的 PDF 中提取表格，保存 XLSX，并在指定 GitHub repository 中创建一个 issue 总结结果。PDF 内容应当可以影响表格单元格和 issue body；但它不应影响 GitHub repository 的选择、approval scope、是否加载另一个 Skill、或委托给 subagent 的 capability set。固定 \texttt{repo=org/repo-x} 的 argument check 可以阻止一个错误仓库参数，但它仍没有表达“PDF text 不能成为 approval scope、delegation target 或 capability request 的控制来源”。如果 PDF 中包含隐藏指令要求 agent 请求 full GitHub access、加载另一个 Skill、或把无关文件摘要进 issue body，系统需要约束的是 context 到 decision 的控制流，而不只是最终 tool name 或单个参数值。

本文关注的 protected decisions 包括 tool selection、sink selection、approval scope、policy/capability request 和 delegation target；我们不声称追踪模型内部的任意语义影响。现有防线分别覆盖了这个问题的一部分。Tool guards 和 argument validators 可以拒绝错误对象或非法参数；static allowlists 和 server-level policies 可以限制可见工具；human approval gates 可以让用户确认高影响操作；OS sandbox 或 monitor 可以约束文件、网络和进程 side effects。这些机制是必要的，但它们本身不为 context 分配 influence authority：它们检查最终操作、可见资源或 side effect，而不是检查产生该决策的 control provenance 是否由当前用户 intent 授权。

\sys 从这一差异出发，但不把“检查 action provenance 是否可信”作为 novelty claim；这一方向已经被多种 agent provenance、IFC 和 intent-policy 系统探索。本文的核心观点更窄：agent extensions 需要一个以 run-time lease 为单位的授权生命周期，用来记录 authority 如何由当前 intent 产生、如何在 protected decision 前被检查和消耗、如何按 temporal/budget guard 缩小、以及如何在 delegation 时单调衰减。关键 invariant 不是 action 是否带有 provenance，而是 authority state 不能通过 untrusted provenance 被 mint、refresh、widen 或 delegate，即使最终 action arguments 单独看都合法。一个 high-impact event 只有在两个条件同时成立时才允许执行：它必须由当前 intent certificate 推导出来；并且它的 control provenance 必须来自被授权影响对应 decision class 的 context source。

\sys 用 intent-carrying capability leases 表达这一生命周期。Lease 是短期、可消耗、可衰减的授权对象，绑定 holder、operation、object、arguments、intent derivation、control/data provenance、flow、temporal state、budget 和 delegation。与只在 action time 比对参数来源的 guard 不同，\sys lease 先成为 future decision authority 的对象：它定义哪些 context 可以影响哪些安全相关字段、何时有效、可以被调用几次、能否传给 subagent、以及如何 lowering 到不同 extension boundary。因此，同一个 document 可以被授权 parameterize 输出数据，但不能 authorize 新工具、扩大 approval scope 或选择外部 sink。

这一设计带来两个实现挑战。第一，lease 不能由 LLM 单独决定，否则攻击 context 仍可能通过 prompt 影响授权本身。第二，系统不能只在工具调用口检查权限，因为 untrusted context 可能先影响 plan、prompt construction、delegation 或 approval request。\sys 因此采用 LLM-assisted compiler 与 deterministic checker 的结构：LLM 只提出 plan 和 candidate leases；checker 根据 intent certificate、context labels、effect IR 和 global policy 验证 proof-carrying leases；runtime adapters 再将 leases lowering 到 context constructor、tool gateway、MCP broker、本地执行 sandbox 和 delegation monitor。这个定位把 \sys 与参数级 provenance guard 区分开来：后者判断已观察到的 action 是否可信，\sys 还显式管理未来决策权限的 mint、check、consume、attenuate 和 delegate 过程。一个具体残差例子是一次性 approval lease：用户确认“在 \texttt{org/repo-x} 创建一个 issue”后，第一次 \texttt{create\_issue} 消耗该 lease；随后 PDF text 诱导模型复用同一合法 repo、请求 full GitHub scope，或把 issue authority 交给 subagent。Exact-argument ACL 可以检查 repo，action-provenance guard 可以检查某些参数来源，但除非重新引入类似 \sys 的 authority object，它们不会同时保证 consumed lease 不能被 untrusted context 重新 mint、widen、refresh 或 delegated beyond parent authority。

本文贡献如下：

\begin{enumerate}[leftmargin=*]
  \item 我们提出一个 stateful run-time lease calculus，将 protected decision authority 表示为 intent-derived lease，并定义 mint、check、consume、expire、attenuate 和 no-amplification transitions。
  \item 我们实现一个 proof-carrying compiler/checker/runtime prototype，使 LLM 生成的计划和权限提议保持在 TCB 之外，并把通过的 leases lowering 到 context、tool、MCP、本地执行和 delegation 边界。
  \item 我们给出 trace-level safety properties，并通过实证评估隔离 lifecycle state、no-amplification 和 delegation attenuation 何时拒绝 ACL、action-provenance 和 IFC-style baselines 会接受的攻击，同时报告当前 utility/recovery 边界。
\end{enumerate}

\section{威胁模型与目标}

\sys 面向可扩展 agent runtime。Agent 可以读取外部文档、加载 Skills、连接 MCP servers、运行本地脚本、调用工具，并将部分任务委托给 subagents。攻击者可以控制部分输入 context，例如 PDF、网页、邮件、issue comment、MCP tool result、Skill instruction、script output 或 subagent summary。攻击目标是让 agent 超出用户当前意图：读取无关私有文件、向错误 sink 发送数据、创建错误 repository 的 issue、请求更宽 approval、修改策略、跳过检查，或将权限委托给恶意组件。

我们不假设 LLM 会可靠遵守安全指令；不信任外部文档、工具结果、MCP annotations 或 unsigned Skill instructions；也不假设静态分析能完美理解任意脚本。\sys 依赖的 TCB 包括 intent certificate issuer、deterministic checker、provenance tracker、audit log 和 enforcement adapters。OS sandbox 或 ActPlane-style monitor 可以作为执行后端，但不是本文的核心贡献；\sys 关注的是上游 policy synthesis 和 context-authority checking。

设计目标包括：

\begin{itemize}[leftmargin=*]
  \item 将 high-impact authority 绑定到当前用户意图、显式选择、管理员策略或 fresh approval。
  \item 区分 data dependency 和 control dependency，允许不可信 context 作为数据使用，同时阻止其影响 protected decisions。
  \item 将权限铸造成短期、可衰减、可审计的 leases，而不是安装期或 server 级别的 ambient permission。
  \item 让 LLM 只提出计划和 lease 候选，不让其决定最终授权。
  \item 将同一 lease language lowering 到 context、tool、MCP、本地执行和 delegation 边界。
\end{itemize}

非目标同样重要。\sys 不声称解决 prompt injection，不声称自动 compiler 永远正确，不替代 sandbox，也不保证所有 agent 决策都能被完整追踪。它的安全性质只覆盖已建模的 decision classes 和已接入的 enforcement boundaries。

\section{系统设计}

\sys 的设计目标是把“模型想做什么”和“系统允许什么”分开。LLM 可以读取任务、扩展说明和上下文，并提出 plan、effects 和 candidate leases；但它不能铸造最终权限。最终 authority 只来自 trusted intent certificate，并由 deterministic checker 根据 context provenance 和 global policy 验证。

系统的数据流分成五步。首先，trusted issuer 将用户请求、显式对象选择、sink 选择、管理员策略和 fresh approval canonicalize 成 intent certificate。其次，context labeler 为用户输入、文档、Skill instructions、MCP metadata、tool results、script outputs 和 subagent summaries 标注 origin、integrity、confidentiality 和 influence modes。第三，LLM-assisted compiler 从扩展元数据、tool schemas、local scripts 和当前 plan 中生成 security-relevant effect IR，并提出短期 leases。第四，checker 验证每个 lease 和事件是否满足 intent derivation、argument constraints、control/data provenance、flow、temporal、budget 和 delegation rules。最后，runtime adapters 将通过的 leases lowering 到 context construction、tool/MCP invocation、本地执行和 subagent delegation 边界。

这个结构把 \sys 和普通 tool allowlist 区分开来。Allowlist 只决定哪个对象可见；\sys 还决定哪些 context source 可以影响哪个 protected decision。即使某个工具和参数在对象层面都匹配，checker 仍会拒绝 control provenance 来自无权影响该 decision class 的 context。

\subsection{Intent Certificates}

Intent certificate 是结构化安全对象，而不是 prompt 中的一段自然语言。它记录用户目标、选择的对象、允许的 sinks、禁止的 sinks、approval tokens 和 expiry。对于 PDF 处理任务，certificate 会命名选中的 PDF 路径、输出目录、唯一授权的 GitHub repository、禁止任意网络访问的约束，以及创建 issue 所需的 fresh approval。LLM 可以提出 certificate fields，但 trusted parser、UI、administrator policy 或 approval step 必须 canonicalize 这些字段。Certificate 是后续 capability derivation 的根。

Certificate 的粒度决定后续 leases 的最大 authority。若用户只选择了两个 PDF，后续 file-read leases 只能覆盖这些对象或它们的安全派生物；若用户只确认了一个 GitHub repository，repo selection 就不能由 PDF text、Skill instruction 或 MCP result 改写。模糊字段不会被解释为宽权限；系统要么将其保守地标记为 unknown，要么要求 fresh approval。

\subsection{Context Authority}

每个 context cell 都带 provenance label，记录 origin、integrity、confidentiality、purpose、time 和 endorsements。\sys 的关键增加是为 context source 指定 influence modes。PDF text 可以被允许 quote、summarize 或 parameterize spreadsheet cells，但禁止 plan、delegate 或 authorize。Signed Skill instructions 可以在其 declared workflow scope 内 instruct extraction procedure，但不能为自己请求额外工具。MCP tool results 和 annotations 必须按照 server trust 处理；MCP specification 已经提醒 tool behavior annotations 不能默认可信~\cite{mcp-spec}，\sys 将这一原则变成 deterministic provenance rule。

Influence modes 总是相对于 decision class 解释。一个 source 可以被允许影响 output content，却不能影响 recipient、repository、network destination、approval scope 或 delegated capability set。这种 decision-specific authority 是 \sys 的核心：它允许系统保留有用的数据流，同时切断同一 context 对 authority-bearing fields 的控制流。

\subsection{Effect IR and Leases}

Agent plan 被编译成安全相关 effect graph。IR 不追求表达完整程序语义，而是捕捉会影响 authority 的事件：context use、tool call、MCP call、file read/write、subprocess execution、network connection、approval request 和 subagent delegation。每个 effect 记录 data provenance 和 control provenance。这个区分是 \sys 的关键：不可信 PDF bytes 可以是 output data 的 data dependency，但不应是 sink selection 的 control dependency。

Lease 绑定 operation、object、arguments、intent、influence、flow、temporal、budget、expiry 和 delegation。一个 PDF extraction lease 可以允许脚本读取两个选中文件并写入输出目录，但禁止网络、secrets 和额外 subprocess；GitHub issue lease 可以允许一次 \texttt{create\_issue(repo=org/repo-x)}，但要求 repo control provenance 只能来自 trusted user intent。Lease solving 的目标不是全局证明所有可能 plan 的最小权限，而是在给定 effect plan、extension catalog 和 global policy 下生成风险更低、可检查、可衰减的 authority set。

\subsection{Compiler and Checker}

\sys 使用 LLM-assisted 但 checker-verified 的 pipeline。Compiler frontend 总结 Skill metadata、MCP schemas、local scripts、observed traces 和 policy；推导 plan 与 security-relevant effects；并为当前 run 提出 candidate leases。Frontend 不在 TCB 内。Deterministic checker 验证每个 lease 是否满足 global policy、intent certificate、provenance constraints、flow rules、temporal guards、budgets 和 delegation monotonicity。

Compiler 的输出必须携带 proof object，说明每个 protected field 来自哪些 intent fields、哪些 context sources 可以作为 data/control dependency、哪些 sources 被排除、以及 lease 在什么 temporal state 和 budget 下有效。Checker 不信任 proof 的自然语言解释，只检查其中的结构化 predicates。若 proof 缺失、argument predicate 过宽、control provenance 不明、或 lease 试图把 parent 不具备的 authority 委托给 child，checker 返回结构化 denial。

Denial 是 runtime refinement 的输入，而不是 authority escape hatch。LLM 可以根据 denial reason 重新规划，例如选择更窄的 sink、补充 read-only proof probe、或请求用户 fresh approval；但新 proposal 仍要经过同一个 checker。这一设计使 recovery 可以改善 utility，而不会把失败路径变成 broad permission fallback。

\subsection{Runtime Lowering}

Accepted leases 被分发到多个 enforcement boundaries。Context constructor 决定哪些 labeled context 可以进入 trusted instruction、untrusted data 或 policy feedback 区域。Tool gateway 只暴露 active leases 覆盖的 tools，并在调用前验证 arguments 和 control provenance。MCP broker 将 server-level access 收窄为 method-level leases，并把 server metadata 与 tool results 标为 server-dependent context。Execution sandbox 将 file、process、network 和 secrets constraints 下发到底层环境。Delegation monitor 确保 subagent 只接收 parent lease 的 attenuation。

这些 adapters 不需要共享同一种底层 sandbox；它们共享的是 lease language 和 checker verdict。若某个 boundary 尚未接入 adapter，\sys 不声称覆盖该 side effect。这个限制保留了 OS-level monitor 的价值，也使本文的贡献保持在 policy synthesis and checking，而不是新的 syscall monitor。

\section{形式化模型}

\sys 的形式化目标不是为任意自然语言推理建模，而是为上一节 checker pipeline 的输入和输出提供可检查语义。Compiler、gateway 和 runtime adapters 负责把模型计划、工具调用、MCP 调用、本地执行和 delegation 转换成安全相关事件；checker 只对这些事件、active leases、intent certificate 和 provenance labels 做确定性判断。因此，模型刻意不讨论 LLM 内部是否“理解”指令，而是讨论一个被 instrumentation 暴露出来的 authority-bearing event 是否应被接受。

一个 agent execution 被表示为事件 trace：

\[
\tau = e_1, e_2, \ldots, e_n .
\]

事件语言只覆盖会影响 authority 的边界：

\[
\begin{aligned}
e ::=~& ctx(c,m,d)
  \mid call(t,args)
  \mid mcp(s,t,args)
  \mid read(o)
  \mid write(o) \\
  & \mid exec(p,args)
  \mid send(sink,msg)
  \mid spawn(a,K)
  \mid approve(u,a)
  \mid lease(\kappa).
\end{aligned}
\]

每个事件带有两类 provenance：

\[
prov(e)=\langle prov_D(e), prov_C(e) \rangle ,
\]

其中 \(prov_D\) 表示数据依赖，\(prov_C\) 表示控制依赖。\sys 只在 protected decisions 上约束 \(prov_C\)，例如 tool selection、sink selection、approval scope、policy update 和 delegation target。这样可以表达同一段 PDF text 可以作为 issue body 的 data dependency，却不能作为 GitHub repository 参数的 control dependency。

每个 context cell \(c\) 具有标签：

\[
\ell(c)=\langle origin, integrity, confidentiality, purpose, time, endorsements\rangle .
\]

标签进一步映射到 context authority：

\[
auth(c) \subseteq Modes \times Decisions .
\]

例如，PDF text 可以拥有 \((parameterize, spreadsheet\_cells)\) 和 \((summarize, issue\_body)\)，但不拥有 \((authorize, capability\_request)\) 或 \((sink\_select, github\_repo)\)。实现可以将 \(Modes\) 做成偏序或命名集合；论文中的性质只要求 checker 能判断某个 context 是否被授权影响某个 decision class。

Intent-carrying lease 的抽象形式为：

\[
\kappa =
\langle holder, op, obj, args, I, influence, flow, temporal, budget, expiry, delegation\rangle .
\]

其中 \(I\) 是当前 intent certificate，\(influence\) 约束 control/data provenance，\(flow\) 约束 confidentiality 到 sink 的流向，\(temporal\) 和 \(budget\) 约束事件顺序与调用次数，\(delegation\) 约束可传递给 subagent 的能力集合。

Lease lifecycle 是 \sys 与静态 policy rule 或单次 action provenance check 的主要区别。Lease 是 future protected decision 的有状态授权对象，而不只是具体 operation 的 permission：它表达某个 context source 在 intent \(I\) 下可以影响 decision class \(d\)，直到某个状态转移使其被消耗、过期或衰减。Runtime state \(\sigma\) 至少包含 active/exhausted lease set、consumption counters、temporal automata、mint proof records 和 parent-child delegation relation。Checker 将 lease 作为有状态对象处理，而不是把它当作一组静态 predicates：

\begin{itemize}[leftmargin=*]
  \item \textbf{Mint.} 只有 trusted issuer 可以通过 \(lease(\kappa)\) 事件把新 lease 加入 \(K\)。Mint rule 要求 \(\kappa.intent\) 可由 \(I\) 中的 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导；untrusted context 不能成为 mint 或 widening 的 control dependency。
  \item \textbf{Check.} 每个 protected event 在 side effect 发生前必须引用一个 active lease \(\kappa \in K\)，且 operation、object、arguments、influence、flow 和 temporal predicates 同时满足。没有 active lease 的 event 即使 action arguments 看似可信也被拒绝。
  \item \textbf{Consume.} 若 \(\kappa.budget\) 是 one-shot 或 bounded count，accepted event 会递减 counter；counter 为零时 \(\kappa\) 从 active set 中移除或进入 exhausted state。后续相同 tool/argument 的复用必须重新 mint。
  \item \textbf{Expire.} Temporal automata 可以在 task completion、approval window 结束、required predecessor missing 或 forbidden successor 出现时使 lease invalid。Expired lease 不再授权 event。
  \item \textbf{Attenuate.} Delegation event \(spawn(a,K_c)\) 只在 \(K_c \sqsubseteq K_{parent}\) 时被接受；child lease 的 objects、arguments、budgets、sinks、influence modes 和 delegation rights 均不得宽于 parent lease。
  \item \textbf{Deny amplification.} Skill instructions、MCP results、documents、script outputs 或 subagent summaries 可以作为 data/control provenance 参与普通 decisions，但除非被 trusted issuer endorsement，它们不能 mint fresh high-impact lease、扩大 existing lease，或把 exhausted/expired lease 恢复为 active。
\end{itemize}

这些规则也可以写成紧凑的状态转移。令 \(K_\sigma\) 表示 \(\sigma\) 中的 active leases，\(budget_\sigma(\kappa)\) 表示剩余调用次数，\(\sqsubseteq\) 表示 capability attenuation：

\[
\begin{array}{ll}
\textsc{Mint}:&
  \Gamma;I;\sigma \vdash trusted\_mint(\kappa)
  \Rightarrow \sigma[K_\sigma \leftarrow K_\sigma \cup \{\kappa\}] \\
\textsc{Check}:&
  \Gamma;I;K_\sigma;\sigma \vdash e:\kappa
  \land active_\sigma(\kappa)
  \Rightarrow \sigma_1 \\
\textsc{Consume}:&
  budget_{\sigma_1}(\kappa)=1
  \Rightarrow \sigma_1[K \leftarrow K_{\sigma_1}\setminus\{\kappa\}] \\
&
  budget_{\sigma_1}(\kappa)>1
  \Rightarrow \sigma_1[budget(\kappa)\leftarrow budget_{\sigma_1}(\kappa)-1] \\
\textsc{Expire}:&
  expired_\sigma(\kappa)
  \Rightarrow \sigma[K \leftarrow K_\sigma\setminus\{\kappa\}] \\
\textsc{Attenuate}:&
  spawn(a,K_c) \land K_c \sqsubseteq K_\sigma
  \Rightarrow \sigma[child(a)\leftarrow K_c] \\
\textsc{NoAmp}:&
  c \notin trusted\_issuers
  \Rightarrow \neg mint_c(\kappa') \land \neg widen_c(\kappa,\kappa').
\end{array}
\]

这些转移表达的不可约 invariant 是：authority state 不能通过 untrusted provenance 被创建、刷新、扩大或委托；若一个 provenance graph 或 stateful policy baseline 要表达同一性质，它必须引入等价的 trusted mint、budget consumption、expiration 和 monotone delegation state。

Checker judgment 写作：

\[
\Gamma; I; K; \sigma \vdash e \Rightarrow \sigma' ,
\]

其中 \(\Gamma\) 是全局策略，\(I\) 是 intent certificate，\(K\) 是 active leases，\(\sigma\) 是 runtime state。事件被接受当且仅当存在 \(\kappa \in K\) 使得以下条件同时成立：operation 匹配；object 和 arguments 满足 lease predicate；event purpose 可由 \(I\) 推导；\(prov_C(e)\) 中的每个 context 都拥有影响该 decision class 所需的 authority；\(prov_D(e)\) 满足 flow policy；temporal guard 和 budget 尚未过期；delegated leases 不宽于 parent leases。

一个 trace 被接受，表示其中每个 protected event 都按上述 judgment 顺序通过 checker，并且每个通过事件对 \(\sigma\) 的更新会消耗预算、推进 temporal state、记录 provenance，并更新 active leases。下面三个性质是这一 checker semantics 的直接安全后果；它们不是关于任意 prompt 的全局声明，而是关于 accepted traces 的 trace-level properties。

\paragraph{Property 1: No Unauthorized Influence.}
对任意 accepted trace \(\tau\)，如果事件 \(e \in \tau\) 属于 protected decision class \(d\)，则对所有 \(c \in prov_C(e)\)，都有 \((mode_d,d) \in auth(c)\)。换言之，未被授权影响 \(d\) 的 context 不能成为该决策的控制依赖。

\paragraph{Property 2: Intent-Bounded Authority.}
对任意 accepted high-impact event \(e\)，存在 active lease \(\kappa\)，且 \(\kappa\) 的 intent predicate 可由 trusted user input、explicit object/sink selection、administrator policy 或 fresh approval 推导。系统不会仅凭 Skill identity、MCP server identity 或工具描述铸造 high-impact authority。

\paragraph{Property 3: Delegation Monotonicity.}
对任意 accepted delegation event \(spawn(a,K_c)\)，child capability set \(K_c\) 是 parent active capability set 的子集或更窄 attenuation。除非 trusted issuer 基于新的 intent certificate 创建 fresh lease，subagent、Skill、MCP tool 或 script 都不能扩大自身权限。

这些性质是 trace-level safety properties。它们依赖于两个工程前提：runtime 必须把 protected decisions 接入 checker，并且 provenance tracker 必须保守地标记事件控制依赖。若某个 OS-level side effect 没有经过任何 adapter，\sys 无法单独观察它；这正是 sandbox 或 ActPlane-style backend 仍然有价值的原因。

\section{实现}

我们的 prototype 是一个 Python-based agent authorization harness。它的目标不是替换生产级 MCP server、agent runtime 或 OS sandbox，而是验证三件事：lease language 是否足以表达 intent/context authority，deterministic checker 是否能在统一 event model 上执行这些约束，以及这些约束能否被 lowering 到 trace replay、live callable gateway 和 benchmark-derived workflows。

\paragraph{Checker and policy core.}
Checker core 位于 \texttt{src/intentcap/}，实现 context labels、effect events、leases、checker state 和 proof objects。当前 checker 支持 exact、prefix、suffix 和 one-of argument predicates，支持 lease matching、holder scoping、control/data provenance checks、context-label influence-mode checks、temporal prerequisites、per-lease invocation budgets、proof-carrying intent/approval derivation，以及 delegated capability attenuation。这个 core 是本文的主要 TCB：LLM 输出、Skill 描述、MCP annotations 和 tool results 只能产生 candidate effects 或 candidate leases，不能绕过 checker 直接获得 authority。

\paragraph{Trace and live gateways.}
\texttt{TraceGateway} 用同一个 checker replay benchmark-derived traces，并把每个事件记录为 execute/block decision。它用于把 AgentDojo、MCPTox、InjecAgent 和 tau2/tau3 reference artifacts 转换成统一的 protected-decision events，从而比较不同 policy variants 的 false accept、authority breadth 和 oracle distance。\texttt{LiveToolGateway} 则包裹本地 Python callables：只有 checker 接受的 event 才会真正调用注册函数；被拒绝的事件在 side effect 发生前停止。这一区分很重要：trace replay 提供大规模 policy evidence，live gateway 验证同一 lease semantics 可以保护真实 callable boundary。

\paragraph{Compiler probes and runtime binding.}
Prototype 还包含 local llama.cpp/Qwen frontend，用来模拟不可信 lease compiler 和 action proposer。模型可以根据 task text、tool schemas、已有 tool-result previews 或 checker feedback 生成 candidate leases/actions；系统只把通过 checker 的结果交给 gateway。对于 tau2/tau3-style stateful workflows，runtime binding 会从已经执行的 tool-result evidence 中铸造 exact one-shot leases，并重新检查参数值是否出现在当前可见 task/tool-result context 中。High-impact write candidates 还必须满足 structured value proof：例如列表参数需要逐项绑定到结构化上下文，不能从原始文本窗口或 sibling objects 中借用 authority。

\paragraph{Artifact boundary.}
当前 artifact 已经实现 checker/gateway、live local callable boundary、benchmark-derived replay adapters、local Qwen/llama.cpp compiler probes、runtime binding、structured value proof、read/write activation、expert-oracle labeling/scoring/audit pipeline，以及 E1 security/utility wrapper matrices。这些实现足以支撑机制性 claim：在已建模 traces 上，\sys 能拒绝 unauthorized context-to-decision influence；run-time exact leases 可以缩小 object/tool authority surface；checker 可以阻止 LLM-proposed broad/runtime authority 直接进入执行路径。它还不能单独支撑更强的端到端 claim：full MCP broker、生产级 execution sandbox、fresh user-simulator loop、独立 human expert-oracle replication 和 benchmark-scale utility/recovery evaluation 仍是评估完成前的 open items。

\section{评估设计}

本文评估围绕四个可证伪问题组织。Run IDs、pilot probes 和 residual analyses 只作为这些问题的证据来源，不作为论文主实验编号。评估的共同原则是固定底层模型、任务顺序、工具描述和 prompt budget，尽量只改变授权层；当某个 baseline 只有 trace-level 或概念性复现时，结果必须标记为 partial comparison，不能写成完整数值胜利。所有 high-impact denials 使用 protected-decision oracle 判定：若某个 protected decision 的 control provenance 来自缺少相应 influence mode 的 context source，则该事件应被拒绝。

\subsection{E1: End-to-End Security and Utility}

E1 检验 \sys 是否在真实 agent workflows 中降低攻击成功率，同时不把 benign task progress 降到不可用。Primary comparison 是同一模型和同一任务集下的 vanilla agent、static ACL、exact-tool ACL、MCP/server allowlist、tool-call guard、OS/sandbox-only wrapper 和完整 \sys。Workloads 覆盖 AgentDojo/InjecAgent-style document or tool-result injection、MCPTox-style MCP poisoning，以及 tau2/tau3 或 MCP utility slices。

主要指标是 attack success、wrong-sink rate、data exfiltration、approval/delegation violations、benign completion、false denial、recovery rate 和 tool errors。Oracle 来自 benchmark security/utility checks 与 \sys protected-decision labels。成功条件是：\sys 相比 access-control baselines 明显降低危险动作，并且 utility loss 有界、可由 denial/recovery 日志解释；若 utility collapse，论文只能主张 \sys 是安全 enforcement layer；若攻击仍通过，则必须定位到缺失的 provenance 或 enforcement boundary。

当前 security-side evidence 已经可以形成一张 same-event matrix：在已保存的 InjecAgent 和 MCPTox protected events 上，完整 \sys 检查 control provenance；exact/task ACL、server allowlist 和 broader static ACL 只检查对象或工具集合。初步结果显示，exact/task ACL 即使暴露的平均工具数量与 \sys 相同，仍会接受大量 poisoned-description 或 injected-context 控制的 protected decisions。因此 E1 的剩余风险不是“安全侧是否完全没有信号”，而是缺少同一模型和任务设置下的 online trajectory utility comparison。

当前 utility-side evidence 可以形成一张 reference-action proxy table：在 tau2/tau3 assistant reference actions 上，\sys 和静态 wrapper families 都覆盖全部 benign reference actions；\sys 的 exact reference replay 还能通过几乎全部适用的 tool oracle。这个结果说明窄 leases 并不会必然阻断 benchmark-labeled benign actions，但它还不是 same-model online utility comparison。Static wrapper rows 只说明它们不会挡 reference actions，同时也暴露了更宽 authority；它们不能证明真实模型在这些宽权限下仍然安全或高效。

第三块 E1 evidence 使用现有 local-Qwen trace 标注，而不是再跑一个模型。我们把 R035/R038 中 58 个 saved model-proposed calls 按 wrapper family 反事实 replay：\sys 的 saved gateway 和 exact lease 都接受 11/11 个 exact proposals、接受 0/47 个 off-reference proposals；task-reference tool ACL 会接受 22 个 same-tool/wrong-argument proposals；broad proposed-tool ACL 会接受全部 47 个 off-reference proposals，其中 25 个是 wrong 或 hallucinated tools。这个结果说明同一模型已经提出过的错误调用会被 object-only wrappers 放行，而 \sys 的 exact event lease 会拒绝它们；但它仍是 action-level saved replay，不测 denial 之后的重规划、approval burden 或完整 task success。

第四块 E1 evidence 使用已经真实执行过的 local-Qwen task-loop trajectories。我们比较同一固定 tau2 切片上的 all-tool exposure 和 leased-tool exposure：后者只向模型暴露 active leased tools，但仍使用 exact gateway 检查。这个 saved-online comparison 显示 leased exposure 改变了模型实际行为：exact executions 从 1 增到 6，gateway blocks 从 19 降到 6，wrong/hallucinated tool proposals 从 11 降到 0。负面结果同样重要：tool-oracle pass tasks 从 1 降到 0，action-reward pass tasks 也从 1 降到 0。因此 \sys 当前可以证明缩小工具可见面改善 proposal/gateway surface，但不能证明 tool hiding alone 足以保持端到端 utility。

\subsection{E2: Lease Quality and Authority Reduction}

E2 检验 \sys 生成的 run-time leases 是否比 static extension/server policies 更接近 least privilege。实验单位不是单个工具调用，而是一个 run 在完成任务时暴露的 authority surface。样本来自 security traces、MCP poisoning、stateful utility tasks 和 Skill-style workflows。理想 oracle 是 blinded expert leases：专家在不看 \sys 输出的情况下写 minimal leases，经 adjudication 后形成 reference policy。当前第一版结果使用 24 个 Codex-adjudicated labels 作为 project-author oracle；所有 labels 通过 schema validation，且没有 placeholder、missing sample 或 ambiguous sample。我们还加入了一个复现审计：逐条检查 InjecAgent、MCPTox 和 tau2/tau3 labels 的 operation、object、argument predicate、budget、forbidden authority、provenance constraints，以及 policy-distance scoring invariants。审计结果是 24/24 samples 通过、0 failed checks、0 missing/extra labels。这个结果可作为 E2 的第一版 audited evidence，但强 expert-oracle claim 仍需独立 human reviewer 复核。

这里不能直接把现有 trace 标注当作 expert lease oracle。Benchmark 标注通常说明攻击是否成功、reference action 是否应该出现、或模型是否调用了某个工具；它们很少标注最小授权对象本身，例如 context source 可影响的 decision class、argument predicate 的最小范围、temporal/budget guards、delegation bounds 或 approval scope。因此现有标注适合 E1/E3 的安全与 utility 判断，但 E2 的 least-privilege claim 仍需要独立 lease 标注。

比较对象包括 domain/global ACL、exact-tool ACL、MCP server allowlist、Skill manifest-style policy、Progent/PCAS/AgentSpec-style policy 和 \sys。指标包括 risk-weighted authority score、exposed tools/methods/sinks/paths/arguments、context-influence breadth、delegation breadth 和 oracle distance。在第一版 E2 scoring 中，三类 \sys policy family 的 oracle distance 均为 0，而所有非 \sys family 都有非零 distance：最近的 task/exact-tool policies 仍有 authority 或 provenance mismatch，global/server/catalog policies 的距离更大。机器审计确认 IntentCap rows 保持 zero-distance，所有 non-IntentCap rows 保持 positive-distance。成功条件是独立复核后 \sys 仍在保持授权 data use 的同时降低 authority breadth 和 oracle distance；若 symbolic policy baselines 同样窄，则论文应把 least-privilege claim 收窄为 proof-carrying checker operationalization。

\subsection{E3: Context-Influence Mechanism Ablation}

E3 隔离 \sys 的核心机制是否真的是 context influence capability，而不是普通 tool permission 或 taint/provenance tracking。实验比较完整 \sys、no intent certificate、no influence lattice、no control/data split、object-only checker、no-provenance checker，以及 AuthGraph/PACT/AIRGuard-style provenance-authority 和 IFC/taint-style labelers。Workloads 包括 document injection、MCP poisoning、malicious Skill/tool-result 和 residual workflow suites。

关键指标是 protected decisions 中 unauthorized context control 的接受率，以及每个 denial 是否只能由 lease semantics、temporal guards、budget、delegation 或 proof constraints 解释。成功条件是 ablations 或 access-control baselines 接受完整 \sys 拒绝的 protected-decision violations，并且 residual cases 不能被最强 provenance/taint baseline 完全解释。若最强 baseline 已解释所有 denials，论文必须放弃“新的 security boundary”表述，转而主张 \sys 是一种 proof-carrying lease compiler/checker organization。

\subsection{E4: Compiler, Checker, and Recovery Practicality}

E4 检验 LLM-assisted compiler 在 deterministic checker 下是否可用。这里的对象不是模型能力本身，而是一个安全 pipeline：LLM 生成 candidate plans/leases，checker 拒绝 broad/runtime policies 或 missing-proof writes，runtime 返回 structured denial，planner 再生成更窄的替代动作。比较对象包括 LLM-only accept、checker strict、checker+feedback、proof-only/rank-only/filter-only variants 和 checker+CEGAR planner。

指标包括 invalid leases accepted、valid leases rejected、broad/runtime proposals rejected、proof completeness、exact-next precision/recall、denial recovery、dangerous execute、tool errors 和 benchmark task reward。成功条件不是简单增加 executed calls，而是在不激活 broad/runtime authority、零 dangerous execute、零 tool error 的前提下，提高 task-correct executions 或 reward。若 recovery 仍停滞，论文应把 utility claim 收窄为 bounded safety/proof completeness，并将 planner/candidate generation 明确列为 artifact gap。

\section{证据状态与局限}

本节的作用不是提前报告最终结果，而是限定哪些 claim 已经有机制性证据，哪些必须等待 E1--E4 的主实验闭环。这样写的原因很直接：\sys 的核心风险不是机制讲不清，而是把 trace replay、oracle replay 或局部 compiler probes 误写成完整 end-to-end agent 结果。

\paragraph{当前可支撑的结论.}
已有 artifact 足以支撑一个保守的 system claim：\sys 可以作为 agent decision authorization layer，把 LLM-generated plans/leases 放在 TCB 之外，并在 checker-enforced boundaries 内拒绝违反 intent、argument、provenance、temporal、budget 或 delegation constraints 的事件。Saved benchmark traces 和 live callable gateway 证明同一 checker semantics 可以作用于 replay events 与真实本地调用边界；local Qwen/llama.cpp probes 证明模型输出可以作为 untrusted compiler input，由 checker 拒绝 broad/runtime authority 和 missing-proof writes；runtime binding 与 structured value proof 证明短期 exact leases 可以从已执行的 tool-result evidence 中安全铸造，而不是从模型自由文本中直接产生。

最清楚的当前安全结果是 same-event wrapper matrix：在 3,746 个保存的 protected events 上，\sys 接受 0 个 dangerous events；exact/task ACL 在同一事件集合上接受 2,149 个，toolkit/server allowlist 接受 4,283 个。这个结果说明 object/tool exposure 的窄并不等于 authority decision 的安全；缺少 control-provenance check 时，系统仍会让 untrusted context 决定 sink、tool 或 authorization-relevant fields。对应的 utility proxy 显示 \sys 覆盖 3,813/3,813 个 benign assistant reference actions，并在 direct reference replay 中通过 2,554/2,556 个适用 tool-oracle tasks；更宽的 static policies 同样覆盖 reference actions，但以大量额外 tool slots 为代价。Saved local-Qwen proposal replay 进一步显示，\sys 保留 11/11 个 exact proposals，同时拒绝 47/47 个模型实际提出过的 off-reference proposals；tool-only 或 broad wrappers 则会接受 22--47 个 off-reference proposals。Saved-online trajectory comparison 再说明 leased exposure 可以把 wrong/hallucinated proposals 从 11 降到 0，但这个小切片里 task utility 仍不提升。这个结论应写入“安全收益、authority cost 与 utility 边界”结果，而不是写成 run history。

\paragraph{尚不能支撑的结论.}
现有证据还不能写成“\sys 在完整 agent benchmarks 上保持接近 vanilla 的 utility”。Utility 结果仍主要来自固定 slices、reference-action accounting、saved proposal replay、saved-online trajectory comparison、局部 recovery 和 bounded activation；它们能说明机制边界和 failure modes，但不足以替代 fresh model/user-simulator loop。Least-privilege 结论已有第一版 adjudicated oracle、policy-distance scoring 和机器审计，但它仍是 single project-author adjudication with machine audit，不应被表述为双专家盲标结论；最终主张还需要独立复核。Context-influence residual 已经在 workflow suites 中出现，但若要主张它超越 provenance/taint baselines，E3 还必须在 benchmark-derived 或 model-loop setting 中复现这些 residual cases。

\paragraph{最终 claim gate.}
论文最终应按实验结果选择 claim 强度。如果 E1 同时显示低 attack success 和可接受 utility，\sys 可以主张端到端安全收益；如果 E1 的 utility 不足，则应把 \sys 写成 safety enforcement layer，并把 planner/recovery 作为 open engineering gap。如果 E2 的 expert-oracle scoring 显示 leases 明显更窄，才能主张 least-privilege synthesis；否则只能主张 proof-carrying checker operationalization。如果 E3 的 ablation 显示普通 ACL、provenance 或 IFC baselines 接受完整 \sys 拒绝的 protected-decision violations，才能强调 context influence capability 是新的安全边界。若 E4 不能提升 task-correct executions 或 reward，则 recovery 只能作为 diagnostic/refinement mechanism，而不是 utility-preservation result。

\section{相关工作}

\paragraph{应用扩展隔离与底层 enforcement.}
传统扩展系统和 sandbox 关注扩展代码能访问哪些宿主资源。EIM/bpftime 将应用扩展表示为 host resources 和 capabilities，并强调安全、高效的 userspace extension execution~\cite{eim-bpftime}。ActPlane 将 agent policy lowering 到 OS-level enforcement，以覆盖 tool interception 看不到的 file、process 和 network side effects~\cite{actplane}。这些机制适合作为 \sys 的 enforcement backend 或系统对照，但它们不决定 policy 的来源。\sys 的问题在更上层：某个 tool choice、sink selection、approval scope 或 delegation decision 是否由当前用户 intent 授权，并且是否被不可信 context 非法影响。

\paragraph{Prompt-injection, provenance, and information flow.}
CaMeL、Prompt Flow Integrity、FIDES、RTBAS 和 NeuroTaint 等工作从 control/data flow、IFC、dependency screening 或 taint tracking 的角度约束 untrusted content~\cite{camel,prompt-flow-integrity,fides,rtbas,neurotaint}。AuthGraph、PACT 和 AIRGuard 进一步把 user intent、argument provenance 和 action-time authority control 引入 agent security~\cite{authgraph,pact,airguard}。这些工作是 \sys 最接近的思想背景，也意味着本文不能声称“intent/provenance authorization”本身是新的。本文把差异放在授权对象的粒度和生命周期上：这些系统主要检查 observed action、argument 或 side effect 是否与 intent/provenance 对齐；\sys 额外把未来决策权限表示为短期 lease，显式记录 authority 如何由 intent mint、如何在 protected decision 前被 check/consume、如何按 temporal/budget guard 缩小、如何在 delegation 中 attenuation，以及如何 lowering 到 Skills、MCP tools、本地执行和 subagents。一个与 \sys 等价的系统必须同时维护四类状态：trusted mint provenance、per-lease consumption/expiration、protected-decision influence modes 和 parent-child capability partial order。现有系统分别覆盖其中若干子集；\sys 的 claim 是 agent-specific authorization granularity and lifecycle composition，而不是单独发明 provenance label、taint tracking 或 attenuable capability。换言之，provenance label 是 checker 的输入，lease 是可被铸造、消耗和委托的授权对象。

\paragraph{Agent policy languages and access control.}
Progent、PCAS、AgentSpec、AgentGuard 和 AgentBound 等工作提供 programmable privilege control、policy compilation、runtime DSL、ABAC 或 MCP access-control enforcement~\cite{progent,pcas,agentspec,agentguard,agentbound}。这些系统回答“某个工具、参数、server 或 action 是否被 policy 允许”。\sys 与它们的关系是互补和竞争并存：它同样需要确定性 checker，但 policy object 不是静态工具规则或 server policy，而是从当前 run 的 intent 和 context provenance 推导出的短期 leases。E2 和 E3 因此必须比较 \sys 与这些 policy families 在 authority breadth、oracle distance 和 protected-decision false accepts 上的差异。

\paragraph{Skill and MCP permission frameworks.}
SkillGuard 和 SkillScope 直接面向 Agent Skills 的权限、行为规范和 least-privilege enforcement~\cite{skillguard,skillscope}；MCP 规范也强调 tool metadata 和 behavior annotations 不能默认被当作可信事实~\cite{mcp-spec}。这些工作说明 Skill/MCP ecosystem 已经需要权限边界，本文不应被写成“第一个 agent permission system”。\sys 的范围是 run-centric 和 cross-extension 的：一个 lease 可以同时约束 Skill instruction 对 workflow 的影响、MCP result 对后续 sink 的影响、本地脚本的 file/network authority，以及 subagent 获得的 attenuated capability set。与 package-level manifest 或 server-level grant 不同，\sys 的 authority root 是当前 intent certificate；同一个 Skill 或 MCP server 在不同 run 中会得到不同、可消耗、不可自扩张的 leases。这个定位把 \sys 放在 Skill/MCP permissions 之上，作为一个 intent-derived lease compiler/checker，而不是替代 Skill/MCP 的包级权限系统。
