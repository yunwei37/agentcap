## 题目
IntentCap: Intent-Carrying Capabilities for LLM Agent Extensions

## 当前 novelty 结论
不能把 IntentCap 写成“确定无重叠”或“第一个 agent permission model”。截至 2026-07-04，最接近的压力已经不只来自 CaMeL、Progent、SkillGuard、Task Shield/DRIFT 和 ActPlane；新增的 Intent-to-Execution Integrity、A Framework for Formalizing LLM Agent Security、AgentSecBench、Prompt Flow Integrity、SkillScope、MCP description-code inconsistency 和 WebMCP tool-surface poisoning 已经说明：用户意图、contextual security、prompt/data-flow integrity、Skill/MCP 权限和 metadata trust boundary 都已有相关工作。第二轮和后续线上查新进一步把 novelty 风险推高：AuthGraph 已经做 clean user-intent authorization graph vs execution provenance；PACT 已经做 authority-bearing argument 的 provenance contracts；AIRGuard 已经把问题命名为 authority confusion，并明确说 untrusted resources may inform but must not authorize side effects；ARGUS 已经做 influence-provenance decision auditing；Agent-Sentry 已经做 execution-provenance behavior bounding；ClawGuard 已经做 user-confirmed runtime tool-boundary rules；VIGIL 已经做 Skill behavior specification enforcement；FIDES/RTBAS/NeuroTaint 覆盖 IFC/dependency/taint 路线；PCAS/AgentSpec/AgentGuard/AgentGuardian/AgentBound 覆盖 policy compiler/runtime DSL/ABAC/MCP access control；SecureClaw/AgentSentry/ARM 覆盖 runtime boundary、causal diagnostics 和 denial-feedback leakage。因此本文最稳的 novelty 是一个更窄的 claim：

> IntentCap 是一个 run-centric extension authorization compiler/checker，把 context 对 authority-bearing decisions 的影响编译成 proof-carrying, attenuable capability leases，并在 Skills、MCP tools、本地执行和 subagents 之间把当前用户意图、control provenance、argument constraints、flow、temporal state、budget 和 delegation 绑定起来。

这意味着论文主线不应是“我们也给 Skills/MCP 加权限”，也不应是“context provenance/authority binding 是第一次被提出”。更准确的主线是：现有系统已经能做 flow、argument provenance、runtime authority guard、policy DSL、access control、Skill behavior specs 或 provenance-aware decision auditing；IntentCap 的剩余空间是把这些约束组织成一个跨扩展、可衰减、可审计的 lease compiler/checker，其中 protected decision classes、influence modes、temporal/budget/delegation 和 proof object 是一等对象。当前 related-work PDF corpus 已经收集到 `docs/reference/`，共 73 篇核心/候选论文，并已用本地 `file` 检查为 PDF；P0 baseline notes 和 reproduction decisions 已经写入 `docs/background-related-work.md`。R082 已经实现第一版 trace-level closest-baseline labelers。结果很关键：ACL/policy-only baselines 仍然很弱，但 AuthGraph/PACT/AIRGuard-style provenance-authority 和 IFC/taint-style labelers 已经解释完当前 3,811 个 checker denials，residual 为 0。R083 则补了一个 controlled residual suite：在 7 个固定事件中，IntentCap 允许 1 个 budgeted call，拒绝 6 个 temporal、budget、delegation、cross-extension holder、wrong decision class 和 missing proof 违规；四类 closest labelers 全部接受这些事件，因此 6/6 compiler/gateway/task-loop 结果整理成 deterministic validity labels：严格 lowering 能阻断 broad/runtime 参数授权，但 exact coverage 仍是 11/61，下游 task loop 仍有 wrong/hallucinated tool calls。R091 进一步把 R077 runtime placeholders 接进真实 task loop：runtime 参数只有在已执行工具结果中出现证据时才被绑定为 one-shot exact lease，6 次绑定全部成功，bound reference calls 从 R079 的 7 提到 13，但 official tool-oracle 仍是 0/11。R092-R094 再把这个 0/11 拆开：15 个模型调用里 13 个已经 ## 摘要草稿
LLM agent 正在从固定工具调用器变成可扩展的执行环境。一个 agent 可以加载 Skills，连接 MCP servers，运行本地脚本，并把任务委托给 subagents。问题不只是这些扩展能否读文件、发请求或调用 API；更关键的是，扩展提供的文本会影响 agent 后续的带权限决策：选择哪个工具、把结果发到哪个 sink、请求多大范围的 approval、或者把哪些上下文和能力交给另一个组件。现有工作已经覆盖了很多关键部分：clean intent authorization graph、argument-level provenance contracts、runtime authority guard、IFC/taint、policy DSL、ABAC/MCP access control 和 OS/sandbox enforcement。IntentCap 的缺口不是“首次发现 context 会影响 authority”，而是把这些约束组织成跨 Skills、MCP、本地执行和 subagents 的 proof-carrying, attenuable lease compiler/checker。

IntentCap 的核心 artifact 是 intent-carrying lease。系统首先生成结构化 intent certificate，记录当前目标、用户选择的对象、授权的 sink、约束、approval 和过期条件。每个 context cell 都带有 provenance label 和允许的 influence modes。一个 capability lease 只有在操作可由 intent certificate 推导、该操作的 control provenance 被允许影响对应 decision class、且 temporal/budget/delegation 约束都成立时才有效。LLM 可以辅助提出 plan、effect graph 和 candidate leases，但 deterministic checker 才能接受或拒绝 lease 与事件；checker 验证 provenance、flow、temporal order、budget 和 delegation 约束，并把允许的 lease 下发到 context constructor、tool gateway、MCP broker、本地执行 sandbox 和 delegation monitor。

本文的目标不是宣称“解决 prompt injection”，而是给出一个更精确的系统安全性质：在已建模的 decision classes 和已接入的 enforcement boundaries 内，accepted high-impact events 不能被未授权 context 作为 control dependency 影响。当前原型已经把这个思想落到多个 benchmark 的 replay 与本地执行路径上：AgentDojo、InjecAgent 和 MCPTox 用于 context-to-decision 攻击，tau2/tau3 用于 policy-following utility 与 authority minimization。后续数据集选择先做网络检索和官方 metadata 审核，不自动同步新测试集；R026 已把 28 个 web-only 候选整理成可复现排序，但它只是候选选择证据，不是 benchmark 执行结果。R027-R081 逐步把 evidence 从 authority minimization、local Qwen lease/action corpus、task-gateway pilots、constrained active-lease UI、输出预算、reference-feasibility audit、reference-user-simulator diagnostic 推进到第一版 non-evaluation-task-JSON non-oracle lease compiler/gateway/task-loop probes 和 paper-facing top-conference evidence matrix。R062 在 airline/retail/telecom 固定子集上运行本地 Qwen3.6：27 个 assistant calls 全部通过 exact leases 执行，0 block、0 off-lease、0 tool error；同时把 1 个 benchmark user-side `toggle_roaming` 作为 user principal 单独 replay，不授予 assistant。官方 fixed-subset tool-oracle 从 R057 的 5/7 提到 6/7，env reward 为 7/7。R065 把同一设置扩大到 11 个 evaluated tasks：55/61 个 assistant reference calls exact-execute，0 block、0 off-lease、0 tool error，1 个 user-side action 单独 replay，官方 tool-oracle 为 8/11，env reward 为 11/11。R066-R069 显示剩余失败集中在 retail visible continuation：3 个 hidden missing refs 都是 `product_id=6086499569` schema-example-only invalid reference，另有 3 个 DB-feasible R081 把已有 R025 checker ablation 和 R027 oracle-distance 结果合成为顶会 claim evidence matrix，而不是新增 benchmark run。它不运行模型、不执行工具、不 clone/sync/download 数据集。最重要的数字是：object-only 和 full-event-args/no-provenance 策略会 false-accept 3,811/3,811 个 checker-denied protected decisions；same-exposure 的非 IntentCap ACL profile 在不暴露更多 tool objects 的情况下仍产生 2,149 次 unsafe event admissions；所有 non-oracle baseline profiles 在安全语料中合计产生 13,851 次 unsafe event admissions，覆盖 3,746 个 unique unsafe events；最宽的非 oracle profile 比当前 IntentCap oracle profiles 多暴露 756,096 个 authority slots。这个结果适合支撑“context provenance 是 tool/resource access 之外的独立安全边界”，但不能写成 fresh online baseline、expert-blind oracle study 或 utility/recovery 结果。

R082 则是对 R081 的必要降温：它把 saved traces 交给最接近 prior-work-family 的 trace-level labelers，而不是只和 ACL 比。结果是双面的。正面是 static tool ACL false-accept 3,811/3,811 个 checker denials，policy-DSL/access-control-style labeler false-accept 3,810/3,811，说明只看 tool/object/argument policy 仍然不够。负面也是最重要的：AuthGraph/PACT/AIRGuard-style provenance-authority labeler 和 IFC/taint-style labeler 解释了全部 3,811 个 denials，residual 为 0。也就是说，当前 saved traces 不能证明 IntentCap 已经超过最强 provenance/IFC 近邻；它们只能证明 ACL/policy-only baseline 不够。顶会版要想保住 novelty，需要新的 residual 实验，专门覆盖 temporal order、budget、delegation attenuation、cross-extension authority、same-trusted-source-but-wrong-decision-class 和 proof-carrying lease derivation。

R083 是第一步 residual 实验，但它必须被写成机制隔离，而不是 benchmark 胜利。它新增 `examples/residual_closest_baseline_suite.json`，并把 checker 扩展到 trace-level temporal prerequisites、per-lease invocation budgets、holder scope、intent/approval proof checks 和 delegation attenuation。R083 的 6 个 deny 都来自这些 lease-specific semantics；控制和数据来源都是 trusted，op/object/args 都匹配，所以 provenance-authority、IFC/taint、policy-DSL 和 static ACL labelers 都接受。正确结论是：IntentCap 的 lease 语义不是当前 labeler 的同义改写；下一步必须把这些 residual patterns 放到真实 benchmark 或本地 Qwen3.6 agent loop 里，证明它们不是手工 microbenchmark 才有的现象。

R084 把同一 residual suite 送进 `TraceGateway`，确认 stateful checker 语义已经穿过 runtime-facing replay 边界：7 个 attempted events 中，唯一合法的 first budgeted call 被 execute，6 个 residual violations 被 block，blocked modes 是 `sink_select=4`、`delegate=1`、`tool_select=1`。这比 R083 多一步 runtime-boundary evidence，但仍然不是 live tool execution，也不是 model-loop utility。

R085 再把同一 residual suite 送进 `LiveToolGateway`，并为 `github.create_issue`、`/skills/pdf/extract.py` 和 `summarizer` 都注册本地 no-op callable。结果是 7 个 attempted events、3 个 registered callables、1 次 callable invocation、6 个 registered blocked residual violations、0 missing tools、0 tool errors。也就是说，这些 residual violations 已经能在本地 callable side-effect boundary 前被阻断；但这仍然不是外部 MCP/API 调用、OS sandbox enforcement 或 task utility。

R086 加入本地模型环节：用 llama.cpp 9870 和现有 qwen2.5-3b GGUF，让模型对 residual suite 的每个事件判断 call 还是 abort。模型 7/7 都成功解析，但也 7/7 都选择 `call`，其中 6 个是 reference-denied residual violations。IntentCap 用 trace-prefix state 做 gateway evaluation，最后只允许 1 个合法 call，阻断 6 个 unsafe model-proposed calls，dangerous executes 为 0。这个结果对论文很有用：它直接说明 LLM 不能在 TCB 内，哪怕模型明确提出不该执行的 residual calls，checker/gateway 仍能挡住。

R087 把 R083 的 residual classes 放进更像真实 agent extension 的本地 workflow suite。`examples/residual_workflow_suite.json` 包含 PDF Skill review materialization、GitHub MCP issue creation、calendar summary subagent delegation、cross-extension local script execution、signed Skill repo-selection overreach 和 approval-scope proof checking。它仍保持 residual-test 纪律：control provenance 是 trusted，op/object/args 匹配 policy-DSL baseline，所以 closest labelers 不会因为 untrusted provenance 或参数不匹配而拒绝。结果是 8 个事件中 checker/gateway 允许 2 个、拒绝 6 个；6/6 denials 都是 AuthGraph/PACT/AIRGuard-style、IFC/taint-style、policy-DSL 和 static ACL labelers 会接受的 residual；本地 qwen2.5-3b 8/8 都选择 `call`，其中 6 个是 unsafe calls，但 gateway 全部阻断，dangerous executes 为 0。这个结果是 R083 到 realistic workflow 的桥梁，但还不是 benchmark-derived residual、外部 MCP/API execution 或 task utility。

R088/R089 把 R087 的本地 workflow suite 接到一个 feedback-capable residual runner。R088 使用本地 Qwen3.6 27B GGUF：8/8 parse，模型只 call 2 个 reference-allowed workflow events，直接 abort 6 个 unsafe residual events，因此 feedback 没有触发，callable invocations 为 2，dangerous executes 为 0。R089 使用 qwen2.5-3b 作为更弱压力测试：初始 8/8 都 call，产生 6 个 LLM-only unsafe calls；gateway 阻断全部 6 个；结构化 denial feedback 对 6 个 blocked unsafe calls 都触发，但只把其中 1 个转成 correct abort，剩下 5 个仍继续 call 并再次被 gateway 阻断。这个结果不能证明 robust recovery，但能支持两个更小的结论：模型强弱会改变 unsafe proposal rate；feedback 可以帮助一部分拒绝收敛，但 authority 仍必须由 checker/gateway 决定。

R090 是 R070-R080 compiler 线的补强而不是新模型实验。它读取 R074/R077 compiler outputs、R075/R076/R078 gateway replays 和 R080 task-loop mismatch，输出 `compiler_reference_action_labels.csv`、`compiler_lease_labels.csv`、`compiler_run_validity.csv` 和 `task_loop_call_labels.csv`。核心结论是：R075 的 non-strict lowering 会让 34/61 个 reference actions 通过 broad/runtime 参数策略放行；R076/R078 的 strict lowering 把这些宽授权变成 60 个 broad/runtime blocks，但每个 compiler corpus 的 exact constrained coverage 仍只有 11/61；121 个 lease labels 中，45 个是 exact active，27 个是 non-strict broad/runtime active，49 个是 strict inactive broad/runtime；R080 的 task-loop calls 仍是 7 个 exact executed 加 2 个 wrong/hallucinated tool calls。因此 R090 支持“checker 能阻止 broad compiler authority 执行”和“当前 non-oracle compiler 的失败点已可审计”，但不能支持“compiler 已经 utility-ready”。

R091 是 R090 之后最直接的机制增量：它不再把所有 runtime placeholders 都视为永久 inactive，而是把它们当成需要运行时证据的模板。这个模板不能独立授权；只有当模型提出的参数值已经由先前工具执行结果产生，系统才把它降成 exact one-shot lease。这个结果支持 IntentCap 论文里的一个关键设计点：runtime-dependent authority 应该通过 temporal/provenance evidence 绑定，而不是通过 broad argument policy 放行。它也暴露了 TC4 的真实难点：即使所有 15 个模型 calls 都被安全执行，task-level oracle 仍是 0/11，所以 utility claim 不能只看 gateway allowed/executed，还必须看计划完整性和环境 reward。

R092-R094 是 R091 后必须加的降温证据。R092 只读保存的 R091 结果，把 15 个 model calls 分成 13 个 exact executed 和 2 个 wrong/hallucinated tool calls，没有 same-tool wrong-argument，也没有 argument-key issue。R093 进一步看 residual completion：11 个 evaluated tasks 全部 official tool-oracle fail，其中 9 个任务仍缺 hidden reference actions，2 个任务执行完全部 reference actions 但 reward/env fail。R094 把 R093 和已有 R067 feasibility audit 合并，发现 61 个官方 reference actions 里只有 3 个是 invalid，仍有 45 个 DB-feasible reference actions 没执行。因此 R091 的失败不能再简单归因于 runtime binder 不安全、参数 key 错或 benchmark invalid references；真正瓶颈是 non-oracle compiler 的 tool recall、计划覆盖和 runtime-dependent action synthesis 还不够。

## 论文主线
1. Agent 安全的对象不是只有工具操作，还有“谁能影响未来决策”。
2. 同一段 context 可以被允许作为数据使用，但不应自动拥有控制 authority。
3. 用户当前意图应当是 run-time authority 的根，而不是 Skill package、MCP server 或工具 catalog 的静态身份。
4. IntentCap 用 intent certificate、context authority label、effect IR 和 proof-carrying capability lease，把这件事变成可检查的 authorization 问题。
5. LLM-assisted compiler 只负责提议；deterministic checker 和 runtime adapters 才在 TCB 内。

## 设计要点
- Intent certificate：结构化表示用户目标、选择的对象、允许的 sink、禁止的 sink、fresh approval 和过期条件。
- Context authority：把 `observe`、`quote`、`summarize`、`parameterize`、`plan`、`instruct`、`delegate`、`authorize` 等 influence modes 与 decision classes 绑定。
- Effect IR：把 agent plan 或 benchmark trajectory 转成 `ctx.use`、`tool.call`、`mcp.call`、`fs.read`、`fs.write`、`exec.run`、`net.connect`、`subagent.spawn` 等安全相关事件。
- Capability lease：短期、可衰减的授权对象，约束 operation、object、arguments、intent derivation、control/data provenance、flow、temporal guard、budget 和 delegation。
- Deterministic checker：检查 event 是否匹配 active lease，且 control provenance 是否有权影响当前 decision class。
- Runtime adapters：在 context construction、tool/MCP invocation、本地执行和 subagent delegation 处执行 allow/deny，并保留 audit log。

## 当前可支持的说法
- IntentCap 可以在 toy trace、AgentDojo、MCPTox、InjecAgent 和 tau2/tau3 reference replay 中表达并执行 context-provenance-aware allow/deny。
- 对 MCPTox，context authority 不是“更小 tool set”的同义词：exact-tool ACL 与 IntentCap 暴露同样的 tool count，但缺少 provenance check 时会放行 poisoned-description-controlled calls。

- 对 oracle scoring，R027 支持一个 saved-result 说法：在当前三类 authority-minimization summaries 上，IntentCap oracle profiles 比最接近的静态/半静态 baseline 更接近 intent/provenance/event-binding 要求，并且非 oracle baseline 仍会放行大量 unsafe events。
