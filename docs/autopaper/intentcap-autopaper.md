## 题目
IntentCap: Intent-Carrying Capabilities for LLM Agent Extensions

## 当前 novelty 结论
不能把 IntentCap 写成“确定无重叠”或“第一个 agent permission model”。截至 2026-07-04，最接近的压力已经不只来自 CaMeL、Progent、SkillGuard、Task Shield/DRIFT 和 ActPlane；新增的 Intent-to-Execution Integrity、A Framework for Formalizing LLM Agent Security、AgentSecBench、Prompt Flow Integrity、SkillScope、MCP description-code inconsistency 和 WebMCP tool-surface poisoning 已经说明：用户意图、contextual security、prompt/data-flow integrity、Skill/MCP 权限和 metadata trust boundary 都已有相关工作。第二轮和后续线上查新进一步把 novelty 风险推高：AuthGraph 已经做 clean user-intent authorization graph vs execution provenance；PACT 已经做 authority-bearing argument 的 provenance contracts；AIRGuard 已经把问题命名为 authority confusion，并明确说 untrusted resources may inform but must not authorize side effects；ARGUS 已经做 influence-provenance decision auditing；Agent-Sentry 已经做 execution-provenance behavior bounding；ClawGuard 已经做 user-confirmed runtime tool-boundary rules；VIGIL 已经做 Skill behavior specification enforcement；FIDES/RTBAS/NeuroTaint 覆盖 IFC/dependency/taint 路线；PCAS/AgentSpec/AgentGuard/AgentGuardian/AgentBound 覆盖 policy compiler/runtime DSL/ABAC/MCP access control；SecureClaw/AgentSentry/ARM 覆盖 runtime boundary、causal diagnostics 和 denial-feedback leakage。因此本文最稳的 novelty 是一个更窄的 claim：

> IntentCap 是一个 run-centric extension authorization compiler/checker，把 context 对 authority-bearing decisions 的影响编译成 proof-carrying, attenuable capability leases，并在 Skills、MCP tools、本地执行和 subagents 之间把当前用户意图、control provenance、argument constraints、flow、temporal state、budget 和 delegation 绑定起来。

这意味着论文主线不应是“我们也给 Skills/MCP 加权限”，也不应是“context provenance/authority binding 是第一次被提出”。更准确的主线是：现有系统已经能做 flow、argument provenance、runtime authority guard、policy DSL、access control、Skill behavior specs 或 provenance-aware decision auditing；IntentCap 的剩余空间是把这些约束组织成一个跨扩展、可衰减、可审计的 lease compiler/checker，其中 protected decision classes、influence modes、temporal/budget/delegation 和 proof object 是一等对象。当前 related-work PDF corpus 已经收集到 `docs/reference/`，共 73 篇核心/候选论文，并已用本地 `file` 检查为 PDF；P0 baseline notes 和 reproduction decisions 已经写入 `docs/background-related-work.md`。R082 已经实现第一版 trace-level closest-baseline labelers。结果很关键：ACL/policy-only baselines 仍然很弱，但 AuthGraph/PACT/AIRGuard-style provenance-authority 和 IFC/taint-style labelers 已经解释完当前 3,811 个 checker denials，residual 为 0。R083 则补了一个 controlled residual suite：在 7 个固定事件中，IntentCap 允许 1 个 budgeted call，拒绝 6 个 temporal、budget、delegation、cross-extension holder、wrong decision class 和 missing proof 违规；四类 closest labelers 全部接受这些事件，因此 6/6 R152-R159 对上面这条线的意义是“安全机制增量”的 intermediate 版本，不是最终 utility 结果，也不是当前 paper-facing 结果。R152-R155 先证明 structured list/leaf proof 和 typed option discriminator 可以让 previously blocked high-impact runtime writes 在不放宽 authority 的情况下获得 one-shot exact leases；但审查发现第一版 proof 会把同一 list 里 sibling item 的结构化 context 合并得过宽。R156-R159 改成每个 semantic runtime leaf 都必须有自己的 local structured context，但第二轮审查发现每个 leaf 仍混入了 1200 字符 raw text window，临近 sibling 的 discriminator token 仍可能污染 proof。R164-R167 恢复了当前安全边界：R164 删除 raw text-window fallback、拒绝裸 scalar-list sibling context 作为 local proof 后重跑同一路径，记录 51/54 gateway-allowed executions、3 gateway blocks、38/41 runtime-binding successes、3 missing-value-proof blocks、0 tool errors、44 bound-reference calls，official tool-oracle 仍是 0/11。R168-R172 是当前版本的下一步：R168 把 R166 的 visible-state recovery map 累积进 R144 map，R169 用本地 Qwen3.6/llama.cpp、同一 tau2 本地 artifact、R074/R077 compiler corpora、no-text-window value proof 和 repair-map priority 执行，记录 50/53 calls allowed/executed、3 gateway blocks、37/40 runtime-binding successes、3 missing-value-proof blocks、0 tool errors、43 bound-reference calls，official tool-oracle 仍是 0/11。R170-R172 说明 utility 没有恢复：DB-feasible misses 是 18，8 个 visible-state generated runtime candidates 进入下一轮输入，3 个 high-impact write value-proof blocks 仍存在，另有 2 个 repeated/consumed-state residuals、3 个 tool-not-active/not-compiled blockers 和 5 个 upstream-action-needed blockers。R173 把 repeated/consumed-state residual 再拆细：一个同参只读 `get_product_details` 已经执行过，可以只在 adjusted accounting 中记为已观察；另一个 `get_order_details` exact candidate 没有执行，仍是 planner-selection residual。R174 又把 tool-activation bucket 拆细：两个 airline `get_user_details` 是 read-only、schema-backed 且 `user_id` 已在 saved context 可见，可以作为下一轮 one-shot read activation 候选；retail `return_delivered_order_items` 缺完整 item-id 可见证据，不能激活。调整后 DB-feasible misses 只是潜在可从 17 降到 15，official utility 不变。因此它强化的是 checker-safe lease refusal 和 compiler/planner residual localization claim，而不是 benchmark-derived realistic utility claim。

R175 把 R174 的两个 read-only activation 候选真正接进 task loop。实现上，runner 只读取 `results/eval/R174/tool_activation_candidates.csv` 中 activation-ready 的 read rows；这些工具不会进入模型可见的 active tool schema 列表。调度器在对应 step 合成带 marker 的 `get_user_details` call，binder 再重新检查 `user_id` 是否出现在当前 task/tool-result visible state 中，只有通过后才铸造 invocation budget 为 1 的 exact lease，并交给普通 `LiveToolGateway` 执行。R175 运行同一 R169 配置加 `--stepwise-tool-activation-csv` 和 `--stepwise-tool-activation-priority`，记录 2/2 tool-activation binding successes、0 tool errors、bound-reference calls 从 43 提到 47、all-reference-executed/action-reward tasks 从 6/11 提到 7/11，exact sequence match 仍为 5/11。这个结果说明“missing read tool + visible exact argument evidence”可以安全变成 one-shot lease，并给固定切片带来一个真实 task-level improvement；但它仍不是 benchmark-scale non-oracle utility，因为 high-impact write proof、retail missing-item evidence、upstream planning、repeated/consumed-state selection 和 reward/env residual 都还没解决。

R176-R179 把 R175 之后的 residual 重新审计成下一步实验输入。R176 读取 R175、R170、R168、R131 和 R067 保存产物，不运行模型或工具，显示 DB-feasible missing references 从 18 降到 14，说明 R175 实际关闭 4 个 missing reference actions。剩余 14 个分为 7 个 runtime-candidate-generation gaps、6 个 tool-activation gaps 和 1 个 candidate-selection/planning gap，另有 7 个 reward residual tasks。R177 把 7 个 runtime-candidate-generation gaps 转成下一轮 visible-state runtime candidates，其中 3 个 high-impact。R178 确认 3 个 high-impact `modify_pending_order_items` 仍被 value proof 阻断，1 个 repeated/consumed-state residual 仍存在，6 个 not-ready residual 都是 tool-not-active/not-compiled。R179 再审计 tool activation：5 个 airline task 4 的 `get_reservation_details(reservation_id=...)` 是 read-only、schema-backed、参数可见、activation-ready；1 个 retail `return_delivered_order_items` 仍是 write/high-impact activation，必须要求 structured value proof。这个审计直接产生 R180 的 cumulative read-activation 输入，而不是扩大静态 tool catalog。

R180-R184 已经把这一步做完。R180 先把 R174 的 2 个 `get_user_details` read candidates 和 R179 的 5 个 `get_reservation_details` read candidates 合并成 `results/eval/R180/cumulative_tool_activation_candidates.csv`，再用同一 local Qwen3.6/llama.cpp、同一 tau2 本地 artifact、同一 R074/R077 compiler corpora、同一 runtime-binding/value-proof/repair-map-priority 配置运行 task loop。所有 7 个 read activation 都在 runtime 重新检查 visible arguments 后铸造成 exact one-shot leases，并通过普通 gateway 执行。结果是：tool-activation binding 7/7 成功，executed calls 58，gateway blocks 3，tool errors 0，bound-reference calls 从 R175 的 47 提到 52，all-reference-executed/action-reward tasks 从 7/11 提到 8/11，exact sequence match 从 5/11 提到 6/11。R181-R184 随后只读保存产物：DB-feasible missing references 从 14 降到 9；剩余 9 个分为 7 个 runtime-candidate-generation gaps、1 个 candidate-selection/planning gap、1 个 tool-activation gap；R182 生成 7 个 visible-state runtime candidates，潜在后验上界可到 1；R183 确认 3 个 high-impact write value-proof blocks 和 1 个 repeated/consumed-state residual 仍在；R184 确认当前没有 read-only activation candidates，唯一 activation blocker 是 retail `return_delivered_order_items`，必须走 structured value proof。这个结果把 read activation 从“可行机制”推进到“当前 bucket 已耗尽”，但仍然不是 end-to-end utility claim。

R185-R192 把 R180 后的 write-proof blocker 变成一个更清楚的机制结果。R185 先说明 R182 的 recovery map 已经被 R168 覆盖，继续重放 repair map 没有新增候选；R186/R190 只在 adjusted accounting 中抵扣一个已执行过的同参只读 `get_product_details`。真正的实现变化是把模型可见的 `tool_result_preview` 和 checker 使用的完整 `tool_result_evidence` 分开，并在 high-impact value proof 中使用 parent-scoped product-variant context：命中的 variant 可以继承父 product 的 name/product_id，但不能继承 sibling variants 或 raw text window。R187 在同一 local tau2/Qwen3.6 设置上把 3 个 `modify_pending_order_items` value-proof blocks 全部清掉，gateway blocks 从 3 降到 0，runtime binding 从 38/41 升到 41/41，tool errors 仍为 0。这个结果不能写成 utility 提升，因为 bound-reference calls 仍是 52，all-reference/action-reward 仍是 8/11；它支持的是 proof completeness 和 checker evidence design。R188-R192 显示新的 residual frontier 是 6 个 DB-feasible missing（adjusted accounting 后 5 个）：4 个 runtime-candidate-generation、1 个 candidate-selection/planning、1 个 write/high-impact activation blocker `return_delivered_order_items`；high-impact write-proof blocks 现在为 0。

R193 进一步把“再跑一个 recovery map”排除掉：把 R189 的 5 行 visible-state recovery map 与 R168 cumulative map 合并后，5 行全部按 domain/task/event/tool/args key 去重，输出仍是 29 行、27 个 eligible rows。也就是说，当前 cumulative repair-map queue 没有新输入；下一步必须改变 planner/candidate generator、repeated-state selection、retail write/high-impact activation 或 reward/env debugging，而不是继续重放 repair map。

R194 把 residual 解释再收窄了一步：新的 repeated-state accounting 不只看显式 repeated residual CSV，也扫描 R187 保存的 action log。规则仍然保守，只对同一个 task 中已经成功执行过、参数完全相同、且工具是 read-only `get_*` 的调用给 adjusted credit；write 和未执行候选都不给 credit。结果 6 个 DB-feasible missing 中有 5 个其实是 duplicate observed reads，adjusted DB-feasible missing 从 6 降到 1。official tau2 score 和 8/11 task utility 不变；真正剩下的 adjusted mechanism blocker 是 `retail:2:2_11 return_delivered_order_items` 的 write/high-impact activation。

R195 进一步审计这个唯一剩余 blocker，而不是直接放开 write activation。脚本复用 runtime binder 的 structured value-proof predicate，只读取 R192 activation row 和 R187 action log，不运行模型、不执行工具、不同步数据集。结果是：`return_delivered_order_items` 的 item/order/payment values 都有结构化上下文，return 子意图的 `cleaner|headphone|smart|watch` tokens 在 item list 上也被 collective 覆盖；但当前 per-leaf semantic threshold 要求每个 `item_id` 自己携带足够多的 intent tokens，所以 cleaner 和 headphone 单项被挡住。结论是 current rule 保守且安全；下一步要设计 grouped-list value proof，证明一个 item_ids list 作为整体满足多项用户意图，同时不能重新引入 sibling/raw-window contamination。

R196 已经实现这个 grouped-list value proof，并且只在很窄的场景生效：return-style 的 list-valued item arguments；`new_`、`target_`、`to_` 这类 replacement/target 参数仍走严格 per-leaf 证明。新规则要求每个 list leaf 都有自己的 local structured context，每个 leaf 至少匹配一个 intent discriminator token，所有 leaf 的匹配 token 合起来覆盖完整 multi-item intent。R196 重新跑 `analyze_tau2_write_activation_proof.py` 后，`retail:2:2_11 return_delivered_order_items` 变成 `write_activation_value_proof_complete`，`potential_adjusted_missing_after_complete_write_activation` 从 1 变成 0。这个结果仍然不是 utility：没有模型运行、没有工具执行、没有在 task loop 铸造 write lease、official tau2 score 不变。它只说明最后一个 adjusted write activation 现在可以进入 bounded one-shot activation 实验。

R197 把这个 proof-readiness 变成真实 task-loop 机制结果。实现上，runner 新增 `--stepwise-write-activation-proof-csv`，只读取 R196 中 `write_activation_candidate_ready=True` 且 `proof_gap_class=write_activation_value_proof_complete` 的 write/high-impact rows；真正执行前仍要重新检查所有 argument values 是否出现在当前模型可见 state 中，并重新跑 structured value proof。R197 在和 R187 相同的 local tau2/Qwen3.6 compiler-corpus 配置上运行，只额外加入 `results/eval/R196/write_activation_proof.csv`，没有同步新数据集。结果是：8/8 tool activation bindings 成功，其中 `retail:2:2_11 return_delivered_order_items` 通过 `saved_value_proof_write_activation_candidate` 在 step 10 获得 `tool-activation-live:retail:2:9:return_delivered_order_items` one-shot exact lease 并执行；gateway blocks 为 0，tool errors 为 0，runtime binding 41/41 成功，missing-value-proof blocks 为 0。这个结果支持的 claim 很窄但重要：proof-complete high-impact write activation 可以在 task loop 中安全 lowering。它不支持 utility claim：bound-reference calls 是 51，比 R187 的 52 还少 1，因为 retail sequence drift 抵消了这个 write；all-reference-executed/action-reward tasks 仍是 8/11，exact-sequence tasks 仍是 6/11，official tool-oracle 仍是 0/11。R197 当时把下一步收窄为 post-activation residual audit、planner/CEGAR、reward/env debugging 和 expert/proof-completeness labels；其中 residual audit 已由 R198 完成。

R198 已经完成 post-R197 residual audit。它没有运行模型、没有执行工具、没有同步数据集，只读取 R197 和之前保存的 residual/candidate/feasibility artifacts。结果把 R197 的 utility failure 定位得更清楚：raw DB-feasible missing references 是 7，分成 3 个 candidate-selection/planning gaps 和 4 个 runtime-candidate-generation gaps；重复读 accounting 仍能 credit 5 个已经观察过的 read-only `get_*` calls，adjusted 后剩 2 个 DB-feasible misses；tool activation gaps 是 0，write/high-impact activation blockers 也是 0。新增的 `analyze_tau2_post_activation_delta.py` 进一步说明为什么 R197 安全执行了 write 但 utility 没提升：相对 R187，R197 确实 gain 了 `retail:2:2_11 return_delivered_order_items`，但 lost 了两个 planner-selected read references：`retail:3:3_6` 和 `retail:3:3_8`。所以当前 blocker 已经不是 read activation、grouped-list proof、单个 write activation 或 post-activation audit，而是 planner-confirmed candidate generation、upstream CEGAR、reward/env residual debugging 和 expert/proof-completeness labels。

R131 把这个负结果量化到候选层：它只读取 R125 保存的 runtime-evidence candidates，不运行模型、不执行工具、不同步数据集。73 个候选里只有 5 个是 exact-next references；top-1 exact-next precision 是 1/15，ranked-fallback exact-next precision 是 0/6；44/73 个 proof-complete candidates 其实是 non-reference 或 same-tool wrong-argument candidates。正确解释是：proof completeness 和 rank 可以作为安全/证据信号，但不能直接当作 next-action correctness proof。顶会 claim 如果要继续推进 TC4，必须增加候选正确性谓词、planner/CEGAR、exact argument synthesis、tool recall/exposure 或 blinded expert oracle，而不是继续把高分候选自动激活。

R132 进一步把这个判断变成 filter-policy upper bound。它仍然只读取保存的 R131/R125 产物，不运行模型、不执行工具、不同步数据集。结果是：R125 ranked fallback replay 的 exact-next precision 是 0/6；`rank_score >= 50` 且 margin 至少 1 的非 oracle filter 只选择 1/15 个 step，而且仍然选错；margin 至少 11 的严格 filter 直接 0/15 abstain；即使使用后验 oracle exact-next selector，当前候选池也只能覆盖 5/15 个 hint-bearing steps。这个结果证明 filter/ranker 不是充分解法，但它还没有区分“候选池真的缺 DB-feasible action”和“reference 本身无效”。

R133 把 R132 的 no-exact-next-candidate 结论再细分一次。它只读取保存的 R125/R131 artifacts 和 R067 feasibility labels，不运行模型、不执行工具、不同步数据集，也不把 hidden reference 暴露给 planner。结果是：15 个 hint-bearing steps 中，5 个已有 exact-next candidate；另外 10 个没有 exact-next candidate 的 step 全部对应已知 `product_id=6086499569` invalid schema-example reference。因此在这个 R125 runtime-evidence hint pool 内，DB-feasible exact-next upper bound 仍是 5/15，继续为这 10 个 step 生成 candidate 的 DB-feasible gain 是 0。正确解释要更窄：当前候选池的 selector/filter 仍然不能带来 utility，但这 10 个 no-candidate step 不是要修的 DB-feasible generation gap。下一轮 TC4 实验应该转向更广的 37-45 个 DB-feasible missing references、tool recall/exposure、exact argument synthesis、evidence-gathering 和 planner/CEGAR continuation，而不是在这个 invalid-reference 子集上过拟合。

R134 把这个“更广的 missing references”进一步变成可执行实验设计。它只读取保存的 R129 missing-reference rows、R131 runtime-evidence candidate labels、R067 direct feasibility labels 和 R128 task-level adjusted accounting，不运行模型、不执行工具、不同步数据集。结果是：R129 的 41 个 missing actions 中，3 个是 invalid schema-example references，38 个是 DB-feasible misses；这 38 个 DB-feasible misses 中，没有一个已经在 R131 当前候选池里有 exact runtime candidate。它们的 actionability split 是：11 个 runtime candidate-generation gaps，18 个 argument-evidence gaps，2 个 tool-activation gaps，6 个 upstream-planning gaps，1 个 complete compiler hint not called；另有 2 个 task 属于 feasible references 已完成但 reward/env 仍失败。这个结果说明下一轮最重要实验不是 rank/filter/selector，而是 closed-loop compiler/planner：能不能为 visible-tool/visible-argument misses 生成 exact candidates，为 visible-tool/no-argument-evidence misses 主动收集或抽取证据，为 argument-visible/tool-hidden misses 激活工具，为 no-evidence misses 做上游规划，并把这些候选转化成 official utility，同时仍保持 checker/gateway 不扩大 authority。

## 摘要草稿
LLM agent 正在从固定工具调用器变成可扩展的执行环境。一个 agent 可以加载 Skills，连接 MCP servers，运行本地脚本，并把任务委托给 subagents。问题不只是这些扩展能否读文件、发请求或调用 API；更关键的是，扩展提供的文本会影响 agent 后续的带权限决策：选择哪个工具、把结果发到哪个 sink、请求多大范围的 approval、或者把哪些上下文和能力交给另一个组件。现有工作已经覆盖了很多关键部分：clean intent authorization graph、argument-level provenance contracts、runtime authority guard、IFC/taint、policy DSL、ABAC/MCP access control 和 OS/sandbox enforcement。IntentCap 的缺口不是“首次发现 context 会影响 authority”，而是把这些约束组织成跨 Skills、MCP、本地执行和 subagents 的 proof-carrying, attenuable lease compiler/checker。

IntentCap 的核心 artifact 是 intent-carrying lease。系统首先生成结构化 intent certificate，记录当前目标、用户选择的对象、授权的 sink、约束、approval 和过期条件。每个 context cell 都带有 provenance label 和允许的 influence modes。一个 capability lease 只有在操作可由 intent certificate 推导、该操作的 control provenance 被允许影响对应 decision class、且 temporal/budget/delegation 约束都成立时才有效。LLM 可以辅助提出 plan、effect graph 和 candidate leases，但 deterministic checker 才能接受或拒绝 lease 与事件；checker 验证 provenance、flow、temporal order、budget 和 delegation 约束，并把允许的 lease 下发到 context constructor、tool gateway、MCP broker、本地执行 sandbox 和 delegation monitor。

本文的目标不是宣称“解决 prompt injection”，而是给出一个更精确的系统安全性质：在已建模的 decision classes 和已接入的 enforcement boundaries 内，accepted high-impact events 不能被未授权 context 作为 control dependency 影响。当前原型已经把这个思想落到多个 benchmark 的 replay 与本地执行路径上：AgentDojo、InjecAgent 和 MCPTox 用于 context-to-decision 攻击，tau2/tau3 用于 policy-following utility 与 authority minimization。后续数据集选择先做网络检索和官方 metadata 审核，不自动同步新测试集；R026 已把 28 个 web-only ## 论文主线
1. Agent 安全的对象不是只有工具操作，还有“谁能影响未来决策”。
2. 同一段 context 可以被允许作为数据使用，但不应自动拥有控制 authority。
3. 用户当前意图应当是 run-time authority 的根，而不是 Skill package、MCP server 或工具 catalog 的静态身份。
4. IntentCap 用 intent certificate、context authority label、effect IR 和 proof-carrying capability lease，把这件事变成可检查的 authorization 问题。
5. LLM-assisted compiler 只负责提议；deterministic checker 和 runtime adapters 才在 TCB 内。

## 设计要点
- Intent certificate：结构化表示用户目标、选择的对象、允许的 sink、禁止的 sink、fresh approval 和过期条件。
- Context authority：把 `observe`、`quote`、`summarize`、`parameterize`、`plan`、`instruct`、`delegate`、`authorize` 等 influence modes 与 decision classes 绑定。
- Effect IR：把 agent plan 或 benchmark trajectory 转成 `ctx.use`、`tool.call`、`mcp.call`、`fs.read`、`fs.write`、`exec.run`、`net.connect`、`subagent.spawn` 等安全相关事件。
- Capability lease：短期、可衰减的授权对象，约束 operation、object、arguments、intent derivation、control/data provenance、flow、temporal guard、budget 和 delegation。
- Deterministic checker：检查 event 是否匹配 active lease，且 control provenance 是否有权影响当前 decision class。
- Runtime adapters：在 context construction、tool/MCP invocation、本地执行和 subagent delegation 处执行 allow/deny，并保留 audit log。

## 评估计划
论文正文不应该把 R001-R198 或旧的 E1-E6 当成一串并列实验来讲。顶会版本应收敛成四个核心实验，每个实验对应一个可被 falsify 的主 claim：

这个收敛不是排版问题，而是 claim 纪律：run ID 只是证据来源，不是实验编号。主文最多应有四个结果块，每个块对应一个核心问题、一组强 baseline、一个明确 oracle 和一个失败后必须收窄的 claim。新的小 run 只有在能改变 E1、E2、E3 或 E4 的结论时才进入正文；否则只能作为 appendix/provenance、negative control 或后续调试记录。

现有 benchmark/trace 标注要按用途分开使用，不能混成一个“万能 oracle”。InjecAgent、MCPTox 和 AgentDojo 的现有标注可以支持 E1/E3 的安全判断：攻击是否成功、某个 tool call 是否由 injection/poisoning 触发、某个 reference action 是否应出现。它们不能直接替代 E2 的 expert lease oracle，因为它们通常不标注最小授权对象：哪些 context source 可以影响哪个 decision class、argument predicate 应该多窄、budget/temporal/delegation 怎么设、approval scope 应该是什么。tau2/tau3 reference actions 也只说明任务期望动作，不说明一个 run 的最小 capability lease。因此论文可以大量使用现有 trace 标注做安全/utility oracle，但 least-privilege claim 仍需要 blinded expert leases。

- **E1 端到端安全与效用。** 在 AgentDojo/InjecAgent/MCPTox 安全工作流和 tau2/tau3 或 MCP utility 工作流上，比较 vanilla agent、static ACL、exact-tool ACL、MCP/server allowlist、tool-call guard、OS/sandbox-only 和 IntentCap。核心指标是 attack success、wrong sink、exfiltration、approval/delegation violation、benign completion、false denial、recovery 和 tool errors。这是论文第一张主结果表，回答“系统整体是否真的有用”。
- **E2 lease quality / authority reduction。** 对 10-30 个跨 security traces、tau2/tau3 或 MCP utility、Skill-style workflow 的任务写 blinded expert leases，比较 static/domain/global ACL、exact-tool ACL、MCP server allowlist、Skill manifest/SkillGuard-style policy、Progent/PCAS/AgentSpec-style policy 和 IntentCap。核心指标是 risk score、exposed tools/methods/sinks/paths/args、influence-mode breadth、delegation breadth 和 distance to expert oracle。这是论文第二个主结果，证明 IntentCap 不是只“挡攻击”，而是真的降低授权面。
- **E3 context-influence 机制消融。** 把文档注入、MCP poisoning、malicious Skill/tool-result、subagent/delegation residual suites 放在一起，比较完整 IntentCap、去掉 intent certificate、去掉 influence lattice、去掉 control/data split、object-only、no-provenance、provenance-authority/IFC/taint labelers。核心指标是 protected decisions 中有多少被 unauthorized context 控制，以及哪些 residual denials 只有 lease semantics 能解释。这是最关键的 novelty 实验，证明本文不是普通 tool permission。
- **E4 compiler/checker recovery。** 把 R070-R198 的 local Qwen/llama.cpp lease compiler、tau2/tau3 task loop、runtime binding、value proof、read/write activation、post-activation residual audit 组织成一个实验，而不是许多小实验。比较 LLM-only accept、checker strict、checker+feedback、proof-only/rank-only/filter-only、checker+CEGAR planner。核心指标是 invalid accepted、valid rejected、proof completeness、exact-next precision/recall、denial recovery、dangerous execute 和 official task reward。
