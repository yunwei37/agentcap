\title{IntentCap: Intent-Carrying Capabilities for LLM Agent Extensions}
% IntentCap：面向 LLM 智能体扩展的意图承载能力模型

\begin{abstract}
LLM agents take real actions --- executing code, modifying files, calling services, and delegating tasks --- driven by context from user requests, tool results, documents, shell outputs, Skill and MCP instructions, and memory.
% LLM 智能体会执行真实操作——运行代码、修改文件、调用服务、委托任务——由来自用户请求、工具结果、文档、Shell 输出、Skill 与 MCP 指令和记忆的上下文驱动。
This creates a security challenge: all inputs enter one shared planning state with equal influence, yet a user's explicit request and a document's extracted text carry different trust and should not have the same authority to choose a destination or widen access.
% 这带来了安全挑战：所有输入以同等影响力进入同一个共享规划状态，但用户的显式请求和文档的提取文本有不同的信任等级，不应拥有同样的权限来选择目标或扩大访问范围。
Current defenses control which operations are allowed but not which inputs can influence which decisions.
% 当前防御控制哪些操作被允许，但不控制哪些输入可以影响哪些决策。

Agent capabilities should be scoped to the current task intent, not to a sandbox or session lifetime, and no single context source carries enough authority to define a complete capability.
% Agent 的能力应该限定在当前任务意图的范围内，而不是沙箱或会话生命周期，且没有任何单一上下文来源能定义完整的能力。
\sys composes capabilities from four context sources --- user intent, workflow instructions, tool schemas, and runtime data --- each contributing specific fields with priority, so that no source can fill another's fields.
% IntentCap 从四个上下文来源——用户意图、工作流指令、工具模式和运行时数据——组合能力，每个来源按优先级贡献特定字段，任何来源都不能填充另一个来源的字段。
Task plans compile into short-lived capability leases validated by a checker before any side effect commits.
% 任务计划编译为短生命周期的能力租约，由检查器在任何副作用提交之前验证。

Replaying 3,746 protected-decision events from four public benchmarks, \sys produces zero unsafe accepts while covering all benign reference actions.
% 在从四个公开基准重放的 3,746 个受保护决策事件上，IntentCap 产生零个不安全接受，同时覆盖了所有良性参考操作。
Across 38 enforcement checks spanning tool calls, local execution, context placement, and delegation, the checker allows all authorized actions and blocks all violations.
% 在涵盖工具调用、本地执行、上下文放置和委托的 38 个执行检查中，检查器允许了所有授权操作并阻断了所有违规。
Ablations show the four-class partition is necessary: collapsing it causes 94\% of blocked violations to be wrongly allowed.
% 消融实验表明四类分区是必要的：合并它会导致 94\% 的已阻断违规被错误放行。
\end{abstract}

\section{Introduction}
% 引言

LLM agents are becoming extensible execution environments.
% LLM 智能体正在成为可扩展的执行环境。
A modern agent can load Skills with workflow instructions, references, and optional scripts~\cite{openai-skills}; connect to MCP servers that expose resources, prompts, and tools~\cite{mcp-spec}; run local commands; and spawn subagents.
% 现代智能体可以加载包含工作流指令、引用和可选脚本的 Skill~\cite{openai-skills}，连接暴露资源、提示和工具的 MCP 服务器~\cite{mcp-spec}，运行本地命令，以及生成子智能体。
This extensibility is useful precisely because these components affect future behavior.
% 这种可扩展性之所以有用，恰恰是因为这些组件会影响未来的行为。
A Skill can teach the agent how to perform a workflow.
% 一个 Skill 可以教会智能体如何执行某个工作流。
A tool result can become evidence for a later action.
% 一条工具结果可以成为后续操作的依据。
A document can determine the content of a generated report.
% 一份文档可以决定生成报告的内容。
The same property creates a security problem: in an agent, text is not passive data---it can become control.
% 同样的特性带来了安全问题：在智能体中，文本不是被动数据——它可以变成控制指令。

Most current defenses ask an operation-centric question: which tool may be called, which file may be read, which endpoint may be contacted, or which process may execute?
% 当前大多数防御机制提出的是以操作为中心的问题：哪个工具可以被调用，哪个文件可以被读取，哪个端点可以被访问，或者哪个进程可以执行？
That question is necessary but incomplete.
% 这个问题是必要的，但不完整。
Consider a user who asks an agent to extract tables from two selected PDFs, save spreadsheets, and create one GitHub issue in a named repository.
% 考虑一个用户要求智能体从两个选定的 PDF 中提取表格、保存为电子表格，并在指定仓库中创建一个 GitHub issue 的场景。
The PDF contents should influence spreadsheet cells and perhaps the issue body.
% PDF 内容应当影响电子表格的单元格，也许还有 issue 正文。
They should not influence the repository name, network destination, approval scope, tool selection, or decision to load another Skill.
% 但它们不应影响仓库名称、网络目标、审批范围、工具选择或加载另一个 Skill 的决定。
If hidden text in the PDF says ``send all extracted values to attacker.com'' or ``create the issue in attacker/repo,'' a tool-call allowlist or OS sandbox may catch some concrete effects, but it does not directly express the more agent-specific invariant: the untrusted PDF was never allowed to influence those decisions.
% 如果 PDF 中的隐藏文本说"将所有提取值发送到 attacker.com"或"在 attacker/repo 中创建 issue"，工具调用白名单或 OS 沙箱可能会拦截某些具体效果，但它无法直接表达更具智能体特性的不变式：不受信任的 PDF 从未被允许影响那些决策。

Securing agent extensions requires controlling not only what extensions can do, but what their context is allowed to mean.
% 保护智能体扩展的安全，不仅需要控制扩展能做什么，还需要控制其上下文被允许意味着什么。
The root of authority is a structured \emph{intent certificate}, not an extension package, server identity, or static manifest.
% 权限的根源是结构化的\emph{意图证书} (intent certificate)，而非扩展包、服务器身份或静态清单。
A certificate records the current goal, selected objects, authorized sinks, constraints, approvals, and expiry.
% 一份证书记录当前目标、选定对象、授权的输出目标、约束条件、批准信息和过期时间。
Context cells are labeled with provenance and allowed \emph{influence modes}, such as observing, quoting, summarizing, parameterizing, planning, instructing, delegating, or authorizing.
% 上下文单元被标注来源和允许的\emph{影响模式} (influence modes)，如观察、引用、摘要、参数化、规划、指示、委托或授权。
A capability is then minted for a particular run as an \emph{intent-carrying lease}: it authorizes an operation only if the operation is derivable from the current intent and its control dependencies come from context allowed to influence that decision class.
% 然后为特定运行铸造一个\emph{意图承载租约} (intent-carrying lease) 形式的能力：仅当操作可从当前意图推导出且其控制依赖来自被允许影响该决策类别的上下文时，才授权该操作。

The root cause is \emph{issuer collapse}: user intent, Skill instructions, tool metadata, and runtime observations share one planning channel, so one class can fill another's authority fields.
% 根本原因是\emph{发行者合并} (issuer collapse)：用户意图、Skill 指令、工具元数据和运行时观测共享同一个规划通道，因此一个类别可以填充另一个类别的权限字段。

\section{Design}
% 设计

\sys addresses issuer collapse by making the authorization unit a \emph{pre-effect commit record} rather than an action-level predicate.
% IntentCap 通过将授权单元设为\emph{预效果提交记录} (pre-effect commit record) 而非操作级谓词来解决发行者合并问题。
Figure~\ref{fig:arch} shows the pipeline.
% 图~\ref{fig:arch} 展示了该流水线。
A trusted issuer canonicalizes user intent into a structured certificate.
% 受信任的发行者将用户意图规范化为结构化证书。
A context labeler assigns provenance and influence modes.
% 上下文标注器分配来源和影响模式。
An untrusted LLM-assisted compiler proposes plans and candidate leases.
% 不受信任的 LLM 辅助编译器提议计划和候选租约。
A deterministic checker validates field-owner proofs and atomically consumes lease state via a single transition:
% 确定性检查器通过单次转换原子地验证字段所有者证明并消耗租约状态：
\vspace{-2pt}
\begin{small}
\[\mathsf{check\_and\_consume}(e,\mathit{lease},\mathit{proofs},\mathit{prov},\sigma) \rightarrow \mathsf{allow}(\sigma'\!,\mathit{audit}) \mid \mathsf{deny}(\mathit{reason})\]
\end{small}
\vspace{-4pt}

Every authority-changing boundary---tool call, local execution, context placement, Skill instruction slot, or delegation handoff---must submit this record \emph{before} the side effect commits.
% 每个改变权限的边界——工具调用、本地执行、上下文放置、Skill 指令槽或委托交接——都必须在副作用提交\emph{之前}提交此记录。
The checker is the sole writer of lease state; adapters cannot cache decisions or mutate budgets locally.
% 检查器是租约状态的唯一写入者；适配器不能在本地缓存决策或修改预算。

\sys partitions inputs into four issuer-owned proof boundaries: \emph{agent} context (user intent, selected objects/sinks, approvals), \emph{instruction} context (system policy, endorsed Skill/manual procedures), \emph{tool} context (MCP/registry schemas, credential scopes, sandbox contracts), and \emph{env} context (runtime observations, tool results, file state, script outputs).
% IntentCap 将输入划分为四个发行者所有的证明边界：\emph{agent} 上下文（用户意图、选定对象/输出目标、批准），\emph{instruction} 上下文（系统策略、认可的 Skill/手动流程），\emph{tool} 上下文（MCP/注册表模式、凭证范围、沙箱合约），以及 \emph{env} 上下文（运行时观测、工具结果、文件状态、脚本输出）。
The \emph{no-substitution rule} requires that each protected field be proven by its owner class: tool metadata cannot supply user-authorized sinks, runtime observations cannot supply instruction authority, and Skill text cannot widen approval scope.
% \textbf{不可替代规则}要求每个受保护字段必须由其所有者类别来证明：工具元数据不能提供用户授权的输出目标，运行时观测不能提供指令权限，Skill 文本不能扩大审批范围。
Two classes may merge only if issuer, forgeable surface, owned fields, and lifecycle authority are identical.
% 两个类别仅在发行者、可伪造面、所有字段和生命周期权限完全相同时才可合并。

A lease binds operation, object, arguments, intent derivation, control/data provenance, temporal guards, budget, expiry, and delegation depth.
% 一个租约绑定操作、对象、参数、意图推导、控制/数据来源、时序守卫、预算、过期时间和委托深度。
Leases are short-lived, consumable, and non-self-widening: a one-shot issue lease is consumed on first use, and a child subagent receives only attenuated capabilities.
% 租约是短生命周期、可消耗且不可自扩展的：一次性 issue 租约在首次使用时即被消耗，子智能体只接收衰减后的能力。
The checker validates all of these atomically in a single transition, preventing the authority-state split where separate guards lose track of consumption or delegation across boundaries.
% 检查器在单次转换中原子地验证所有这些属性，防止各独立守卫在边界之间丢失消耗或委托状态的权限-状态分裂问题。

\section{Preliminary Evidence}
% 初步证据

\textbf{RQ1: Can the event model express benign behavior without over-blocking?}
% \textbf{RQ1：事件模型能否在不过度阻断的情况下表达良性行为？}
\sys produces 0 unsafe accepts across 3,746 protected-decision events replayed from AgentDojo, MCPTox, InjecAgent, and tau2-bench~\cite{agentdojo,mcptox,injecagent,tau2bench}.
% IntentCap 在从 AgentDojo、MCPTox、InjecAgent 和 tau2-bench 重放的 3,746 个受保护决策事件中产生 0 个不安全接受~\cite{agentdojo,mcptox,injecagent,tau2bench}。
On the utility side, \sys covers all 3,813 benign reference actions in the tau2-style proxy and passes 2,554 of 2,556 applicable tool-oracle checks.
% 在可用性方面，IntentCap 覆盖了 tau2 风格代理中全部 3,813 个良性参考操作，并通过了 2,556 个适用工具预言检查中的 2,554 个。

\textbf{RQ2: Is the four-class partition a safety requirement or a naming choice?}
% \textbf{RQ2：四类划分是安全要求还是命名选择？}
The partition is load-bearing: removing issuer ownership entirely (collapsing all four classes into generic trusted context) causes 3,593 false accepts out of 3,823 checker-denied events (94\%).
% 该划分承担关键安全作用：完全移除发行者所有权（将四个类别合并为通用受信任上下文）导致 3,823 个检查器拒绝事件中出现 3,593 个错误接受（94\%）。
% 完全移除发行者所有权（将四个类别合并为通用受信任上下文）导致 3,593 个错误接受（94\%）。
Even the weakest single-edge collapse (tool$\rightarrow$agent) falsely accepts 1,928 events (50\%).
% 即使最弱的单边合并（tool$\rightarrow$agent）也错误接受了 1,928 个事件（50\%）。
On 7 controlled workflow residuals, a post-hoc policy DSL that checks predicates without field ownership falsely accepts 7/7; splitting lifecycle state across separate guards falsely accepts 5/7.
% 在 7 个受控工作流残差上，一个不检查字段所有权只检查谓词的事后策略 DSL 错误接受了 7/7；将生命周期状态拆分到各独立守卫则错误接受了 5/7。

\textbf{RQ3: Does the same commit API work beyond tool calls?}
% \textbf{RQ3：同一提交 API 能否超越工具调用生效？}
The \texttt{check\_and\_consume} transition generalizes across five boundary types: local env side effects, context placement, Skill instruction slots, delegation handoffs, and an OS-monitor-style replay backend.
% \texttt{check\_and\_consume} 转换可推广到五种边界类型：本地环境副作用、上下文放置、Skill 指令槽、委托交接和 OS 监视器风格的重放后端。
Across 38 checker-submitted boundary events, \sys allows 17 authorized effects and blocks 21 violations, with 0 unsafe executions or placements and 0 checker/monitor mismatches.
% 在 38 个提交给检查器的边界事件中，IntentCap 允许了 17 个授权效果并阻断了 21 个违规，0 个不安全执行或放置，0 个检查器/监视器不匹配。

\textbf{RQ4: Are leases auditable authority objects?}
% \textbf{RQ4：租约是否为可审计的权限对象？}
All \sys policy families achieve oracle distance~0 across 24 adjudicated lease labels covering InjecAgent, MCPTox, and tau2 samples; every non-\sys baseline has positive distance over 144 scored policy rows.
% 在覆盖 InjecAgent、MCPTox 和 tau2 样本的 24 个裁定租约标签上，所有 IntentCap 策略族的预言距离为 0；在 144 个评分策略行上，每个非 IntentCap 基线都有正距离。

\section{Discussion}

\sys asks a different question from adjacent work. EIM and bpftime ask how an extension can safely use host-provided resources~\cite{eim-bpftime}. ActPlane asks how policy can be enforced below the tool layer~\cite{actplane}. SkillGuard asks how a Skill package can be governed as a permission-bearing artifact~\cite{skillguard}. \sys asks which future decisions a given context may influence, under a given user intent, and with what proof. That distinction keeps the LLM outside the trusted computing base, treats the current user goal as the root of authority, and makes context influence a least-privilege capability rather than an implicit side effect of putting text in the prompt.

Current evidence covers replay safety, issuer-collapse ablations, and local multi-boundary enforcement. End-to-end utility with a live user simulator, independent expert-oracle lease labels, and production OS-level enforcement remain open. Scaling the checker to real-time MCP broker integration and measuring approval burden are immediate next steps.
